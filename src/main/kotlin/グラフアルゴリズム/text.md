# 第8章 グラフアルゴリズム

## 8.1 はじめに ・ Phase 0 で蒔いた種を回収する

Phase 0 第8章で、 グラフの基本用語を学びました。 頂点と辺、 隣接、 次数、 パス、 サイクル、 連結、 隣接行列、 隣接リスト ・・・ そして「BFS の触り」 まで。

あれから Phase 1 を半分以上進めてきましたが、 グラフ自体はずっと脇役でした。 ここでようやく **主役** として登場します。

なぜグラフが重要なのか? 答えはシンプルです。 **現実世界の多くの問題が、 グラフの問題として定式化できる** からです。

- **地図と経路**: 地点が頂点、 道路が辺。 最短経路は「最短のパス」 問題
- **SNS の人間関係**: ユーザーが頂点、 友達関係が辺。 「友達の友達」 は「2ステップで到達」 問題
- **インターネット**: ルータが頂点、 物理接続が辺。 パケットの転送はパス問題
- **タスク依存**: タスクが頂点、 依存が辺(有向)。 ビルド順序は「トポロジカルソート」
- **ファイルシステム**: ディレクトリが頂点、 親子関係が辺。 探索は DFS / BFS
- **電力網、 鉄道網、 流通網**: ノードと接続。 ボトルネック発見、 障害復旧経路

これらすべてに対して、 グラフ理論のアルゴリズムが応用されています。

この章で学ぶアルゴリズム:

1. **BFS(幅優先探索)**: 近い順に探索 ・ キュー(第4章)を使う
2. **DFS(深さ優先探索)**: 行けるところまで行ってから戻る ・ スタックまたは再帰
3. **ダイクストラ法**: 重み付きグラフの最短経路 ・ ヒープ(第7章)を使う
4. **最小全域木**: 「全頂点を最小コストで繋ぐ」 問題

ここまで Phase 1 で積み上げてきた **すべてのデータ構造** が、 グラフアルゴリズムを支えます。 まさに「データ構造とアルゴリズム」 の集大成。

## 8.2 グラフの表現 ・ コードに落とす

実装に入る前に、 Phase 0 第8章でやった「グラフの表現」 を復習します。 グラフをコードで扱うとき、 主に 2 つの表現方法があります。

### 隣接行列(Adjacency Matrix)

頂点数 n × n の行列で、 「頂点 i と j に辺があれば 1、 なければ 0」 と書く。

```
頂点 0 ─ 1
│     │
3 ─── 2

隣接行列:
    0 1 2 3
  0 0 1 0 1
  1 1 0 1 0
  2 0 1 0 1
  3 1 0 1 0
```

Kotlin での表現:

```kotlin
val n = 4
val adjMatrix = Array(n) { IntArray(n) }
// 辺の追加(無向グラフ)
adjMatrix[0][1] = 1; adjMatrix[1][0] = 1
adjMatrix[0][3] = 1; adjMatrix[3][0] = 1
adjMatrix[1][2] = 1; adjMatrix[2][1] = 1
adjMatrix[2][3] = 1; adjMatrix[3][2] = 1
```

特性:
- 「ある2頂点に辺があるか」 の判定: **O(1)** (`adjMatrix[i][j]`)
- メモリ使用量: **O(V²)** ( V = 頂点数)
- 辺が密(多い)グラフ向き

### 隣接リスト(Adjacency List)

各頂点ごとに「隣接する頂点のリスト」 を持つ。

```
0 → [1, 3]
1 → [0, 2]
2 → [1, 3]
3 → [0, 2]
```

Kotlin での表現:

```kotlin
val n = 4
val adjList = List(n) { mutableListOf<Int>() }
adjList[0].add(1); adjList[1].add(0)
adjList[0].add(3); adjList[3].add(0)
adjList[1].add(2); adjList[2].add(1)
adjList[2].add(3); adjList[3].add(2)
```

特性:
- 「頂点 i の隣接頂点をすべて列挙」 が **次数の分だけ** で済む(行列のように 0 を含めて走査しない)
- メモリ使用量: **O(V + E)** (E = 辺の数)
- 辺が疎(まばら)なグラフ向き
- **実務ではこちらの方が一般的**(現実のグラフは疎なことが多い)

### 重み付きグラフの表現

辺に「重み」 (距離、 コスト、 容量など)がある場合は、 隣接リストで `(隣接頂点, 重み)` のペアを持ちます。

```kotlin
data class Edge(val to: Int, val weight: Int)

val n = 4
val adjList = List(n) { mutableListOf<Edge>() }
adjList[0].add(Edge(1, 5))
adjList[1].add(Edge(0, 5))
adjList[0].add(Edge(3, 9))
adjList[3].add(Edge(0, 9))
// ...
```

この章の以降では基本的に **隣接リスト** を使って実装していきます。

## 8.3 BFS(幅優先探索)・ キューの本領発揮

Phase 0 第8章で触りだけ学んだ BFS を、 ここでガッツリ実装します。

### BFS のアイデア

「**近い順から探索する**」。

具体的には:
1. 開始頂点 s をキューに入れる
2. キューから頂点を1つ取り出す
3. その隣接頂点のうち、 まだ訪問していないものをキューに入れる
4. キューが空になるまで繰り返す

「近いところから順に」 が「キューに入れた順に処理」 という FIFO の性質と完全に一致するので、 **キューが BFS の自然な道具** になります。 第4章でやったキューの真の応用例です。

### 動きを追ってみる

```
グラフ:
0 ─ 1 ─ 2
│       │
3 ─ 4 ─ 5

開始: 頂点 0
```

ステップごとに追います。 訪問済みフラグと、 キューの中身。

```
初期: 訪問済み = {0}、 キュー = [0]、 距離 = {0: 0}

ステップ1: キューから 0 を取り出す。 隣接は 1, 3
  → 1, 3 を訪問済みにして、 キューに入れる
  → 距離 = {0: 0, 1: 1, 3: 1}
  → 訪問済み = {0, 1, 3}、 キュー = [1, 3]

ステップ2: キューから 1 を取り出す。 隣接は 0, 2
  → 0 は訪問済み、 2 を訪問済みにして キューに入れる
  → 距離 = {0: 0, 1: 1, 3: 1, 2: 2}
  → 訪問済み = {0, 1, 2, 3}、 キュー = [3, 2]

ステップ3: キューから 3 を取り出す。 隣接は 0, 4
  → 4 を訪問済みにして キューに入れる
  → 距離 = {0: 0, 1: 1, 2: 2, 3: 1, 4: 2}
  → キュー = [2, 4]

ステップ4: 2 を取り出す。 隣接 1, 5
  → 5 を訪問済みに
  → 距離 = {..., 5: 3}
  → キュー = [4, 5]

ステップ5: 4 を取り出す。 隣接 3, 5 → どちらも訪問済み
  → キュー = [5]

ステップ6: 5 を取り出す。 隣接 2, 4 → どちらも訪問済み
  → キュー = []  (空)、 終了
```

各頂点の「開始点からの距離」 は:
- 0 → 0
- 1, 3 → 1 (1 ステップ)
- 2, 4 → 2 (2 ステップ)
- 5 → 3 (3 ステップ)

「距離が近いものから順に発見される」 のが BFS の本質。 各頂点で最初に発見された距離は、 **無重みグラフの最短距離** に等しい。

### Kotlin での実装

```kotlin
fun bfs(adjList: List<List<Int>>, start: Int): IntArray {
    val n = adjList.size
    val distance = IntArray(n) { -1 }   // -1 は未訪問の意味
    distance[start] = 0
    val queue = ArrayDeque<Int>()
    queue.addLast(start)

    while (queue.isNotEmpty()) {
        val u = queue.removeFirst()
        for (v in adjList[u]) {
            if (distance[v] == -1) {   // 未訪問なら
                distance[v] = distance[u] + 1
                queue.addLast(v)
            }
        }
    }
    return distance
}

fun main() {
    // 上の例のグラフ
    val n = 6
    val adj = List(n) { mutableListOf<Int>() }
    fun addEdge(u: Int, v: Int) { adj[u].add(v); adj[v].add(u) }
    addEdge(0, 1); addEdge(0, 3)
    addEdge(1, 2)
    addEdge(2, 5)
    addEdge(3, 4); addEdge(4, 5)

    val dist = bfs(adj, 0)
    println(dist.toList())  // [0, 1, 2, 1, 2, 3]
}
```

シンプルな実装です。 50 行未満で書ける。

### 計算量

- 各頂点は最大1回キューに入る → 全頂点で **O(V)** の処理
- 各辺は最大1回見られる(両端から1回ずつなので、 無向で2回)→ **O(E)**
- **全体 O(V + E)**

これは「グラフのサイズに線形」 という、 大変効率的な計算量です。

### BFS の用途

- **無重みグラフの最短経路**(辺の重みが全部1のとき)
- **連結成分の発見**(BFS で訪問できる頂点群が1つの連結成分)
- **木の階層的な走査**(レベルごとに処理)
- **6次の隔たり**(知人関係を BFS で辿る)

> **重み付きグラフの最短経路** には BFS は使えません(BFSは辺の重みを考慮しない)。 そこでダイクストラ法が登場します(8.5 節)。

## 8.4 DFS(深さ優先探索)・ 行けるところまで行ってみる

BFS と対になるのが DFS。 「**行けるところまで突き進んで、 行き止まりになったら戻る**」 という探索戦略です。

### DFS のアイデア

迷路で「右手の壁伝いに進む」 戦略を想像してください。

1. 開始頂点から、 隣接する未訪問の頂点に進む
2. そこから、 さらに未訪問の隣接頂点に進む
3. 進める頂点がなくなったら、 戻って別の道を試す

「最後に進んだところから戻る」 という性質が **スタック** と一致します。 BFS がキュー(FIFO)なら、 DFS はスタック(LIFO)。 これも第4章の応用です。

### 動きを追ってみる(再帰版)

```
グラフ:
0 ─ 1 ─ 2
│       │
3 ─ 4 ─ 5

開始: 頂点 0
DFS の訪問順(隣接リストの順による): 0 → 1 → 2 → 5 → 4 → 3
```

頂点 0 から始める:
- 0 に行く
  - 1 に行く
    - 2 に行く
      - 5 に行く
        - 4 に行く
          - 3 に行く (3 の隣接 0 は訪問済み、 行き止まり)
        - 戻る(4 の他の選択肢なし)
      - 戻る
    - 戻る
  - 戻る
- 終了

訪問順: 0 → 1 → 2 → 5 → 4 → 3

### Kotlin での実装(再帰版)

再帰関数を使えば、 とても自然に書けます。 関数呼び出しのコールスタックが、 そのまま DFS のスタックの役割を果たすからです(第4章の話)。

```kotlin
fun dfs(adjList: List<List<Int>>, start: Int): List<Int> {
    val n = adjList.size
    val visited = BooleanArray(n)
    val order = mutableListOf<Int>()

    fun visit(u: Int) {
        visited[u] = true
        order.add(u)
        for (v in adjList[u]) {
            if (!visited[v]) {
                visit(v)
            }
        }
    }

    visit(start)
    return order
}
```

驚くほどシンプル。 BFS よりさらに短い。

### Kotlin での実装(スタック版)

「再帰だと深いとスタックオーバーフローする」 という心配がある場面では、 明示的なスタックを使います。

```kotlin
fun dfsIterative(adjList: List<List<Int>>, start: Int): List<Int> {
    val n = adjList.size
    val visited = BooleanArray(n)
    val order = mutableListOf<Int>()
    val stack = ArrayDeque<Int>()
    stack.addLast(start)

    while (stack.isNotEmpty()) {
        val u = stack.removeLast()
        if (visited[u]) continue
        visited[u] = true
        order.add(u)
        // 隣接を逆順にスタックに入れると、 再帰版と同じ訪問順になる
        for (v in adjList[u].reversed()) {
            if (!visited[v]) stack.addLast(v)
        }
    }
    return order
}
```

BFS のコードと見比べると、 違うのは **`ArrayDeque` を「キューとして」使うか「スタックとして」使うか** だけ。 これが第4章で「ArrayDeque は両端キュー、 両方に使える」 と書いた話の意味です。

### 計算量

BFS と同じく **O(V + E)**。

### DFS の用途

- **連結成分の発見**(BFS でもできるが、 DFS の方が再帰で書きやすい)
- **サイクル検出**(訪問中の頂点を再度訪問しようとしたらサイクル)
- **トポロジカルソート**(DAG で順序を決める)
- **強連結成分の検出**(タージャン法、 コサラジュ法など)
- **迷路の解法**、 **数独ソルバー**、 **バックトラッキング全般**

### BFS と DFS の使い分け

| 状況 | おすすめ |
|---|---|
| 最短ステップ数を知りたい | BFS |
| 全頂点を訪問したい(順不問) | どちらでも |
| 経路を1つ見つければ良い | DFS(枝刈りしやすい) |
| サイクル検出 | DFS |
| トポロジカルソート | DFS |
| 「近い順に処理」 が大事 | BFS |

## 8.5 ダイクストラ法 ・ 重み付き最短経路

ここからが第8章の本番。 重み付きグラフの最短経路を求めるアルゴリズムです。 「**ダイクストラ法**」 と呼ばれ、 オランダのコンピュータ科学者ダイクストラが 1956 年に考案しました。

### 何が問題?

地図アプリで「現在地から目的地まで最短時間で行きたい」 を考えてみてください。 単純な BFS では、 各辺の重み(=距離や時間)を考慮できないので使えません。

```
A ─5─ B ─5─ C
│              │
└──── 11 ────┘

A から C への最短距離は?
ルート1: A → B → C = 5 + 5 = 10
ルート2: A → C = 11

→ ルート1 の方が短い。 これを発見したい。
```

BFS で愚直に行くと「A → C を直接行く」 のが最短だと答えてしまう(辺数が少ないので)。 重みを考慮する仕組みが必要。

### ダイクストラ法のアイデア

「**現時点で確定している頂点の中で、 開始点から最短距離の頂点を選び、 そこから隣接頂点の距離を更新する**」

そして、 「現時点で最短距離の頂点を素早く取り出す」 のに **優先度キュー(=ヒープ、 第7章)** を使います。 ここでヒープが大活躍します!

### アルゴリズム

1. すべての頂点の距離を ∞ で初期化、 開始頂点の距離は 0
2. 優先度キュー(最小ヒープ)に開始頂点を `(距離 0, 頂点)` で入れる
3. キューから最小距離の頂点 u を取り出す
4. u の隣接頂点 v について、 `(uまでの距離) + (u→v の重み)` が現在の v の距離より小さければ更新し、 キューに入れる
5. キューが空になるまで繰り返す

### 動きを追ってみる

```
グラフ:
A ─5─ B ─2─ C
│      │       │
3      4       1
│      │       │
D ─1─ E ─7─ F

開始: A
```

初期:
- 距離: A=0, B=∞, C=∞, D=∞, E=∞, F=∞
- キュー: [(0, A)]

ステップ1: (0, A) を取り出す
- A の隣接: B (5), D (3)
- B: 0 + 5 = 5 < ∞ → B = 5、 キューに (5, B)
- D: 0 + 3 = 3 < ∞ → D = 3、 キューに (3, D)
- 距離: A=0, B=5, C=∞, D=3, E=∞, F=∞
- キュー: [(3, D), (5, B)]

ステップ2: (3, D) を取り出す(キューの中で最小)
- D の隣接: A (3), E (1)
- A は既に確定(=訪問済み)
- E: 3 + 1 = 4 < ∞ → E = 4、 キューに (4, E)
- 距離: A=0, B=5, C=∞, D=3, E=4, F=∞
- キュー: [(4, E), (5, B)]

ステップ3: (4, E) を取り出す
- E の隣接: B (4), D (1), F (7)
- B: 4 + 4 = 8、 既存 5 のほうが小さい → 更新しない
- D は確定済み
- F: 4 + 7 = 11 < ∞ → F = 11、 キューに (11, F)
- 距離: A=0, B=5, C=∞, D=3, E=4, F=11
- キュー: [(5, B), (11, F)]

ステップ4: (5, B) を取り出す
- B の隣接: A, C (2), E (4)
- C: 5 + 2 = 7 < ∞ → C = 7、 キューに (7, C)
- E は確定済み(距離 4 で既に処理済み)
- 距離: A=0, B=5, C=7, D=3, E=4, F=11
- キュー: [(7, C), (11, F)]

ステップ5: (7, C) を取り出す
- C の隣接: B, F (1)
- F: 7 + 1 = 8 < 11 → F = 8、 キューに (8, F)
- 距離: A=0, B=5, C=7, D=3, E=4, F=8
- キュー: [(8, F), (11, F)]

ステップ6: (8, F) を取り出す
- F の隣接: C, E → どちらも確定済み(または現在の距離より大)
- 距離は変わらず

ステップ7: (11, F) を取り出す
- F の現在距離は 8、 取り出した値 11 より小さい → **古い情報なのでスキップ**

キュー空、 終了。

最終距離: A=0, B=5, C=7, D=3, E=4, **F=8**

### Kotlin での実装

```kotlin
import java.util.PriorityQueue

data class Edge(val to: Int, val weight: Int)

fun dijkstra(adjList: List<List<Edge>>, start: Int): IntArray {
    val n = adjList.size
    val dist = IntArray(n) { Int.MAX_VALUE }
    dist[start] = 0

    // (distance, vertex) のペアを distance でソート
    val pq = PriorityQueue<IntArray>(compareBy { it[0] })
    pq.offer(intArrayOf(0, start))

    while (pq.isNotEmpty()) {
        val (d, u) = pq.poll()
        if (d > dist[u]) continue   // 古い情報はスキップ

        for (edge in adjList[u]) {
            val newDist = d + edge.weight
            if (newDist < dist[edge.to]) {
                dist[edge.to] = newDist
                pq.offer(intArrayOf(newDist, edge.to))
            }
        }
    }
    return dist
}

fun main() {
    // 上の例のグラフ
    val n = 6   // A=0, B=1, C=2, D=3, E=4, F=5
    val adj = List(n) { mutableListOf<Edge>() }
    fun addEdge(u: Int, v: Int, w: Int) {
        adj[u].add(Edge(v, w))
        adj[v].add(Edge(u, w))
    }
    addEdge(0, 1, 5)  // A-B 5
    addEdge(1, 2, 2)  // B-C 2
    addEdge(0, 3, 3)  // A-D 3
    addEdge(1, 4, 4)  // B-E 4
    addEdge(2, 5, 1)  // C-F 1
    addEdge(3, 4, 1)  // D-E 1
    addEdge(4, 5, 7)  // E-F 7

    val dist = dijkstra(adj, 0)
    println(dist.toList())   // [0, 5, 7, 3, 4, 8]
}
```

### 計算量

- 各頂点が最大 |E| 回ヒープに入る(最悪)
- ヒープ操作が O(log V)
- **全体 O((V + E) log V)**

ヒープを使わずにナイーブに実装すると O(V²) になります。 ヒープによる高速化の威力がここで出ます。 グラフが大きくなるほど差が顕著。

### ダイクストラ法の制約

ダイクストラ法には **重要な制約** があります:

> **負の重みの辺がある場合は使えない**

理由を厳密に説明すると長くなりますが、 直感的には:「最短距離が確定したらもう更新しない」 という前提が、 負の辺があると崩れるからです。 後から「マイナスのショートカット」 が見つかるかもしれない、 ということが起きうる。

負の辺がある場合は **ベルマン-フォード法** という別のアルゴリズムを使います。 計算量は O(VE) で遅くなります。 興味があれば調べてみてください。

## 8.6 最小全域木(MST)・ クラスカル法

最後にもう一つ重要なグラフアルゴリズムを学びます。 「**最小全域木**(Minimum Spanning Tree、 MST)」。

### 何が問題?

ある町に N 個の家があって、 すべての家を電線で繋ぎたい。 どの2軒を繋ぐにもコスト(距離)が違う。 **すべての家を繋ぐ最小コスト** はどうやって求める?

```
グラフ:
  A ─4─ B
  │     │
  3     2
  │     │
  C ─5─ D
   \  / 
    1
    │
    E

すべての頂点を最小コストで繋ぐには?
```

「全部の頂点を繋ぐ、 でもサイクルは作らない、 そして合計重みが最小」。 つまり **木で覆う** 形になります(木は n 頂点で n-1 辺、 Phase 0 第8章で学びましたね)。

これを「**全域木**」 と呼び、 そのうちコストが最小のものを「**最小全域木**」 と呼びます。

### 用途

- ネットワーク設計(光ファイバ、 道路、 電力網)
- クラスタリング(似たもの同士をまとめる)
- 画像処理(セグメンテーション)
- 巡回セールスマン問題の近似解法

### クラスカル法のアイデア

シンプルです:

1. **すべての辺を重みの小さい順にソート**
2. **小さい順に追加していく。 ただし、 サイクルを作る辺はスキップ**
3. n-1 本の辺を追加したら終了

この戦略は「**貪欲法(greedy)**」 と呼ばれます。 第10章で詳しくやりますが、 「その時点で最良に見える選択をする」 という方針。 ここでは「常に最も軽い辺を選ぶ」 だけで MST が得られる、 という美しい事実があります。

### Union-Find ・ 「サイクルかどうか」 の判定

クラスカル法のキモは「**この辺を追加するとサイクルになるか?**」 の判定です。 これは **Union-Find(またはDisjoint Set Union、 DSU)** というデータ構造で高速にできます。

Union-Find の発想: 「2つの頂点が既に繋がっているグループにいる」 なら、 その間に辺を追加するとサイクルになる。

```kotlin
class UnionFind(n: Int) {
    private val parent = IntArray(n) { it }   // 自分自身が親
    private val rank = IntArray(n)

    fun find(x: Int): Int {
        if (parent[x] != x) {
            parent[x] = find(parent[x])   // 経路圧縮
        }
        return parent[x]
    }

    fun union(x: Int, y: Int): Boolean {
        val rx = find(x)
        val ry = find(y)
        if (rx == ry) return false   // 既に同じグループ → ユニオン不要
        // 小さい木を大きい木にぶら下げる
        if (rank[rx] < rank[ry]) {
            parent[rx] = ry
        } else if (rank[rx] > rank[ry]) {
            parent[ry] = rx
        } else {
            parent[ry] = rx
            rank[rx]++
        }
        return true
    }
}
```

`find` で「自分の属するグループのルート」 を求め、 `union` で2つのグループを結合する。 これらの操作はほぼ O(1)(厳密にはアッカーマン関数の逆関数 α(n) という驚くほど遅い増加関数で、 実用上はずっと定数とみなせる)。

### クラスカル法の実装

```kotlin
data class WeightedEdge(val u: Int, val v: Int, val weight: Int)

fun kruskal(n: Int, edges: List<WeightedEdge>): List<WeightedEdge> {
    // 重みで昇順ソート
    val sorted = edges.sortedBy { it.weight }
    val uf = UnionFind(n)
    val mst = mutableListOf<WeightedEdge>()

    for (edge in sorted) {
        if (uf.union(edge.u, edge.v)) {
            // 異なるグループだった → サイクル発生なし → 採用
            mst.add(edge)
            if (mst.size == n - 1) break   // n-1 本集まれば完成
        }
    }
    return mst
}

fun main() {
    // 上の例: A=0, B=1, C=2, D=3, E=4
    val edges = listOf(
        WeightedEdge(0, 1, 4),  // A-B 4
        WeightedEdge(0, 2, 3),  // A-C 3
        WeightedEdge(1, 3, 2),  // B-D 2
        WeightedEdge(2, 3, 5),  // C-D 5
        WeightedEdge(2, 4, 1),  // C-E 1
        WeightedEdge(3, 4, 1),  // D-E 1
    )
    val mst = kruskal(5, edges)
    println(mst)
    println("Total weight: ${mst.sumOf { it.weight }}")
    // 採用される辺: (C-E, 1), (D-E, 1), (B-D, 2), (A-C, 3)
    // 合計: 7
}
```

### 計算量

- 辺のソート: O(E log E)
- Union-Find の各操作: ほぼ O(1)(厳密には O(α(V)))
- **全体 O(E log E)** ≒ O(E log V)

### プリム法という選択肢(参考)

クラスカル法のほかに **プリム法** という別の MST アルゴリズムもあります。

- クラスカル法: 辺中心、 Union-Find を使う
- プリム法: 頂点中心、 ヒープ(優先度キュー)を使う

実装の構造が **ダイクストラ法に似ています**。 計算量は同じく O(E log V)。 興味があれば実装してみてください。

## 8.7 実務との接続

### ケース1: 地図アプリの経路探索

Google マップなどの経路検索は、 ダイクストラ法の発展版(A* アルゴリズムなど)を使っています。 「現在地から目的地」 という1対1の最短経路の場合、 ダイクストラ法を **ゴール側からも同時に探索する**(双方向ダイクストラ)などの工夫が入っています。

日本中の道路網だと頂点数が数百万、 辺数も同程度。 ヒープを使った O((V+E) log V) のおかげで、 ミリ秒単位で経路が出てくる。

### ケース2: ソフトウェアのビルドシステム

Make、 Bazel、 Gradle などのビルドシステムは、 「タスクの依存関係」 をグラフで表現し、 **トポロジカルソート** で実行順を決めています。

```
タスクA (コンパイル) → タスクB (テスト) → タスクC (デプロイ)
                    ↓
                  タスクD (ドキュメント生成)
```

依存グラフはDAG(有向非巡回グラフ、 Phase 0 第8章)。 DFS でトポロジカルソートを実装するのが定石。 演習問題で扱います。

### ケース3: SNS の「友達おすすめ」

SNS で「友達の友達」 (2ステップで到達)を推薦する仕組み。 BFS で2ステップ先を取得 → ランキングしておすすめ。

ただし、 ユーザー数が数億のスケールだと愚直な BFS では破綻するので、 グラフデータベース(Neo4j など)や近似アルゴリズム、 事前計算が使われます。 でも基礎はやはり BFS。

### ケース4: ネットワークルーティング

インターネットのパケットルーティング(OSPF プロトコルなど)は、 ダイクストラ法系のアルゴリズムでルーティング表を計算しています。 ルータ間の「通信コスト」 を辺重みとして、 各ルータが「どこに送れば最短か」 を計算する。

### ケース5: 依存解決(パッケージマネージャ)

npm、 pip、 Maven などのパッケージマネージャは、 ライブラリの依存グラフを管理します。 「サイクルがあると無限ループ」「複数バージョンが衝突する」 などの問題を、 グラフアルゴリズムで解決しています。

### ケース6: コンパイラの命令スケジューリング

最近の CPU は命令を並列実行できる。 「どの命令を、 どの順で発行すれば最速か」 はトポロジカルソート + スケジューリング問題。 コンパイラの最適化の重要な一角。

## 8.8 まとめ

- グラフは現実世界の関係性を表現する万能ツール。 表現は隣接行列・隣接リスト
- **BFS**: キュー(第4章)で「近い順」 に探索 ・ 無重みグラフの最短経路、 O(V+E)
- **DFS**: スタック or 再帰で「深く」 探索 ・ サイクル検出、 トポロジカルソート、 O(V+E)
- **ダイクストラ法**: ヒープ(第7章)で重み付きグラフの最短経路 ・ O((V+E) log V)
- **クラスカル法**: 辺のソート + Union-Find で最小全域木 ・ O(E log V)
- 負の辺がある場合はダイクストラ法は使えない → ベルマン-フォード法へ
- Union-Find は「グループ管理」 のための高速データ構造、 ほぼ O(1)
- 実務での活用は無数: 地図アプリ、 ビルドシステム、 SNS、 ネットワーク、 依存解決、 コンパイラ
- Phase 1 で学んだすべてのデータ構造(キュー、 スタック、 ヒープ、 配列)が、 グラフアルゴリズムを支える

<summary>問題3 の解答</summary>

**方法1: DFS による実装**

```kotlin
fun topologicalSort(adjList: List<List<Int>>): List<Int> {
    val n = adjList.size
    val visited = BooleanArray(n)
    val result = mutableListOf<Int>()

    fun dfs(u: Int) {
        visited[u] = true
        for (v in adjList[u]) {
            if (!visited[v]) dfs(v)
        }
        result.add(u)   // ここで自分を追加(子孫を訪問してから)
    }

    for (u in 0 until n) {
        if (!visited[u]) dfs(u)
    }
    return result.reversed()   // 逆順にする
}
```

**動作の直感**: DFS で「全ての子孫を訪問してから自分を記録」 すると、 「子孫が先に記録される」 リストができる。 これを逆にすれば「自分が子孫より先」 = トポロジカル順序。

**方法2: BFS による実装(Kahn のアルゴリズム)**

```kotlin
fun topologicalSort(adjList: List<List<Int>>): List<Int> {
    val n = adjList.size
    val inDegree = IntArray(n)
    for (u in 0 until n) {
        for (v in adjList[u]) inDegree[v]++
    }

    val queue = ArrayDeque<Int>()
    for (u in 0 until n) {
        if (inDegree[u] == 0) queue.addLast(u)
    }

    val result = mutableListOf<Int>()
    while (queue.isNotEmpty()) {
        val u = queue.removeFirst()
        result.add(u)
        for (v in adjList[u]) {
            inDegree[v]--
            if (inDegree[v] == 0) queue.addLast(v)
        }
    }
    return result
}
```

**動作の直感**: 「入次数 0(誰にも依存されていない)頂点」 から処理する。 処理したら、 そこから出る辺を取り除き(=次の頂点の入次数を減らす)、 新たに入次数 0 になったものをキューに入れる。

**計算量**: どちらも O(V + E)。

**ポイント**:
- DFS 版は再帰で書きやすい
- BFS 版は「サイクル検出も同時にできる」(最後に入次数が 0 にならない頂点があればサイクル)
- 実務では DAG にサイクルがないことを検証したい場合が多いので、 Kahn 法が好まれる

<summary>問題4 の解答</summary>

**(a) 定式化**

- タスクを頂点として、 依存関係を有向辺で表す(A → B は「A の後に B」)
- 各頂点に「処理時間」 という重みを付ける

- 全体の答えは「全頂点の中の最大の最長距離」(=クリティカルパス)

**(b) アルゴリズム**

DAG 上の最長経路は、 **トポロジカルソート + 動的計画法** で O(V + E) で解ける。

```kotlin
fun earliestCompletionTime(adjList: List<List<Int>>, duration: IntArray): Int {
    val topo = topologicalSort(adjList)
    val n = adjList.size
    val earliest = IntArray(n)

    for (u in topo) {
        val finishU = earliest[u] + duration[u]
        for (v in adjList[u]) {
            if (finishU > earliest[v]) {
                earliest[v] = finishU
            }
        }
    }
    return (0 until n).maxOf { earliest[it] + duration[it] }
}
```

**(c) 計算量**: **O(V + E)**

**ポイント**:
- 「最長」 だからダイクストラ法は **使えない**(ダイクストラは最短専用)
- でも DAG なら「トポロジカル順に処理すれば、 各頂点を1回見れば最長が確定」 という特性が使える
- これは **クリティカルパス法(CPM)** と呼ばれ、 プロジェクト管理ツール(Microsoft Project など)で使われる古典的手法

</details>

<summary>問題5 の解答</summary>

**(a) 正しく動くか**: **動く**(ただし負の辺がない前提)。 結果は正しい。

**(b) 計算量**:

- 外側ループ: n 回(全頂点)
- 「最小距離の頂点を探す」 内側ループ: O(n)
- 辺の確認: 全辺で合計 O(E)
- **全体 O(V² + E) = O(V²)**

**(c) 改善**:

「最小距離の頂点を探す」 部分が O(V) でボトルネック。 ここに **優先度キュー(ヒープ)** を使えば O(log V) になる。

```kotlin
fun dijkstra(adjList: List<List<Edge>>, start: Int): IntArray {
    val n = adjList.size
    val dist = IntArray(n) { Int.MAX_VALUE }
    dist[start] = 0

    val pq = PriorityQueue<IntArray>(compareBy { it[0] })
    pq.offer(intArrayOf(0, start))

    while (pq.isNotEmpty()) {
        val (d, u) = pq.poll()
        if (d > dist[u]) continue
        for (edge in adjList[u]) {
            val newDist = d + edge.weight
            if (newDist < dist[edge.to]) {
                dist[edge.to] = newDist
                pq.offer(intArrayOf(newDist, edge.to))
            }
        }
    }
    return dist
}
```

これで **O((V + E) log V)**。

**いつ元の O(V²) 版が良いか**:
- 密グラフ(E ≒ V²)では、 O(V²) と O((V+E) log V) ≒ O(V² log V) なので、 むしろ元の方が速い
- 疎グラフ(E ≒ V)では、 ヒープ版が圧勝(O(V²) vs O(V log V))
- 実務のグラフは疎なことが多いので、 ヒープ版が定番

**ポイント**:
- 「正しさ」 と「効率」 は別の話。 正しいけど遅い、 はよくある
- グラフの性質(密 vs 疎)で選択が変わる
- 「ヒープを使うことで計算量を改善」 はダイクストラ法の典型的な最適化パターン

</details>

<summary>問題6 の解答</summary>

```kotlin
fun countConnectedComponents(adjList: List<List<Int>>): Int {
    val n = adjList.size
    val visited = BooleanArray(n)
    var count = 0

    fun dfs(u: Int) {
        visited[u] = true
        for (v in adjList[u]) {
            if (!visited[v]) dfs(v)
        }
    }

    for (u in 0 until n) {
        if (!visited[u]) {
            count++
            dfs(u)   // この頂点から DFS で繋がっている全頂点を訪問
        }
    }
    return count
}
```

**ポイント**:
- 「未訪問の頂点を見つけたら、 そこから DFS(または BFS)で繋がっている頂点をすべて訪問」 という発想
- 「DFS を開始した回数」 = 連結成分の数
- 計算量: O(V + E)

**別解 ・ Union-Find による実装**:

```kotlin
fun countConnectedComponents(n: Int, edges: List<Pair<Int, Int>>): Int {
    val uf = UnionFind(n)
    for ((u, v) in edges) uf.union(u, v)
    return (0 until n).map { uf.find(it) }.toSet().size
}
```

Union-Find で「グループ数」 を数える。 これも O((V + E) α(V)) ≒ O(V + E)。

実装としては DFS 版の方がシンプル、 でも Union-Find 版は「辺が動的に追加されていく」 シナリオに強い。 グラフが変化していく問題で重宝する。

</details>
