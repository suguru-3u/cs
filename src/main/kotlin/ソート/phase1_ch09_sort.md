# 第9章 ソートアルゴリズム

## 9.1 はじめに ・ なぜ今さらソートを学ぶのか

Phase 1 を通じて、 ソートには何度も出会ってきました。

- 第1章: 「ソートしてから二分探索すれば O(log n)」
- 第2章: 「`list.sorted()` を使えばよい」
- 第8章: 「クラスカル法は辺をソートして始める」

普段は `sorted()` を呼べば終わり。 では、 **なぜ今さらソートを学ぶ必要があるのか?**

理由は3つあります。

### 理由1: ソートは「アルゴリズム設計の教科書」

ソートアルゴリズムには、 アルゴリズム設計のあらゆる重要パターンが詰まっています。

- **分割統治法**(マージソート、 クイックソート)
- **データ構造の利用**(ヒープソート)
- **計算量解析**(再帰式、 期待値、 最悪と平均)
- **理論的下限**(なぜ O(n log n) より速くできないのか)
- **トレードオフ**(メモリ vs 時間、 安定性 vs 速度)

ソートを通じてこれらを学ぶと、 他のアルゴリズムを設計・分析する力が格段に上がります。

### 理由2: 実務で「適切なソートを選ぶ」 場面がある

普段使う `sorted()` は内部で **Timsort** という特殊なソートを使っています。 でも、 場面によっては:

- 「メモリが厳しいので in-place ソートを使いたい」
- 「同じキーの順序を保ちたい(安定ソートが必要)」
- 「整数のソートだけなので、 もっと速い方法は?」
- 「ほぼソート済みのデータが頻繁に来る」

など、 ソートの選択が問題になります。 内部を知らないと判断できません。

### 理由3: 面接やコーディングテストの定番

LeetCode、 競技プログラミング、 技術面接で「ソート系の問題」 は超頻出。 「マージソートを書ける」「クイックソートのピボット選択を知っている」 は、 ある程度のレベルで期待される基礎力です。

### この章で学ぶこと

1. ソートを評価する4つの観点(計算量、 安定性、 in-place、 比較ベース)
2. 基本のソート(バブル・選択・挿入)・ 全部 O(n²) だが教育的価値あり
3. **マージソート** ・ 分割統治の代表
4. **クイックソート** ・ 分割統治の本命、 実務で最も使われる
5. 比較ソートの理論的下限 O(n log n)
6. 比較なしのソート(基数ソート、 カウントソート)
7. 実務での Timsort

それでは始めましょう。

## 9.2 ソートを評価する観点

「どのソートが優秀か」 を考えるには、 単純に「速さ」 だけでなく、 いくつかの観点があります。

### 観点1: 計算量(時間・空間)

時間計算量と空間計算量。 これは説明不要ですね(第1章)。

ただしソートでは、 「**最悪・平均・最良**」 が分かれることがよくあります。

| ソート | 最良 | 平均 | 最悪 |
|---|---|---|---|
| バブル | O(n) (既ソート時) | O(n²) | O(n²) |
| 選択 | O(n²) | O(n²) | O(n²) |
| 挿入 | O(n) (既ソート時) | O(n²) | O(n²) |
| マージ | O(n log n) | O(n log n) | O(n log n) |
| クイック | O(n log n) | O(n log n) | O(n²) (悪い場合) |
| ヒープ | O(n log n) | O(n log n) | O(n log n) |

クイックソートは平均が速いけど最悪が O(n²)、 マージソートは最悪も O(n log n) で安定、 など特性が分かれます。

### 観点2: 安定性(stability)

**同じキーの要素の順序が、 ソート前後で保たれるか** という性質。

```
ソート前: [(A, 3), (B, 1), (C, 3), (D, 2)]
キー(2番目)でソートしたい。

安定ソートの結果:
[(B, 1), (D, 2), (A, 3), (C, 3)]   ← A が C より先(元の順序が保たれる)

不安定ソートの結果(の可能性):
[(B, 1), (D, 2), (C, 3), (A, 3)]   ← C が A より先(順序が崩れた)
```

これが大事になるのは:
- 複数キーでソートしたいとき(例: 「部署でソート、 同じ部署内では入社順を保ちたい」)
- UIで結果の見え方が変わるとき(ユーザーが「意外な順序」 になると混乱)

| ソート | 安定性 |
|---|---|
| バブル | 安定 |
| 選択 | 不安定 |
| 挿入 | 安定 |
| マージ | 安定 |
| クイック | 通常は不安定(実装による) |
| ヒープ | 不安定 |
| Timsort(Kotlin/Java の sort) | **安定** |

### 観点3: in-place(その場で並び替え)

**追加のメモリをほとんど使わずに、 元の配列をその場で書き換える** ソートを **in-place** と呼びます。

- in-place: 追加メモリ O(1)〜O(log n)
- 非 in-place: 追加メモリ O(n) など

大きなデータをソートするときは、 「メモリを2倍使う」 ことが許容できないことがあります(数十 GB のログとか)。

| ソート | in-place? |
|---|---|
| バブル、 選択、 挿入 | in-place |
| マージ | **非 in-place** (補助配列が必要) |
| クイック | 概ね in-place |
| ヒープ | in-place |

### 観点4: 比較ベースか否か

ほとんどのソートは「2つの要素を比べる」 ことで動きます(`a < b` で順序判定)。 これを **比較ソート** と呼ぶ。

実は「比較ソートには O(n log n) より速くできない」 という **理論的下限** があります(9.7 節で証明)。

一方、 比較せずに動くソート(基数ソート、 カウントソート)もあって、 これらは特定の条件下で O(n) で動きます。 9.8 節で扱います。

### この観点で章を読む

これ以降、 各ソートを学ぶときに **「計算量・安定性・in-place・比較ベース」** の4観点で評価していきます。

## 9.3 基本のソート ・ O(n²) の3兄弟

まずは「シンプルだけど遅い」 ソート3つを見ていきます。 実務で直接使うことはありませんが、 **ソートのアイデアを掴むには絶好の教材** です。

### バブルソート

「**隣の要素と比べて、 大きい方を後ろに送る**」 を繰り返す。 大きい要素が「泡(bubble)のように」 後ろへ上がっていく様子から名付けられました。

```kotlin
fun bubbleSort(arr: IntArray) {
    val n = arr.size
    for (i in 0 until n - 1) {
        for (j in 0 until n - 1 - i) {
            if (arr[j] > arr[j + 1]) {
                // 入れ替え
                val tmp = arr[j]
                arr[j] = arr[j + 1]
                arr[j + 1] = tmp
            }
        }
    }
}
```

各パスで「最大値が右端に確定」 する。 n パスで完了。

評価: O(n²) / 安定 / in-place / 比較ベース

### 選択ソート

「**未ソート部分の最小値を探して、 先頭と入れ替える**」 を繰り返す。

```kotlin
fun selectionSort(arr: IntArray) {
    val n = arr.size
    for (i in 0 until n - 1) {
        var minIdx = i
        for (j in i + 1 until n) {
            if (arr[j] < arr[minIdx]) {
                minIdx = j
            }
        }
        // i 番目と最小値を入れ替え
        val tmp = arr[i]
        arr[i] = arr[minIdx]
        arr[minIdx] = tmp
    }
}
```

各パスで「最小値が左端に確定」 する。

評価: O(n²) / **不安定** / in-place / 比較ベース

> 不安定な理由: 「i 番目と minIdx 番目を入れ替える」 操作が、 同じ値の他の要素を飛び越えて移動させてしまうから。

### 挿入ソート

「**手元のトランプを並び替える**」 のと同じ。 新しいカードを取って、 既にソート済みの部分の正しい位置に挿入する。

```kotlin
fun insertionSort(arr: IntArray) {
    val n = arr.size
    for (i in 1 until n) {
        val key = arr[i]
        var j = i - 1
        // key より大きい要素を1つずつ右にずらす
        while (j >= 0 && arr[j] > key) {
            arr[j + 1] = arr[j]
            j--
        }
        arr[j + 1] = key
    }
}
```

評価: O(n²) / 安定 / in-place / 比較ベース

### 挿入ソートの隠れた強み

挿入ソートは O(n²) で遅いですが、 **「ほぼソート済み」 のデータには異様に速い**(O(n) に近い)という特性があります。

「ほぼソート済み」 とは、 各要素がせいぜい数個分しかズレていない状態。 このとき、 内側の while ループがほとんど回らないので O(n) で済む。

この特性のおかげで、 挿入ソートは **他のソートの部分パーツとして** 今でも使われます。 Timsort もクイックソートの実装も、 配列が小さくなったら挿入ソートに切り替えるのが定石。

## 9.4 マージソート ・ 分割統治の代表

ここから「実用的なソート」 の世界に入ります。 まずは **マージソート**。

### 分割統治のアイデア

マージソートは「**分割統治法(divide and conquer)**」 の代表例です。 分割統治とは:

1. 問題を小さな部分問題に **分割** する
2. 各部分問題を **再帰的に解く**
3. 解いた結果を **統合** する

ソートに当てはめると:

1. 配列を2つに分ける
2. それぞれをソートする(再帰)
3. ソート済み2つを「マージ」 して1つの ソート済み配列にする

```
[5, 3, 8, 1, 4, 7, 2, 6]
↓ 分割
[5, 3, 8, 1]    [4, 7, 2, 6]
↓ それぞれ再帰的にソート
[1, 3, 5, 8]    [2, 4, 6, 7]
↓ マージ
[1, 2, 3, 4, 5, 6, 7, 8]
```

「2つのソート済み配列を1つにマージする」 のは、 各配列の先頭を見て小さい方を取っていけば O(n) でできる。

### 動きを追ってみる

```
[5, 3, 8, 1, 4, 7, 2, 6]

分割: [5, 3, 8, 1] と [4, 7, 2, 6]
  
  [5, 3, 8, 1] の分割: [5, 3] と [8, 1]
    [5, 3] の分割: [5] と [3]
      [5] と [3] をマージ → [3, 5]
    [8, 1] の分割: [8] と [1]
      [8] と [1] をマージ → [1, 8]
    [3, 5] と [1, 8] をマージ → [1, 3, 5, 8]
  
  [4, 7, 2, 6] の分割: [4, 7] と [2, 6]
    [4, 7] の分割: [4] と [7]
      → [4, 7]
    [2, 6] の分割: [2] と [6]
      → [2, 6]
    [4, 7] と [2, 6] をマージ → [2, 4, 6, 7]
  
  [1, 3, 5, 8] と [2, 4, 6, 7] をマージ → [1, 2, 3, 4, 5, 6, 7, 8]
```

### Kotlin での実装

```kotlin
fun mergeSort(arr: IntArray): IntArray {
    if (arr.size <= 1) return arr   // ベースケース: 要素1個以下はソート済み

    val mid = arr.size / 2
    val left = mergeSort(arr.copyOfRange(0, mid))    // 左半分を再帰的にソート
    val right = mergeSort(arr.copyOfRange(mid, arr.size))  // 右半分も
    return merge(left, right)
}

fun merge(left: IntArray, right: IntArray): IntArray {
    val result = IntArray(left.size + right.size)
    var i = 0   // left のインデックス
    var j = 0   // right のインデックス
    var k = 0   // result のインデックス

    while (i < left.size && j < right.size) {
        if (left[i] <= right[j]) {
            result[k++] = left[i++]
        } else {
            result[k++] = right[j++]
        }
    }
    // 残りをそのまま追加
    while (i < left.size) result[k++] = left[i++]
    while (j < right.size) result[k++] = right[j++]
    return result
}

fun main() {
    val arr = intArrayOf(5, 3, 8, 1, 4, 7, 2, 6)
    val sorted = mergeSort(arr)
    println(sorted.toList())   // [1, 2, 3, 4, 5, 6, 7, 8]
}
```

「`<=`」 で「左の方が先に取られる」 ようにすると **安定ソート** になります。

### 計算量の解析(再帰式)

マージソートの計算量を漸化式で表すと:

```
T(n) = 2 × T(n / 2) + O(n)
       ↑              ↑
   半分のサイズを     マージにかかる時間
   2回再帰
```

これを **解く**(Phase 0 第5章の漸化式の話、 ここでも登場):

```
T(n) = 2 T(n/2) + n
     = 2 (2 T(n/4) + n/2) + n = 4 T(n/4) + 2n
     = 4 (2 T(n/8) + n/4) + 2n = 8 T(n/8) + 3n
     ...
     = 2^k T(n/2^k) + k n
```

`n/2^k = 1` のとき再帰の底に到達するので、 `k = log₂ n`。

```
T(n) = n × T(1) + n log₂ n ≒ n log n
```

つまり **O(n log n)**。 最悪・平均・最良すべて O(n log n)。 安定して速いのがマージソートの強み。

> **マスター定理という道具**: `T(n) = a T(n/b) + f(n)` という形の漸化式に対する一般解法があります(マスター定理)。 マージソートは a=2, b=2, f(n)=n → O(n log n) と一発で出ます。 興味があれば調べてみてください。

### 評価

- 計算量: **O(n log n)**(最悪も)
- 安定性: **安定**
- in-place: **非 in-place**(補助配列が要る、 空間 O(n))
- 比較ベース

### マージソートの強み

- **最悪計算量も O(n log n)**(クイックソートと違って退化しない)
- **安定**
- **並列化しやすい**(左右の再帰は独立しているので並列実行可能)
- **連結リストのソートに向いている**(配列のインデックスアクセスを使わない)

## 9.5 クイックソート ・ 分割統治の本命

実務で最も多く使われる比較ソートが **クイックソート** です。 平均的にマージソートよりも速く、 in-place で動くのが強み。

### クイックソートのアイデア

「**ピボット(基準値)を選んで、 ピボットより小さいものを左に、 大きいものを右に分ける**」 を繰り返す。

```
[5, 3, 8, 1, 4, 7, 2, 6]
ピボット = 5 を選ぶ

5 より小さいもの: [3, 1, 4, 2]
5 より大きいもの: [8, 7, 6]

→ [3, 1, 4, 2, 5, 8, 7, 6]

これで「5 はもう正しい位置にいる」 状態。
あとは左半分と右半分を再帰的にソート。
```

### Kotlin での実装

```kotlin
fun quickSort(arr: IntArray, low: Int = 0, high: Int = arr.size - 1) {
    if (low < high) {
        val pivotIndex = partition(arr, low, high)
        quickSort(arr, low, pivotIndex - 1)
        quickSort(arr, pivotIndex + 1, high)
    }
}

// Lomuto 法のパーティション
fun partition(arr: IntArray, low: Int, high: Int): Int {
    val pivot = arr[high]   // 右端の要素をピボットに
    var i = low - 1
    for (j in low until high) {
        if (arr[j] <= pivot) {
            i++
            val tmp = arr[i]; arr[i] = arr[j]; arr[j] = tmp
        }
    }
    val tmp = arr[i + 1]; arr[i + 1] = arr[high]; arr[high] = tmp
    return i + 1
}

fun main() {
    val arr = intArrayOf(5, 3, 8, 1, 4, 7, 2, 6)
    quickSort(arr)
    println(arr.toList())   // [1, 2, 3, 4, 5, 6, 7, 8]
}
```

### 計算量の解析

**平均**: O(n log n)

ピボットが「中央値に近い」 と、 マージソートと同じ構造で O(n log n)。

**最悪**: O(n²)

ピボットが「最小値または最大値」 だと、 分割が「1 vs n-1」 になり、 再帰の深さが n まで増える。

```
T(n) = T(n-1) + O(n) → O(n²)
```

例: 既にソート済みの配列に対して、 「最後の要素をピボットに」 という戦略だと毎回最大値が選ばれる → 最悪。

### ピボット選択 ・ 最悪を避ける工夫

最悪を避けるためのよくある対策:

- **ランダム選択**: ピボットをランダムに選ぶ。 最悪が来る確率は天文学的に小さい
- **中央値の中央値**: 3つの要素の中央値をピボットに
- **配列が小さくなったら挿入ソートに切り替える**: 小さい配列では挿入ソートのほうが定数倍速い

実用版のクイックソートはこれらを組み合わせています。

### 評価

- 計算量: 平均 O(n log n)、 最悪 O(n²)、 ランダム化で O(n log n) の期待値
- 安定性: 通常は **不安定**(値が同じ要素の順序が崩れる可能性)
- in-place: **概ね in-place**(再帰のスタックで O(log n) は使う)
- 比較ベース

### クイックソートの強み

- **平均的に最速**(定数倍が小さい)
- **in-place**(マージソートと違ってメモリ効率が良い)
- **キャッシュ効率が良い**(配列を局所的にアクセスするので)

### クイックソート vs マージソート

| 観点 | クイックソート | マージソート |
|---|---|---|
| 平均速度 | 速い | やや遅い(でも O(n log n) は同じ) |
| 最悪 | O(n²) (退化) | O(n log n) |
| 安定性 | 不安定 | 安定 |
| メモリ | in-place | 補助配列 O(n) |
| 並列化 | やや難しい | しやすい |
| ほぼソート済み | 普通 | O(n log n) |

「速度重視 + メモリ気にする」 ならクイック、 「安定性が必要 + メモリは問題ない」 ならマージ。

## 9.6 ヒープソートの復習

第7章で扱ったヒープソートも、 改めて整理しておきます。

### アルゴリズム

1. すべての要素をヒープに入れる
2. ルート(最小値または最大値)を取り出して、 配列の最後に置く
3. これを繰り返す

```kotlin
import java.util.PriorityQueue

fun heapSort(arr: IntArray): IntArray {
    val heap = PriorityQueue<Int>()
    for (x in arr) heap.offer(x)
    return IntArray(arr.size) { heap.poll() }
}
```

### 評価

- 計算量: O(n log n)(最悪も)
- 安定性: **不安定**
- in-place: **可能**(配列をヒープとして使う実装)
- 比較ベース

「最悪も O(n log n)」 という意味でマージソートに似ていますが、 in-place で動くのが強み。 ただし定数倍が遅め、 キャッシュ効率もマージ・クイックに劣るので、 純粋なソートとしてはあまり選ばれない。

ヒープソートが真に活きるのは「**Top K**」 「**ストリーミングデータの中央値**」 など、 ソート以外の場面です(第7章のとおり)。

## 9.7 比較ソートの理論的下限 ・ なぜ O(n log n) より速くできないのか

ここまで見たマージ・クイック・ヒープは、 すべて O(n log n)。 「もっと工夫すれば O(n) のソートが作れるのでは?」 と思いたくなりますが、 答えは **No**(比較ベースでは)。

これは数学的に証明できます。 厳密にやらずに、 直感的な説明だけ。

### 直感的な証明

n 個の要素を並べる順序は n! 通り。 ソートとは「この n! 通りの中から、 正しい1つを特定する」 こと。

比較1回で得られる情報は「a < b か a ≥ b」 という **1 bit**。 つまり、 各比較で **可能性の半分を排除** できる。

n! 通りから1つを特定するのに必要な比較回数:

```
n! ≤ 2^(比較回数)
log₂(n!) ≤ 比較回数
```

スターリングの近似で `log₂(n!) ≒ n log₂ n` なので:

```
比較回数 ≥ n log n
```

つまり **比較ソートはどんなに頑張っても最低 n log n 回の比較が必要**。 これが理論的下限です。

> Phase 0 第6章の組み合わせ(n!)と、 第1章の Big-O が交わる地点。 アルゴリズムの限界を「数学的に証明する」 という、 まさに CS の真髄。

### 重要な意味

これは「比較ソート」 に対する下限。 つまり、 **比較せずにソートする方法** があれば、 O(n) も可能性がある。

それが次節の話。

---

## 9.8 比較しないソート ・ O(n) の世界へ

「比較ベースの下限が O(n log n)」 を超えるには、 **比較以外の方法** を使う必要があります。

### カウントソート

整数の配列があるとき、 値の範囲が分かっている場合に使える。

```kotlin
fun countingSort(arr: IntArray, maxValue: Int): IntArray {
    val count = IntArray(maxValue + 1)
    for (x in arr) count[x]++
    val result = IntArray(arr.size)
    var idx = 0
    for (v in 0..maxValue) {
        repeat(count[v]) {
            result[idx++] = v
        }
    }
    return result
}
```

「値ごとの出現回数を数えて、 0 から順に書き戻す」 だけ。

評価:
- 計算量: **O(n + k)** (k は値の範囲)
- 安定: **安定**(やや工夫すれば)
- in-place: 非 in-place
- 比較ベース: **比較しない**

### 制約と適用範囲

カウントソートは「値の範囲 k が n に対して小さい」 ときに強力です。

- 0〜100 のスコアが 100 万件 → O(10^6) で爆速
- 0〜10^9 のランダム整数が 100 件 → メモリが 10^9 必要、 破綻

つまり「**値が整数で、 範囲が限られている**」 ことが条件。

### 基数ソート(radix sort)

カウントソートを応用して、 もっと広い範囲の整数をソートする方法。

考え方: 「**桁ごとに** カウントソートする」。 1の位 → 10の位 → 100の位 ・・・ と桁ごとに安定なカウントソートを繰り返す。

例: [170, 45, 75, 90, 802, 24, 2, 66] を基数ソートで:

```
1の位でソート: [170, 90, 802, 2, 24, 45, 75, 66]
10の位でソート: [802, 2, 24, 45, 66, 170, 75, 90]
100の位でソート: [2, 24, 45, 66, 75, 90, 170, 802]
```

評価:
- 計算量: **O(d × (n + k))** (d は桁数、 k は基数 ≒ 10)
- 安定: 安定
- 比較ベース: 比較しない

桁数 d が定数なら、 実質 **O(n)** で動く。 ただし「整数限定」 「桁数が決まっている」 という制約があります。

### バケットソート

「データが特定の範囲に均等に分布している」 という前提のもと、 範囲を等分した「バケット」 に振り分け、 各バケット内でソートする方法。

例: 0〜1 の実数を 10 個のバケットに振り分け、 各バケット内を挿入ソート。

平均 O(n) で動くことがある(分布の仮定が満たされれば)。

### まとめ: O(n) ソートの限界

「O(n) で動くソートが存在する」 のは事実です。 ただし:

- 値の型や範囲に制約がある(整数限定、 範囲が n に対して小さい、 など)
- 範囲を満たさないと爆発する(メモリ、 時間)

なので、 「汎用ソート」 としては比較ベースの O(n log n) が王道。 特殊な状況下でだけ、 非比較ソートが活きる。

## 9.9 Kotlin/Java のソート ・ Timsort の凄み

Kotlin の `list.sorted()` や Java の `Arrays.sort()`(オブジェクト配列に対して)は、 内部で **Timsort** という独自のソートを使っています。

> Tim Peters さんが 2002 年に開発したことから「Tim sort」。 Python の標準ソートとしても採用されている。

### Timsort の特徴

Timsort は「マージソートと挿入ソートを組み合わせた」 ハイブリッド型のソートです。

ポイント:
- 計算量: **最悪 O(n log n)**、 最良 **O(n)** (既ソート時)
- **安定**
- 実データに頻出する「**部分的に既ソート**」 のパターン(=run と呼ぶ)を活用
- 小さい部分配列には挿入ソートを使う(定数倍が小さい)
- 大きな部分配列はマージソート

### なぜ Timsort が採用されたか

現実のデータは「**完全にランダム** ではない」 ことがほとんど。

- 「既にソート済みの古いデータ」 + 「新規追加された小数の未ソートデータ」
- 「逆順だが整列している部分」 を含む
- 「ほとんどソート済みだが少数が入れ替わっている」

こういうデータに対して Timsort は **O(n) 近い性能** を出します。 純粋なクイックソートやマージソートは、 こういう場合でも O(n log n) かかってしまう。

実世界のデータに対する最適化が組み込まれているのが Timsort の強み。

### Kotlin のコード上での扱い

```kotlin
val list = listOf(5, 3, 8, 1, 4)
val sorted = list.sorted()   // 内部で Timsort
println(sorted)   // [1, 3, 4, 5, 8]

// プリミティブ配列の場合
val arr = intArrayOf(5, 3, 8, 1, 4)
arr.sort()   // 内部で Dual-Pivot Quicksort(Java の Arrays.sort(int[]))
println(arr.toList())   // [1, 3, 4, 5, 8]

// 注意: オブジェクト配列の場合は Timsort
val strArr = arrayOf("banana", "apple", "cherry")
strArr.sort()   // 内部で Timsort
```

> **小ネタ**: Java では `int[]` のような **プリミティブ配列** のソートはクイックソート系(Dual-Pivot Quicksort)、 **オブジェクト配列** や `List` は Timsort を使います。 理由は「プリミティブには『安定性』 の概念が不要(同じ int は区別がない)」 なので、 速いクイックソートで十分だから。 一方オブジェクトは「内容は同じでもインスタンスは別」 なので、 安定性が重要 → Timsort。

### 実務的な指針

「自前でソートを実装する必要があるか?」 → **ほぼない**。

`list.sorted()` を使えば、 内部で最適化された Timsort が動きます。 自前で書くと、 ほぼ確実に標準ライブラリより遅く、 バグも多い。

例外:
- 教育目的、 学習
- 特殊な制約(例: 整数限定で O(n) が必要)
- 競技プログラミング・面接

## 9.10 まとめ

- ソートを評価する観点: **計算量・安定性・in-place・比較ベース**
- 基本ソート(バブル・選択・挿入): O(n²) で遅いが教育的、 **挿入ソートは「ほぼソート済み」 で異様に速い**
- **マージソート**: O(n log n)、 安定、 非 in-place、 分割統治の代表
- **クイックソート**: 平均 O(n log n)、 最悪 O(n²)、 in-place、 実務で最速
- ヒープソート: O(n log n)、 不安定、 in-place
- **比較ソートの理論的下限は O(n log n)**(数学的に証明可能)
- 非比較ソート(カウント・基数・バケット): 条件付きで O(n)、 値の制約あり
- **Kotlin/Java の sort は Timsort(プリミティブ以外)** ・ 実データに最適化されている
- 実務では `list.sorted()` を使えば良い、 自前実装は教育/特殊用途

```kotlin
fun mergeAndCountInversions(left: IntArray, right: IntArray): Pair<IntArray, Long> {
    val result = IntArray(left.size + right.size)
    var i = 0; var j = 0; var k = 0
    var inversions = 0L

    while (i < left.size && j < right.size) {
        if (left[i] <= right[j]) {
            result[k++] = left[i++]
        } else {
            // right[j] を取った → 残っている left の全要素より小さい
            // つまり、 left.size - i 個の反転が発生
            result[k++] = right[j++]
            inversions += (left.size - i).toLong()
        }
    }
    while (i < left.size) result[k++] = left[i++]
    while (j < right.size) result[k++] = right[j++]
    return Pair(result, inversions)
}

fun countInversions(arr: IntArray): Long {
    if (arr.size <= 1) return 0
    val mid = arr.size / 2
    val leftInv = countInversions(arr.copyOfRange(0, mid))
    val rightInv = countInversions(arr.copyOfRange(mid, arr.size))
    val (_, mergeInv) = mergeAndCountInversions(
        arr.copyOfRange(0, mid).also { it.sort() },
        arr.copyOfRange(mid, arr.size).also { it.sort() }
    )
    return leftInv + rightInv + mergeInv
}
```

**ポイント**:
- マージ中に「右の要素を取る」 ということは、 「左の残りすべて > 右の現在」 を意味する
- なので、 残っている左の要素の数だけ反転が発生
- 各マージで O(n)、 全体で O(n log n)
- 愚直に全ペアをチェックすると O(n²)、 マージソートの構造を使うと O(n log n) に下がる

これは LeetCode の「Count of Smaller Numbers After Self」 など、 様々な変種で出題される定番テクニック。 「分割統治を別の問題に応用する」 例として教育的。

**計算量の違い**:
- `toSortedSet()`: 内部 TreeSet で O(n log n)、 メモリ O(n)
- `distinct().sorted()`: HashSet O(n) + ソート O(n log n) = O(n log n)
- `toSet().sorted()`: 同様 O(n log n)

どれも計算量は同じ O(n log n) だが、 「重複除去」 と「ソート」 をライブラリに任せた方が簡潔・バグも少ない。

**ポイント**:
- Kotlin の標準ライブラリには `distinct`、 `toSet`、 `toSortedSet` などの便利関数がある
- 「やりたいこと」 がライブラリでカバーされていれば、 自前ループは避けるのがベター
- 第5章で学んだ HashSet/TreeSet の知識が、 こういう簡潔な書き方に繋がる

</details>
