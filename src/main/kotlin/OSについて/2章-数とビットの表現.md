## 2.1 なぜ「0と1」なのか

そもそも、なぜコンピュータは 0 と 1 だけで世界を表すのでしょうか。10進数(0〜9)のほうが人間には自然なのに。

理由はシンプルで、**電気で動くから**です。電気の状態を確実に区別しようとすると、「電圧が高い/低い」「スイッチが ON/OFF」のような **2択がいちばん間違えにくい**のです。

たとえば「電圧で 0〜9 の10段階を表す」ことを考えてみてください。0.5V きざみで10段階を区別する必要があり、ノイズで少し電圧が揺れただけで「3 のつもりが 4 に見える」事故が起きます。一方「ある電圧より上なら 1、下なら 0」の2択なら、多少ノイズが乗ってもまず間違えません。

電灯のスイッチを思い浮かべてください。「ちょっとだけ点いてる」は不安定ですが、「点いてる/消えてる」ははっきりしています。コンピュータはこの「はっきりした2択」を、気の遠くなるほどたくさん並べることで動いています。

この **ON/OFF の1個分の情報** を **ビット(bit)** と呼びます。すべての出発点です。

> **腑に落とすための一言**
> 「0と1しかないなんて不便そう」と思うかもしれません。でも、桁をたくさん並べれば 0と1 でもどんな数でも表せます。私たちが普段 0〜9 の10種類の数字で大きな数(例: 4,294,967,296)を表せるのと同じ理屈です。種類が少ない分、桁数で稼ぐわけです。

## 2.2 2進数 ―― 0と1で数を数える

### まず10進数を「分解」して思い出す

私たちが普段使う10進数。たとえば `253` という数を、改めて分解してみます。

```
 253 = 2×100 + 5×10 + 3×1
     = 2×10² + 5×10¹ + 3×10⁰
```

各桁には「位(くらい)」があり、右から 1の位、10の位、100の位 …… と **10倍ずつ** 大きくなっていきます。10進数は「10種類の数字(0〜9)」と「10倍ずつの位」でできている。だから "deci(10)" mal なんですね。

### 2進数は「2倍ずつの位」

2進数はこれの「10」を「2」に変えただけです。使える数字は **0と1の2種類だけ**、位は **2倍ずつ** 大きくなります。

```
位:    ... 16   8   4   2   1
            2⁴  2³  2²  2¹  2⁰
```

たとえば 2進数の `1101` は:

```
1101 = 1×8 + 1×4 + 0×2 + 1×1
     = 8 + 4 + 0 + 1
     = 13   (10進数)
```

「その位が立っている(1)なら足す、立っていない(0)なら足さない」だけ。これが2進数の読み方のすべてです。

### 10進数 → 2進数(余りを拾う)

逆に 10進数を 2進数にするには、**2で割って余りを下から拾う**のが定番です。13 でやってみます。

```
13 ÷ 2 = 6 余り 1   ┐
 6 ÷ 2 = 3 余り 0   │  下から上へ読む
 3 ÷ 2 = 1 余り 1   │
 1 ÷ 2 = 0 余り 1   ┘
→ 余りを下から並べて 1101
```

最初は手で何回かやると感覚がつかめます。が、**毎回手計算する必要はありません**。普段は Kotlin に変換させればいいので、「仕組みはこうなっている」とわかれば十分です。

```kotlin
// Kotlin で確認
println(0b1101)                       // → 13   (0b で2進数リテラル)
println(13.toString(2))               // → "1101"
println(13.toString(radix = 2))       // 同じ
```

> `0b1101` のように `0b` を付けると、Kotlin では2進数で数を書けます。`13.toString(2)` は「13 を2進数の文字列で表して」という意味です。

## 2.3 16進数 ―― 人間がビットを読みやすくするための工夫

2進数は機械には自然ですが、人間には**桁が長すぎて読みにくい**のが難点です。たとえば 1バイト分の `11111111` を見て一瞬で「255」とは思えませんよね。

そこで登場するのが **16進数(hex)** です。16進数は **2進数を4桁ずつまとめて、1文字で書く**ための表記です。なぜ4桁かというと、2進数4桁(2⁴ = 16通り)がちょうど16進数1桁にぴったり対応するからです。

16進数では 0〜9 の後が足りないので、10〜15 を **A〜F** で表します。

| 10進 | 2進(4桁) | 16進 |
|------|-----------|------|
| 0    | 0000      | 0    |
| 5    | 0101      | 5    |
| 9    | 1001      | 9    |
| 10   | 1010      | A    |
| 13   | 1101      | D    |
| 15   | 1111      | F    |

この対応のおかげで、長い2進数も **4桁ずつ区切って1文字に置き換える**だけで読めます。

```
2進数  1111 1111
        ↓    ↓
16進数   F    F      → 0xFF (10進数で 255)

2進数  0001 1010
        ↓    ↓
16進数   1    A      → 0x1A (10進数で 26)
```

実務で 16進数を見る場面はけっこうあります。色の指定(`#FF0000` は赤)、メモリのアドレス表示、ハッシュ値、デバッガの出力など。「16進数は2進数の読みやすい服を着たもの」と思っておけば、見ても怯まなくなります。

```kotlin
// Kotlin で確認(0x で16進数リテラル)
println(0xFF)                  // → 255
println(0x1A)                  // → 26
println(255.toString(16))      // → "ff"
println(26.toString(16))       // → "1a"
```

## 2.4 ビットとバイト ―― KB・MB の正体

ビット(bit)が ON/OFF 1個分の情報でした。これが **8個まとまった単位** を **バイト(byte)** と呼びます。

```
1 バイト = 8 ビット = 00000000 〜 11111111
        → 表せる値は 2⁸ = 256 通り(0〜255)
```

なぜ8個なのかは歴史的な事情が大きいのですが、「だいたい英数字1文字を表すのにちょうどいいサイズだった」と思っておけば実用上は困りません。

ファイルサイズやメモリ量で見る KB・MB・GB は、このバイトをまとめた単位です。コンピュータは2進数の世界なので、**1000倍ではなく1024倍(= 2¹⁰)** きざみが基本です。

```
1 KB ≒ 1024 バイト   (2¹⁰)
1 MB ≒ 1024 KB       (2²⁰)
1 GB ≒ 1024 MB       (2³⁰)
```

(厳密には「1000きざみ(KB)」と「1024きざみ(KiB)」を区別する流儀もありますが、ここでは深入りしません。「だいたい1024倍ずつ」で十分です。)

### Kotlin/JVM の整数型はサイズで決まっている

ここが JVM 接続の最初のポイントです。Kotlin の整数型は、**何バイト使うか=何ビットか** で種類が分かれています。

| 型     | ビット数 | 表せる範囲 |
|--------|----------|-----------|
| `Byte` | 8        | -128 〜 127 |
| `Short`| 16       | -32,768 〜 32,767 |
| `Int`  | 32       | 約 -21億 〜 +21億 |
| `Long` | 64       | 約 -922京 〜 +922京 |

```kotlin
println(Int.MAX_VALUE)   // → 2147483647    (約21億)
println(Int.MIN_VALUE)   // → -2147483648
println(Long.MAX_VALUE)  // → 9223372036854775807
```

「`Int` は32ビット = 4バイト分の箱」というイメージが、次の2.5・2.6 でとても効いてきます。**箱のサイズが決まっている**ことが、後で「桁あふれ」を生む原因になるからです。

## 2.5 負の数 ―― 2の補数(これが意外と面白い)

ここまでで 0 以上の数(正の整数)は表せました。では **マイナス** はどう表すのでしょう? ビットは 0 と 1 しかないので、マイナス記号「−」は存在しません。

### 素朴なアイデアと、その問題点

「いちばん左のビットを符号にして、0なら+、1なら−にすればいい」と思いつきます(符号ビット方式)。これは半分正解なのですが、素朴にやると2つ問題が出ます。

1. **+0 と −0 ができてしまう**(`00000000` と `10000000`)。0が2種類あるのは気持ち悪い
2. **足し算の回路が複雑になる**。符号を見て場合分けが必要

そこで実際のコンピュータが採用しているのが **2の補数(two's complement)** という、ちょっと天才的な方式です。

### 2の補数の作り方

ある数のマイナス版を作る手順は、たった2ステップです。

```
手順:  ① 全ビットを反転する(0↔1)
       ② 1 を足す
```

8ビットで `-5` を作ってみましょう。

```
  5         = 00000101
  ① 反転    → 11111010
  ② +1      → 11111011   ← これが -5
```

### なぜこれで「マイナス」になるのか

天才的なのはここです。`5` と `-5(=11111011)` を **普通に足してみてください**。

```
  00000101   ( 5)
+ 11111011   (-5)
-----------
 100000000   ← 9桁になった!
```

9桁目に繰り上がりますが、**箱は8ビットしかない**ので、はみ出した1は捨てられます。残るのは下8桁の `00000000`、つまり **0**。

```
5 + (-5) = 0  ✓
```

足し算の回路が **正の数とまったく同じまま** でマイナスを扱える ―― これが2の補数の最大の利点です。CPU は「引き算」を「2の補数を足す」ことで実現しています(`a - b` は `a + (-b)`)。引き算専用の回路を作らなくて済む。

```kotlin
// Kotlin で -5 のビットパターンを覗く(下位8ビットだけ表示)
val x = -5
println((x and 0xFF).toString(2))   // → "11111011"
println(x.toUInt().toString(2))     // 32ビット全体: 11111111111111111111111111111011
```

> **7割理解ポイント**
> 「2の補数の作り方(反転して+1)」と「これでマイナスが表せる」の2点さえ持って帰れれば十分です。「なぜ繰り上がりが消えてうまくいくのか」は、何度か手で足し算すると自然と腑に落ちます。今わからなくても気にせず先へ。

## 2.6 整数オーバーフロー ―― Int がある日突然マイナスになる

2.4 で「`Int` は32ビットの箱」、2.5 で「いちばん上のビットは符号(2の補数)」と学びました。この2つが組み合わさると、実務でハマりがちな現象が説明できます。

`Int.MAX_VALUE`(約21億)に **1 を足す** と、どうなるでしょうか。

```kotlin
println(Int.MAX_VALUE)       // → 2147483647
println(Int.MAX_VALUE + 1)   // → ???
```

直感では「21億 + 1 = 21億1」ですが、実際は:

```kotlin
println(Int.MAX_VALUE + 1)   // → -2147483648  (Int.MIN_VALUE!)
```

**いきなり最小値(マイナス約21億)に化けます。** これが **整数オーバーフロー(桁あふれ)** です。

### なぜマイナスに飛ぶのか

ビットで見ると一瞬で納得できます。`Int.MAX_VALUE` は「符号ビット以外が全部1」の状態です。

```
Int.MAX (2147483647) = 0111...1111  (先頭0=正、残り31桁が全部1)
                  +1
              =  1000...0000  ← 先頭が1に繰り上がった
```

先頭ビットは符号です。それが 0(正)から 1(負)に変わった瞬間、値は一気に **最小の負数** へ飛びます。メーターが振り切れて反対側に回り込む、いわゆる「桁あふれの折り返し」です。

### 実務での事故と対策

これは絵空事ではなく、現実に事故ります。

- **ミリ秒や件数を `Int` で積算していて、いつの間にか21億を超えてマイナスになる**
- **`a + b` の合計が `Int` の範囲を超える**(個々は範囲内でも合計で溢れる)

対策はシンプルで、**大きくなりうる値は最初から `Long` を使う**ことです。`Long` は64ビットなので約922京まで持ち、まず溢れません。

```kotlin
val big: Long = Int.MAX_VALUE.toLong() + 1   // → 2147483648 (正しい!)
```

また、Kotlin/Java には溢れたら例外を投げてくれる安全な計算もあります。

```kotlin
// 溢れたら ArithmeticException を投げる(黙って化けるより安全)
val safe = Math.addExact(Int.MAX_VALUE, 1)   // → 例外が発生する
```

> **実務接続**: 「金額」「カウンタ」「タイムスタンプ(ミリ秒)」のように **積み上がっていく数は `Long` で持つ**、が基本の防御です。`System.currentTimeMillis()` が `Long` を返すのも、`Int` では2038年あたりで溢れるからです(有名な「2038年問題」)。

## 2.7 浮動小数点 ―― 小数のワナ(0.1 + 0.2 != 0.3)

ここまで整数を見てきました。次は **小数** です。そして、ここが実務で一番びっくりする落とし穴です。

次の Kotlin を実行してみてください。

```kotlin
println(0.1 + 0.2)            // → ???
println(0.1 + 0.2 == 0.3)     // → ???
```

結果はこうなります。

```kotlin
println(0.1 + 0.2)            // → 0.30000000000000004
println(0.1 + 0.2 == 0.3)     // → false
```

**`0.1 + 0.2` が `0.3` にならない。** バグでも JVM の欠陥でもありません。これは小数を2進数で表す仕組み(**IEEE 754 浮動小数点**)の宿命です。

### なぜズレるのか ―― 2進数で「0.1」が割り切れない

10進数でも `1 ÷ 3 = 0.3333…` と無限に続いて正確に書けませんよね。同じことが2進数でも起きます。

`0.1`(10進)を2進数にしようとすると:

```
0.1 (10進) = 0.0001100110011001100110011... (2進、無限ループ)
```

`0.1` は2進数では**割り切れず、無限に続く**のです。でもコンピュータの箱(`Double` は64ビット)は有限なので、**どこかで打ち切って近似**するしかありません。その「ほんのわずかな誤差」が、足し算を重ねると表面化して `0.30000000000000004` のような形で見えてくる、というわけです。

### `Double` / `Float` は「近似値」だと心得る

| 型      | ビット数 | 用途 |
|---------|----------|------|
| `Float` | 32       | 精度低め・省メモリ |
| `Double`| 64       | 標準。Kotlin の小数リテラルは既定で Double |

ポイントは、**`Double` は「だいたいの値」を高速に扱うための型** だということ。科学計算やグラフィック、機械学習のように「多少の誤差は許容できて、速度が大事」な場面には最適です。

その代わり、**「ぴったり正確な値」が求められる場面には向きません**。次の2.8 がまさにそれです。

### 小数の比較は `==` で書かない

誤差があるので、`Double` 同士を `==` で比較するのは危険です。実務では「十分近いか」で判定します。

```kotlin
val a = 0.1 + 0.2
val b = 0.3
println(a == b)                          // → false(危険)

val epsilon = 1e-9
println(Math.abs(a - b) < epsilon)       // → true(誤差を許容して比較)
```

## 2.8 お金に Double を使うな ―― 実務で一番やりがちな事故

2.7 を踏まえると、**お金の計算に `Double` を使ってはいけない**理由がそのまま説明できます。

```kotlin
// やってはいけない例
val price = 0.1
val tax = 0.2
println(price + tax)        // → 0.30000000000000004
```

`0.30000000000000004` 円。1円の誤差すら許されないお金の世界で、こんな値が出るのは致命的です。何百万件と積み重なれば、誤差も無視できない額に育ちます。

### 正しい選択肢

**① 整数(`Int`/`Long`)で「最小単位」で持つ**

お金を「円」ではなく「最小単位(円や銭)」の整数で扱う方法です。多くの決済系システムがこれを採用しています。

```kotlin
// 金額を「銭(1/100円)」など最小単位の Long で持つ → 誤差ゼロ
val priceInSen: Long = 10   // 0.10円 を 10銭 として保持
val taxInSen: Long = 20
println(priceInSen + taxInSen)   // → 30 (= 0.30円、ぴったり)
```

**② `BigDecimal` を使う**

正確な10進小数をそのまま扱える型です。ただし生成方法に注意があります。

```kotlin
import java.math.BigDecimal

// ✕ Double から作ると、Double の誤差をそのまま引き継いでしまう
val bad = BigDecimal(0.1)        // → 0.1000000000000000055511...

// ◯ 文字列から作れば正確
val a = BigDecimal("0.1")
val b = BigDecimal("0.2")
println(a + b)                   // → 0.3 (ぴったり)
```

> **実務接続(超重要)**
> 「お金・通貨・正確な小数 → `Double` 禁止。整数の最小単位か `BigDecimal`(文字列から生成)」 ―― これは Phase 2 で持ち帰る知識の中でも、**明日から実務で効く**ものです。`BigDecimal` を `new BigDecimal(0.1)` のように Double から作ると台無しになる点も、レビューでよく指摘される定番ポイントです。

## 2.9 ビット演算 ―― and / or / xor / shl の使いどころ

最後は **ビット演算** です。ビットを直接いじる操作で、「省メモリで複数のフラグを管理する」場面などで使います。

### Kotlin のビット演算子は「名前」で書く

ここは Kotlin 特有の注意点です。Java や C は `&` `|` `^` `<<` といった記号を使いますが、**Kotlin はこれらを記号ではなく英単語の中置関数で書きます**。

| 演算       | Kotlin   | 意味 | 例 |
|------------|----------|------|-----|
| AND        | `and`    | 両方1なら1 | `0b1100 and 0b1010` → `0b1000` |
| OR         | `or`     | どちらか1なら1 | `0b1100 or 0b1010` → `0b1110` |
| XOR        | `xor`    | 違うとき1 | `0b1100 xor 0b1010` → `0b0110` |
| NOT(反転) | `inv()`  | 0↔1 を反転 | `0.inv()` → `-1`(全ビット1) |
| 左シフト    | `shl`    | 左へずらす(×2ずつ) | `1 shl 3` → `8` |
| 右シフト    | `shr`    | 右へずらす(符号を保つ) | `-8 shr 1` → `-4` |
| 符号なし右  | `ushr`   | 右へずらす(0で埋める) | `-1 ushr 28` → `15` |

```kotlin
println(0b1100 and 0b1010)   // → 8  (0b1000)
println(0b1100 or  0b1010)   // → 14 (0b1110)
println(0b1100 xor 0b1010)   // → 6  (0b0110)
println(1 shl 3)             // → 8  (1を左に3つ→1000)
```

`shl`(左シフト)は「2倍ずつする」操作、`shr`(右シフト)は「半分にする」操作だと捉えるとイメージしやすいです(`1 shl 3` = 1×2³ = 8)。

> **`shr` と `ushr` の違い**: `shr` は符号を保つ(負数のまま)ように上位を符号ビットで埋め、`ushr` は問答無用で 0 を詰めます。`-8 shr 1 = -4`(マイナスのまま半分)に対し、`-1 ushr 28 = 15` になるのはこのためです。実務で `ushr` が要る場面は限られるので、まずは「2種類ある」とだけ覚えておけば十分です。

### 実務例:フラグ管理(権限ビット)

ビット演算の代表的な使い道が **フラグ管理** です。「読み取り可」「書き込み可」「実行可」のような ON/OFF を、1つの `Int` の各ビットに割り当てて、まとめて持ちます(Linux のファイル権限 `rwx` がまさにこれ)。

```kotlin
// 各権限を「2のべき乗」= 立てるビットが1つだけの値にする
const val READ    = 1   // 0b001
const val WRITE   = 2   // 0b010
const val EXECUTE = 4   // 0b100

var flags = 0   // 最初は権限なし (0b000)

// ① 立てる(セット):OR を使う
flags = flags or WRITE          // 0b010
flags = flags or READ           // 0b011 (READ と WRITE)

// ② 立っているか調べる:AND を使う
val canWrite = (flags and WRITE) != 0   // → true
val canExec  = (flags and EXECUTE) != 0 // → false

// ③ 落とす(クリア):AND と inv() を組み合わせる
flags = flags and WRITE.inv()   // WRITE のビットだけ 0 に → 0b001

// ④ 反転(トグル):XOR を使う
flags = flags xor READ          // READ が立っていれば落ちる、なければ立つ
```

整理すると、フラグ操作の4点セットはこうです。

```
セット   :  flags = flags or  FLAG      (立てる)
チェック :  (flags and FLAG) != 0       (立ってるか)
クリア   :  flags = flags and FLAG.inv() (落とす)
トグル   :  flags = flags xor FLAG      (反転)
```

> **実務接続**: 複数の真偽値を `Boolean` 変数で個別に持つ代わりに、1つの整数のビットに詰め込めば、省メモリかつ「まとめて1個の値として保存・送信」しやすくなります。設定ビット、状態フラグ、権限管理などで今も現役の手法です。とはいえ可読性は下がるので、**Kotlin では基本 `Set<Enum>` や個別の `Boolean` で十分**で、ビットフラグは「外部仕様・プロトコル・性能要件でそう決まっているとき」に読めればいい、くらいの温度感で OK です。

## この章のまとめ

- コンピュータが 0と1(**ビット**)を使うのは、電気の ON/OFF が**いちばん間違えにくい**から
- **2進数**は「2倍ずつの位」、**16進数**は「2進数4桁を1文字にまとめた読みやすい表記」。8ビット = 1**バイト** = 0〜255
- 負数は **2の補数**(反転して+1)で表す。おかげで引き算を足し算の回路で実現できる
- 箱のサイズが決まっているので **整数オーバーフロー** が起きる。`Int.MAX_VALUE + 1` はマイナスに化ける → 積み上がる値は `Long`
- 小数の `Double` は2進数の近似なので **`0.1 + 0.2 != 0.3`**。比較は `==` でなく誤差許容で
- **お金に `Double` は禁止**。整数の最小単位か `BigDecimal`(文字列から生成)を使う
- **ビット演算**(`and`/`or`/`xor`/`inv`/`shl`/`shr`/`ushr`)はフラグ管理などで使う。Kotlin は記号でなく英単語で書く

実務に直結する持ち帰りは2つ。**「積み上がる数は Long」「お金に Double は禁止」**。この2つだけでも今日から効きます。

## 演習の解答

### 問1
1. `101010` = 32+8+2 = **42**
2. `200` = `0xC8`(200 = 12×16 + 8 = C8)
3. `0x2F` = 2×16 + 15 = **47**
```kotlin
println(0b101010)            // → 42
println(200.toString(16))    // → "c8"
println(0x2F)                // → 47
```

### 問2
```kotlin
println(a + b)            // → -294967296  (オーバーフロー!40億は Int に収まらない)
println(a.toLong() + b)   // → 4000000000  (Long なら正しい)
```
20億 + 20億 = 40億 は `Int.MAX_VALUE`(約21億)を超えるため、(1) は桁あふれしてマイナスに化けます。`toLong()` で先に `Long` にしておけば (2) のように正しく計算できます。**「合計が範囲を超えるかも」と思ったら、足す前に `Long` へ。**

### 問3
```kotlin
println(0.1 + 0.2 == 0.3)                    // → false
println(0.5 + 0.25 == 0.75)                  // → true
println(Math.abs((0.1+0.2) - 0.3) < 1e-9)    // → true
```
**(2) が `true` になる理由**: `0.5 = 1/2`、`0.25 = 1/4`、`0.75 = 3/4` は、いずれも分母が2のべき乗です。2進数では `0.1`(2進で 0.1)、`0.01`、`0.11` ときれいに割り切れるため、誤差が出ません。一方 `0.1`(10進)や `0.2`(10進)は2進数で割り切れず無限に続くので、(1) では誤差が表面化します。「2進数で割り切れる小数(分母が2のべき乗)なら誤差は出ない」がポイント。

### 問4
```kotlin
println(0.7 + 0.1)   // → 0.7999999999999999  (0.8 にならない!)
```
誤差なく `0.8` を得る方法の例:
```kotlin
// 方法A: 最小単位の整数で持つ
val a = 7; val b = 1
println((a + b))                 // → 8 (= 0.8 を 1/10 単位の整数で)

// 方法B: BigDecimal を文字列から生成
import java.math.BigDecimal
println(BigDecimal("0.7") + BigDecimal("0.1"))   // → 0.8
```

### 問5
```kotlin
// (1) READ(0b001) or EXECUTE(0b100) = 0b101
// (2) さらに WRITE(0b010) を立てて 0b111、その後 READ を落として 0b110
// (3) WRITE ビット(0b010)は立っているので true
```
- (1) `0b101`(=5、READ と EXECUTE)
- (2) `0b110`(=6、WRITE と EXECUTE)
- (3) `true`

```kotlin
// 実際に動かして確認するなら
const val READ = 1; const val WRITE = 2; const val EXECUTE = 4
var flags = 0
flags = flags or READ
flags = flags or EXECUTE
println(flags.toString(2))          // → "101"
flags = flags or WRITE
flags = flags and READ.inv()
println(flags.toString(2))          // → "110"
println((flags and WRITE) != 0)     // → true
```
