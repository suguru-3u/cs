## 9.1 I/O とは ―― アプリの外の世界とのやり取り

**I/O(Input/Output、入出力)** とは、プログラムが **自分の外の世界とやり取りすること** です。ファイルの読み書き、ネットワーク通信、キーボード入力、画面表示 ―― これらは全部 I/O。

ここまでの章(CPU、メモリ)は、いわば**プロセスの "内側"** の話でした。この章は、その内側から**外の世界へ手を伸ばす**話です。そして外の世界には、内側と決定的に違う2つの性質があります。

1. **外は遅い**。第5章のメモリ階層を思い出してください。ディスクやネットワークは、あの図の**一番遠い端**(「別の街」「海の向こう」)でした。CPU やメモリの速さとは桁が違います。
2. **外は自分のものではない**。ディスクもネットワークカードも、他のアプリと共有するハードウェアです。だから**勝手に触れず、OS に仲介してもらう**必要があります(第6章の隔離、以前話したユーザ/カーネルモード)。

この2つ ―― 「遅い」と「OS を通す」 ―― が、この章の全トピックの通奏低音です。

## 9.2 ファイルとディレクトリ ―― OS が与える「見え方」

まず身近な「ファイル」から。あなたが `data.txt` というファイルを開くとき、その実体はストレージ(SSD/HDD)上の**ただのバイトの並び**が、あちこちのブロックに散らばって記録されているだけです。でもあなたは「◯番ブロックと△番ブロックにデータがある」なんて意識しません。「`data.txt` という名前のファイル」として扱えます。

これは第1章で学んだ **抽象化** です。**OS が、生のストレージの複雑さを隠して、「ファイル」という分かりやすい見え方を与えてくれている**。「ファイル = 名前のついた、順に読み書きできるバイトの列」という便利な幻を、OS が作っているわけです。

そして **ディレクトリ(フォルダ)** は、ファイルを整理する入れ物で、**木構造(ツリー)** をなしています ―― これは **Phase 0 で学んだ木構造** そのものです。

```
/(ルート)
├── home/
│   └── you/
│       ├── data.txt
│       └── projects/
│           └── app.kt
└── etc/
    └── config
```

`/home/you/data.txt` という**パス**は、根(ルート)から木をたどる道順に他なりません。Phase 0 の抽象的な木が、ファイルシステムという形で目の前にあった ―― これも分野をまたいだ伏線回収です。

> **7割理解ポイント**: ファイルシステムの内部実装(inode、ブロック管理等)は今は不要。「OS が生のストレージを隠して "ファイルと木構造のフォルダ" という見え方を与えている」で十分です。

## 9.3 ユーザモードとカーネルモード ―― 特権の壁

ファイルを読むには、結局ストレージという**ハードウェアに触れる**必要があります。でも第6章と、以前の CPU/OS の話で見たように、**アプリはハードに直接触れません**。ここで、以前話した「CPU の2つのモード」を正式に据えます。

```
ユーザモード  : アプリ(あなたのJVM等)が動く。できることが制限されている。ハードに直接触れない
カーネルモード: OS が動く特権モード。ハードを直接操作できる。何でもできる
```

なぜ壁を設けるのか ―― もしどんなアプリもディスクやネットワークを直接叩けたら、他人のファイルを勝手に読み書きしたり、ハードを占有したりし放題で、安全も安定も保てません。だから**危険な操作(ハードへのアクセス)は、すべて OS(カーネルモード)に集約**し、アプリは「OS にお願いする」形に限定されています。

では、ユーザモードのアプリは、どうやってカーネルモードの OS に仕事を頼むのか。その**唯一の正式な窓口**が、次のシステムコールです。

---

## 9.4 システムコール ―― OS へのお願いの窓口(約束の回収)

以前、「アプリは OS に依頼してハードを触る。その仕組みがシステムコールで、第9章の主役」と約束しました。回収します。

**システムコール(system call / syscall)** とは、**ユーザモードのアプリが、カーネルモードの OS に特権的な仕事を依頼するための、制御された "扉"** です。流れはこうです。

```
アプリ(ユーザモード)
   │  「このファイルを読んで」= read システムコール
   ▼
CPU がカーネルモードに切り替わり、OS のコードに入る    ← 特権の壁を、正規の扉で越える
   │  OS がストレージへの読み取りを実行(ハードを操作)
   ▼
結果(読んだデータ)を持って、ユーザモードのアプリに戻る
```

これが、以前話した「CPU の主導権がアプリと OS を行き来する」の、I/O 版の正体です。代表的なシステムコールには、`open`(ファイルを開く)、`read`(読む)、`write`(書く)、`close`(閉じる)、`socket`(通信の口を作る)などがあります。

あなたが Kotlin で書く高水準のコードは、水面下で必ずこれらに行き着きます。

```kotlin
val text = File("data.txt").readText()
//  ↓ 水面下では、おおよそ
//  open("data.txt")  → read(...) → read(...) → ... → close()
//  という一連のシステムコールに変換されている
```

### システムコールはタダではない

重要なのはここ。**システムコールには手間(オーバーヘッド)があります**。ユーザ↔カーネルのモード切り替えや、OS 内部の処理が挟まるため、ただのメソッド呼び出しよりずっと重い。**1回のシステムコールは、CPU から見ると小さな "遠出" のようなもの**。だから「システムコールをむやみに大量発行しない」ことが、I/O 性能の鍵になります ―― これが 9.6 のバッファリングにつながります。

## 9.5 割り込み ―― ハードから OS への「呼び出しベル」

もう一つの基礎、**割り込み(interrupt)**。第7章で「タイマー割り込みで OS が CPU を取り戻す」と出てきましたが、割り込みはもっと一般的な仕組みです。

**割り込みとは、ハードウェアが CPU/OS に「何か起きたよ」と知らせるための信号**です。

```
ディスクの読み取りが完了した   → 割り込み → OS「データが届いた」
キーボードが押された           → 割り込み → OS「入力が来た」
ネットワークにパケットが届いた  → 割り込み → OS「通信が来た」
タイマーが時間切れになった      → 割り込み → OS「スレッドを交代させよう」(第7章)
```

割り込みのありがたさは、**CPU が "できたか?できたか?" と問い続けなくて済む**こと。もし割り込みがなかったら、CPU はディスクに「もう読めた?」と延々と聞き続ける(ビジーウェイト)しかなく、その間は他の仕事ができません。割り込みがあるおかげで、**CPU は依頼を出したら別の仕事に移り、完了したら割り込みで呼び戻される**。この「頼んで、待たずに他をやり、できたら知らされる」構造が、9.8 の非同期 I/O の土台です。

## 9.6 バッファリング ―― まとめて運ぶ(実測・演習の核心)

いよいよ実務の核心、**バッファリング**です。9.4 で「システムコールは重い」と言いました。では、8MB のファイルを **1バイトずつ** 読んだらどうなるでしょう?

```kotlin
// 無バッファ:1バイト読むごとに read システムコール
FileInputStream(f).use { input ->
    var b = input.read()          // ← この read() 1回ごとに OS へ遠出
    while (b != -1) { /* 使う */ ; b = input.read() }
}
```

8MB = 800万バイトなので、**800万回のシステムコール**。遠出を800万回。これは悲惨です。

対して **バッファリング** は、**一度に大きなかたまり(例: 8KB)をメモリ上の "バッファ" に読み込んでおき、1バイトずつの要求はそのバッファから返す**。バッファが空になったら、また8KBまとめて読む。

```kotlin
// バッファあり:8KBまとめて1回のシステムコール → あとはメモリから配る
FileInputStream(f).buffered().use { input ->   // .buffered() を挟むだけ
    var b = input.read()
    while (b != -1) { /* 使う */ ; b = input.read() }
}
```

これなら、システムコールは 800万回 → **約1000回**(8MB ÷ 8KB)に激減します。**実測**すると:

```
① 無バッファ (1バイトずつ) : 2256 ms
② バッファあり            :  207 ms
→ バッファありは 約 11 倍 速い(読んだ合計は一致)
```

**同じ8MBを読むのに、11倍の差。** 違いは「OS への遠出を何回したか」だけです。

> **買い物のたとえ**: 冷蔵庫の材料を、**1品ごとにスーパーへ往復**する(無バッファ)か、**カート1台にまとめて1往復**する(バッファ)か。往復(システムコール)そのものが重いので、まとめ買いが圧勝します。

Kotlin/JVM では、`buffered()`、`bufferedReader()`、`BufferedInputStream`/`BufferedOutputStream` などがこれを提供します。**実務では、ファイルやネットワークの読み書きはバッファを挟むのが基本**。フレームワークやライブラリの多くは内部でバッファリングしていますが、生のストリームを直接1バイトずつ扱うと、この罠にはまります。

## 9.7 なぜ I/O は遅いのか ―― 整理

「I/O は遅い」の理由を、2つに分けて整理します。

1. **デバイス自体が遅い**(第5章)。ディスクやネットワークは、CPU・メモリに比べて桁違いに遅い。これは物理的な事実で、変えられません。
2. **1操作あたりのシステムコール(モード切替)のオーバーヘッド**。これは 9.6 のバッファリングで**減らせます**。

つまり I/O 高速化の基本戦略は、「**遠くて遅い相手(1)への往復回数(2)を、できるだけ減らす**」こと。バッファリングは (2) への対策、そして次の非同期 I/O は「(1) の待ち時間を無駄にしない」ための対策です。

---

## 9.8 ブロッキング vs ノンブロッキング I/O ―― 待ち方の違い(第7・8章の回収)

ここで、第7章の「スレッド」と第8章の「並行」が、I/O と合流します。この章で一番実務に効くところです。

### ブロッキング I/O ―― 待っている間、スレッドは固まる

普通に `read()` を呼ぶと、**データが届くまで、そのスレッドは停止して待ちます**。これを **ブロッキング I/O** と呼びます。書き方は素直ですが、**待っている間、そのスレッドは何もできず拘束されます**。

```kotlin
val data = socket.read()   // データが来るまで、このスレッドはここで固まって待つ
```

1件2件なら問題ありません。でも ―― **1万件の接続を同時にさばく Web サーバ**を考えてください。各接続がブロッキング I/O で待つなら、**1万本のスレッドが、それぞれ待ちのために拘束**されます。第7章の実測を思い出してください。プラットフォームスレッドは**約3.2万本が限界**で、各1MB のスタックを食う。1万本の "待つだけのスレッド" は、資源の大浪費であり、スケールしません。「I/O バウンドな仕事(待ちが多い)」に素朴なスレッドを当てると破綻する、という第7章の話が、ここで具体的になりました。

### ノンブロッキング / 非同期 I/O ―― 待ちにスレッドを縛りつけない

そこで発想を変えます。9.5 の割り込みを思い出してください ―― **「頼んで、待たずに他をやり、できたら知らされる」**。この仕組みを使えば、**待ちのたびにスレッドを1本ずつ張り付ける必要がなくなります**。

少数のスレッドが「今どの接続の準備ができたか」を OS に教えてもらい(準備できた接続だけを処理する)、**1本のスレッドで何千もの接続を面倒みる**。待ちの間、スレッドは別の接続の仕事をする。これが **ノンブロッキング / 非同期 I/O** です。

### そして Coroutines が嬉しい理由

第7・8章で見た **Coroutines(や仮想スレッド)** は、まさにこれを**書きやすく**してくれます。コルーチンの中で I/O 待ちに入ると、そのコルーチンは **サスペンド(中断)** して、**下地の OS スレッドを手放します**。手放されたスレッドは、別のコルーチンの仕事を進める。I/O が完了すると、コルーチンは**再開**されます。

```kotlin
// 見た目はブロッキングのように素直に書けるのに…
suspend fun fetch(id: Int): Data {
    val data = httpClient.get(id)   // ← ここで "待つ" が、スレッドは固まらず手放される
    return process(data)
}

// 1万タスクを起動しても、下地は少数のスレッドで回る(第7章:100万コルーチンも可能)
coroutineScope {
    repeat(10_000) { id -> launch { fetch(id) } }
}
```

**「ブロッキングのように素直に書けるのに、実際にはスレッドを拘束しない」** ―― これが Coroutines が I/O 処理で嬉しい理由です。第7章の「軽量」(スレッドを1対1で使わない)、第8章の「構造化並行性」(後始末が保証される)、そしてこの章の「I/O 待ちでスレッドを手放す」が、ここで一本につながりました。

> **実務接続のまとめ**: I/O バウンド(待ちが多い)な処理 ―― 多くの Web バックエンドがこれ ―― では、**ブロッキング I/O + スレッド大量生成**は破綻する。**非同期 I/O / Coroutines** なら、少数のスレッドで大量の待ちをさばける。「なぜ async や Coroutines を使うのか」に、原理から答えられるようになりました。

## 9.9 全体像の締め ―― OS はアプリとハードの仲介役

最後に、ぐっと引いて Phase 2 全体を一枚の絵にします。あなたが `val text = File("data.txt").readText()` と書いて、それを処理する ―― この何気ない1行に、Phase 2 の全部が詰まっています。

```
  あなたの Kotlin コード                                    (第1章 アプリ層)
       │
   ・文字列やオブジェクトは "ビット" として表現され           (第2章)
   ・コードは バイトコード→機械語 になって CPU が実行         (第3・4章)
   ・CPU は OS のスケジューラに CPU 時間を借りて動く          (第7章)
       │
   ・データはメモリに置かれ、キャッシュ階層で速さが決まり       (第5章)
   ・仮想記憶で他プロセスから隔離され、ヒープ/スタックに住み     (第6章)
   ・複数スレッドで扱うなら競合に注意し、同期で守る            (第8章)
       │
   ・ファイルを読むには、ユーザ→カーネルの壁を                (第9章)
     システムコール という扉で越えて OS に依頼し
   ・OS がハード(ストレージ)を操作し、割り込みで完了を知り
   ・遅い I/O は バッファ でまとめ、待ちは 非同期 でしのぐ
       ▼
  結果があなたのコードに返ってくる
```

貫いているのは、**「あなたのアプリはユーザモードの一住人にすぎず、CPU を使うにも(OS がスケジュールし)、メモリを使うにも(OS が仮想記憶で管理し)、外の世界に触れるにも(OS にシステムコールで頼む)、すべて OS が仲介している」** という構図です。第1章で「OS はハードとアプリの仲介役」と一言で言ったことの、中身が全部見えました。

そして第1章の「なぜか本番が遅い」障害対応 ―― あのとき挙がった容疑者(GC ポーズ、スワップ、CPU の取り合い、I/O 待ち…)を、あなたは今、**どの層の話なのか・なぜ起きるのか、原理から説明できます**。コンピュータは、もう中身の見えないブラックボックスではありません。

## 演習問題

**バッファあり/なしの実測**が主役です。`strace` が使えるなら、システムコールの回数も覗いてみましょう。

### 問1(推奨・必須:バッファの効果)
9.6 のコードで、無バッファ(`FileInputStream` を直接)とバッファあり(`.buffered()`)のファイル読み込み時間を比較してください。何倍の差が出ましたか? なぜ差が出るのかを、「システムコール」の回数で説明してください。

### 問2(オプション・strace)
Linux 環境があれば、システムコールの回数を実際に覗けます。無バッファ版とバッファあり版のプログラムを、それぞれ次のように実行して、`read` システムコールの回数を比べてください。
```bash
strace -c -e trace=read java YourProgram
```
無バッファ版で `read` が桁違いに多いことが確認できるはずです。(strace がない環境ならスキップで OK)

### 問3(システムコールの理解)
「ユーザモードのアプリは、なぜファイルを直接読めず、システムコールを経由しなければならないのか」を説明してください。(第6章の隔離・安全とつなげて)

### 問4(ブロッキング I/O とスレッド)
「1万件の接続を同時にさばく Web サーバを、接続ごとにブロッキング I/O で待つプラットフォームスレッドを1本ずつ立てて作る」設計の問題点を、第7章の知識(スレッドの数・コスト)と結びつけて説明してください。また、非同期 I/O / Coroutines がこれをどう解決するかを述べてください。

### 問5(Phase 2 総まとめ)
第1章で出てきた「本番のレスポンスが特定の時間帯だけ急に遅くなる。コードは変えていない」という障害。Phase 2 で学んだ知識から、**どの層に・どんな容疑者がいるか**を3つ以上挙げてください。(各章を思い出しながら)

---

## 演習の解答

### 問1(解答例)
バッファありが数倍〜十数倍速くなります(実測例では約11倍)。無バッファでは `read()` を呼ぶたびに `read` システムコールが発行され、8MB を1バイトずつ読むと800万回もの「ユーザ↔カーネルの遠出」が発生します。バッファありは、一度に大きなかたまり(例: 8KB)を1回のシステムコールで読み、以降はメモリ上のバッファから配るため、システムコールが約1000回に激減します。差の正体は「OS への遠出の回数」です。

### 問2(解答例)
`strace -c` はシステムコールを集計します。無バッファ版は `read` の回数がファイルのバイト数に近い莫大な数になり、バッファあり版は「バイト数 ÷ バッファサイズ」程度に収まります。回数の桁違いの差が、そのまま実行時間の差の原因だと目で確認できます。

### 問3(解答例)
ハードウェア(ストレージ等)は全アプリで共有する資源であり、どのアプリも直接触れると、他人のファイルを勝手に読み書きしたりハードを占有したりできてしまう(安全・安定が崩れる)。そこで OS はアプリをユーザモード(制限つき)に閉じ込め、ハードへのアクセスという特権操作はカーネルモードの OS に集約する。アプリはシステムコールという制御された扉を通してのみ OS に依頼できる。この壁が、第6章のプロセス隔離と合わせて、システムの安全と安定を支えている。

### 問4(解答例)
- **問題点**: 各接続がブロッキング I/O で待つと、1接続=1スレッドが「待つだけ」で拘束される。1万接続なら1万スレッド。だがプラットフォームスレッドは各約1MB のスタックを持ち、上限(実測で数万本)に近づく/超える。大量の待ちスレッドはメモリを浪費し、コンテキストスイッチも増えて、スケールしない。
- **解決**: 非同期 I/O は、待ちのたびにスレッドを張り付けず、少数のスレッドで多数の接続を面倒みる(準備できた接続だけ処理する)。Coroutines なら、I/O 待ちでコルーチンがサスペンドして下地のスレッドを手放すため、少数のスレッドで1万タスクの待ちをさばける。素直に書けて、かつスレッドを拘束しない。

### 問5(解答例・複数可)
- **アプリ層/JVM(第6章)**: GC が動いてアプリが一瞬止まった(GC ポーズ)。あるいはメモリリークでヒープが膨らみ、GC 頻発・ポーズ長期化。
- **メモリ/OS 層(第6章)**: メモリ不足で OS がスワップし、遅いディスクへの退避・読み戻しが発生(第5章:ディスクは桁違いに遅い)。
- **CPU/OS 層(第7章)**: その時間帯だけ別のプロセス/スレッドが CPU を食い、自分のアプリに CPU の順番が回ってこない(スケジューリング)。
- **並行(第8章)**: 特定の負荷でロック競合が増え、スレッドが待ち合ってスループットが落ちる。
- **I/O 層(第9章)**: その時間帯に I/O(DB・外部 API・ディスク)が遅く、ブロッキングでスレッドが待ち続ける。
- こうして「アプリのコードは正しいのに遅い」を、層ごとに切り分けて当たりをつけられる ―― これが Phase 2 の最大の成果です。

> 問1 でバッファの効果を体感し、問5 で「遅いの容疑者を層ごとに挙げられた」なら、Phase 2 の締めくくりとして完璧です。
