# 第8章 並行性と同期

> **この章のねらい**
> ロードマップで **「実務で一番刺さる章」** と位置づけた、Phase 2 の山場です。第7章で「スレッドはヒープを共有する ―― 協力しやすいが、ぶつかると事故る」と予告しました。この章では、その **"事故" が実際にどう起きるか** を自分の目で見て(**カウンタが実測でずれます**)、**なぜ再現しにくいのか**、そして **どう防ぐか** を身につけます。

## 8.0 この章で解けること

- なぜ「テストは通るのに本番でたまに数字が合わない」が起きるのか
- `synchronized`、`@Volatile`、`AtomicInteger` は、それぞれ何を保証して何を保証しないのか
- デッドロック(処理が永遠に止まる)はなぜ起き、どう避けるのか
- Coroutines を使えば競合は起きないのか?(答え: いいえ)

まずは諸悪の根源、**競合状態**から。実物を見るのが一番なので、いきなり事故を起こします。

## 8.1 競合状態 ―― 共有メモリの事故

第7章で、同じプロセス内のスレッドは**ヒープ(データ)を共有**すると学びました。では、**複数のスレッドが、同じ変数を同時に書き換えたら**どうなるでしょう。

こんな単純なコードで確かめられます。8本のスレッドが、共有カウンタを各10万回ずつ増やす。期待値は 8 × 100,000 = **800,000** です。

```kotlin
import kotlin.concurrent.thread

var counter = 0   // 共有カウンタ

fun main() {
    val threads = List(8) {
        thread { repeat(100_000) { counter++ } }   // 各スレッドが10万回 ++
    }
    threads.forEach { it.join() }
    println("期待値 800000 → 実際 $counter")
}
```

これを **同じコードのまま3回**実行した実測結果:

```
実行1 : 749065   (50,935 件が消えた)
実行2 : 800000   (たまたま正しい!)
実行3 : 312175   (487,825 件も消えた)
```

**同じコードなのに、毎回ちがう。しかも半分近く消えることもある。** なぜでしょう?

### `counter++` は「1つの操作」ではない

犯人はここです。`counter++` は、人間には一瞬に見えますが、CPU にとっては**3つの操作**です(第4章を思い出してください)。

```
counter++  の正体:
  ① 読む   : counter の現在値をレジスタに読み込む
  ② 足す   : レジスタで +1 する
  ③ 書く   : 結果を counter に書き戻す
```

この3ステップの**途中で、コンテキストスイッチ(第7章)が割り込む**と、事故が起きます。カウンタが 100 のときに、2本のスレッドが重なった例:

```
時間  スレッドA              スレッドB              counter
 │   ①読む(100)                                   100
 │                          ①読む(100)             100  ← Aの+1前に読んでしまった
 │   ②足す→101                                     100
 │                          ②足す→101              100
 │   ③書く(101)                                    101
 │                          ③書く(101)             101  ← 上書き!2回 ++ したのに +1 しか増えてない
```

**2回インクリメントしたのに、結果は +1 だけ。** 1回ぶんの更新が**消えた(lost update)**。これが大量に積み重なって、80万が74万や31万に化けるのです。

このように、**複数のスレッドが共有データに同時アクセスし、タイミング次第で結果が変わってしまう状態**を **競合状態(race condition)** と呼びます。「レース(競争)」の名のとおり、どのスレッドが先に書き込むかの**早い者勝ちレース**の結果次第で、答えが変わってしまう。

## 8.2 なぜ並行バグは再現しにくいのか

さっきの実測で、**実行2だけ偶然 800,000(正しい)** になったのを覚えていますか。ここに、並行バグの一番いやらしい性質が表れています。

事故が起きるかどうかは、**コンテキストスイッチが "たまたま" 3ステップの途中で割り込むか**にかかっています。そして切り替えのタイミングは、OS のスケジューラ(第7章)がその時々の状況で決めるもので、**実行のたびに変わり、プログラム側からは制御も予測もできません**。だから:

- **運がよければ事故が起きず、テストが通ってしまう**(実行2)
- 本番で負荷が上がり、スレッドの重なりが増えると**急に顕在化する**
- バグを捕まえようとログを仕込むと、そのわずかな遅延でタイミングが変わり、**症状が消える**(観測すると挙動が変わるので "ハイゼンバグ" とも呼ばれます)

「テストは通るのに本番で時々データがおかしい、しかも再現できない」 ―― 多くのエンジニアを苦しめるこの現象の正体が、この**非決定性**です。**再現しにくいのはバグが軽いからではなく、むしろ潜んでいる証拠**。だからこそ、次に見る「そもそも事故が起きない書き方」で**予防**するのが鉄則になります。

## 8.3 クリティカルセクション ―― 同時に入ってはいけない部屋

事故は、共有データを**「読んで・変えて・書く」その区間**で起きていました。この、**一度に1本のスレッドしか実行してはいけない危険な区間**を **クリティカルセクション(critical section / 危険領域)** と呼びます。

解決の原理はシンプルです ―― **クリティカルセクションには、同時に1本のスレッドしか入れないようにする**。これを **相互排他(mutual exclusion)** と言います。

> **たとえ**: 1人用のトイレ。中に入ったら**鍵をかける**。次の人はドアが開くまで外で待つ。誰も「中に人がいるのに同時に入る」ことができない。クリティカルセクションを、この "1人用の部屋" にしてしまえばいい。

その「鍵」を提供するのが、次の**ロック**です。

---

## 8.4 ロック / ミューテックス ―― 鍵をかける

**ロック(lock)**、別名 **ミューテックス(mutex = mutual exclusion)** は、まさにトイレの鍵です。クリティカルセクションに入る前に鍵を取り(acquire)、出るときに返す(release)。鍵は1つしかないので、**取れなかったスレッドは、返されるまで外で待ちます**。

Kotlin/JVM では `synchronized` ブロックがこれを提供します。

```kotlin
import kotlin.concurrent.thread

val lock = Any()    // 鍵(何でもいいのでオブジェクトを1つ用意)
var counter = 0

fun main() {
    val threads = List(8) {
        thread {
            repeat(100_000) {
                synchronized(lock) {   // ← 鍵を取ってから入る。ここがクリティカルセクション
                    counter++
                }                      // ← ブロックを抜けると鍵を返す
            }
        }
    }
    threads.forEach { it.join() }
    println("実際 $counter")
}
```

`synchronized(lock) { ... }` の中は、**同時に1本のスレッドしか実行できません**。他のスレッドは鍵が返るまで待つので、`counter++` の3ステップが割り込まれることがなくなります。

**実測**すると、`synchronized` 版は **3回とも 800,000**。事故が消えました。

### ロックの代償

万能ではありません。ロックは、待たせることで事故を防ぐ = **並行性を犠牲にしています**。クリティカルセクションが長かったり、あちこちで鍵を取り合ったりすると、スレッドが待ってばかりで**並行にした意味がなくなる**(最悪、1本ずつ順番に実行しているのと変わらない)。だから、**鍵をかける範囲は必要最小限に、短く**が鉄則です。そして、鍵を2つ以上使い始めると ―― 後述のデッドロックという新たな罠が待っています。

---

## 8.5 アトミック操作 ―― 分割できない一撃

カウンタを増やすだけのために、いちいちロック(部屋の鍵)を持ち出すのは大げさです。もっと軽い手があります。

そもそもの問題は「`counter++` が3ステップに分かれ、途中で割り込まれる」ことでした。なら、**3ステップを "分割不可能な1回の操作" にまとめてしまえばいい**。これを **アトミック操作(atomic operation)**(atomic = それ以上分けられない)と呼びます。ハードウェア(CPU)が「読んで・比べて・書く」を割り込ませず一撃で行う命令を提供していて、それを使います。

Kotlin/JVM では `AtomicInteger` が代表です。

```kotlin
import java.util.concurrent.atomic.AtomicInteger
import kotlin.concurrent.thread

val counter = AtomicInteger(0)

fun main() {
    val threads = List(8) {
        thread { repeat(100_000) { counter.incrementAndGet() } }  // ++ を1回のアトミック操作で
    }
    threads.forEach { it.join() }
    println("実際 ${counter.get()}")
}
```

`incrementAndGet()` は「読む・足す・書く」を**割り込まれない一撃**で行うので、更新が消えません。**実測でも3回とも 800,000**。しかも、ロックのように「待たせる」のではなく `synchronized` より軽く済むことが多い。

**使い分け**: 単一の変数を増減する・入れ替えるといった**単純な操作**なら **アトミック操作(`AtomicInteger` 等)**。複数の変数をまとめて整合的に更新するような**複雑なクリティカルセクション**なら **ロック(`synchronized`)**。まず「アトミックで済むか?」を考え、無理ならロック、という順番が実務的です。

## 8.6 @Volatile ―― 「消える更新」とは別の、「見えない更新」問題

ここで、競合状態とは**別種の、もっと気づきにくい**並行の落とし穴を紹介します。「更新が消える」のではなく「**更新が他スレッドから見えない**」問題です。

第5章を思い出してください。各 CPU コアは、データを自分のキャッシュに持っています。すると、こんなことが起きえます ―― **スレッドAがある変数を書き換えても、その新しい値がAのキャッシュ(やレジスタ)に留まったままで、スレッドBには古い値のまま見えている**。

典型例が「停止フラグ」です。

```kotlin
var running = true    // ← これだと危ない

// スレッド1(ワーカー): running が false になったら止まりたい
while (running) { doWork() }

// スレッド2: 停止させたい
running = false       // ← この変更が、スレッド1に "見えない" ことがある
```

スレッド2が `running = false` にしても、スレッド1がその更新を見られず、**ループが止まらない**ことがあります(スレッド1は自分のキャッシュ上の古い `true` を見続ける)。

これを防ぐのが **`@Volatile`** です。`@Volatile` を付けた変数は、**書き込みが必ず主記憶まで届き、読み込みも必ず最新を取りに行く**ので、スレッド間で最新の値が見えることが保証されます。

```kotlin
@Volatile var running = true   // これで停止フラグが他スレッドに見える
```

### 超重要な誤解ポイント

`@Volatile` が保証するのは **可視性(見えること)だけ** で、**アトミック性(不可分であること)は保証しません**。つまり:

```kotlin
@Volatile var counter = 0
counter++   // ← @Volatile を付けても、これは依然として競合する!
```

`counter++` は相変わらず「読む・足す・書く」の3ステップ。`@Volatile` は各ステップで最新値をやり取りさせるだけで、**3ステップが割り込まれるのは防げません**。だから **`@Volatile` はカウンタには使えない**。「1本が書き、他が読むだけのフラグ」には `@Volatile`、「複数が読み書きするカウンタ」には `AtomicInteger` かロック ―― この住み分けが要です。よくある誤りなので、ここは押さえておく価値があります。

## 8.7 デッドロック ―― 待ち合いの膠着

ロックで事故は防げます。が、**ロックを2つ以上使う**と、新しい災難が生まれます ―― **デッドロック(deadlock)**。**複数のスレッドが、互いに相手の持っている鍵を待ち続けて、永遠に動けなくなる**状態です。

2つの鍵 `lockA`, `lockB` を、2本のスレッドが**逆の順序**で取ろうとすると起きます。

```kotlin
// スレッド1: A → B の順で取る
thread {
    synchronized(lockA) {
        Thread.sleep(50)
        synchronized(lockB) { /* ... */ }   // B を待つ
    }
}
// スレッド2: B → A の逆順で取る ← これが原因
thread {
    synchronized(lockB) {
        Thread.sleep(50)
        synchronized(lockA) { /* ... */ }   // A を待つ
    }
}
```

**実測**すると、こうなりました(3秒待っても両方止まったまま):

```
T1: lockA を取得、次に lockB を待つ…
T2: lockB を取得、次に lockA を待つ…
→ 3秒たっても両スレッドが停止 = デッドロック発生!
```

```
スレッド1 : [lockA を保持] ──→ lockB がほしい ──┐
                                                │ 互いに相手待ち
スレッド2 : [lockB を保持] ──→ lockA がほしい ──┘   永遠に解けない
```

> **たとえ**: 狭い橋の両端から2人が同時に踏み込み、真ん中で鉢合わせ。どちらも「相手が下がってくれないと進めない」と待ち、両方が意地を張って一歩も引かない。永遠に膠着。

### デッドロックの4条件と、その破り方

デッドロックは、次の**4つの条件がすべて同時に成り立つ**ときにだけ起きます(コフマン条件)。

```
① 相互排他     : 資源(鍵)は同時に1人しか持てない
② 保持と待機   : 鍵を1つ持ったまま、別の鍵を待つ
③ 横取り不可   : 持っている鍵を、強制的に奪えない
④ 循環待機     : 「AはBを待ち、BはAを待つ」という待ちの輪ができている
```

裏を返せば、**この4つのうち1つでも崩せば、デッドロックは起きません**。実務で一番使いやすいのが **④循環待機を崩す**方法 ―― **すべてのスレッドで、鍵を取る順序を統一する**。

```kotlin
// 両方のスレッドが「必ず A → B の順」で取る → 待ちの輪ができない
thread { synchronized(lockA) { synchronized(lockB) { /* ... */ } } }
thread { synchronized(lockA) { synchronized(lockB) { /* ... */ } } }
```

全員が同じ順で鍵を取れば、「Aを持つ人がBを待ち、Bを持つ人がAを待つ」という循環が構造上できません。橋の例で言えば「渡るときは必ず北側から」と全員で決めておくようなもの。複数ロックを扱うときの**最重要の心得**です。

## 8.8 実務での防ぎ方 ―― 優先順位つきの心得

ここまでの道具を、実務で使う順に整理します。**上ほど根本的で安全**です。

1. **そもそも共有可変状態を持たない(最強)**。書き換えられるデータを複数スレッドで共有しなければ、競合は起きようがありません。**イミュータブル(不変)なデータ**を使う、状態を**1つのスレッド/コルーチンに閉じ込める**、スレッド間は**メッセージの受け渡し**でやり取りする ―― これらが効きます。「ロックで守る」より「守るべき共有をなくす」ほうが上位の戦略です。
2. **既製の並行データ構造を使う**。自前でロックを書く前に、`ConcurrentHashMap` や `java.util.concurrent` の部品を使う。プロが正しく作った安全な部品に任せるのが賢い。
3. **単純な更新はアトミック操作**(`AtomicInteger` 等)。
4. **どうしても複数変数をまとめて守るならロック**。ただし**範囲は最小限・短く**、**複数ロックは取得順序を統一**。

「並行処理は自分でロックを書いて頑張る」のは最後の手段です。**まず "共有しない・既製品に任せる" を考える**のが、事故を減らす一番の近道です。

## 8.9 Coroutines の構造化並行性 ―― 触り

第7章で、Coroutines を「軽量スレッド」として紹介しました。ここではその**扱いやすさを支える考え方 ―― 構造化並行性(structured concurrency)** を軽く触れます(深入りは Coroutines 専門の学習に譲り、"地図" だけ持ち帰りましょう)。

### 生のスレッドの困りごと

生のスレッドは「投げっぱなし」にできてしまいます。スレッドを起動した後、それが**いつ終わるか・エラーで死んでいないか**を、起動した側が把握しづらい。終わり損ねたスレッドが裏で生き続ける(第6章のリーク的な問題)、エラーが握りつぶされる、といったことが起きます。

### 構造化並行性の考え方

Coroutines の `coroutineScope` は、**「あるスコープの中で起動した子は、そのスコープを抜ける前に必ず全部終わる(か、まとめてキャンセルされる)」**ことを保証します。親子関係が構造(ネスト)として表現され、迷子のコルーチンが生まれない。

```kotlin
import kotlinx.coroutines.*

suspend fun main() = coroutineScope {
    launch { delay(100); println("タスク1 完了") }
    launch { delay(200); println("タスク2 完了") }
    // coroutineScope は、中で launch した2つが両方終わるまで、ここで待ってから抜ける
    println("2つのタスクを起動した(まだ待っている)")
}
// このブロックを抜けた時点で、子は全て完了済み。1つがエラーなら他もキャンセルされ、エラーが伝播する
```

「起動したものの後始末が、スコープの構造によって保証される」 ―― これが、大量の並行処理を**破綻なく**扱える理由です(第7章の「10万タスクが普通に書ける」の、書きやすさの側面)。

### 大事な注意 ―― Coroutines でも競合は起きる

誤解されがちですが、**Coroutines を使っても競合状態はなくなりません**。コルーチンも、複数が**同じ可変データを共有**すれば、この章の冒頭とまったく同じカウンタずれが起きます。軽量なだけで、共有の危険は同じ。

対処もこの章の延長です ―― 共有可変状態を避ける、コルーチン用のロック(`kotlinx.coroutines.sync.Mutex`)で守る、あるいは**以前あなたが質問した `Semaphore`(同時アクセス数の制限)**で下流資源への流量を絞る。あのセマフォの話は、まさにこの「共有資源をどう安全に扱うか」というテーマの一部でした。

> **7割理解ポイント**: ここは「構造化並行性 = 起動した並行処理の後始末をスコープが保証してくれる仕組み」「Coroutines でも共有可変状態には注意」の2点だけで十分。詳細は、いずれ Coroutines を本格的に使うときに深掘りすれば大丈夫です。

## この章のまとめ

- **競合状態**: 複数スレッドが共有データに同時アクセスし、タイミング次第で結果が変わる。`counter++` は「読む・足す・書く」の3ステップで、途中で割り込まれると更新が消える(実測: 80万が74万や31万に化ける)
- **再現しにくい**のは、事故が起きるかどうかがコンテキストスイッチのタイミング(制御不能・非決定的)に依存するから。テストが通っても潜んでいる ―― だから**予防**が鉄則
- **クリティカルセクション**(同時に1本しか入れない区間)を **相互排他** で守る
- **ロック/ミューテックス**(`synchronized`)= 鍵。事故は消えるが、待たせるので**範囲は最小限に**(実測: 常に正しい 80万)
- **アトミック操作**(`AtomicInteger`)= 分割できない一撃。単純な更新に最適で軽い(実測: 常に 80万)
- **`@Volatile`** は **可視性のみ**保証、**アトミック性は保証しない**。フラグには使えるが `counter++` には使えない(頻出の誤解)
- **デッドロック**: 複数の鍵を逆順で取ると、互いに相手待ちで永久膠着(実測: 3秒でも解けず)。**4条件のどれか(特に循環待機)を崩す** = **ロック取得順序を統一**
- 実務の優先順位: **①共有可変状態を持たない → ②既製の並行部品 → ③アトミック → ④最小限のロック(順序統一)**
- **Coroutines の構造化並行性**は後始末を保証してくれるが、**共有可変状態の競合は依然起きる**

これで Phase 2 の山場を越えました。「テストは通るのに本番で時々おかしい」の正体に名前がつき、防ぎ方の地図も手に入りました。次はいよいよ最終章、**ファイル・I/O・システムコール** ―― 「アプリはどうやって OS に頼み、OS はどうハードを仲介するのか」という、Phase 2 全体を締めくくる話です。

## 演習問題

**競合を自分で起こして、直す**のが主役です。1コアのマシンでも(むしろ)再現するので、ぜひ手を動かしてください。

### 問1(推奨・必須:競合を起こす→直す)
8.1 のカウンタコードを動かし、**複数回実行**してください。毎回同じ値になりますか? 期待値(800000)とずれますか? 次に `synchronized` または `AtomicInteger` で直し、**今度は毎回正しくなる**ことを確認してください。なぜ最初はずれ、直すと合うのかを、`counter++` の3ステップを使って説明してください。

### 問2(再現性)
問1 で、直す前のコードを10回実行したら、たまたま1〜2回だけ正しい値(800000)が出ることがあります。**なぜ "たまに正しくなる" のか**、そして **なぜこれが並行バグを厄介にするのか** を説明してください。

### 問3(@Volatile の誤解)
ある人が「カウンタが競合するなら `@Volatile var counter = 0` にすれば直る」と言いました。これは正しいですか? 正しくない場合、理由と正しい対処を述べてください。

### 問4(デッドロック)
次のコードはデッドロックの危険があります。(1) なぜ危険か、(2) 4条件のどれを崩せば防げるか、(3) 具体的な修正を答えてください。
```kotlin
fun transfer(from: Account, to: Account, amount: Int) {
    synchronized(from) {
        synchronized(to) {
            from.balance -= amount
            to.balance += amount
        }
    }
}
// スレッドX: transfer(accountA, accountB, 100)
// スレッドY: transfer(accountB, accountA, 50)   ← 逆順!
```

### 問5(Coroutines)
「Coroutines は軽量で安全だから、共有変数を複数のコルーチンから `++` しても競合しない」 ―― これは正しいですか? 理由を述べ、もし共有が必要なら何を使うべきか挙げてください。

## 演習の解答

### 問1(解答例)
直す前は、実行のたびに値が変わり、期待値 800000 を下回ります(更新が消えるため)。`synchronized`/`AtomicInteger` にすると毎回 800000。理由: `counter++` は「①読む・②足す・③書く」の3ステップ。無防備だと、あるスレッドが①②の途中で別スレッドが割り込んで同じ値を読み、両者が同じ結果を書き戻して**片方の +1 が消える**。ロックは「同時に1本しか入れない」ことで、アトミック操作は「3ステップを割り込めない一撃にする」ことで、この割り込みを封じるため、更新が消えなくなる。

### 問2(解答例)
事故が起きるかは、コンテキストスイッチが `counter++` の3ステップの途中に割り込むかどうか次第。その割り込みタイミングは OS のスケジューラが決め、実行ごとに変わる**非決定的**なもの。運よく割り込みが悪いタイミングに入らなければ、更新が消えず正しい値になる ―― だから「たまに正しい」。これが厄介なのは、**テストが偶然通ってしまい、バグが潜伏したまま本番に出て、負荷が上がって初めて顕在化し、しかも再現困難**だから。「たまに正しい」は「安全」ではなく「運が良かっただけ」。

### 問3(解答例)
**正しくない。** `@Volatile` が保証するのは**可視性(最新値が他スレッドに見えること)だけ**で、**アトミック性は保証しない**。`counter++` は依然「読む・足す・書く」の3ステップのままで、途中で割り込まれれば更新は消える。正しい対処は、`AtomicInteger`(単純なカウンタなら最適)か `synchronized`(ロック)を使うこと。`@Volatile` が有効なのは「1本が書き、他は読むだけのフラグ」のようなケース。

### 問4(解答例)
1. **なぜ危険か**: スレッドX は `accountA → accountB` の順でロックを取り、スレッドY は `accountB → accountA` の**逆順**で取る。X が A を持って B を待ち、Y が B を持って A を待つと、互いに相手待ちの**循環**ができてデッドロック。
2. **崩す条件**: ④**循環待機**(取得順序を統一すれば輪ができない)。
3. **修正例**: すべての `transfer` で、**ロックを取る順序を一定のルールで統一**する。たとえば口座に一意な ID を振り、**常に ID の小さいほうから先にロックする**。
```kotlin
fun transfer(from: Account, to: Account, amount: Int) {
    val (first, second) = if (from.id < to.id) from to to else to to from
    synchronized(first) {
        synchronized(second) {
            from.balance -= amount
            to.balance += amount
        }
    }
}
```
これで、どの2口座間でも全スレッドが同じ順序でロックを取るため、循環が生じずデッドロックしない。

### 問5(解答例)
**正しくない。** Coroutines は軽量なだけで、**共有可変状態への競合はスレッドと同様に起きる**。複数のコルーチンが同じ変数を `++` すれば、`counter++` の3ステップが割り込まれて更新が消えうる。対処: 共有可変状態を避ける(不変データ、状態を1コルーチンに閉じ込める)、コルーチン用ロック `Mutex`(`withLock { }`)で守る、`AtomicInteger` を使う、下流資源への流量なら `Semaphore` で絞る、など。「軽量 ≠ 競合しない」を区別することが肝心。

> 問1 で「自分の手でカウンタをずらし、`synchronized`/`AtomicInteger` で直せた」ら、この章 ―― そして Phase 2 の山場 ―― は大成功です。あとは「競合は予防が鉄則」「@Volatile は可視性だけ」「複数ロックは順序統一」の3つが残れば、実務で十分戦えます。
