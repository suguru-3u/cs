# 第5章 メモリ階層とキャッシュ

> **この章のねらい**
> ここから **第2部・メモリ編** です。そしてこの章は、**Phase 1 との合流点**。Phase 1 で「配列は速い、連結リストは遅い」と習ったとき、どこか腑に落ちなかったのではないでしょうか。どちらも要素を順に辿るだけ、計算量(Big-O)も同じ O(n) なのに、なぜ実際の速度はそんなに違うのか?
> その答えが **キャッシュ** です。この章を終えると、その差を**自分の手で実測**し(実際に**100倍以上**の差が出ます)、「なぜそうなるのか」を原理から説明できるようになります。第2部は実務に効くゾーン。ここは少し気合いを入れていきましょう。

---

## 5.0 この章で解ける謎

第4章で、CPU はこう動くと学びました ―― **メモリからデータをレジスタに運び、計算し、メモリに書き戻す**。計算そのもの(ALU)は爆速です。では、何がボトルネックになるのか?

答えは **「メモリからデータを運んでくる時間」** です。そしてこの「運んでくる時間」を劇的に左右するのが、**データがメモリ上にどう並んでいるか**。並び方しだいで、まったく同じ処理が100倍遅くなる ―― これがこの章の主題です。

---

## 5.1 メモリは「遠い」 ―― CPU が待たされている

意外な事実から始めます。現代の CPU は、**仕事時間の大半を "データ待ち" で消費している**ことがよくあります。計算が遅いのではなく、計算する材料が届くのを待っているのです。

なぜか。CPU は GHz(秒間何十億回)の速さで動きますが、メインメモリ(RAM)はそれに比べると**ずっと遅い**のです。CPU が「このデータちょうだい」とメモリに頼んでから届くまで、CPU は何百回分ものクロックを**手持ち無沙汰で待つ**ことになります。

> **料理人のたとえ**
> 超一流のシェフ(CPU)がいて、包丁さばきは一瞬。でも食材は**街の向こうの倉庫(RAM)**にあって、一品ごとに使いを走らせて取りに行かせる。シェフの腕がいくら速くても、食材待ちで手が止まる。これが CPU とメモリの関係です。

この「待ち」をなんとか減らす工夫が、これから見る **メモリ階層** と **キャッシュ** です。

---

## 5.2 メモリ階層 ―― 速さと大きさのトレードオフ

「全部レジスタみたいに速いメモリにすればいい」と思いますよね。でもできません。**速いメモリは、高価で、小さくしか作れない**からです。逆に大容量で安いメモリは遅い。「速い・大きい・安い」は同時に成り立たない、これがハードウェアの宿命です。

そこでコンピュータは、**速くて小さいものを CPU の近くに、遅くて大きいものを遠くに**、段階的に配置します。これが **メモリ階層** です。

```
         速い・小さい・近い・高価
   ▲   ┌─────────────┐
   │   │ レジスタ      │  CPU の中。数十個だけ。最速(第3章)
   │   ├─────────────┤
   │   │ L1 キャッシュ  │  CPU直近。数十KB
   速   │ L2 キャッシュ  │  少し外。数百KB〜数MB
   さ   │ L3 キャッシュ  │  さらに外。数十MB(複数コアで共有)
   │   ├─────────────┤
   │   │ メインメモリ   │  RAM。数GB〜。電源切ると消える
   │   │  (RAM)        │
   │   ├─────────────┤
   │   │ ストレージ     │  SSD/HDD。数百GB〜。電源切っても残る
   ▼   └─────────────┘
         遅い・大きい・遠い・安価
```

数字で「遠さ」を感じてもらうために、**レジスタへのアクセスを「1秒」だとしたら**、各階層はだいたいこのくらいの体感になります(あくまでイメージのたとえです)。

```
レジスタ   : 1秒          ← 手元のメモ
L1         : 数秒          ← 机の上
L2         : 十数秒        ← 引き出しの中
L3         : 1分前後       ← 部屋の棚
メインメモリ : 数分          ← 倉庫へ取りに行く ← ここで急に「遠く」なる
SSD        : 数日           ← 別の街
HDD        : 数ヶ月〜       ← 海の向こう
```

注目すべきは、**メインメモリ(RAM)で急に「遠く」なる**段差です。CPU から見ると、RAM はもう「倉庫まで使いを出す」感覚。だから、できるだけ **L1〜L3 の "近い棚" でデータを間に合わせたい**。その「近い棚」が **キャッシュ** です。

> **キャッシュとは**: よく使うデータを、CPU の近くの速いメモリ(L1〜L3)に**コピーして置いておく**しくみ。倉庫(RAM)まで毎回取りに行かず、机の上(キャッシュ)で済ませるための工夫です。

## 5.3 局所性 ―― なぜキャッシュがうまくいくのか

ここで疑問。「よく使うデータを近くに置く」と言っても、CPU は**次に何を使うか分からない**はずでは? でたらめにアクセスされたら、キャッシュは役に立たないのでは?

ところが、実際のプログラムのアクセスには**強い偏り(クセ)**があります。これを **参照の局所性(locality)** と呼び、2種類あります。

### 時間的局所性(temporal locality)

**一度使ったデータは、すぐまた使われやすい**。

```kotlin
var sum = 0
for (i in 0 until n) {
    sum += data[i]   // sum と i は毎回くり返し使われる
}
```

`sum` や `i` は、ループのたびに何度も登場します。「さっき使ったものは、また使う」 ―― だからキャッシュに置いておけば次もヒットする。

### 空間的局所性(spatial locality)

**あるデータを使ったら、その "隣" のデータも使われやすい**。

上のループの `data[i]` は、`data[0]`, `data[1]`, `data[2]`… と**隣どうしを順に**舐めていきます。「`data[5]` を使ったなら、次は `data[6]` だろう」 ―― この予想がよく当たる。

キャッシュは、この2つの「クセ」に**賭けて**います。そして次の節の「キャッシュライン」が、空間的局所性に賭ける具体的な仕掛けです。

---

## 5.4 キャッシュライン ―― キャッシュは「1個ずつ」運ばない(最重要)

ここがこの章の心臓部です。

CPU が RAM から「`data[5]` が欲しい」と頼んだとき、**RAM は `data[5]` 1個だけを返すのではありません**。`data[5]` を含む**一定サイズのかたまり**(典型的には **64バイト**)を、まるごとキャッシュに運び込みます。このかたまりを **キャッシュライン** と呼びます。

```
CPU「data[5] ちょうだい」
RAM はその周辺ごと(例: data[4]〜data[11] の64バイト分)をまとめてキャッシュへ運ぶ

      ┌────────────────── 1本のキャッシュライン(64バイト)──────────────────┐
RAM:  │ data[4] │ data[5] │ data[6] │ data[7] │ data[8] │ ... │ data[11] │
      └──────────────────────────────────────────────────────────────────┘
                  ↑ 欲しかったのはこれだけ。でも隣もまとめて来る
```

なぜ周辺ごと運ぶのか? ―― **空間的局所性に賭けているから**です。「`data[5]` を使ったなら、次は隣の `data[6]` を使うだろう。なら一緒に持ってきておこう」というわけです。

この一点から、すべてが説明できます。

### 連続して並んだデータ(配列)は速い

`data` が**メモリ上で隙間なく連続**して並んでいれば、`data[5]` を取りに行ったついでに `data[6]`〜`data[11]` までキャッシュに入ります。次に `data[6]` を使うとき、**もう手元(キャッシュ)にある**のでRAMまで行かずに済む。**1回の "倉庫行き" で、複数要素ぶんを稼げる**わけです。

### 散らばったデータは遅い

逆に、使うデータが**メモリ上のあちこちに散らばっている**と、`data[5]` のために運ばれた周辺(`data[6]`等)は使われず無駄になり、次に必要なデータはまた別の遠い場所にある。**毎回 "倉庫行き" が発生**します。同じ64バイトを運んでも、使うのは1個だけ。残り63バイト分の運賃が丸損です。

**これが「配列は速く、連結リストは遅い」の正体です。** 5.7 で本題として深掘りします。

## 5.5 キャッシュヒットとミス、そしてプリフェッチ

用語を整理します。

- **キャッシュヒット**: 欲しいデータが、もうキャッシュ(近い棚)にある状態。速い。
- **キャッシュミス**: キャッシュになく、より遅い階層(RAM など)まで取りに行く必要がある状態。遅い。

プログラムの性能は、ざっくり言えば **「いかにヒットを増やし、ミスを減らすか」** で決まります。計算量(Big-O)が同じでも、ミスだらけのコードは桁違いに遅くなります。

もう一つ、ハードウェアには **プリフェッチ(先読み)** という賢い仕掛けがあります。「`data[0]`, `data[1]`, `data[2]`… と順番にアクセスしているな。じゃあ次は `data[3]` だろう」と**規則性を見抜いて、頼まれる前に先回りでキャッシュに運んでおく**機能です。

ここが重要で、**プリフェッチは "順番に・規則的に" アクセスするときだけ効きます**。でたらめ(ランダム)な順でアクセスすると、ハードウェアは次を予測できず、先読みが効きません。順次アクセスが圧倒的に速いのは、キャッシュラインの恩恵に加えて、このプリフェッチも効くからです。

## 5.6 【実測】順次 vs ランダム ―― 100倍の差を自分の目で

理屈はここまで。**本当に差が出るのか、測ってみましょう。** これがこの章の手を動かすパートです。

実験の設計はこうです。「次に辿るインデックス」を入れた配列 `next` を用意し、それを6400万回辿って合計を取ります。**アクセスする回数はまったく同じ**。違うのは並び方だけ。

- **順次**: `next[i] = i+1`(0→1→2→… と隣へ進む。配列を順に舐めるのと同じ)
- **ランダム**: 全要素をでたらめな順で1周する輪(メモリ上を飛び回る。散らばった連結リストを辿るのと同じ)

```kotlin
// Kotlin 版(あなたのマシンで試せます。IntArray は連続したプリミティブ配列)
import kotlin.random.Random

fun buildSequential(n: Int) = IntArray(n) { (it + 1) % n }

fun buildRandomCycle(n: Int): IntArray {
    val perm = IntArray(n) { it }
    val r = Random(42)
    for (i in n - 1 downTo 1) {                 // シャッフル
        val j = r.nextInt(i + 1)
        val t = perm[i]; perm[i] = perm[j]; perm[j] = t
    }
    val next = IntArray(n)
    for (i in 0 until n) next[perm[i]] = perm[(i + 1) % n]
    return next
}

fun chase(next: IntArray, steps: Int): Long {   // next を steps 回辿る
    var idx = 0; var acc = 0L
    repeat(steps) { idx = next[idx]; acc += idx }
    return acc
}

fun main() {
    val n = 64_000_000                            // 256MB(L3 を大きく超える)
    val seq = buildSequential(n)
    val rnd = buildRandomCycle(n)
    repeat(3) { chase(seq, n); chase(rnd, n) }    // ウォームアップ(第4章!JIT を効かせる)

    var t = System.nanoTime(); chase(seq, n)
    println("順次   : ${(System.nanoTime() - t) / 1_000_000} ms")
    t = System.nanoTime(); chase(rnd, n)
    println("ランダム: ${(System.nanoTime() - t) / 1_000_000} ms")
}
```

第4章のウォームアップをちゃんと入れているのがポイントです(温める前に測ると JIT 前の遅い数字を見てしまう)。

### 実測結果

あるマシン(L1=32KB / L2=1MB / L3=33MB)で、**アクセス回数を6400万回に固定**したまま配列サイズだけ変えて測ると、こうなりました。

```
配列サイズ        順次アクセス   ランダムアクセス   ランダム / 順次
[1] 16KB (L1内)     約 100ms        約 100ms          1.0 倍   ← 差なし
[2] 800KB (L2内)    約 100ms        約 365ms          3.7 倍
[3] 16MB (L3内)     約 100ms        約 9700ms        97   倍
[4] 256MB (L3超)    約 105ms       約 13500ms       128   倍   ← 100倍超!
```

(数値はマシンによって変わりますが、**この傾向は普遍的**です。手元で実行すると、自分のマシンのキャッシュ事情が透けて見えます。)

この表は、メモリ階層のすべてを物語っています。読み解きます。

- **順次アクセスは、サイズによらず一定(約100ms)。** どんなに配列が大きくても速い。キャッシュライン(隣もまとめて運ぶ)とプリフェッチ(先読み)が完璧に効いているからです。
- **ランダムアクセスは、データがキャッシュからあふれるほど急激に遅くなる。**
  - 16KB は L1 に丸ごと収まる → ランダムでも全部キャッシュ内 → 差なし(1.0倍)
  - 大きくなるにつれ、L1 → L2 → L3 → RAM と、ミスのたびに辿る階層が遠くなる
  - L3 を超えると毎回 RAM(あの「倉庫行き」)→ **128倍**

**同じ回数アクセスして、同じ合計を計算しているのに、128倍。** この差は、計算量(Big-O では両方とも O(n))には一切現れません。**Big-O が見ているのは "操作の回数"、現実の速度を左右するのは "メモリのどこを触るか"** なのです。

## 5.7 Phase 1 との合流 ―― 配列 vs 連結リストの正体

お待たせしました。Phase 1 の宿題に答えを出します。

### なぜ配列は速いのか

配列(Kotlin の `IntArray` や、`ArrayList` の内部)は、**全要素がメモリ上に隙間なく連続して並んでいます**。

```
配列(連続):
  [a][b][c][d][e][f][g][h] ...   ← メモリ上でぴったり隣り合っている
```

順に舐めると、キャッシュライン(隣もまとめて運ぶ)とプリフェッチ(先読み)が両方効いて、**ほぼキャッシュヒットだけで走り抜けます**。さっきの実験の「順次アクセス(約100ms 一定)」がまさにこれです。

### なぜ連結リストは遅いのか

連結リスト(`LinkedList`)は、各要素が **`Node` という別々のオブジェクト**で、`next` ポインタで「次はあそこ」とつながっています。問題は、**これらの Node がメモリ上のどこに置かれるかは保証されず、たいてい散らばっている**こと。

```
連結リスト(散らばり):
  メモリ:  ...[Node_c]......[Node_a].........[Node_e]...[Node_b]...
            辿る順:  Node_a → Node_b → Node_c → ...(毎回メモリ上を飛ぶ)
```

`next` を辿るたびに**メモリ上の遠い別の場所**へジャンプする ―― これは、さっきの実験の **「ランダムアクセス」そのもの**です(実際あの実験の `buildRandomCycle` は、散らばった連結リストを辿る動きを再現しています)。だから毎回キャッシュミスし、**最大100倍以上遅くなる**。

> **核心**: 配列の順次走査 = 実験の「順次」。連結リストの走査 = 実験の「ランダム」。あの128倍の差が、そのまま「配列 vs 連結リスト」の差です。Phase 1 の「なんとなく配列が速い」が、ここで物理的な理由とともに腑に落ちます。

### JVM での注意点(実務で効く)

Kotlin/JVM では、さらに2つ知っておくと得をします。

**① `LinkedList` はほぼ使わない**。理論上は「先頭への挿入が O(1)」などの利点がありますが、現実にはキャッシュミスの遅さがそれを帳消しにすることが多く、**ほとんどの場面で `ArrayList`(= 連続配列)が速い**です。「リスト?とりあえず `ArrayList`」が実務の基本姿勢で問題ありません。

**② プリミティブ配列(`IntArray`)と、ボックス化されたリスト(`List<Int>`)は別物**。これは見落としがちです。

```kotlin
val a = IntArray(n)        // int が連続して並ぶ → キャッシュに優しい・速い
val b = Array<Int>(n){0}   // Integer "オブジェクト" への参照が並ぶ → 実体は散らばる
val c = List(n){0}         // 同上(中身は Integer オブジェクト)
```

`IntArray` は 32ビット整数が**そのまま連続**して並びます。一方 `Array<Int>` や `List<Int>` は、各要素が `Integer` という**オブジェクト**になり(ボックス化)、配列に並ぶのは**そのオブジェクトへの参照(住所)**です。実体のオブジェクトはヒープ上に散らばるので、辿ると連結リストに近い「散らばりアクセス」になります。**数値を大量にぶん回すホットな処理では、`List<Int>` ではなく `IntArray`/`LongArray`/`DoubleArray` を使う** ―― これは性能調査でよく効く一手です。

(なぜ「参照」「ヒープ」「散らばる」のかは、第6章でメモリの中身を見るとさらにスッキリします。今は「プリミティブ配列は連続、オブジェクトのリストは散らばる」で十分。)

## 5.8 実務接続 ―― この章の持ち帰り

- **「リストは基本 `ArrayList`(連続配列)」**。`LinkedList` は理論上の O(1) に釣られず、まず疑う。
- **数値の重い処理は `IntArray`/`DoubleArray` などプリミティブ配列**。`List<Int>` のボックス化はキャッシュ性能を落とす。
- **Big-O がすべてではない**。計算量が同じでも、メモリの触り方しだいで実速度は桁違い。性能調査では「このループ、メモリ上を飛び回っていないか?」という視点を持つ。
- **データは "使う順に・固めて" 並べると速い**。バラバラのオブジェクトを大量に辿る設計より、まとまった配列を順に処理する設計のほうがキャッシュに優しい(列指向データやゲーム開発の「データ指向設計」が速いのも根は同じ理由です)。

これらは「毎回意識する」ものではありません。普段は `ArrayList` で気楽に書き、**"ここが性能の急所だ" という箇所だけ**この章の知識を引っぱり出せれば十分です。

## この章のまとめ

- CPU は速いが **RAM は遠い**。CPU は「データ待ち」で時間を食う。だから速い小さなメモリを近くに置く **メモリ階層**(レジスタ→L1→L2→L3→RAM→ストレージ)がある
- **キャッシュ** = よく使うデータを CPU 近くの速いメモリに置いておく工夫。**局所性**(時間的=また使う/空間的=隣も使う)に賭けている
- **キャッシュライン**: メモリは1個ずつでなく **64バイトのかたまり**で運ばれる → 連続データ(配列)は隣もまとめて手に入って速い、散らばったデータは毎回ミスして遅い
- **プリフェッチ**(先読み)は順次アクセスのときだけ効く。ランダムアクセスでは効かない
- 実測で、アクセス回数が同じでも **順次は一定・ランダムは最大100倍以上遅い**。Big-O には現れない差
- **配列が速く連結リストが遅い**のは、配列=連続(順次アクセス)、連結リスト=散らばり(ランダムアクセス)だから。Phase 1 の謎が解けた
- 実務:**リストは `ArrayList`、数値は `IntArray`**、Big-O だけで判断しない

第2部に入った最初の章で、いきなり実務に直結する話でした。次章はメモリの "中身" ―― あなたの Kotlin オブジェクトが実際どこに置かれ、なぜ `OutOfMemoryError` や `StackOverflowError` が起きるのか ―― に踏み込みます。

## 演習問題

実測を体感する章なので、**手を動かす問1が主役**です。

### 問1(推奨・実測)
5.6 の Kotlin コードを自分のマシンで動かし、「順次」と「ランダム」の実行時間を測ってください。何倍の差が出ましたか? さらに余裕があれば、配列サイズ `n` を小さく(例: `100_000`)して測り直し、**差が縮む**ことを確認してください。なぜ小さいと差が縮むのか、説明してみてください。

### 問2(局所性の判定)
次の (1)(2) のうち、**キャッシュに優しい(速い)**のはどちらでしょう。理由も添えてください。
```kotlin
val grid = Array(1000) { IntArray(1000) }   // 1000×1000 の2次元配列

// (1) 行優先で走査
for (i in 0 until 1000) for (j in 0 until 1000) sum += grid[i][j]

// (2) 列優先で走査
for (j in 0 until 1000) for (i in 0 until 1000) sum += grid[i][j]
```
**ヒント**: `grid[i]` は1行ぶんが連続した配列。`i` を固定して `j` を回すと、メモリ上ではどう動く?

### 問3(Phase 1 との接続)
「`ArrayList` と `LinkedList` は、要素を全部順に辿る処理(走査)の計算量がどちらも O(n) なのに、実際は `ArrayList` のほうがずっと速いことが多い」 ―― これはなぜですか。この章の言葉(キャッシュライン・局所性・散らばり)を使って説明してください。

### 問4(プリミティブ vs ボックス化)
1億個の整数を合計する処理を書くとき、`List<Int>` と `IntArray` のどちらを選ぶべきですか。理由を、メモリ上の並び方の違いから説明してください。

### 問5(考える・実務)
ある性能調査で、「計算量は O(n) で理論上は十分速いはずのループが、実測ではやけに遅い」とわかりました。この章の知識から、どんな原因を疑い、どう確かめますか? 思いつくことを書いてください。

---

## 演習の解答

### 問1(解答例)
大きい配列(L3 を超えるサイズ)では、ランダムが順次の数十〜100倍以上遅くなるはずです。`n` を小さく(キャッシュに収まるサイズに)すると差が縮むのは、**データ全体がキャッシュに乗ってしまえば、ランダムに飛んでも結局すべてキャッシュヒットで済む**から。差が出るのは「キャッシュに収まりきらず、ランダムアクセスのたびに遅い階層(RAM など)へ取りに行く」ときだけです。あなたのマシンで差が急に開くサイズは、そのマシンの L2/L3 容量を反映しています。

### 問2(解答例)
**(1) 行優先が速い。** `grid[i]` は1行ぶんが連続した配列です。`i` を固定して `j` を 0→1→2… と回すと、**連続したメモリを順に舐める**ので、キャッシュライン(隣もまとめて運ぶ)とプリフェッチが効きます。一方 (2) は `j` 固定で `i` を回すため、毎回 `grid[i]`(別の行=別の配列)へ飛び、行から1要素だけ使ってまた別の行へ。**散らばりアクセス**になりキャッシュミスが増えます。「2次元配列は、外側ループを行(連続している側)にする」が定石です。

### 問3(解答例)
`ArrayList` の中身は**連続した配列**なので、順に辿るとキャッシュラインで隣の要素もまとめて手に入り、プリフェッチも効いて、ほぼキャッシュヒットで走れます。`LinkedList` は各 `Node` がメモリ上に**散らばって**いて、`next` を辿るたびに遠い場所へジャンプ → 毎回キャッシュミス。操作の回数(O(n))は同じでも、1操作あたりのメモリ待ち時間が桁違いに長いので、実速度で大きく負けます。Big-O は回数を数えますが、キャッシュミスのコストは数えないのです。

### 問4(解答例)
**`IntArray` を選ぶ。** `IntArray` は 32ビット整数がメモリ上に**連続**して並ぶので、合計ループは順次アクセスとなりキャッシュに優しく高速。`List<Int>` は各要素が `Integer` オブジェクトにボックス化され、リストに並ぶのは**そのオブジェクトへの参照**で、実体はヒープに散らばります。辿ると散らばりアクセスになり遅い。大量の数値処理ではプリミティブ配列が正解です。

### 問5(解答例)
- **疑うこと**: メモリ上を飛び回る「散らばりアクセス」になっていないか。具体的には ―― 連結リストや、オブジェクトのリスト(`List<SomeObject>`)を大量に辿っていないか / 2次元配列を列優先で舐めていないか / `List<Int>` のようなボックス化されたコレクションを数値処理に使っていないか。
- **確かめ方**: データ構造を連続配列(`ArrayList`/`IntArray`)に置き換えて測り直す、アクセス順を「メモリの並び順」に揃える、プロファイラでキャッシュミス率(取れる環境なら)を見る、など。「計算量は正しいのに遅い → メモリの触り方を疑う」が、この章で身につけてほしい嗅覚です。

> 問1 を実際に動かして「100倍の差」を自分の目で見られたら、この章は大成功。あとは「リストは ArrayList、数値は IntArray、Big-O だけで判断しない」という実務感覚が残れば十分です。
