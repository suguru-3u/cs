## 7.1 プロセス ―― 実行中のプログラム、自分専用の部屋

**プロセス**とは、ざっくり「**実行中のプログラム1つ**」です。あなたが JVM アプリを起動すれば、それは1つのプロセス。ブラウザは別のプロセス。音楽プレーヤーもまた別のプロセス。

第6章を思い出してください。OS は各プロセスに **自分専用の仮想アドレス空間**(自分だけのヒープ、スタック、コード領域…)を与え、互いに隔離していました。だからプロセスは「**自分専用の部屋を持った住人**」のようなものです。

```
プロセスA (JVMアプリ)      プロセスB (ブラウザ)
 ┌────────────┐            ┌────────────┐
 │ 専用のヒープ  │            │ 専用のヒープ  │
 │ 専用のスタック │            │ 専用のスタック │
 │ 専用のコード  │            │ 専用のコード  │
 └────────────┘            └────────────┘
   互いのメモリには触れない(隔離されている)
```

この隔離のおかげで、片方のアプリがクラッシュしても、もう片方は巻き込まれにくい(安全・安定)。反面、**プロセスどうしが情報をやり取りするのは "重い"** です。別々の部屋なので、OS を仲介して郵便を送るような手間がかかります。「同じ仕事を協力してやりたい」ときに、いちいちプロセス間通信をするのは大変。そこで登場するのが、次の**スレッド**です。

---

## 7.2 スレッド ―― 一つの部屋の中の、複数の働き手

**スレッド**は、**1つのプロセスの中で動く "実行の流れ"** です。ポイントはここ ―― **同じプロセス内のスレッドたちは、そのプロセスのメモリ(特にヒープ)を共有します**。ただし、各スレッドは**自分専用のスタック**を持ちます。

```
プロセス(1つのオフィス)
 ┌──────────────────────────────────────┐
 │        共有のヒープ(みんなの資料棚)          │  ← 全スレッドで共有!
 │  [オブジェクトA] [オブジェクトB] [オブジェクトC]   │
 ├──────────────────────────────────────┤
 │ スレッド1     スレッド2     スレッド3          │
 │ 専用スタック   専用スタック   専用スタック       │  ← 各自のメモ帳(専用)
 └──────────────────────────────────────┘
```

> **オフィスのたとえ**: プロセスは1つのオフィス。ヒープは**全員で使う共有の資料棚やホワイトボード**。スレッドはそこで働く**複数のスタッフ**で、各自が**自分の机とメモ帳(スタック)**を持つ。スタッフどうしは同じ資料棚を見られるので、協力(情報共有)が簡単。

この「**ヒープを共有する**」性質が、スレッドの強力さの源であると同時に、**危険の源**でもあります。複数のスタッフが同じホワイトボードに同時に書き込んだら、内容がぐちゃぐちゃになりかねない ―― これが次章(第8章)の主役、**競合状態(race condition)** の芽です。今は「スレッドはメモリを共有するから、協力しやすいが、ぶつかると事故る」とだけ覚えておいてください。

そしてもう一つ、第6章とのつながり。**各スレッドは自分専用のスタックを持つ = スレッドを1本作るごとにメモリ(スタックぶん)を消費する**。この事実が、後半の「作りすぎると遅い」に効いてきます。

## 7.3 コンテキストスイッチ ―― 1個の CPU で複数を回すからくり

ここで、以前の質問への約束を回収します。第3〜4章で学んだとおり、**1個の CPU コアは、一度に1つの命令の流れしか実行できません**。なのに、私たちのマシンでは何十ものプロセス・スレッドが「同時に」動いているように見える。どうやって?

答えは、**OS が高速にバトンタッチしている**からです。

```
時間 →
コア:  [スレッドA を少し][スレッドB を少し][スレッドA を少し][スレッドC を少し]...
              ↑切替        ↑切替          ↑切替
     人間の目には「全部同時に動いている」ように見える(実は超高速の切替)
```

スレッドAを少し実行 → **Aの状態(レジスタの中身、プログラムカウンタ=第3・4章)を保存** → Bの状態を復元 → Bを少し実行 → …。この**「今のスレッドの状態を保存し、次のスレッドの状態を復元する」切り替え**を **コンテキストスイッチ(context switch)** と呼びます。

切り替えのきっかけは、以前説明した**タイマー割り込み**です。OS は「一定時間たったら私に戻ってこい」とタイマーを仕掛けてスレッドに CPU を貸します。時間が来るとタイマーが鳴り、CPU は強制的に OS のコードに戻り、OS が「次は誰にするか」を決めて、また貸す。この繰り返しです。

### コンテキストスイッチはタダではない

大事な注意。**切り替えには手間(コスト)がかかります**。状態の保存・復元そのものの時間に加えて、第5章で学んだ**キャッシュが冷える**問題があります。スレッドAで温まっていたキャッシュ(Aのデータ)は、Bに切り替わるとBのデータで上書きされ、Aに戻ったときには**またキャッシュミスから**やり直し。この「切り替えの度に払う税金」が、後半の「スレッド多すぎると遅い」の一因です。

(補足: 同じプロセス内の**スレッド間**の切り替えは、**プロセス間**の切り替えより軽いです。スレッドはメモリ空間を共有しているので、切り替えても仮想アドレスの対応表などを丸ごと入れ替えずに済むからです。)

## 7.4 スケジューリング ―― 誰に CPU を渡すか

「次はどのスレッドに CPU を貸すか」を決めるのが、OS の **スケジューラ** です。限られた CPU 時間を、多数のスレッドに**公平に(かつ優先度を考慮して)**配分するのが仕事。

各スレッドに割り当てられる一回ぶんの持ち時間を **タイムスライス** と呼びます。持ち時間を使い切る(かタイマーが鳴る)と、スケジューラが介入して次のスレッドに交代させます。

重要なのは、この交代が **強制的(preemptive)** であること。あるスレッドが「まだ譲りたくない」と思っても、OS はタイマー割り込みで**強制的に取り上げられます**。だから、1本のスレッドが無限ループに陥っても、OS ごと固まらずに他のスレッドは動き続けられる(OS がちゃんと CPU を取り戻すので)。

> **7割理解ポイント**: スケジューリングのアルゴリズム(ラウンドロビン、優先度付き等)の詳細は今は不要です。「OS が持ち時間を区切って公平に配り、強制的に交代させている」というイメージで十分。

## 7.5 並行 vs 並列 ―― 似て非なる二つ(最重要)

ここがこの章の核心で、実務でも混同されやすい区別です。

### 並行(concurrency)

**複数のタスクを "同時期に" 進めること**。ある期間で見れば複数の仕事が進行しているが、**ある一瞬を切り取ると動いているのは1つ**かもしれない。1個の CPU コアでも、コンテキストスイッチで細切れに切り替えれば実現できます。「複数の仕事を**さばく段取り**」の話。

### 並列(parallelism)

**複数のタスクを "本当に同じ瞬間に" 実行すること**。これには**複数の CPU コアが物理的に必要**です。「複数の仕事を**同時に手を動かして**やる」話。

```
並行 (1コアでも可):  一人のシェフが、パスタを茹でながら合間にサラダを刻み、
                     ソースを煮る…と、複数の料理を切り替えながら進める
                     → 手は1つ。でも複数の料理が「進行中」

並列 (複数コア必要):  シェフが3人いて、それぞれパスタ・サラダ・ソースを
                     同時に手を動かして作る
                     → 手が3つ。本当に同時
```

**並行は "構造・段取り"、並列は "物理的な同時実行"**。並行だからといって並列とは限りません。

### この違いが実務で決定的な理由

「並行 ⊃ 並列」の関係と、**どういう仕事に効くか**を押さえてください。

- **I/O バウンドな仕事**(ネットワーク・DB・ディスクの**応答待ち**が主。多くの Web バックエンドがこれ)には、**並行が効きます**。待っている間は CPU が暇なので、その間に別のタスクを進められる。コアが1個でも、待ち時間を埋めるだけで大きく改善します。
- **CPU バウンドな仕事**(重い計算が主)には、**並列が効きます**。計算そのものを速くするには、複数コアで手分けするしかない。ただし**コア数を超えてスレッドを増やしても速くなりません**(手は増えないので)。

> **実測でわかる注意**: この確認に使ったマシンは **1コア**です。1コアでは、重い計算を2スレッドに分けても**2倍速くはなりません**(むしろコンテキストスイッチのぶん遅くなることも)。並列には物理的なコアが要る、を地で行く例です。一方で、後述のように「待ち」が多い仕事なら1コアでも並行で改善できます。

## 7.6 JVM スレッドと OS スレッド ―― 伝統的には1対1

では Kotlin/JVM のスレッドは、ここまでの「OS のスレッド」とどう関係するのか。

伝統的に、**JVM のスレッド1本 = OS のスレッド1本**でした(**プラットフォームスレッド**と呼びます)。あなたが Kotlin で `Thread { ... }` を作ると、その裏で**本物の OS スレッドが1本生成**されます。

```kotlin
val t = Thread {
    println("別スレッドで実行: ${Thread.currentThread().name}")
}
t.start()   // OS スレッドが1本生まれ、この処理がそこで走る
t.join()    // t が終わるのを待つ
```

つまり JVM スレッドは、OS スレッドのコストをそのまま引き継ぎます ―― **各スレッドが専用スタック(既定で約1MB、第6章)を持ち、コンテキストスイッチのコストもかかる**。ここから「作りすぎ問題」が生まれます。

## 7.7 なぜスレッドを作りすぎると遅い/危ないのか(実測)

「並行にしたいなら、タスクごとにスレッドをどんどん作ればいい」 ―― これは罠です。理由は2つ。

### ① そもそも作れる数に上限がある

各プラットフォームスレッドは専用スタック(約1MB)などの資源を食うため、**無限には作れません**。実際に「作れなくなるまで作り続ける」実験をすると:

```kotlin
fun main() {
    var count = 0
    try {
        while (true) {
            Thread { Thread.sleep(Long.MAX_VALUE) }.start()  // 寝かせておく
            count++
        }
    } catch (e: OutOfMemoryError) {
        println("$count 本でスレッド生成に失敗: ${e.message}")
    }
}
```

あるマシンでの実測結果:

```
約 31,800 本で OutOfMemoryError: unable to create native thread
(各スレッドのスタックサイズ = 1024k = 1MB)
```

数万本で頭打ちです。「リクエストごとに無制限にスレッドを作る」設計は、負荷が上がるとこの壁にぶつかって**スレッドを作れずエラー**になります。

### ② 上限に達する前に、切り替えコストで遅くなる

もっと手前の問題として、**動いているスレッドがコア数を大きく超えると、コンテキストスイッチ(7.3)ばかりに時間を取られて逆に遅くなります**。切り替えのたびにキャッシュが冷える(第5章)ぶんも積み重なる。

```
狭い厨房に、シェフを100人詰め込む
→ 手を動かす時間より、譲り合い・ぶつかり合い(切替)の時間のほうが増えて、料理が進まない
```

「並行にすれば速い」ではなく、「**適切な数**の並行が速い」。多すぎる並行はむしろ害、というのが実務の勘所です。

## 7.8 スレッドプール ―― 作らずに使い回す

上の2つの問題への定番の答えが **スレッドプール** です。発想はシンプル ―― **タスクごとにスレッドを作って捨てるのをやめ、あらかじめ決まった本数の "働き手" を用意しておいて、タスクを順に割り当てて使い回す**。

```
      ┌─ タスクの待ち行列(キュー)─┐
      │ [task1][task2][task3]...   │
      └────────────┬──────────────┘
                    │ 空いた働き手が取りに来る
      ┌─────────────┴─────────────┐
      │ 固定数のスレッド(例: 4本)を使い回す │
      │  [worker1][worker2][worker3][worker4] │
      └───────────────────────────┘
```

```kotlin
import java.util.concurrent.Executors

fun main() {
    val pool = Executors.newFixedThreadPool(4)   // 4本だけ用意して使い回す
    repeat(100) { id ->
        pool.submit {                            // 100タスクを投入(でもスレッドは4本)
            println("task $id → ${Thread.currentThread().name}")
        }
    }
    pool.shutdown()
}
```

100個のタスクを投げても、スレッドは4本のまま。**スレッド数に上限があり(爆発しない)、生成コストもかからず(使い回し)、キューに溜めることで急な負荷も受け止められる**。あなたが使う Web フレームワークやライブラリが内部でスレッドプールを持っているのは、このためです。

**本数の目安**: CPU バウンドな処理のプールは「**コア数と同じくらい**」が基本(それ以上増やしても並列度は上がらず切替が増えるだけ)。I/O バウンド(待ちが多い)なら、待っている間は他が動けるので、もう少し多めでも有効です。

## 7.9 「軽量スレッド」への布石 ―― 仮想スレッド / Coroutines(実測・第8章への橋)

ここまでの話には、根っこの制約がありました ―― **プラットフォームスレッドは "重い"**(1本=OSスレッド1本=約1MB スタック、切替も OS 任せ)。だから数万本が限界だし、プールで数を絞る必要があった。

では、**「待つ」だけの仕事**を大量にさばきたいときはどうする?(例: 1万件の外部 API を並行に呼び、それぞれ応答を待つ)。プラットフォームスレッドでは、1万本作るのも一苦労です。

そこで生まれたのが **軽量スレッド** という発想。**OS ではなくランタイム(JVM や言語)が管理する、ごく軽いスレッド**を大量に作り、それらを**少数の OS スレッドの上でうまく多重化(乗り合い)する**。待ちに入ったものは OS スレッドを手放し、別の軽量スレッドに譲る。切り替えも OS のコンテキストスイッチではなくユーザ空間で安く行う。

これを実測で見ると、対比が鮮烈です。

```
プラットフォームスレッド : 約 31,800 本が限界(前述)
仮想スレッド(Java 21)  : 100 万本を難なく完走(各100ms スリープ、合計 約15秒)
```

同じ「スレッド」でも、**軽量なら100万本**。この「軽量スレッド」を、

- **Java 21 の仮想スレッド(Virtual Threads)**
- **Kotlin の Coroutines(コルーチン)**

が提供します。Coroutines が「軽量」「数十万の並行処理が普通に書ける」と言われるのは、まさに**OS スレッドに1対1で対応せず、少数のスレッドの上で多数を多重化する**から。第7章で見た「プラットフォームスレッドの重さ」を回避する仕組みなのです。

```kotlin
// Java 21 仮想スレッド:100万本でも平気(プラットフォームでは不可能)
Executors.newVirtualThreadPerTaskExecutor().use { ex ->
    repeat(1_000_000) { ex.submit { Thread.sleep(100) } }
}
```

**その "軽さ" が具体的にどう実現され、どう使い、どんな落とし穴があるか(競合状態や構造化並行性)は、次の第8章の主役**です。この章は「なぜ軽量スレッドが必要とされたのか」という土台を作りました。

---

## 7.10 実務接続 ―― この章の持ち帰り

- **仕事の性質を見極める**。I/O バウンド(待ちが多い=多くの Web 処理)なら **並行**が効く。CPU バウンド(計算が重い)なら **並列**が効くが、コア数が上限。
- **スレッドを無制限に作らない**。「リクエストごとに新スレッド」は負荷でスレッド枯渇(実測:数万本で限界)や切替過多で破綻する。**スレッドプール**で数を絞って使い回す。
- **プールの本数**は、CPU バウンドならコア数前後、I/O バウンドなら多め、が出発点。
- **大量の "待ち" をさばくなら軽量スレッド**(Coroutines / 仮想スレッド)。プラットフォームスレッドの重さから解放される。
- これらを知っていると、「負荷を上げたらスレッドが増えて逆に遅くなった」「スレッドプールのサイズをどう決めるか」「なぜ async / Coroutines を使うのか」に、原理から答えられます。

## この章のまとめ

- **プロセス** = 実行中のプログラム。自分専用のメモリ空間を持ち、他と隔離される(安全だが、プロセス間のやり取りは重い)
- **スレッド** = プロセス内の実行の流れ。**ヒープを共有**し(協力しやすい/ぶつかると事故る=第8章)、各自**専用スタック**を持つ(=1本ごとにメモリを消費)
- **コンテキストスイッチ**: 1個のコアで複数を回すため、OS が状態を保存・復元して高速にバトンタッチ(タイマー割り込みが契機)。タダではない(切替コスト+キャッシュが冷える)
- **スケジューラ**が持ち時間(タイムスライス)を公平に配り、**強制的に**交代させる
- **並行(段取り、1コアでも可)** と **並列(物理的な同時、複数コア必要)** は別物。I/O バウンドは並行が、CPU バウンドは並列(コア数まで)が効く
- 伝統的に **JVM スレッド=OS スレッド1対1(プラットフォームスレッド、約1MB)**。**作りすぎると**、上限(実測 約3.2万本)に達する/切替過多で遅くなる
- だから **スレッドプール**で数を絞って使い回す
- 大量の待ちには **軽量スレッド(Coroutines / 仮想スレッド)**。OS スレッドに1対1で縛られず、実測で**100万本**も可能 ―― これが「軽量」の正体(詳細は第8章)

OS の本丸の入口として、「同時に動く」の仕組みが見えてきました。次の第8章は、この「メモリを共有するスレッド」が**同じデータに同時に触れたときに起きる事故**(競合状態・デッドロック)と、その防ぎ方 ―― **Phase 2 で実務に一番刺さる章**です。

## 演習問題

複数スレッドを実際に動かして、並行の挙動を体感するのが主役です。

### 問1(推奨・手を動かす:並行を見る)
次を動かして、出力を観察してください。`A` と `B` はきれいに交互になりますか? それとも入り乱れますか? なぜそうなるか、コンテキストスイッチとスケジューリングの言葉で説明してください。
```kotlin
fun main() {
    val a = Thread { repeat(5) { println("A-$it"); Thread.sleep(1) } }
    val b = Thread { repeat(5) { println("B-$it"); Thread.sleep(1) } }
    a.start(); b.start()
    a.join(); b.join()
}
```

### 問2(並行 vs 並列)
次の (1)(2) の仕事は、それぞれ「並行が効く」「並列が効く」のどちらでしょう。理由も。
1. 1000件の外部 API を呼び、各レスポンス(平均200ms待ち)を集める。
2. 1億個の数値を全部2乗して合計する(純粋な計算)。
また、(2) を **1コアのマシン**で4スレッドに分けたら、速くなると思いますか?

### 問3(作りすぎ問題)
「Web サーバで、リクエストが来るたびに新しいプラットフォームスレッドを作って処理する」設計があります。平常時は問題なくても、アクセスが急増したときに何が起きうるか、この章の知識で2つ挙げてください。また、その対策は?

### 問4(プール本数)
CPU バウンドな重い計算をたくさんさばくためのスレッドプールを作ります。コア数が8のマシンで、プールの本数を「8本」にする場合と「800本」にする場合、どちらが速そうですか? 理由を説明してください。

### 問5(Coroutines への布石)
「Kotlin の Coroutines なら、10万個の並行タスク(それぞれネットワーク待ち)を普通にさばける。でも同じことをプラットフォームスレッド10万本でやろうとすると難しい」 ―― この違いは何に由来しますか。この章の言葉(OSスレッド、スタック、多重化)で説明してください。

---

## 演習の解答

### 問1(解答例)
多くの場合、`A` と `B` は**完全な交互にはならず、まとまったり入り乱れたり**します。実行順序は OS のスケジューラが決めるもので、プログラム側からは保証されないからです。各スレッドは短い持ち時間ごとにコンテキストスイッチで切り替わり、`sleep` で待ちに入ると他方に CPU が渡る。その細かなタイミングは実行のたびに変わりうるため、出力順は非決定的になります。「**並行処理の実行順序は保証されない**」を体感するのが狙いです(これが第8章の競合状態の伏線)。

### 問2(解答例)
1. **並行が効く**。API 呼び出しは大半が「応答待ち」で CPU は暇。待っている間に別の呼び出しを進められるので、1コアでも大幅に改善します(I/O バウンド)。
2. **並列が効く**。純粋な計算なので、速くするには複数コアで手分けするしかない(CPU バウンド)。
- **1コアで (2) を4スレッドに分けても速くなりません**。手(コア)は1つのままで、むしろコンテキストスイッチのぶん遅くなることも。並列には物理的なコアが必要です。

### 問3(解答例)
- **起きうること**: ① スレッド数が上限(実測で数万本)に達し、`unable to create native thread` でスレッドを作れず処理が失敗する。② 上限前でも、動作スレッドがコア数を大きく超え、コンテキストスイッチとキャッシュ冷えで**スループットが落ちる**(増やしたのに遅くなる)。
- **対策**: スレッドプールで同時実行数に上限を設け、あふれたリクエストはキューで待たせる(または軽量スレッド/Coroutines を使う)。

### 問4(解答例)
**8本のほうが速い(に近い)。** CPU バウンドの計算は、同時に手を動かせる数=コア数(8)が上限です。800本にしても、実際に計算できるのは同時に8つだけで、残り792本は待ちながらコンテキストスイッチのコスト(状態の保存復元、キャッシュ冷え)を増やすだけ。むしろ遅くなります。CPU バウンドのプールはコア数前後が定石です。

### 問5(解答例)
プラットフォームスレッドは **OS スレッドと1対1**で、各本が約1MB のスタックなどの資源を持つ「重い」存在。10万本は資源的にほぼ不可能。一方 Coroutines は **軽量**で、OS スレッドに1対1で縛られず、**少数の OS スレッドの上で多数を多重化**します。ネットワーク待ちに入ったタスクは下地のスレッドを手放して別のタスクに譲るため、待ちが大量にあっても少ないスレッドで回せる。だから10万個の「待つ仕事」を安く並行にさばけます。

> 問1 を動かして「実行順は保証されない」を目撃し、問2 で「並行(待ち)と並列(計算)の効きどころ」を区別できたら、この章の目的は達成です。あとは「スレッドは無制限に作らずプール/軽量スレッド」が残れば十分。
