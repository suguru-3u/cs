## 6.0 この章で解けること

第1章の「なぜか本番が遅い」障害対応を覚えていますか。犯人候補に「GC が動いて一瞬止まった」「メモリが足りずスワップした」が挙がっていました。この章を終えると、その2つが何を言っているのか分かります。加えて、

- 深い再帰でよく出る `StackOverflowError` の正体(Phase 1 の再帰とつながる)
- ヒープが尽きる `OutOfMemoryError` の正体、そして**メモリリーク**の仕組み
- GC(ガベージコレクタ)が「捨てていいオブジェクト」をどう見分けているか

が説明できるようになります。まずは大枠、「仮想記憶」から。

## 6.1 仮想記憶 ―― 各プロセスは「メモリを独り占め」していると思い込む

第1章で OS は「限りあるハードウェアを複数のアプリに公平に貸し出す管理人」だと学びました。メモリ(RAM)も、その貸し出し対象です。

ここで問題。1台の PC で、ブラウザも JVM アプリも音楽プレーヤーも同時に動いています。もしそれぞれが**物理メモリの番地を直接指定**して使ったら、どうなるでしょう?

```
JVM   :「番地 1000 番に俺のデータを置くぞ」
ブラウザ:「番地 1000 番に俺のデータを置くぞ」
        → 衝突!互いのデータを上書きして大惨事
```

これを防ぐために OS が使う魔法が **仮想記憶(virtual memory)** です。OS は各プロセスに、**「あなた専用の、0番地から始まる広大なメモリ」があるという幻**を見せます。これを **仮想アドレス空間** と呼びます。

```
JVM が見ている世界          ブラウザが見ている世界
 0番地 ┌─────────┐          0番地 ┌─────────┐
       │ 俺のデータ │                │ 俺のデータ │
       └─────────┘                └─────────┘
   どちらも「0番地から自分専用」だと思っている(が、実体は別々の物理位置)
```

各プロセスは「メモリは全部自分のもの」と思い込んで、他人のことを気にせずプログラムを書ける。実際にはその仮想の番地が物理メモリのどこに対応するかを、OS が裏でこっそり管理・変換しています。

> **アパートのたとえ**: どの棟にも「101号室」がある。住人は自分の棟の「101号室」だけ意識すればいい。郵便(OS)が「A棟101号室 = 実際の住所◯◯」と変換して届けてくれる。住人どうしが番地でぶつかることはない。

この仕組みの恩恵は2つ。**① 隔離**: あるプロセスは他プロセスのメモリに触れない(だからアプリが1つクラッシュしても他は巻き込まれにくいし、セキュリティ上も安全)。**② 単純さ**: プログラムは「自分は0番地から使える」前提で書けて、物理配置を気にしなくていい。

## 6.2 ページング ―― 仮想と物理をつなぐ

では、仮想の番地と物理の番地は、どう対応づけられるのか。1番地ずつ対応表を作ると膨大になるので、**一定サイズのかたまり単位**で管理します。この単位を **ページ**(典型的には 4KB)と呼び、仕組みを **ページング** と言います。

OS は「仮想ページ ⇔ 物理メモリの置き場所」の対応表(**ページテーブル**)を持ち、CPU がメモリにアクセスするたびに、この表をひいて物理番地へ変換します。

ここで実務に効く事実が1つ。物理メモリ(RAM)が足りなくなると、OS は **今すぐ使わないページを一時的にディスクへ退避**させて、RAM に空きを作ります。これが **スワップ(swap)** です。

そして第5章を思い出してください ―― **ディスクは RAM より桁違いに遅い**(「別の街まで取りに行く」レベル)。スワップされたページを使おうとすると、ディスクから読み戻す必要があり、**急に激烈に遅くなる**。第1章の「メモリが足りずスワップしていた → だから遅かった」は、これのことでした。

> **7割理解ポイント**: ページングの細部(多段ページテーブル等)は今は不要です。「仮想と物理はページ単位で対応づけられる」「RAM が足りないとディスクに退避(スワップ)して激遅になる」の2点だけ持って帰れば十分。

## 6.3 プロセスのメモリの中身 ―― スタックとヒープ

さて、各プロセスに与えられた「自分専用の仮想アドレス空間」。その中は、用途ごとに区画に分かれています。ざっくりこうです。

```
仮想アドレス空間(1プロセスぶん)
  ┌──────────────┐ 高位アドレス
  │   スタック    │  ↓ 下へ伸びる   ← 関数呼び出しのたびに積む
  │      ⋮        │
  │  (空き領域)   │
  │      ⋮        │
  │    ヒープ     │  ↑ 上へ伸びる   ← new したオブジェクトが住む
  ├──────────────┤
  │ 静的領域      │  グローバル/定数など
  ├──────────────┤
  │ コード(命令) │  プログラムの機械語(第4章)
  └──────────────┘ 低位アドレス
```

コード領域(第4章の機械語)や静的領域もありますが、この章の主役は **動的に伸び縮みする2つ ―― スタックとヒープ** です。この2つの違いこそ、`StackOverflowError` と `OutOfMemoryError` を分ける境界線です。順に見ます。

---

## 6.4 スタック ―― 関数呼び出しの積み木

**スタック**は、**関数呼び出しを管理する領域**です。第4章で「CPU はプログラムカウンタで実行位置を追う」「関数を呼ぶと処理が飛ぶ」と学びましたね。関数を呼ぶと、その関数のための小箱 ―― **スタックフレーム** ―― がスタックに1つ積まれます。フレームには、その関数の**ローカル変数・引数・戻り先アドレス**が入ります。

```kotlin
fun a() { val x = 1; b() }   // a を呼ぶと a のフレームが積まれる
fun b() { val y = 2; c() }   // b を呼ぶと b のフレームが上に積まれる
fun c() { val z = 3 }        // c も積まれる
```

```
c() 実行中のスタック:
  ┌─────────┐
  │ c: z=3   │ ← 今ここ(一番上)
  ├─────────┤
  │ b: y=2   │
  ├─────────┤
  │ a: x=1   │
  └─────────┘
関数が return すると、上のフレームから順に取り除かれる(積み木を崩す)
```

スタックの特徴は、**後入れ先出し(LIFO)で、積む・崩すが自動かつ高速**なこと。関数に入れば積まれ、抜ければ崩れる。管理は「先頭位置を示すポインタを動かすだけ」なので、ものすごく速い。第5章の「レジスタに近い速い部分」の恩恵も受けやすい。

その代わり、**スタックのサイズは固定で、しかも小さめ**です(スレッド1本につき数百KB〜数MB程度)。ここに、次の落とし穴があります。

---

## 6.5 StackOverflowError ―― 積み木を高く積みすぎた

関数が別の関数を呼び、それがまた別の…と、**return せずに呼び続ける**と、フレームがどんどん積み上がります。積み木が天井(スタックの上限)に達すると ―― **`StackOverflowError`** です。

いちばん起きやすいのが **深い再帰**(自分を呼び続ける関数)。Phase 1 で再帰を学んだとき、「再帰は関数を呼び続ける」と習いました。その1回1回がスタックにフレームを積んでいたのです。

```kotlin
var depth = 0
fun recurse() {           // 終了条件のない(または深すぎる)再帰
    depth++
    recurse()             // return しないまま自分を呼ぶ → フレームが積み上がり続ける
}

fun main() {
    try { recurse() }
    catch (e: StackOverflowError) { println("深さ $depth で StackOverflowError") }
}
```

**実測**してみると、あるマシンではこうなりました。

```
デフォルトのスタックサイズ    : 深さ 約 21,900 で StackOverflowError
スタックを小さく (-Xss256k)   : 深さ 約  3,600 で
スタックを大きく (-Xss4m)     : 深さ 約 98,800 で
```

はっきり分かるのは、**「どこまで深く再帰できるか」はスタックサイズで決まる**ということ。フレームの数が上限を超えた瞬間に落ちます。

### Phase 1 とのつながり、そして対策

Phase 1 のアルゴリズムで、深い再帰(例: 巨大な木の探索、長いリストの再帰処理)を書くと、実データではこの上限に当たることがあります。対策は主に3つ。

1. **再帰をループ(反復)に書き換える**。ループはフレームを積まないのでスタックを消費しません。もっとも根本的。
2. **Kotlin の `tailrec` を使う**(末尾再帰の場合)。再帰呼び出しが関数の**最後の操作**なら、`tailrec` を付けるとコンパイラが**自動でループに変換**してくれます(第4章のバイトコードで、再帰がジャンプに化けるのを想像してください)。

```kotlin
tailrec fun sum(n: Int, acc: Long = 0): Long =
    if (n == 0) acc else sum(n - 1, acc + n)   // 末尾再帰 → tailrec でループ化 → スタックを食わない
```

3. **スタックサイズを増やす**(`-Xss`)。対症療法的で、根本解決ではないことが多い。

> **障害対応での見分け方**: `StackOverflowError` のスタックトレースは、**同じ関数名がずらーっと繰り返し並ぶ**のが特徴です。これを見たら「ああ、再帰が深すぎる(あるいは終了条件のバグで無限再帰している)な」とほぼ即断できます。

## 6.6 ヒープ ―― オブジェクトが住む広い領域

もう一方の主役、**ヒープ**。ここは **`new` された(生成された)オブジェクトが住む場所**です。あなたが Kotlin で

```kotlin
val user = User("Alice")          // User オブジェクトの実体はヒープに置かれる
val list = mutableListOf(1, 2, 3) // ArrayList の実体も、内部配列もヒープに
```

と書くと、その実体は**ヒープに確保**されます。ヒープの特徴は、**大きくて、自由なタイミングで確保でき、関数を抜けても生き続けられる**こと。スタックのフレームは関数を抜けると消えますが、ヒープのオブジェクトは**誰かが参照している限り生き残ります**。

### 直前の質問への完全な回答

ここで、あなたが第5章の後に質問した件がスッキリ説明できます。

```kotlin
val list = listOf(userA, userB, userC)   // User オブジェクトのリスト
```

- **オブジェクトの実体**(userA, userB, userC)は **ヒープ** にバラバラに置かれる
- **`list` の内部配列**に並んでいるのは、実体そのものではなく **各実体への "参照"(ヒープ上の住所)**
- だから走査は「連続した参照配列を読む」→「参照の先(散らばった実体)へ飛ぶ」の二段構え

```
スタック            ヒープ(散らばりうる)
 list ─→ [参照配列]
          ├─ref0──→ userA
          ├─ref1──────────→ userB
          └─ref2──→ userC
```

「参照は連続だが実体は散らばる」の正体が、まさにこの**「スタックの参照 → ヒープの実体」という構造**でした。プリミティブの `IntArray` が速いのは、参照を介さず**整数の実体そのものがヒープ上に連続して並ぶ**からです。

## 6.7 参照と到達可能性 ―― GC が「捨てていいもの」を見分ける方法

ヒープに置いたオブジェクトは、いつか片付けないとヒープが埋まります。C 言語では**自分で明示的に解放**(`free`)しますが、これは解放し忘れ(リーク)や二重解放(バグ)の温床でした。

JVM ではこれを **GC(ガベージコレクタ)** が**自動**でやってくれます。では GC は、どのオブジェクトが「もう誰も使わないゴミ」で、どれが「まだ使う」のかを、どう見分けるのか?

答えは **到達可能性(reachability)** です。GC は「**根(ルート)** から参照を辿って行き着けるか」で判定します。ルートとは、今まさに使われている出発点 ―― **スタック上のローカル変数**や**静的フィールド**などです。

```
ルート(スタックの変数など)
   │
   ├─→ objA ─→ objB       ← ルートから辿れる = 到達可能 = 生きている(残す)
   │
   ✕   objC ─→ objD       ← どのルートからも辿れない = 到達不能 = ゴミ(回収!)
```

**ルートから辿り着けるオブジェクト = 生存。辿り着けない孤島 = ゴミ**として回収されます。これを実測で確かめたのが次です。

```kotlin
fun main() {
    repeat(10_000) {
        val tmp = ByteArray(1_000_000)   // 1MB。だが次のループで tmp は上書きされ、
        // ... 使う ...                    // 前の ByteArray はどこからも辿れなくなる(到達不能)
    }
    println("完走")   // 64MB ヒープでも、1MB×1万個=約10GB相当を処理できてしまう
}
```

これは **64MB のヒープで動かしても OOM になりません**(実測で完走)。毎ループ `tmp` が上書きされると、前の `ByteArray` は到達不能になり、GC が回収してヒープを空けるからです。**合計10GB相当を作っても、瞬間ごとに生きているのは1個だけ**。だから溜まらない。「作った総量」ではなく「同時に到達可能な量」がヒープを圧迫する ―― これが GC 言語の勘所です。

## 6.8 GC の基礎とコンパクション ―― 「詰め直す」の正体

GC の仕事は「到達不能なゴミの回収」ですが、多くの現代 GC はもう一歩進んで、**生き残ったオブジェクトを一箇所に寄せ集めて詰め直します**。これを **コンパクション(compaction)** と呼びます。第5章の後の質問への回答で触れた「GC がオブジェクトを詰め直すので、時間とともに近くに並ぶこともある」は、これのことです。

```
回収前:  [生][ゴミ][生][ゴミ][ゴミ][生]   ← 隙間だらけ(断片化)
              ↓ GC が回収+詰め直し(コンパクション)
回収後:  [生][生][生][             空き ]   ← 生存者を前に寄せ、空きをまとめる
```

コンパクションのおかげで、①空きが連続してまとまり大きなオブジェクトを置きやすくなる、②生存オブジェクトが近くに並んで**キャッシュ局所性が改善**しうる(第5章とつながる)、という効果があります。

**世代別 GC(generational GC)** という工夫も知っておくと役立ちます。経験則として「**ほとんどのオブジェクトはすぐ死ぬ**(短命)」ため、GC はヒープを「若い世代」と「古い世代」に分け、**生まれたばかりの若い領域を頻繁かつ安く掃除**し、長生きしたものだけ古い領域へ移します。無駄なく効率的に回収するための仕組みです。

### GC ポーズ(実務で効く)

GC には代償もあります。ゴミを正確に見分けるため、伝統的な GC は**アプリのスレッドを一瞬止めて**掃除します(**stop-the-world ポーズ**)。この停止が、第1章の「GC が動いてアプリが一瞬止まり、レスポンスが遅くなった」の正体です。ヒープが大きい・ゴミが多いほどポーズが長引きがち。近年は停止をごく短く抑える GC(ZGC など)も登場していますが、「**GC はタダではなく、時々アプリを止める**」という感覚は性能調査で効きます。

> **7割理解ポイント**: GC のアルゴリズム詳細(マーク&スイープ、コピー等)は今は不要。「ルートから辿れないものを回収」「詰め直して断片化を解消」「掃除中は時々アプリが止まる(ポーズ)」の3点で十分です。

## 6.9 OutOfMemoryError とメモリリーク ―― 参照を握りっぱなしにする罪

いよいよ `OutOfMemoryError`(OOM)です。これは **ヒープがいっぱいで、GC がどうにか掃除しても十分な空きが作れない**ときに起きます。6.7 の「握らないと溜まらない」の**逆**をやると再現できます。

```kotlin
fun main() {
    val leak = mutableListOf<ByteArray>()   // このリストが全部を参照し続ける
    var count = 0
    try {
        while (true) {
            leak.add(ByteArray(1_000_000))  // 1MBずつ溜め込み、誰も手放さない
            count++
        }
    } catch (e: OutOfMemoryError) {
        println("$count 個溜めて OutOfMemoryError")
    }
}
```

**実測**すると:

```
ヒープ 64MB (-Xmx64m)  : 約 60 個溜めて OutOfMemoryError
ヒープ 128MB (-Xmx128m): 約 125 個溜めて OutOfMemoryError
```

さっきの「握らない版」は10GB相当でも完走したのに、こちらは60MB程度で力尽きます。違いはただ一つ ―― **`leak` リストが全 `ByteArray` を参照し続けている**こと。参照されている = ルートから到達可能 = GC は「まだ使うんだな」と判断して**回収できない**。だから溜まる一方。

### GC 言語のリークは「解放し忘れ」ではない

ここが重要な発想の転換です。C のリークは「`free` を書き忘れた」ですが、**GC 言語のメモリリークは「もう要らないのに、参照を握りっぱなしにしている」**ことです。参照が残っている限り GC は回収できません。よくあるパターン:

- **増え続けるコレクション**: 上の例のように、`static`(全体で共有)なリストやマップに追加し続け、消さない
- **上限のないキャッシュ**: キャッシュに入れっぱなしで、古いものを追い出す仕組みがない
- **解除し忘れたリスナー/コールバック**: 登録したまま解除しないと、そこから参照が生き続ける

いずれも「**不要になった参照を切る**(コレクションから remove する、上限付きキャッシュを使う、リスナーを解除する)」ことで防げます。

### StackOverflowError と OutOfMemoryError の違い(この章の要点)

2つの Error は名前も響きも似ていますが、**起きる場所も原因もまったく別物**です。

```
                StackOverflowError          OutOfMemoryError
─────────────────────────────────────────────────────────────
起きる場所       スタック                    ヒープ
何が溢れた?      関数フレームの積みすぎ       オブジェクトの溜めすぎ
典型的な原因     深い再帰 / 無限再帰          リーク / 本当にメモリ不足
サイズを決める   -Xss(スタックサイズ)       -Xmx(ヒープ最大)
トレースの特徴   同じ関数名が延々と繰り返す    OOM時点の確保処理が出る
対策             再帰をループ化/tailrec       不要な参照を切る/ヒープ調整
```

この表を見分けられるようになるのが、第6章の到達目標です。

## 6.10 実務接続 ―― 障害対応での立ち回り

- **`StackOverflowError` を見たら**: まず再帰を疑う。スタックトレースで同じ関数が繰り返していないか確認。終了条件のバグ(無限再帰)か、データが深すぎるか。ループ化や `tailrec` で対処。
- **`OutOfMemoryError` を見たら**: 「本当にメモリが足りない(データ量に対してヒープが小さい)」のか「リーク(何かが際限なく増えている)」のかを切り分ける。後者なら、増え続けているコレクション・キャッシュ・リスナーを探す。ヒープダンプを取ってプロファイラで「何が大量に生き残っているか」を見るのが定石。
- **「なぜか遅い(第1章)」の一因**: GC ポーズ(掃除でアプリが一瞬止まる)や、メモリ不足によるスワップ(6.2、ディスク退避で激遅)。「アプリのコードは正しいのに遅い」ときの有力容疑者です。

これらは毎日使う知識ではありませんが、**メモリ障害という "怖い・原因が見えにくい" 場面で、落ち着いて当たりをつける**ための地図になります。

## この章のまとめ

- **仮想記憶**: OS が各プロセスに「0番地から始まる自分専用メモリ」の幻を見せる。隔離(安全・安定)と単純さが得られる
- **ページング**: 仮想と物理をページ単位で対応づける。RAM 不足時は**スワップ**(ディスク退避)で激遅になる
- プロセスのメモリの主役は **スタック**(関数呼び出しの積み木、小さく高速)と **ヒープ**(オブジェクトの住処、大きく自由)
- **スタック**にフレームを積みすぎ(深い再帰)→ **`StackOverflowError`**(実測:深さは `-Xss` で決まる)。対策はループ化・`tailrec`
- **ヒープ**: オブジェクトの実体が住む。リストに並ぶのは実体でなく**参照** ―― これが「実体は散らばる」の正体
- **GC** は **到達可能性**(ルートから辿れるか)でゴミを判定し、**コンパクション**で詰め直す(=第5章後の質問の答え)。掃除中の **GC ポーズ** が遅延の一因
- ヒープを溜めすぎ → **`OutOfMemoryError`**。GC 言語のリークは「解放し忘れ」でなく「**不要な参照を握りっぱなし**」(実測:リストが握ると回収されず、握らなければ10GB相当でも完走)
- 2つの Error は **場所(スタック/ヒープ)も原因も別物**。見分けられることが目標

これで第2部・メモリ編は完走です。あなたの Kotlin コードが「メモリ上のどこで、どう生き死にするか」が見えるようになりました。次の第3部からは、いよいよ **OS の本丸 ―― プロセス・スレッド・並行処理**。実務に一番刺さるゾーンに入ります。

## 演習問題

`StackOverflowError` と `OutOfMemoryError` を**実際に出して観察する**のが主役です。OOM 実験はヒープを小さく制限して(`-Xmx64m` など)安全に試してください。

### 問1(推奨・実測:StackOverflow)
6.5 の再帰コードを動かし、何回目の再帰で `StackOverflowError` が出るか確認してください。次に `-Xss` オプションでスタックサイズを変えて(例: 小さく `-Xss256k`、大きく `-Xss4m`)、**深さがどう変わるか**を観察してください。そこから何が言えますか?

### 問2(推奨・実測:リーク vs 非リーク)
次の2つを、どちらも `-Xmx64m`(ヒープ64MB)で動かしてください。片方は OOM で落ち、もう片方は完走するはずです。**なぜ差が出るのか**を、到達可能性の言葉で説明してください。
```kotlin
// A: 握り続ける
fun main() {
    val keep = mutableListOf<ByteArray>()
    while (true) { keep.add(ByteArray(1_000_000)) }
}
// B: 握らない
fun main() {
    repeat(100_000) { val tmp = ByteArray(1_000_000) /* 使って捨てる */ }
    println("完走")
}
```

### 問3(見分ける)
次の症状は、`StackOverflowError` と `OutOfMemoryError` のどちらの可能性が高いですか。理由も。
1. スタックトレースに、同じメソッド名が何百行も繰り返し並んでいる。
2. サーバを起動して数時間〜数日たつと、じわじわ遅くなり、最後にエラーで落ちる。再起動すると直る。

### 問4(Phase 1 との接続)
Phase 1 で書いた「リストを再帰で処理する関数」が、要素100万個のリストで `StackOverflowError` を起こしました。原因を説明し、`StackOverflowError` を避ける書き換え方を2つ挙げてください。

### 問5(参照と実体)
`val users = listOf(userA, userB, userC)` について、「スタック」「ヒープ」「参照」「実体」という言葉を使って、メモリ上に何がどう置かれているか説明してください(第5章の「散らばり」ともつなげてみましょう)。

---

## 演習の解答

### 問1(解答例)
デフォルトでは数万回(マシンにより 2万〜数万)で `StackOverflowError` が出ます。`-Xss` を小さくすると浅い深さで落ち、大きくすると深くまで再帰できます。ここから言えるのは、**「どこまで再帰できるか」はスタックサイズという有限の資源で決まる**ということ。再帰1回ごとにフレームが1つ積まれ、その総量がスタックの上限を超えた瞬間に落ちる、という仕組みが実感できます。

### 問2(解答例)
- **A は OOM で落ちる**: `keep` リストが全 `ByteArray` を参照し続けるため、それらは**ルート(keep)から到達可能**で GC が回収できない。ヒープ(64MB)を超えて溜まり、OOM。
- **B は完走する**: `tmp` は毎ループ上書きされ、前の `ByteArray` は**どこからも到達不能**になる。GC が回収してヒープを空けるので、総量が大きくても**同時に生きている量**は小さく、溜まらない。
- 差の本質は「作った総量」ではなく「**同時に到達可能な量**」。GC 言語のメモリ管理の勘所です。

### 問3(解答例)
1. **`StackOverflowError`**。同じメソッドが延々と繰り返すトレースは、深い/無限再帰の典型的な指紋。
2. **`OutOfMemoryError`(リーク)**。「時間とともにじわじわ悪化し、再起動で直る」は、何かが際限なく溜まり続けている(増え続けるコレクション、上限なしキャッシュ、解除し忘れたリスナー等)古典的なメモリリークのパターン。再起動でヒープがまっさらになるので一時的に直る。

### 問4(解答例)
- **原因**: リストの各要素を再帰で処理すると、要素数ぶんだけ関数フレームが積まれる。100万要素なら100万段の再帰となり、スタックの上限をはるかに超えて `StackOverflowError`。
- **書き換え方(例)**:
  1. **ループ(反復)に書き換える** ―― `for`/`while` で回せばフレームを積まないのでスタックを消費しない。最も確実。
  2. **末尾再帰にして `tailrec` を付ける** ―― 再帰呼び出しを関数の最後の操作にすれば、`tailrec` でコンパイラがループへ自動変換し、スタックを消費しなくなる。

### 問5(解答例)
`users` という**変数**(参照)は、それを使っている関数の**スタックフレーム**に置かれます。その参照が指す先 ―― `ArrayList` 本体とその**内部配列** ―― は**ヒープ**にあります。内部配列に並んでいるのは `userA`/`userB`/`userC` の**実体ではなく、各実体への参照**です。そして `User` の**実体そのもの**は、ヒープ上の(必ずしも隣り合わない)バラバラの場所に置かれます。だから全ユーザーを走査すると「連続した参照配列を読む → 散らばった実体へ飛ぶ」となり、第5章で見たキャッシュ的に不利な「散らばりアクセス」が生じ得ます。プリミティブの `IntArray` なら参照を介さず実体が連続するため、この不利がありません。

> 問1・問2 を実際に動かして、2つの Error を**自分の手で出せた**ら、この章は大成功です。あとは「StackOverflow=スタック=再帰、OutOfMemory=ヒープ=溜めすぎ/リーク」の対比と、「リークは参照の握りっぱなし」が残れば、実務的な目的は達成です。
