# 第4章 機械語とプログラムの実行

> **この章のねらい**
> 第3章で「CPU は足し算や比較ができるスイッチの集まり」だとわかりました。でも疑問が残ります ―― **CPU は "何を・どの順で" やればいいと、どうやって知るのか?** その答えが **命令(instruction)** であり、命令の集まりが **プログラム** です。
> この章は第1部(ハードウェア編)の締めであり、同時に**ハードウェア → JVM → あなたの Kotlin** をつなぐ橋です。「あなたが書いた Kotlin が、最終的にどうやって CPU 上で動くのか」を一本の線でたどります。
> ハードウェアの細部は引き続き **7割理解で OK**。ただし後半の **JVM のしくみ**と **`javap` でバイトコードを覗く演習**は、実務(性能・ウォームアップ・ベンチマーク)に直結するので、ここは少し楽しんでいきましょう。

---

## 4.0 この章でたどる「一本の線」

先に全体像を地図にします。あなたが書いた1行が、どこを通って CPU に届くか。

```
  あなたの Kotlin コード        (4F アプリ)
        │  ① kotlinc がコンパイル
        ▼
  JVM バイトコード (.class)     (3F JVM 向けの "中間言語")
        │  ② JVM が実行(最初は解釈、熱くなったら JIT)
        ▼
  機械語(その CPU 専用)         (1F ハード)
        │  ③ フェッチ→デコード→実行 を秒間何十億回
        ▼
  CPU が実際に計算              (第3章のゲート/ALU)
```

この章は、この線を **下(機械語)から上(Kotlin)へ** 順にたどります。まず CPU が直接食べる「機械語」から。

---

## 4.1 CPU への指示書 ―― 機械語と命令セット

第3章で見たように、CPU ができることは原始的です ―― 「メモリから読む」「足す」「比べる」「メモリに書く」「次はどこを実行するか決める」程度。

この**原始的な操作1つ1つを指示するもの**が **命令(instruction)** です。そして命令も、コンピュータの中身である以上、結局は **0と1の並び(数値)** です。たとえば(架空の例ですが)「`00000001` なら足し算しろ」「`00000010` なら読み込め」というふうに、ビットパターンに操作が割り当てられています。この、CPU が直接理解できる 0と1 の命令を **機械語(machine language / machine code)** と呼びます。

CPU が理解できる命令の「語彙の一覧」を **命令セット(instruction set / ISA)** と言います。これは CPU の種類によって違います。

```
x86 / x86-64   … Intel・AMD の PC やサーバでおなじみ
ARM            … スマホ、Apple Silicon (M シリーズ)、最近のサーバ
RISC-V         … オープンな新興の命令セット
```

ここで大事な事実が1つ。**機械語は CPU の種類ごとに違う**ので、x86 用にコンパイルした機械語は、そのままでは ARM の CPU では動きません。言葉が通じないのです。この「機械語は CPU 依存」という点が、後で JVM の「どこでも動く」というありがたみ(4.5)につながる伏線です。

> **7割理解ポイント**: 個々の命令ビットを覚える必要はゼロです。「命令も結局は数値」「CPU ごとに語彙(命令セット)が違う」の2点だけ持って帰れば十分。

---

## 4.2 フェッチ・デコード・実行 ―― CPU の鼓動

では CPU は、ずらりと並んだ命令を**どうやって順に実行する**のでしょうか。答えは、ひたすら同じ3ステップのループを回しているだけです。

```
┌─→ ① フェッチ (Fetch)    : 次の命令をメモリから取ってくる
│   ② デコード (Decode)   : それが何の命令か解読する
│   ③ 実行    (Execute)  : 実際にやる(ALU で計算/メモリ読み書き 等)
└──────────────────────────┘  これを延々と繰り返す
```

このループが、第3章で出てきた **クロック** の「カチ…カチ…」に合わせて回ります。クロックが鼓動なら、フェッチ・デコード・実行はその鼓動に乗って繰り返される呼吸のようなものです。3GHz の CPU なら、このループ(に相当する処理)を**秒間何十億回**こなしています。

「次にどの命令を実行するか」を覚えているのが、**プログラムカウンタ(PC)** という専用のレジスタ(第3章の手元メモ帳の一種)です。フェッチするたびに PC が次の命令を指すように進み、`if` や `for` で処理が飛ぶときは PC の値が書き換わって別の場所へジャンプします。

> **腑に落とすための一言**: CPU は「プログラム全体を理解して賢く動いている」わけではありません。**目の前の命令を1つ取って、解読して、やる。また次を取る。** これだけを猛烈な速さで繰り返しているだけです。第3章の「CPU は単純作業の超高速な繰り返し」が、ここでより具体的になりました。

## 4.3 アセンブリの雰囲気 ―― 機械語に人間語のラベルを貼る

機械語は 0と1 の羅列なので、人間にはまず読めません。`10110000 01100001` と書かれて意味がわかる人はいません。

そこで、機械語の各命令に**人間が読める短い名前**を付けたものが **アセンブリ言語(assembly)** です。アセンブリと機械語はほぼ1対1で対応します(アセンブリ = 機械語に名札を貼っただけ)。

`a + b` を計算して `result` に入れる処理を、アセンブリの**雰囲気**で書くとこんな感じです(特定の CPU の正確な文法ではなく、イメージです)。

```asm
LOAD   R1, a        ; メモリ上の a をレジスタ R1 へ読み込む
LOAD   R2, b        ; メモリ上の b をレジスタ R2 へ読み込む
ADD    R3, R1, R2   ; R1 + R2 を計算して R3 に入れる(← 第3章の加算器!)
STORE  result, R3   ; R3 の中身をメモリ上の result へ書き戻す
```

見えてくるのは、CPU の世界の基本動作 ―― **「メモリからレジスタへ運ぶ → レジスタ上で計算する → メモリへ書き戻す」** という往復運動です。計算は必ず手元(レジスタ)で行い、データはメモリとの間を行き来する。この「メモリとレジスタの往復」が、第5章の「メモリは遠い・レジスタは近い」という話の前振りになります。

> **7割理解ポイント**: アセンブリを書けるようになる必要はありません。「機械語に人間向けの名前を付けたものがアセンブリ」「CPU はメモリ⇄レジスタを往復しながら計算する」という雰囲気だけで OK です。

## 4.4 コンパイラとインタプリタ ―― ソースが機械語になるまでの2つの道

あなたが書く Kotlin は人間向けの高水準言語で、CPU はそれを直接は理解できません。**誰かが翻訳しなければ**動かない。翻訳の方式には大きく2つあります。

### コンパイラ方式 ―― 事前にまとめて翻訳

プログラム全体を、実行前に**いっぺんに機械語へ翻訳**しておく方式です(C、Rust など)。

- 翻訳済みの機械語を直接走らせるので、**実行が速い**
- 翻訳(ビルド)に時間がかかる。CPU ごとに翻訳し直す必要がある(機械語は CPU 依存だから)

### インタプリタ方式 ―― 実行しながら逐次翻訳

プログラムを**1行ずつ、その場で解釈しながら実行**する方式です(古典的な Python、昔の JavaScript など)。

- 事前ビルド不要で**手軽・柔軟**
- 毎回その場で解釈するぶん、コンパイラ方式より**遅くなりがち**

```
コンパイラ:  [全部まとめて翻訳] → [高速に実行]      (準備に時間、走りは速い)
インタプリタ: [1行訳す→実行→次の行を訳す→…]        (準備なし、走りは遅め)
```

そして、Kotlin/Java が動く JVM は、**この2つの良いとこ取りをしたハイブリッド**です。次の節が本題です。

---

## 4.5 JVM のしくみ ―― Kotlin → バイトコード → 機械語

Kotlin は、いきなり機械語にはなりません。**2段階**を踏みます。

### ① コンパイル:Kotlin → バイトコード

`kotlinc`(Kotlin コンパイラ)が、あなたの `.kt` ファイルを **JVM バイトコード**(`.class` ファイル)に翻訳します。

ここがポイント。バイトコードは **CPU の機械語ではありません**。**「JVM という仮想のマシン向けの機械語」** です。実在しない理想の CPU(JVM)に向けた命令、と言ってもいい。

なぜそんな中間言語を挟むのか? ―― **「一度書けば、どこでも動く(Write once, run anywhere)」** を実現するためです。機械語は CPU ごとに違う(4.1)ので、もし Kotlin を直接機械語にすると、x86 用・ARM 用…と作り分けが必要になります。でもバイトコードにしておけば、**同じ `.class` ファイルを、どの CPU 上の JVM でも動かせる**。CPU の違いは JVM が吸収してくれるわけです。

```
   Sample.kt
      │ kotlinc
      ▼
   SampleKt.class   ← この1ファイルが、x86 でも ARM でも動く
      │
   ┌──┴───┬───────┐
   ▼      ▼       ▼
 x86のJVM ARMのJVM …  ← 各 CPU 用の JVM が、その場で機械語に変換して実行
```

### ② 実行:バイトコード → 機械語(JVM がやる)

実行時、JVM がバイトコードを動かします。このとき JVM は賢い二段構えを取ります。

1. **最初はインタプリタ**としてバイトコードを1命令ずつ解釈実行する(手軽だが遅め)
2. **何度も実行される "ホットな" 部分**を見つけると、**JIT(Just-In-Time)コンパイラ**がそこを**本物の機械語に翻訳して最適化**する(以降は高速)

つまり JVM は「最初はインタプリタ、よく使う所はコンパイラ」という、4.4 の両方式のハイブリッドなのです。この二段構えが、4.7 の「最初は遅いがだんだん速くなる」の正体になります。

## 4.6 【手を動かす】javap でバイトコードを覗く

バイトコードは抽象的に聞くと掴みづらいので、**自分の Kotlin が実際にどんなバイトコードになるか**を覗いてみましょう。これがこの章のハイライトです。

### 準備するもの

- **JDK**(`javap` コマンドが含まれる。`java -version` ではなく `javap -version` が通れば OK)
- **Kotlin コンパイラ**(`kotlinc`)。未インストールなら、まずはブラウザの [Kotlin Playground](https://play.kotlinlang.org/) の「Show Bytecode」機能でも近いことが体験できます。

### やってみる

`Sample.kt` という名前で、次の関数を書きます。

```kotlin
// Sample.kt
fun add(a: Int, b: Int): Int {
    return a + b
}
```

ターミナルでコンパイルして、バイトコードを表示します。

```bash
kotlinc Sample.kt          # → SampleKt.class ができる
javap -c SampleKt          # バイトコードを逆アセンブルして表示
```

すると、こんな出力が得られます(Kotlin のトップレベル関数は、ファイル名 + `Kt` というクラスの中の static メソッドになります)。

```
public final class SampleKt {
  public static final int add(int, int);
    Code:
       0: iload_0      // 1番目の引数 a をスタックに積む
       1: iload_1      // 2番目の引数 b をスタックに積む
       2: iadd         // スタックから2つ取り出して足し、結果を積む
       3: ireturn      // スタックのトップ(=結果)を返す
}
```

### 読み解きのポイント:JVM は「スタックマシン」

注目してほしいのは、第3章のアセンブリ(4.3)と**形が違う**ことです。4.3 では `ADD R3, R1, R2` のように**レジスタ**を名指ししていました。でもバイトコードにはレジスタ名が出てきません。代わりに **スタック(積み重ねる作業台)** にデータを積んだり下ろしたりしています。

```
iload_0 :  [    ]        →  [ a ]              a を積む
iload_1 :  [ a  ]        →  [ a ][ b ]         b を積む
iadd    :  [ a ][ b ]    →  [ a+b ]            2つ下ろして足し、結果を積む
ireturn :  [ a+b ]       →  (return)           トップを返す
```

実在の CPU は**レジスタ**を使う「レジスタマシン」、JVM は**スタック**を使う「スタックマシン」 ―― この設計の違いがあります。JVM がスタック方式なのは、**特定の CPU のレジスタ構成に縛られないため**です(レジスタの数は CPU ごとに違うので、それに依存しない抽象的なモデルとしてスタックを採用している)。実行時には、JIT がこのスタック操作を、実際の CPU のレジスタを使う機械語にうまく変換してくれます。

> **試してみると面白いこと**: `if` や `for` を入れた関数、`when` を使った関数などを `javap -c` で覗くと、条件分岐やループが「どこそこへジャンプせよ」という `goto` 系の命令に化けているのが見えます。「高水準の `for` も、底ではジャンプの繰り返しなんだ」と体感できます。**ここまで見られれば、この章の手を動かすパートは大成功**です。

## 4.7 なぜ JVM は最初遅くて、だんだん速いのか ―― ウォームアップ

4.5 の二段構え(最初インタプリタ → ホットな所を JIT)を理解すると、JVM 特有の有名な現象が説明できます。

> **JVM アプリは、起動直後は遅く、しばらく動かすと速くなる。**

理由はこうです。

```
起動直後   : まだ JIT が走っていない。バイトコードをインタプリタで解釈実行 → 遅い
        ↓   (同じコードが何度も呼ばれて "ホット" になる)
JIT 発動   : ホットな部分を最適化された機械語にコンパイル
        ↓
ウォーム後 : 以降はその機械語を直接実行 → 速い
```

この「速くなるまでの助走期間」を **ウォームアップ(warm-up)** と呼びます。実務では次のような形で顔を出します。

- **デプロイ直後の最初のリクエストが遅い**。サーバを再起動した直後はまだ温まっておらず、しばらく経つと安定して速くなる。これを見越して、起動後にダミーのリクエストを流して**事前に温めておく**運用をすることもあります。
- **マイクロベンチマークで罠になる**。「この関数は速い!」と測ったつもりが、JIT がまだ効いていない(または逆に、JIT が最適化で処理ごと消してしまった)せいで嘘の数字を見ていることがあります。だから JVM のベンチマークは、ウォームアップを挟んで正しく測る専用ツール(JMH など)を使うのが定石です。
- **短命なプロセスは JIT の恩恵を受けにくい**。すぐ終わるバッチや CLI ツールは、温まる前に終わってしまうため JVM の真価が出ません。こうした用途には、起動時に最適化済みの機械語にしておく **AOT(事前)コンパイル**(GraalVM のネイティブイメージなど)という選択肢もあります ―― これは「JIT のウォームアップを捨てる代わりに、起動から速い」という割り切りです。

> **実務接続(この章の一番の持ち帰り)**: 「JVM は最初インタプリタ、ホットな所を JIT で機械語化する」という1点を知っているだけで、**「デプロイ直後が遅い」「ベンチの数字が安定しない」**といった現象に、慌てず原因の見当をつけられます。これは性能調査でじわじわ効く知識です。

## この章のまとめ

- CPU への指示1つ1つが **命令**、その集まりが **プログラム**。CPU が直接食べる 0と1 の命令が **機械語**、語彙の一覧が **命令セット**(x86/ARM など、CPU ごとに違う)
- CPU は **フェッチ→デコード→実行** のループを、クロックに合わせて秒間何十億回繰り返しているだけ
- **アセンブリ** = 機械語に人間向けの名前を貼ったもの。CPU は「メモリ⇄レジスタの往復」で計算する
- 翻訳方式は **コンパイラ**(事前に全部翻訳・速い)と **インタプリタ**(逐次翻訳・手軽)。JVM は両者のハイブリッド
- Kotlin は **kotlinc でバイトコード(.class)** になり、**JVM が実行時に解釈 → ホットな所を JIT で機械語化**。バイトコードのおかげで「どの CPU でも同じ .class が動く」
- `javap -c` で自分の Kotlin のバイトコードが覗ける。**JVM はスタックマシン**(`iload`/`iadd`/`ireturn`)
- 「最初インタプリタ → JIT で高速化」が **ウォームアップ** の正体。デプロイ直後が遅い・ベンチが安定しない、を説明できる

これで第1部(ハードウェアとデータの表現)は完走です。底(ゲート)から、命令、そして Kotlin が機械語になるまでの一本の線がつながりました。**ここまで来れば、もうコンピュータは「中身の見えないブラックボックス」ではありません。** お疲れさまでした。第2部からは、実務にぐっと近づくメモリの話に入ります。

## 演習の解答

### 問1(解答例)
1. **機械語**: CPU が直接理解できる 0と1 の命令。CPU の種類(命令セット)ごとに違う。
2. **バイトコード**: Kotlin/Java が一度翻訳される中間言語。実在 CPU ではなく「JVM という仮想マシン」向けの機械語で、どの CPU 上の JVM でも同じものが動く。
3. **JIT コンパイラ**: 実行中に、何度も使われる "ホットな" バイトコードを、その場で本物の機械語に翻訳・最適化するしくみ。これにより JVM はだんだん速くなる。

### 問2(解答例)
クロックは CPU の鼓動で、3GHz なら**秒間30億回**のリズムを刻みます。CPU はそのリズムに乗って **フェッチ(命令を取る)→ デコード(解読)→ 実行** のループを回し続けます。つまり「3GHz」とは、おおまかに「秒間30億回ペースで命令処理を進める速さ」を意味します(厳密には1命令に複数クロックかかったり、逆に並列実行したりしますが、雰囲気としてはこの理解で十分)。

### 問3
- `add` は引数を積んで足して返すだけ(`iload`/`iadd`/`ireturn`)の直線的なバイトコードでした。
- `maxOf` には **比較命令**(`a > b` を判定)と **ジャンプ命令**(条件が成り立つ/成り立たないで別の場所へ飛ぶ)が現れます。高水準の `if` が、底では「比較してジャンプ」という機械寄りの操作に分解されているのが見て取れます。第3章の「PC を書き換えて処理が飛ぶ」が、ここで具体的なジャンプ命令として姿を現します。

### 問4(解答例)
- **何が起きているか**: デプロイ直後はまだ JIT が効いておらず、バイトコードをインタプリタで解釈実行しているため遅い(ウォームアップ前)。同じコードが何度も呼ばれて "ホット" になると、JIT が機械語に最適化し、以降は安定して速くなる。
- **打ち手の例**: 起動後にダミーリクエストを流して事前に温める(ウォームアップ処理)、ロードバランサに「温まるまで本番トラフィックを流さない」猶予を持たせる、短命プロセスなら AOT コンパイル(GraalVM ネイティブイメージ等)で起動から速くする、など。

### 問5(解答例)
- Kotlin は **CPU 非依存のバイトコード**にコンパイルされ、それを各 CPU 用の JVM が実行時に機械語へ変換します。CPU の違いを JVM が吸収するので、同じ `.class` が Intel でも Apple Silicon でも動きます。
- 一方 C は、ふつうコンパイル時点で**特定の CPU 用の機械語**を直接吐きます。機械語は命令セットごとに違うので、x86 用に作った実行ファイルは ARM ではそのまま動きません。これが「中間言語を挟む(JVM)」か「直接機械語にする(C)」かの設計の違いです。

> 問3 を実際に動かして「`if` がジャンプになっている」を目撃できたら、この章は大成功。問4 のウォームアップの説明ができれば、実務的な目的は達成です。あとは忘れて大丈夫 ―― 第2部からはもっと実務に効く話が続きます。
