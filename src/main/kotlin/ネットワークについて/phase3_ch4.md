# Phase 3・第4章 DNS ― 名前からアドレスへ

> **この章の位置づけ**: ここから **第2部「アプリケーションの通信」**。実務で毎日触る層に入ります。その最初が **DNS(Domain Name System)** ―― `example.com` という**名前**を、`93.184.216.34` のような**住所(IP)**に変換する仕組みです。
>
> 実はこの変換、第1〜3章で **ずっと裏で動いていました**。`ping github.com` が `(20.27.177.113)` と IP を表示したのも、`InetAddress.getByName("example.com")` も、`nc example.com 80` も、全部「まず名前を IP に変換してから」通信していた。今日はその **陰の立役者が主役**になります。
>
> 手を動かすパートは **`dig`**。名前が IP に化ける過程を覗きます。そして DNS は **「繋がらない」の犯人第1位**。実務トラブルの常連なので、ここは実務直結度がとても高い章です。
>
> **7割理解で OK**。レコードの種類を全部覚える必要はありません。掴んでほしいのは **①名前→IP の変換をDNSがやる ②DNSは巨大な木構造 ③キャッシュとTTL(これが実務トラブルの温床)** の3点です。
>
> **目安: 3〜4時間**

## 0. 前章までの続き ― 地図のどこにいるか

第1部(第1〜3章)で、ネットワークの土台を積み上げました。

| 層 | 担当 | 状況 |
|---|---|---|
| **④ アプリケーション層** | **会話の中身。DNS もここの住人** | **← 第2部・今ここ(第4章)** |
| ③ トランスポート層 | TCP / UDP | 第3章 ✅ |
| ② インターネット層 | IP(住所) | 第2章 ✅ |
| ① リンク層 | 電気・電波 | (雰囲気) |

第2章で「通信は IP アドレス(住所)目がけて届く」と学びました。でも、あなたがコードや URL に書くのは **`example.com` という "名前"** であって、`93.184.216.34` という数字ではありません。**この名前を住所に変換する誰か**が必要です。それが DNS。DNS は、アプリケーション層に住む **"インターネットの電話帳"** です。

---

## 1. 身近な例 ― 電話帳(名前で引くと、番号が分かる)

スマホで友人に電話するとき、あなたは **「田中さん」という名前**をタップします。11桁の電話番号を暗記してはいません。連絡先(電話帳)が「田中さん → 090-xxxx-xxxx」と**名前から番号へ変換**してくれるからです。

DNS はこれのインターネット版です。

```
   あなたが使うもの          DNS が変換          実際に通信で使うもの
   example.com        ───────────────→        93.184.216.34
   (名前 / 覚えやすい)    (インターネットの電話帳)   (住所=IP / 機械が使う)
```

人間は名前(`github.com`)を覚え、機械は住所(IP)で通信する。**その橋渡しが DNS** です。第1〜3章で「名前を書いたのに、なぜか IP に届いていた」あの魔法の正体が、これでした。

---

## 2. なぜ「名前」が要るのか ― IP を直接使えばいいのでは?

「最初から IP を書けばいいのに、なぜ名前という一手間を挟むの?」―― 良い問いです。理由は2つあります。

### 理由①:人間は数字を覚えられない
`github.com` は覚えられても、`140.82.114.3` は覚えられません。しかも IPv6(第2章)なら `2001:0db8:...` です。名前があるから、人間はインターネットを使えます。

### 理由②:名前は「変わらない看板」、IP は「変わりうる中身」
こちらが本質的で、実務でも効きます。**サーバの IP は変わります**。

- サーバを新しいマシンに引っ越した → IP が変わる
- 負荷分散で複数のサーバに振り分ける → 1つの名前に複数の IP(第2章の `getAllByName` で見ました)
- 障害時に別のサーバへ切り替える(フェイルオーバー)→ IP が変わる

でも、その間ずっと **`github.com` という名前は変わりません**。利用者は名前だけ知っていればよく、裏で IP が変わっても影響を受けない。

> **Phase 2 / 第1章との接続 ―― またしても抽象化**: 「安定した名前」が「変わりうる住所」を **覆い隠している**。これは第1章・第3章で繰り返した **抽象化**そのものです。上の層(名前を使う人)は、下の実体(IP)が変わっても無傷。DNS は「名前という安定したインターフェースの裏で、実体を差し替え可能にする」層。**同じ抽象化のパターンが、また別の場所で現れました**。この "気づき" の蓄積が、CS を体系立てる面白さです。

## 3. DNS の正体 ― 世界に散らばった巨大な木構造

「電話帳」と言いましたが、**世界中の全ドメインを1冊にまとめた電話帳は存在しません**。そんな巨大なものを1台のサーバが持ったら、壊れたら世界中が止まるし、更新も追いつきません。

代わりに DNS は、**役割を分担した巨大な木構造(ツリー)**でできています。

### ドメイン名は「右から左」に読む木のパス

`www.example.com` を、右から左に読んでみてください。

```
                    . (ルート:木の根っこ)
                    │
          ┌─────────┼─────────┐
        com         jp        org      ← TLD(トップレベルドメイン)
          │
       example                          ← example.com を管理する権威サーバ
          │
        www                             ← 具体的なホスト
```

`www.example.com`(正確には末尾にドットがあり `www.example.com.`)は、この木を **根(.)→ com → example → www** とたどるパスです。**Phase 2 で触ったファイルシステムのディレクトリ木**や、**Phase 0 のグラフ理論の木**と、まったく同じ構造です。ドメイン名は木のパス表現なのです。

### 各階層は「次の階層に丸投げ(委譲)」する

この木の賢いところは、**各階層が「その下は誰が管理しているか」だけを知っていて、丸投げする**点です。

- **ルート**は「.com のことは、com を管理するサーバに聞いて」と教える。全 .com ドメインの中身は知らない。
- **.com のサーバ**は「example.com のことは、example.com の権威サーバに聞いて」と教える。
- **example.com の権威サーバ**が、ようやく「www は 93.184.216.34 だよ」と答える。

**誰も全部は知らない。各自は「自分の担当」と「次にどこへ聞くか」だけ知っている** ―― この構造、第2章のルーティング(バケツリレー:各ルータは次の一手だけ知る)とそっくりではありませんか? **「巨大な問題を、階層に分けて委譲し、誰も全体を抱えない」**という同じ設計思想が、ルーティングでも DNS でも使われています。

> **7割理解ポイント**: ルート・TLD・権威サーバという用語は「そういう3階層がある」程度で OK。大事なのは **「DNS は世界に散らばった木で、各階層が下へ委譲する」**という骨格の感覚です。

## 4. 名前解決の流れ ― リゾルバが木をたどる

では、あなたのマシンが `www.example.com` の IP を知りたいとき、実際に何が起きるか。木をたどる係を **リゾルバ**(通常はあなたの ISP のもの、または `8.8.8.8` のような公開リゾルバ)と呼びます。

```
あなたのPC:「www.example.com の IP 教えて」
   └→ リゾルバ:「よし、木をたどるぞ」
        ① ルートに聞く   →「.com は "com のサーバ" に聞け」
        ② com に聞く     →「example.com は "example.com の権威" に聞け」
        ③ 権威に聞く     →「www は 93.184.216.34 だ」
   ←─ リゾルバ:「93.184.216.34 です」
あなたのPC:「ありがとう、そこに TCP で繋ぐ(第3章)」
```

リゾルバが木を **根から順にたどって**答えにたどり着く。これは Phase 0・Phase 1 で学んだ **木の探索**そのものです。CS の基礎(木構造・探索)が、こんな身近なところで動いています。

> ちなみに、この一連の問い合わせの多く(特に短い1往復)は、第3章で学んだ **UDP** で行われます。「取りこぼしたら聞き直せばよく、TCP のハンドシェイク往復をかけるより軽く速い方が良い」―― 第3章の UDP の使いどころが、まさにここでした。伏線回収です。

## 5. 【最重要】キャッシュと TTL ― 実務トラブルの温床

毎回この木を根からたどっていたら、遅くて仕方ありません。だから DNS の結果は、**あちこちで一時的に覚えられ(キャッシュされ)ます** ―― あなたのブラウザ、OS、リゾルバ、それぞれのレベルで。一度引いた `github.com → 140.82.114.3` を、しばらくの間は木をたどらず即答する。

> **Phase 2 第5章との接続**: これは **キャッシュ**そのものです。Phase 2 で「近くて速いキャッシュを先に見て、遅い遠くのデータを毎回取りに行かない」と学びました。DNS も同じ。「遅い木の探索(遠い I/O)を毎回やらず、手元のキャッシュで済ませる」。**Phase 2 のキャッシュ階層の発想が、そのまま DNS に効いている**わけです。「近い・速いものを先に見る」は、CPU キャッシュでも DNS でも共通の知恵。

### TTL ―「いつまでキャッシュしていいか」の賞味期限

でもキャッシュには問題があります。**サーバの IP が変わったのに、古い IP をキャッシュし続けたら、古いサーバに繋ぎ続けてしまう**。そこで各レコードには **TTL(Time To Live、生存時間)**という「キャッシュしていい秒数」が付いています。

さきほど本物の DNS を引いたら、`github.com` の A レコードの **TTL は 60(秒)** でした。つまり「この情報は60秒間キャッシュしてよい。60秒経ったら引き直せ」という意味です。

### 実務での "あるある":「DNS 変えたのに、まだ古いサーバを見てる」

TTL の存在が、実務の混乱を生みます。

> DNS の設定を新しい IP に変更した。でも、**しばらく古い IP にアクセスが行き続ける**。「変更したのに反映されない!」と焦る。

これは **世界中のキャッシュに残った古い値が、TTL が切れるまで生き続けている**ためです。これを DNS の **伝播(propagation)** と呼び、「変更が世界に行き渡るまで時間がかかる」現象の正体は、この **TTL 分のキャッシュ残留**です。バグではなく仕様。

> **実務のコツ(先取り)**: サーバ移行の前は、**あらかじめ TTL を短く(例:60秒)しておく**と、切り替え時の古い値の残留を短くできます。「TTL を事前に下げてから移行する」は、インフラ運用の定番テクです。今は「そういう理由があるのか」で 7割 OK。

## 6. レコードの種類 ― 電話帳の "項目" いろいろ

DNS の電話帳には、名前に対して何を返すかで **レコードの種類**があります。全部覚える必要はなく、**雰囲気**で。以下は実際に `github.com` を引いた **本物の値**です。

| レコード | 意味 | github.com の実測値 |
|---|---|---|
| **A** | 名前 → IPv4 住所 | `140.82.114.3`(TTL 60) |
| **AAAA** | 名前 → IPv6 住所 | (この時は無し) |
| **CNAME** | 別名(名前 → 別の名前へ転送) | 後述 |
| **MX** | メールの届け先サーバ | `github-com.mail.protection.outlook.com`(TTL 122) |
| **NS** | この名前を管理する権威サーバは誰か | `dns1.p08.nsone.net`, `ns-1283.awsdns-32.org` ほか(TTL 487) |
| **TXT** | 任意のテキスト(所有者確認・SPF 等に使う) | (省略) |

面白い実測が **CNAME** です。`www.github.com` を引くと、こうなっていました(本物):

```
www.github.com.  CNAME  github.com.       ← 「www は github.com の別名だよ」と転送
github.com.      A      140.82.112.4      ← 転送先を引くと、こちらが実体の IP
```

**CNAME は「別名(エイリアス)」**です。`www.github.com` は「私の実体は `github.com` を見て」と転送し、リゾルバは改めて `github.com` の A を引く。**1つの実体に複数の名前を付けたり、名前を別サービスに向けたりする**のに使います(実務で CDN やSaaS に向けるとき頻出)。

> **7割理解ポイント**:よく使うのは **A(→IPv4)** と **CNAME(別名)**、あとは **MX(メール)** の存在くらい。NS/TXT は「そういうのもある」で十分。**「名前を引くと、種類に応じた答えが返る」**という枠だけ掴めば OK。

## 7. 【手を動かす】dig で名前解決を覗く

DNS を調べる定番ツールが **`dig`** です(Windows は標準では入っていないので `nslookup` で代用、または WSL を使う)。**あなたの手元のターミナル**で叩いてみてください。

> 下記の**答えの値は、この教科書のために実際に DNS を問い合わせて得た本物**です(A/MX/NS/CNAME とも §6 の実測)。`dig` はあなたの環境でも同じように叩けます。数値(特に IP や TTL)は時期・場所で変わります。

### 基本:A レコードを引く

```
$ dig github.com A
```

出力の中の **ANSWER SECTION** に、こう出ます(値は本物):

```
;; ANSWER SECTION:
github.com.   60   IN   A   140.82.114.3
                └TTL┘        └── 答えの IP ──┘
```

- 左から:名前 / **TTL(60秒)** / クラス(IN=インターネット、ほぼ固定) / 種類(A) / **答え(IP)**。
- **TTL の数字**に注目 ―― §5 のキャッシュの賞味期限がここに見えます。何度か連続で叩くと、この数字が **カウントダウンしていく**のが見えることがあります(キャッシュの残り時間)。

### 答えだけ欲しいとき:`+short`

```
$ dig +short github.com          →  140.82.114.3
```

### 種類を指定して引く

```
$ dig github.com MX     →  github-com.mail.protection.outlook.com.   （メール届け先。本物)
$ dig github.com NS     →  dns1.p08.nsone.net. ほか複数                （権威サーバ。本物)
$ dig www.github.com    →  CNAME で github.com. に転送されるのが見える  （§6 の本物)
```

### 特定のリゾルバに聞く:`@`

```
$ dig @8.8.8.8 github.com        ← Google の公開リゾルバに直接聞く
$ dig @1.1.1.1 github.com        ← Cloudflare の公開リゾルバに聞く
```

> **これが実務で効く場面**: 「うちの会社のネットワークだと繋がらないが、自宅だと繋がる」ようなとき、`dig @8.8.8.8`(外の公開リゾルバ)と `dig`(会社のリゾルバ)で**答えが違う**ことがあります。すると「**問題は経路ではなく、会社の DNS(社内リゾルバ)が変な答えを返している**」と切り分けられる。**リゾルバを指定して比べる**のは、DNS トラブルの強力な武器です。

## 8. Kotlin/JVM 接続 ― `InetAddress` と、JVM 特有の落とし穴

名前解決は、JVM からは第2章で見た `InetAddress` でできます。

```kotlin
import java.net.InetAddress

fun main() {
    // 名前 → IP(内部で DNS が動く。§4 の木の探索がここで起きている)
    val addr = InetAddress.getByName("github.com")
    println(addr.hostAddress)                 // 例: 140.82.114.3

    // 1つの名前に複数の IP(負荷分散・冗長化。§2 の理由②)
    InetAddress.getAllByName("github.com").forEach { println(it.hostAddress) }
}
```

### ★ JVM 特有の落とし穴:JVM も DNS をキャッシュする

ここは **あなたの実務(Kotlin/JVM)に直結する、重要な罠**です。

**JVM は、自分でも DNS の結果をキャッシュします**(OS やリゾルバのキャッシュとは別に、JVM プロセスの中で)。しかも古い Java では、**成功した名前解決を "プロセスが生きている限りずっと" キャッシュする**設定がデフォルトの時代がありました。

これが何を招くか:

> サーバをフェイルオーバー(別 IP に切り替え)したのに、**あなたの JVM アプリだけ、いつまでも古い IP に繋ぎ続ける**。他のツール(`dig` や `curl`)では新しい IP が見えているのに、アプリだけ古いまま。「アプリを再起動したら直った」という、あの現象。

原因は **JVM 内の DNS キャッシュ**です。キャッシュ時間は次のプロパティで制御します。

```
// JVM の DNS キャッシュ TTL(秒)。-1 は「永久にキャッシュ」で危険
networkaddress.cache.ttl=30        // 例:30秒でキャッシュ切れにする
networkaddress.cache.negative.ttl=10   // 解決失敗のキャッシュ時間
```

> **Phase 2 との接続**: キャッシュは速いが、**「古い情報を掴み続ける」という代償**がある ―― これは Phase 2 のキャッシュ整合性の話と同根です。速さと鮮度はトレードオフ。DNS も、OS も、JVM も、CPU キャッシュも、**「キャッシュは速いが古くなる」**という同じ緊張関係を抱えています。**「速さの裏には必ず鮮度の問題がある」**を、また別の場所で確認できました。

## 9. 【演習】採点・解説つき

**まず自分で考えてから**開いてください。7割で OK です。

### 演習1(ドメインを木として読む)
`mail.google.com.` を、DNS の木の「根からのパス」として、たどる順に書き出してください。

<details><summary>解答・解説</summary>

右から左に読んで:**`.`(ルート)→ `com` → `google` → `mail`**。

`.`(ルート)が「.com は com のサーバへ」、`com` が「google.com は google の権威へ」、`google.com` の権威が「mail は…(IP)」と委譲していく。§3 の木構造と §4 の探索の理解が問われました。ドメインは **右端が根に近い**のがポイント。
</details>

### 演習2(TTL とキャッシュ)★実務頻出
あるサービスの DNS を、金曜の夜に新しいサーバの IP に変更しました。しかし土曜の朝になっても、一部のユーザーからは「まだ繋がらない/古い画面が出る」と報告が来ます。dig で引くと自分には新しい IP が見えています。何が起きていて、事前に何をしておけば影響を減らせましたか?

<details><summary>解答・解説</summary>

**世界中のキャッシュ(ユーザーの OS・ブラウザ・ISP のリゾルバ)に、変更前の古い IP が TTL 分だけ残っている**(§5 の伝播遅延)。あなたの手元のキャッシュは切れて新 IP が見えていても、他の場所ではまだ古い値が生きている。TTL が例えば24時間なら、丸1日この状態が続きうる。

事前策:**移行の数日前に TTL を短く(例:300秒や60秒)しておく**。そうすれば切り替え後、古い値は最大でも数分で消え、伝播が速くなる。「移行前に TTL を下げる」の理由がこれ。バグではなく DNS の仕様、と分かっているだけで、当日パニックにならずに済みます。
</details>

### 演習3(レコードの種類)
「あるドメインのメールだけ別のメールサービス(例:Google Workspace)に向けたいが、Web サイトは今のサーバのまま使いたい」。どのレコードをいじりますか?

<details><summary>解答・解説</summary>

**MX レコード**をメールサービスの届け先に向ける。A レコード(Web の実体)はそのまま。§6 の通り、**MX はメールの届け先、A は Web の届け先**で **別々に設定できる**。同じドメインでも「Web はこのサーバ、メールはあのサービス」と分けられるのは、レコードが用途別に分かれているおかげ。「メールが届かない」トラブルで MX を疑うのは実務の定石です。
</details>

### 演習4(切り分け)★第9章の伏線
`ping github.com` を実行したら `ping: cannot resolve github.com: Unknown host` と出ました。`ping 140.82.114.3`(IP 直打ち)なら正常に返ります。どの層の問題で、次に何を叩きますか?

<details><summary>解答・解説</summary>

**IP を直打ちすれば届く**のだから、経路・IP・TCP(第2・3章)は健全。**名前を IP に変換する段階(DNS)で失敗している**。つまり問題は DNS 層。

次の一手:**`dig github.com`** で名前解決そのものを直接確かめる。さらに `dig @8.8.8.8 github.com`(外の公開リゾルバ)で引けるなら、「自分のリゾルバ(会社/自宅の DNS 設定)がおかしい」と絞れる(§7)。**「IP 直打ちなら通るか?」は、DNS 問題を一発で切り分ける鉄板の手筋**です。第1章から積み上げた "層で切り分ける" が、ここでも効いています。
</details>

## 10. 【実務接続】DNS は「繋がらない」の犯人第1位

エンジニアの間で **"It's always DNS"(どうせ原因は DNS)** という半分ジョークの格言があるほど、DNS は障害の常連です。なぜか ―― DNS は **すべての通信の一番最初**に走り(名前→IP がないと何も始まらない)、しかも **キャッシュ・TTL・複数のリゾルバ**という「古い値が残る」「場所で答えが違う」要素を抱えているからです。

### 症状と切り分けの型

- **「名前だと繋がらないが、IP 直打ちなら繋がる」** → ほぼ確実に DNS(演習4)。`dig` で確認。
- **「DNS 変えたのに古いサーバを見てる」** → TTL / 伝播(§5)。時間が解決するか、キャッシュ手動クリア。
- **「自分は繋がるが、他の人は繋がらない/その逆」** → リゾルバによって答えが違う。`dig @8.8.8.8` と比較(§7)。
- **「他ツールは新 IP なのに、うちの JVM アプリだけ古い IP」** → JVM の DNS キャッシュ(§8)。`networkaddress.cache.ttl` を疑う。

### 切り分けの地図での位置

DNS は、第1章で作った "層の切り分け地図" の **一番手前(最初)**に来ます。名前が IP に変換できて初めて、その先(TCP でポートに繋ぐ=第3章 → TLS 握手=第6章 → HTTP 応答=第5章)に進める。だから「繋がらない」を調べるときは、まず:

```
① dig(この章)      → 名前は IP に変換できる?         ← まずここ
② ping(第2章)      → その IP まで経路は届く?
③ nc(第3章)        → 目的のポートは開いてる? refused/timeout?
④ curl / openssl(第5・6章) → TLS 握手・HTTP 応答は?
```

**上から順に、層ごとに1つずつ潰す**。第9章「つながらない・遅いの調べ方」で、この4段階を1つの実践フローに完成させます。**あなたはもう、①〜③の道具を手にしています**。

## 11. この章のまとめ

- **DNS はインターネットの電話帳**。名前(`example.com`)を住所(IP)に変換する。第1〜3章で裏で動いていた変換の正体。
- **なぜ名前が要るか**:人間は数字を覚えられない/名前は安定した看板で、裏の IP は変わりうる(=**抽象化**。安定した名前が変わる実体を覆い隠す)。
- **DNS は世界に散らばった巨大な木構造**(ルート → TLD → 権威)。各階層は下へ**委譲**し、誰も全体を持たない(第2章ルーティングと同じ設計思想。Phase 0 の木・Phase 1 の探索)。
- **リゾルバが木を根からたどって**名前解決する。短い問い合わせは **UDP**(第3章)で軽く。
- **【最重要】キャッシュと TTL**:毎回木をたどらず結果をキャッシュ(Phase 2 第5章の発想)。TTL はキャッシュの賞味期限。**古い値が TTL 分残る=伝播遅延**が実務トラブルの温床。
- **レコード**:A(→IPv4)、CNAME(別名)が主役。MX(メール)、NS(権威)、TXT も存在。
- **手を動かす**:`dig`(A/MX/NS/CNAME、TTL の読み方、`@` でリゾルバ指定して比較)。
- **JVM の落とし穴**:JVM 自身が DNS をキャッシュする。`networkaddress.cache.ttl`。「アプリだけ古い IP を掴む」の犯人。
- **実務**:DNS は "It's always DNS"。切り分けは **IP 直打ちで通るか?** が鉄板。地図の一番手前。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] DNS が「名前→IP の変換(電話帳)」だと言える
- [ ] なぜ名前が要るか、「安定した名前 / 変わる IP」で言える
- [ ] DNS が木構造で、各階層が下へ委譲することを言える
- [ ] **TTL とキャッシュ**、そして「DNS 変更が反映されない=伝播遅延」を説明できる ← 最重要
- [ ] `dig` で名前解決を確認できる/「IP 直打ちで通るか」で DNS を切り分けられる

**全部完璧でなくて大丈夫**。レコードの種類や JVM プロパティの細部は、必要になったとき戻ってくれば十分。「**名前→IP を DNS がやる / 木構造 / TTL とキャッシュが実務の罠**」の3つが体に入っていれば、この章は 7割クリアです。

## 次章の予告

**第5章 HTTP ― Web の会話のルール**

名前が IP に変換でき(第4章)、その IP:ポートに TCP で繋がった(第3章)。**では、繋がった後、実際に何を喋るのか?** それが **HTTP** です。あなたがフレームワークで毎日書いている、あの `GET` / `POST` / ステータスコード(200・404・500)/ ヘッダ ―― その正体を、正面から扱います。

第3章の `nc` で「HTTP は TCP の上を流れるただのテキスト」だと、もう一度体感しました。第5章ではそのテキストの文法(メソッド・ステータス・ヘッダ・ボディ)を読み解き、**`curl -v` で本物の HTTP のやりとりを丸ごと覗きます**。実務で毎日見る 404 や 500、キャッシュヘッダ、Cookie ―― 「なんとなく使っていたもの」が「分かって使えるもの」に変わる、実務直結度 MAX の章です。
