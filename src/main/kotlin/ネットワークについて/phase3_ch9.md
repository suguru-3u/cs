## 0. これまでの全部が、ここで1つになる

```

第2章  ping/traceroute を手にした(IP・経路)
第3章  nc を手にした(TCP・ポート)

```

現場のトラブルは、たいてい **「繋がらない」「遅い」「たまに失敗する」** の3つに集約されます。この章は、その3つそれぞれに **決まった型**を与えます。型があれば、パニックにならず、当てずっぽうもせず、**順序立てて原因にたどり着ける**。

## 1. 完成版「層で切り分ける地図」

まず、Phase 3 全体の地図の完成形です。**トラブルのとき、この表を上から下へ(または下から上へ)たどる**のが基本です。

| 層 | 何を確認するか | 道具 | 失敗したときの症状 | 学んだ章 |
|---|---|---|---|---|
| **DNS** | 名前→IP に変換できる? | `dig` / `getent` | 名前だと繋がらないが IP直打ちなら繋がる | 第4章 |
| **IP・経路** | その IP まで届く? | `ping` / `traceroute` | 全く応答がない、packet loss、途中で止まる | 第2章 |
| **TCP・ポート** | 目的のポートは開いてる? | `nc -z -v` | refused(閉/拒否)or timeout(無反応) | 第3章 |
| **TLS** | 暗号の握手・証明書は? | `openssl s_client` | 証明書エラー、握手失敗、期限切れ | 第6章 |
| **HTTP** | ステータス・ヘッダは? | `curl -v` | 4xx(自分)/ 5xx(サーバ)/ 502・504 | 第5章 |
| **アプリ・信頼性** | タイムアウト・プール・リトライ | ログ / 監視 | たまに遅い/たまに失敗/固まる | 第7・8章 |

**2つのたどり方**があります。

- **下から上へ(積み上げ確認)**: DNS→IP→TCP→TLS→HTTP と、土台から順に「ここまでは OK」を確かめる。原因が下の層にあるとき効率的。
- **上から一発(curl -v)**: `curl -v` は **DNS→TCP→TLS→HTTP を一気に実行し、どこで止まったかを教えてくれる**。だから、まず `curl -v` を撃って、**止まった場所の層を、その層専用の道具で深掘り**するのが実戦的です。

## 2. 「繋がらない」の調べ方 ― 層を1つずつ潰す

「繋がらないんですけど」と言われたら、この順に潰します。

### まず一撃:`curl -v` でどこまで進むか見る

```
$ curl -v https://api.example.com/health
```

`curl -v` の出力は、**上から順に層をたどる**ので、**どの行で止まったか = どの層で失敗したか**が分かります。

```
* Host api.example.com ... was resolved.        ← ここで止まる → DNS(第4章)
* Trying 203.0.113.5:443...                      ← ここで止まる → IP/経路 or ポート
* Connected to api.example.com                   ← TCP接続OK(第3章クリア)
* TLS handshake ...                              ← ここで止まる → TLS/証明書(第6章)
* SSL certificate verify ok                      ← TLS OK(第6章クリア)
< HTTP/2 503                                     ← 繋がってるが 5xx → サーバ側(第5章)
```

止まった場所で、次の深掘りが決まります。

### 層ごとの深掘り(止まった場所に応じて)

**DNS で止まった(名前が解決できない)**
```
$ dig api.example.com              # 名前解決を直接確認(第4章)
$ dig @8.8.8.8 api.example.com     # 外の公開リゾルバでも引けるか比較 → 引ければ自分のDNS設定が原因
```
鉄板の切り分け:**IP 直打ちなら通るか?** 通れば DNS の問題で確定(第4章)。

**IP/経路で止まった(接続すら始まらない)**
```
$ ping api.example.com             # そもそも IP に届くか、loss はないか(第2章)
$ traceroute api.example.com       # どこまで行って、どこで止まるか(第2章)
```
自宅ルータで止まる/プロバイダで止まる/相手直前で止まる、で原因の場所が変わる。

**TCP/ポートで止まった**
```
$ nc -z -v api.example.com 443     # ポートが開いてるか。refused か timeout か(第3章)
```
**refused = 相手は生きてるがポートが閉/拒否**(サーバ設定寄り)。**timeout = 返事すら来ない**(経路/ファイアウォール寄り)。この読み分けで方向が半分に絞れる(第3章)。

**TLS で止まった**
```
$ openssl s_client -connect api.example.com:443 -servername api.example.com </dev/null
```
証明書の期限切れ・信頼されない CA・ホスト名不一致を確認(第6章)。`Verify return code` を見る。

**繋がったが 4xx/5xx が返る**
```
$ curl -v https://api.example.com/... -H "Authorization: Bearer xxx"
```
**4xx なら自分のリクエスト**(URL・認証・パラメータ)、**5xx ならサーバ側**、**502/504 なら奥との接続・応答待ち**を疑う(第5章・第8章)。

> **繋がらないフローの本質**:「繋がらない」という漠然とした一言を、**層ごとの独立した問い**に分解する。第1章で立てた「層で切り分ける」が、ここで完全な実戦手順になりました。

## 3. 「遅い」の調べ方 ― レイテンシを分解する

「なんか遅い」は、**時間がどこで消えているかを分解**すれば、原因が見えます。主役は `curl` の **タイミング内訳**です。

### curl のタイミング内訳(本物の実測)

`https://pypi.org` への1リクエストを、各段階の累積時間で分解しました(本物):

```
DNS解決(namelookup):        0.008s   ┐
TCP接続(connect):           0.009s   │ 各段階の "累積" 時間
TLS完了(appconnect):        0.056s   │
送信直前(pretransfer):      0.058s   │
最初のバイト(starttransfer): 0.089s   │
全体(total):                0.089s   ┘
```

累積なので、**引き算すると各層が消費した時間**が出ます。ここが読み方の肝です。

| 区間(引き算) | 何の時間か | この時間が大きいと… | 層 |
|---|---|---|---|
| namelookup | DNS 解決 | 名前解決が遅い | DNS(第4章) |
| connect − namelookup ≒ **1ms** | TCP ハンドシェイク | 遠い/RTT が大きい | IP・TCP(第2・3章) |
| appconnect − connect ≒ **47ms** | TLS ハンドシェイク | TLS/証明書のネゴが重い | TLS(第6章) |
| starttransfer − pretransfer ≒ **31ms** | **サーバが考えている時間(TTFB)** | **サーバ処理が遅い** | サーバ側(第5・8章) |
| total − starttransfer ≒ **0ms** | 本体のダウンロード | データが大きい/回線が細い | 転送 |

この実測では、**TLS 握手(47ms)とサーバ処理(31ms)が二大要素**でした。もし「遅い」の相談で:

- **starttransfer(TTFB)が大きい** → **サーバが遅い**(重いクエリ・下流の遅延)。ネットワークは無罪。最頻出の「遅い」原因。第8章の応答タイムアウトが発動するのもここ。
- **appconnect が毎回大きい** → TLS 握手を**毎回やっている** = 接続を使い回せていない(第8章)。クライアント使い回しで改善。
- **namelookup が大きい** → DNS が遅い(第4章)。
- **total − starttransfer が大きい** → ダウンロードが重い(大きな JSON、画像など)。ページング・圧縮を検討。

> **遅いフローの本質**:「遅い」を **DNS / TCP / TLS / サーバ処理 / 転送** の5つに分解し、**どこで時間が消えているか**を1コマンドで特定する。「サーバが遅いのか、ネットワークが遅いのか、握手が遅いのか」を推測でなく**実測で**切り分けられる。これが Phase 3 で身につけた最大の実務力です。

### コマンド化(手元で使える）

```
$ curl -s -o /dev/null -w "dns=%{time_namelookup} tcp=%{time_connect} tls=%{time_appconnect} ttfb=%{time_starttransfer} total=%{time_total}\n" https://api.example.com/
```

この1行をコピペしておけば、いつでもレイテンシを分解できます。**あなたの Phase 3 の集大成の1行**です。

## 4. 「たまに失敗する」の調べ方 ― データを集めてパターンを見る

一番やっかいなのが「**たまに**遅い/失敗する」。狙って再現できないからです。型は **「1回の観察ではなく、たくさん観察してパターンを探す」**。

### 繰り返し計測して、ばらつきと外れ値を見る

同じリクエストを10回計測しました(本物):

```
#1  total=0.053s     #6  total=0.057s
#2  total=0.051s     #7  total=0.063s  ← やや遅い
#3  total=0.064s ←   #8  total=0.057s
#4  total=0.054s     #9  total=0.055s
#5  total=0.053s     #10 total=0.052s
```

大半は 51〜57ms ですが、**#3 と #7 が 63〜64ms** と外れています。「たまに遅い」は、こういう**外れ値(tail)**として現れます。1回だけ見ても分からない。**繰り返して分布を見る**のが鉄則です。

コマンドはループにするだけ:
```
$ for i in $(seq 1 100); do curl -s -o /dev/null -w "%{time_starttransfer}\n" URL; done
```
100回なり回して、**時々混じる遅い回**を探す。ping が使える環境なら `ping -c 100 host` で **packet loss %** を見るのも有効(第2章。loss があれば TCP 再送で「たまに遅い」になる)。

### パターンで原因を絞る

集めたデータを、こう分類します。

- **負荷の高い時間帯だけ遅い/失敗** → サーバ過負荷、または自分側の**スレッドプール/コネクションプール枯渇**(第7・8章)。
- **特定のバックエンドに当たったときだけ** → ロードバランサ配下の**1台だけ不調**。`dig` で複数 IP を確認、当たり外れを疑う。
- **どのタイムアウトが発動したか**(第8章) → 接続タイムアウトなら経路、応答タイムアウトならサーバの遅さ。
- **完全にランダム、低頻度** → ネットワークの一時的揺らぎ(**packet loss**、第2章)。**べき等な呼び出しはリトライ**(第8章)で吸収できる。

### 実務では「監視」で tail を捉える

現場では、毎回手で叩く代わりに **監視(モニタリング)** を使います。**p50 / p95 / p99**(パーセンタイル)という指標で、「99%のリクエストが何ms以内か」を見る。「たまに遅い」は多くの場合 **p99 のtail(裾)** に現れます。平均だけ見ていると「たまに」は埋もれるので、**tail(p99)を見る**のが定石 ―― これは Phase 4(分散システム)や実務の SRE で深掘りする話です。今は「平均でなく tail を見る」とだけ覚えておけば十分。

## 6. Kotlin/JVM 側からの観測 ― アプリに仕込む

外から `curl` で調べるだけでなく、**あなたの Kotlin アプリ自身に "調べる目" を仕込む**のも重要です。ツールは相手が見えているときしか使えませんが、本番の「たまに」はアプリのログにしか残らないことも多い。

- **どのタイムアウトで失敗したかをログに残す**:`HttpConnectTimeoutException`(接続)と `HttpTimeoutException`(応答)を区別してログする(第8章の2種類のタイムアウト)。後から「接続で落ちたのか、応答待ちで落ちたのか」が分かる。
- **各呼び出しの所要時間を計測してログ**:`System.nanoTime()` で外部呼び出しを挟み、遅い呼び出しを記録。§3 の分解のアプリ版。
- **JVM 特有の罠を思い出す**:
  - 「アプリだけ古い IP を掴む」→ **JVM の DNS キャッシュ**(第4章、`networkaddress.cache.ttl`)。
  - 「アプリだけ証明書エラー」→ **JVM のトラストストア**(第6章、`PKIX path building failed`)。
  - 「アプリの外部呼び出しが全部遅い」→ **クライアントを毎回 new していないか**(第8章、プールが効かない)。

> **Phase 2 との接続**: これは Phase 2 で実測しながら学んだ姿勢そのものです。「本当にそうなっているか、計測して確かめる」。ツールで外から、ログで内から ―― **推測ではなく実測で原因に迫る**。Phase 3 で `ping`/`curl`/`openssl` の実測を重ねてきたのは、この姿勢を身につけるためでもありました。

## 7. 総合演習 ― シナリオ切り分け(Phase 3 の集大成)

Phase 3 の全知識を動員します。**まず自分で「どの層を、どの道具で、何を疑うか」を考えてから**開いてください。7割で OK。

### シナリオ1
「昨日まで動いていた外部 API 連携が、今朝から全滅。`curl` すると `could not resolve host`。でも同僚の環境では動く」。

<details><summary>解答・解説</summary>

`could not resolve host` = **DNS で失敗**(第4章)。同僚は動くので、**あなたの環境の DNS 設定・リゾルバ**が怪しい。
- `dig API_HOST` で自分のリゾルバの結果、`dig @8.8.8.8 API_HOST` で外のリゾルバの結果を比較。
- 外なら引けるのに自分のリゾルバで引けないなら、**社内 DNS / VPN / hosts ファイル / DNS キャッシュ**を疑う。
- IP 直打ちで通れば DNS 確定。
「同僚は動くが自分は動かない」+「名前解決失敗」→ ほぼ DNS 環境差。第4章の切り分けそのもの。
</details>

### シナリオ2
「API 呼び出しが**たまに**5〜10秒かかる(普段は50ms)。エラーにはならず、遅いだけ。負荷が高い夕方に多い」。

<details><summary>解答・解説</summary>

エラーでなく「たまに遅い」+「負荷が高い時間帯」→ **サーバ過負荷** か **自分側のプール枯渇**(第7・8章)が最有力。
- `curl -w` の**内訳**を遅い時に取る(§3)。**starttransfer(TTFB)が大きい**なら**サーバが考えている時間が長い** = サーバ側の遅さ(重いクエリ・下流の詰まり)。
- **appconnect が毎回大きい**なら接続を使い回せていない(クライアント毎回 new? 第8章)。
- 自分側なら:コネクションプールのサイズ、スレッドプールの状態を確認。接続待ちで詰まっていないか。
- 100回ループで分布を取り、遅い回の割合と時間帯の相関を見る(§4)。
「たまに・遅いだけ・高負荷時」の三点セットは、過負荷/プール枯渇の典型。まず TTFB を分解して、サーバ側か自分側かを切り分ける。
</details>

### シナリオ3
「ブラウザでは開けるサイトに、Kotlin アプリから繋ぐと `PKIX path building failed`」。

<details><summary>解答・解説</summary>

**JVM のトラストストア問題**(第6章)。証明書を署名した CA が JVM の `cacerts` に無い(社内 CA・中間証明書漏れなど)。ブラウザは OS のトラストストアを使うので通るが、JVM は別物。
- `openssl s_client -connect HOST:443` で証明書チェーンと発行元を確認。
- 対処:正しい CA を JVM のトラストストアに**追加**する(検証を切る trust-all は厳禁=なりすまし対策を捨てる)。
「他ツールは OK・JVM だけ証明書エラー」→ トラストストアの差、が定石。
</details>

### シナリオ4
「決済 API へのリクエストがタイムアウトした。リトライ処理を入れたら、今度は**二重課金**が発生した」。

<details><summary>解答・解説</summary>

**POST(べき等でない)を安易にリトライした**結果の二重実行(第5・8章)。タイムアウトは「成功したか分からない」状態(第8章 §1)で、実は1回目が成功していた。
- 対処:POST のリトライには**冪等キー**を付け、サーバ側で「同じキーは1回だけ実行」する。または、そもそも**べき等な操作以外は自動リトライしない**。
- タイムアウト値自体も見直す(短すぎて成功前に打ち切っていないか、第8章 §2)。
「タイムアウト対策のリトライが二重実行を生む」は超頻出の事故。べき等性(第5章)を踏まえた設計が答え。
</details>

### シナリオ5
「`nc -z -v HOST 443` は `succeeded` なのに、`curl https://HOST` が固まって返ってこない」。

<details><summary>解答・解説</summary>

**TCP は繋がる(nc 成功=第3章クリア)が、その先で止まっている**。TCP より上の層 ―― **TLS 握手**か**サーバの応答**が怪しい。
- `openssl s_client -connect HOST:443`(第6章)で TLS 握手が完了するか。証明書の問題やプロトコル不一致で握手が固まることがある。
- 握手も通るなら、**サーバが HTTP 応答を返さない**(サーバ側が固まっている=応答タイムアウト待ち、第8章)。`curl -v` でどこまで進むか、`--max-time` で打ち切って観察。
「TCPは開くが上で固まる」→ 層を1つ上げて TLS→HTTP と潰す。第9章の "止まった層の1つ上を見る" の実践。
</details>

## 8. Phase 3 全体の振り返り ― 何を手に入れたか

最終章なので、Phase 3 の旅を振り返ります。あなたは9章かけて、**「`client.send()` の裏で何が起きているか」**(第1章の問い)を、下から上まで説明できるようになりました。

### 積み上げた地図

```
名前(example.com)を IP に変換し        …… DNS(第4章)
その IP 目がけてパケットをバケツリレー   …… IP・ルーティング(第2章)
取りこぼさない信頼できるパイプを作り     …… TCP(第3章)
暗号で包んで盗聴・なりすましを防ぎ       …… TLS/HTTPS(第6章)
その上でテキストの会話をする            …… HTTP(第5章)
――これを自分の手で書き(第7章)、
  失敗する前提で守り(第8章)、
  壊れたときに調べ切る(第9章)。
```

### 繰り返し現れた「大きな考え方」

Phase 3 を通して、同じ発想が何度も形を変えて現れました。これらは Phase 3 を超えて効く**設計の型**です(あなたが CS の後に学びたい「設計」の下ごしらえ)。

- **抽象化(下の層の複雑さを、上の層に見せず覆い隠す)**: 層・socket・TCP が IP を覆う・DNS の名前が IP を覆う。**同じ型が何度も**。
- **信頼できない土台の上に、信頼性を作る**: ベストエフォートな IP の上に、TCP が書留を作る。
- **キャッシュ(速いが古くなる)**: DNS の TTL、HTTP キャッシュ、CPU キャッシュ(Phase 2)。「速さ vs 鮮度」のトレードオフ。
- **高コストな資源の使い回し**: スレッドプール(Phase 2)、コネクションプール(第8章)。双子。
- **待ちを手放す**: 非同期・軽量スレッド(第7章)。ネットワークは待ちだらけだから効く。
- **失敗する前提で作る**: タイムアウト・リトライ・サーキットブレーカー(第8章)。

### Phase 2 との合流、そして次へ

Phase 3 は Phase 2 の続きでした。socket = I/O、スレッド上限(実測3.2万)、ブロッキング、キャッシュ ―― Phase 2 の実測知識が、ネットワークで生きました。**CS は地続き**です。

そして次の **Phase 4「DB と分散システム」**は、この Phase 3 の上に立ちます。DB への接続もソケット(コネクションプールは第8章そのもの)、分散システムは「ネットワーク越しの信頼できない呼び出し」の集合体(第8章の失敗前提が主役に)。**Phase 3 で作った "ネットワークは失敗する前提" の感覚が、分散システムの心臓部**です。tail レイテンシ(§4)も p99 も、そこで深掘りします。

## 9. この章のまとめ

- トラブルは **「繋がらない」「遅い」「たまに失敗する」** の3つ。それぞれに**型**がある。
- **繋がらない**:`curl -v` で止まった層を特定 → その層の道具(dig/ping/nc/openssl)で深掘り。層を1つずつ潰す。
- **遅い**:`curl -w` のタイミング内訳で、時間を **DNS/TCP/TLS/サーバ処理/転送** に分解。**TTFB が大きい=サーバが遅い**が最頻出。推測でなく実測で切り分ける。
- **たまに失敗する**:1回でなく**繰り返し計測して分布・外れ値・パターン**を見る。負荷連動→過負荷/プール枯渇、ランダム→packet loss+リトライ。実務では tail(p99)を監視。
- **早見表**(§5)を保存版に。**Kotlin アプリ側にも観測を仕込む**(どのタイムアウトか、所要時間、JVM の3大罠)。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] 「層で切り分ける地図」を、各層と道具のセットで言える
- [ ] 「繋がらない」を curl -v + 各層の道具で順に潰せる
- [ ] 「遅い」を curl のタイミング内訳で分解し、TTFB=サーバの遅さと読める
- [ ] 「たまに失敗する」は繰り返し計測してパターンを見る、と言える
