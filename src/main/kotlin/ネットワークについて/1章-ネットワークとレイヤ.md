# Phase 3・第1章 ネットワークとレイヤ ― なぜ層で考えるのか

> **この章の位置づけ**: Phase 3 の入口です。演習はありません。ロードマップの言葉を借りれば「地図を眺める」章です。ここで **「ネットワークを層(レイヤ)で考える」という一生モノの視点**を手に入れます。この視点は、最終章(第9章「つながらない・遅いの調べ方」)で「どの層が怪しいか」を切り分ける武器になります。
>
> **7割理解で OK**。この章は暗記する章ではなく、地図の全体像を一度眺めて「そういう構造なんだ」と掴む章です。細部は後の章で何度も戻ってきます。

## 0. まず、あなたが毎日書いているコードから

Phase 3 に入るにあたって、いきなり抽象論から始めません。**あなたがフレームワークで普段書いている、こういうコード**から始めます。

```kotlin
import java.net.URI
import java.net.http.HttpClient
import java.net.http.HttpRequest
import java.net.http.HttpResponse

fun main() {
    val client = HttpClient.newHttpClient()
    val request = HttpRequest.newBuilder()
        .uri(URI.create("https://example.com"))
        .build()

    val response = client.send(request, HttpResponse.BodyHandlers.ofString())
    println(response.statusCode())   // 例: 200
    println(response.body().take(100))
}
```

このコード、実務では何十回も書いてきたはずです。`client.send(...)` を呼べば、レスポンスが返ってくる。当たり前に使っています。

でも、**この1行 `client.send(...)` の裏で、実際には何が起きているのか?** これを説明できるでしょうか。

- `example.com` という「名前」は、どうやって世界のどこかのサーバに届く「住所」に変わったのか?(→ **第4章 DNS**)
- `https://` の "s"(暗号化)は、いつ・どうやって成立したのか?(→ **第6章 HTTPS/TLS**)
- そもそも、この JVM プロセスは相手のサーバと、どうやって「線」を繋いだのか?(→ **第3章 TCP**、**第7章 ソケット**)
- 途中で「たまにタイムアウトする」のはなぜか? どこで詰まっている?(→ **第8・9章**)

Phase 3 のゴールは、この `client.send(...)` の裏側を **大まかに説明できる**ようになることです。そして裏側が見えると、「繋がらない」「遅い」というトラブルで **どこを疑えばいいか**が分かるようになります。

> **不安を先回りで解消**: 「こんな低レイヤを知らなくても、これまで開発できてたよね?」という気持ちが湧いたら、それは正常です。実際、知らなくても書けます。ただ、**トラブルが起きた瞬間に、知っている人と知らない人で対応速度が桁違いになる**。Phase 3 はそのための投資です。

## 1. 身近な例:手紙を出すとき、あなたは何を知らずに済んでいるか

ネットワークの「層」という考え方は、実は**郵便**とそっくりです。

あなたが友人に手紙を出すとき、やることはこれだけです。

1. 便箋に手紙を書く
2. 封筒に入れて、宛先の住所と自分の住所を書く
3. ポストに入れる

これで届きます。ここで **あなたが一切考えなくて済んでいること** に注目してください。

- 郵便局員が、どのトラックに載せるか
- そのトラックが、どの高速道路を通るか
- 途中でどの集配センターを経由するか
- 飛行機を使うのか、船を使うのか

**あなたは「宛先の住所」だけ知っていればよく、輸送の中身は完全に隠されている**。これがまさに「層」です。あなた(手紙を書く層)は、下の層(輸送のしくみ)を **信頼して丸投げ**している。下の層は複雑なことをやっているけれど、それをあなたに見せない。

そして逆も成り立ちます。**郵便局は、封筒の中身(手紙の内容)を知らなくても運べる**。ラブレターだろうと請求書だろうと、封筒の外側の住所さえ正しければ、同じように運ぶ。

> ここに層の本質が2つ現れています。
> - **上の層は、下の層の「やり方」を知らなくてよい**(あなたは輸送経路を知らない)
> - **下の層は、上の層の「中身」を知らなくてよい**(郵便局は手紙の内容を知らない)
>
> お互いに **決められた接点(＝住所を書いた封筒)** だけでやりとりする。これが「層に分ける」ということです。

## 2. なぜ、わざわざ層に分けるのか

「便利そう」だけでは弱いので、**層に分けると具体的に何が嬉しいのか**を押さえます。ここが第1章の核心です。

### 嬉しさ①:複雑さを分割して、各層が自分の仕事に集中できる

もし層がなかったら、「手紙を書く人」がトラックの燃料や高速道路の渋滞まで全部考える必要があります。無理です。層に分けることで、**各層は「自分の担当」だけに集中**できます。手紙を書く人は文章に、郵便局は輸送に。

ネットワークも同じです。後で出てくる各層は、それぞれ**担当が1つ**です。

- ある層は「どのマシンに届けるか(住所)」だけを担当(→ IP)
- ある層は「取りこぼしなく順番通りに届けるか」だけを担当(→ TCP)
- ある層は「Web の会話の中身」だけを担当(→ HTTP)

### 嬉しさ②:ある層を取り替えても、他の層が壊れない

これが実務的に一番効きます。

あなたがノート PC で `https://example.com` を開くとき、**WiFi でも、有線 LAN でも、スマホのテザリング(4G/5G)でも、まったく同じコードで動きます**。

なぜ動くのか? **一番下の「物理的な運び方」の層を取り替えても、その上の層(HTTP を話す層)は何も変わらない**からです。HTTP は「下がどう運ぶか」を知らないし、知る必要もない。だから WiFi ↔ 有線を切り替えても、`client.send(...)` のコードは1文字も変えずに動く。

> これは Phase 2 第1章で学んだ **「抽象化」** そのものです。上の層は、下の層が提供する **決まった約束(インタフェース)** だけを使う。下の層の中身が変わっても、約束さえ守られていれば上は無傷。
>
> Phase 2 では「OS がハードウェアの違いを隠す(だからアプリはハードを気にせず書ける)」という抽象化を見ました。ネットワークの層も、まったく同じ発想の繰り返しです。**同じ考え方が、場所を変えて何度も出てくる**。CS を体系的にやる面白さは、この「あ、これ前に見たやつだ」の積み重ねにあります。

### 嬉しさ③:問題の切り分けができる ← 実務で一番効く

手紙が届かなかったとき、原因を **層ごとに切り分け**られます。

- 宛先の住所を書き間違えた?(あなたの層の問題)
- 住所は合っていたが、輸送中に紛失した?(輸送の層の問題)

ネットワークでも同じで、「繋がらない」とき、**どの層で詰まっているか**を順に確認できます。

- 名前(`example.com`)が住所に変換できていない? → **DNS の層**
- 住所は分かったが、そもそも線が繋がらない? → **TCP の層**
- 線は繋がったが、暗号化の握手で失敗? → **TLS の層**
- 全部 OK だが、返ってきた中身がエラー? → **HTTP/アプリの層**

**この「層ごとの切り分け」こそが、Phase 3 全体の実務的なゴール**です(第9章で総まとめします)。今この地図を持っておくと、以降の各章が「地図のどのマスの話か」として頭に入ります。

## 3. Phase 2 との接続:`socket` は「ネットワーク版の I/O」

ここで Phase 2 の記憶を呼び起こします。**この接続が、Phase 3 の一番大事な入口**です。

Phase 2 第9章で、こんなことを学びました。

- プログラムが外の世界(ファイル・ディスクなど)とやりとりするには、**OS に頼む(システムコール)**しかない。
- ファイルを開くと **ファイル記述子(fd)** という「取っ手」がもらえて、それに対して `read` / `write` する。
- そして第9章の最後で、**「`socket` というシステムコールがある」と予告**されていました。

その予告の回収がここです。

> **ソケット(socket)とは、「ネットワーク版のファイル」です。**
>
> ファイルを開くと fd がもらえて read/write できたように、
> **ソケットを開くと、やはり「取っ手」がもらえて、それに read/write するとネットワーク越しに相手とデータをやりとりできる。**

つまり、あなたが `client.send(...)` を呼んだとき、その一番奥では:

1. OS に「ソケットをくれ」とシステムコール(`socket`)する
2. 相手に繋ぐ(`connect`)
3. その取っ手に「GET / HTTP/1.1 ...」と **write** する
4. 返事を **read** する

これが起きています。**ネットワーク通信は、結局のところ「相手が遠くにいる I/O」**なのです。Phase 2 で散々やった read/write が、ファイル相手ではなくネットワーク相手になっただけ。

Kotlin で「生のソケット」を書くと、こんな雰囲気になります(詳しくは第7章でやるので、今は**眺めるだけ**で十分):

```kotlin
import java.net.Socket

fun main() {
    // ①「取っ手」をもらって相手に繋ぐ(socket + connect に相当)
    Socket("example.com", 80).use { socket ->
        val out = socket.getOutputStream()   // write する口
        val input = socket.getInputStream()  // read する口

        // ② 取っ手に「文字」を write する(これがリクエスト)
        val request = "GET / HTTP/1.1\r\nHost: example.com\r\nConnection: close\r\n\r\n"
        out.write(request.toByteArray())
        out.flush()

        // ③ 取っ手から「文字」を read する(これがレスポンス)
        val response = input.readBytes().decodeToString()
        println(response.take(200))
    }
}
```

`getOutputStream()` / `getInputStream()` ―― この **ストリームへの read/write**、Phase 2 でファイルに対してやったことと **まったく同じ形**ですよね。これが「socket = ネットワーク版の I/O」の意味です。

> **接続の確認**:
> - `Socket(...)` を開く = ネットワーク版の「ファイルを開く」(システムコール)
> - `getInputStream()/getOutputStream()` への読み書き = Phase 2 の I/O・read/write そのもの
> - `.use { }`(Kotlin の try-with-resources)= fd と同じで、**ソケットも使い終わったら閉じる資源**。閉じ忘れると「fd リーク」ならぬ「ソケットリーク」になる ← これは第8章のコネクション管理で効いてきます
>
> **Phase 3 は Phase 2 の続き**です。新しい世界に来たのではなく、Phase 2 で覚えた I/O を「遠くの相手」に向けただけ。ここが腑に落ちると、この先ずっと楽になります。

`HttpClient` のような便利なクラスは、この生のソケット操作を **隠してくれている**わけです(また層の話!)。フレームワークが隠しているものの正体が、この生のソケットです。

## 4. データは「パケット」という小分けで運ばれる

もう1つ、この章で掴んでおきたい単位があります。**パケット**です。

大きなデータを送るとき、ネットワークは **一気に丸ごと**は送りません。**小さな塊(パケット)に分割**して、1つずつ運びます。

郵便のたとえに戻ると、**分厚い本を1冊まるごと1つの封筒には入れられない**ので、**ページを何枚かずつ別々の封筒に分けて送り、届いた先で順番通りに綴じ直す**イメージです。

なぜわざわざ分割するのか。ざっくり言うと:

- **途中の道(回線)を、みんなで細切れに共有できる**。1人が巨大なデータで道を独占しないよう、小分けにして交互に流す。
- **一部が壊れても、その小さな塊だけ送り直せばいい**。本を丸ごと送り直すより、失敗したページだけ再送する方が効率的。

ここで疑問が湧くはずです。「小分けにしたら、**順番がバラバラに届いたり、途中で1個なくなったり**しない?」――**します**。そして「それをどう解決するか」が、まさに **第3章 TCP** のテーマです(TCP が順番を揃え、なくなった分を再送してくれる)。今は「データはパケットに分かれて運ばれる」という単位だけ覚えておけば十分です。

> **Phase 2 との接続**: この「大きいデータを小分けにして運ぶ」発想、Phase 2 でも見ました。第9章のバッファ I/O(まとめて運ぶと速い)や、第5章のキャッシュライン(メモリもある大きさの塊で運ぶ)と、根っこは同じ「適切な塊の単位で運ぶ」という話です。

## 5. 層のモデル ― TCP/IP モデルと OSI モデル

さて、ここまで「層」「層」と言ってきました。では**具体的にどんな層があるのか**、地図を広げます。

ネットワークの層を整理するモデルは、有名なものが2つあります。**丸暗記は不要**です。「こういう段重ねになっている」という **雰囲気**だけ掴んでください。

### TCP/IP モデル(実務で実際に使う、4層)

現実のインターネットは、ざっくり **4つの層**の積み重ねでできています。**下から上へ**並べます。

| 層(下から) | 担当していること | 代表的なもの | 学ぶ章 |
|---|---|---|---|
| ④ アプリケーション層 | アプリ同士の「会話の中身」 | HTTP, DNS, TLS | 第4〜6章 |
| ③ トランスポート層 | 「取りこぼさず・順番通り」or「速く」相手プロセスへ | TCP, UDP | 第3章 |
| ② インターネット層 | 「どのマシンに届けるか」(住所) | IP | 第2章 |
| ① リンク層 | 実際に電気・電波で隣へ運ぶ | Ethernet, WiFi | (雰囲気だけ) |

**下の層ほど「物理的な運び方」に近く、上の層ほど「アプリの都合」に近い**。そして各層は、**すぐ下の層のサービスだけを使い**、その下がどうなっているかは気にしません(層の原則!)。

この表は、そのまま **Phase 3 の学習ルート**になっています。第2章で②を、第3章で③を、第4〜6章で④を、下から積み上げていきます。**あなたが毎日書く HTTP は、一番上の④**。その下に TCP・IP・物理が支えとして積まれている、という構造です。

### OSI モデル(教科書によく出る、7層)

もう1つ、**OSI 参照モデル**という **7層**の分け方があります。TCP/IP モデルより細かく分かれていて、教科書や資格試験でよく出てきます。

7層は下から:物理 → データリンク → ネットワーク → トランスポート → セッション → プレゼンテーション → アプリケーション。

ただ **実務で「第7層が…」と番号で会話することは、そう多くありません**(あるとすれば「L4 ロードバランサ」「L7 で振り分け」くらい。L4=トランスポート、L7=アプリの意味)。なので **今は「7層の細かいモデルもある」という存在だけ知っておけば十分**。この教科書は、実務に直結する **TCP/IP の4層**を軸に進めます。

> **7割理解ポイント**: 「4層と7層、どっちを覚えるの?」と迷ったら、**4層(TCP/IP)を優先**。7層は「そういう精密な地図もある」程度で今は流して OK です。番号の暗記より、「**下ほど物理、上ほどアプリ**」「**各層は隣の層としか話さない**」という感覚の方がずっと大事。

## 6. カプセル化 ― 封筒の中に封筒(層はこうやって重なる)

層が「積み重なる」とき、データは物理的にどうなっているのか。ここだけ1つ、具体的なイメージを持っておくと、この先ずっと効きます。**カプセル化(encapsulation)** です。

送信するとき、**各層は、上の層から受け取ったデータを「自分の封筒」に入れて、宛名(ヘッダ)を付けて下の層に渡します**。

イメージは **入れ子の封筒(マトリョーシカ)** です。上から下へ渡るたびに、封筒が1枚ずつ外側に増えていきます。

```
  [ HTTP: GET / ... ]                        ← アプリ層が用意した中身
         ↓ TCP が封筒で包む
  [ TCP頭 | HTTP: GET / ... ]                ← 「ポート番号・順番」の宛名を付けた
         ↓ IP が封筒で包む
  [ IP頭 | TCP頭 | HTTP: GET / ... ]         ← 「相手マシンの住所」の宛名を付けた
         ↓ リンク層が封筒で包む
  [ 物理頭 | IP頭 | TCP頭 | HTTP: ... ]      ← 「隣の機器」の宛名を付けた → 送信
```

そして **受信側では、逆に外側の封筒から1枚ずつ開けていきます**。リンク層の封筒を開け、IP の封筒を開け、TCP の封筒を開け、最後に一番内側の HTTP の中身をアプリが受け取る。

各層は **自分の封筒(ヘッダ)だけを見て**、中身(上の層が入れたもの)には手を出しません。郵便局が封筒の中の手紙を読まないのと同じです。**上の層にとっては、下の層が付けた封筒は見えない**。だから HTTP を書くあなたは、TCP や IP のヘッダを意識せずに済んでいる。

> **なぜこれを今知っておくと良いか**: 第9章でパケットを覗く(`tcpdump` 等の雰囲気)とき、「HTTP の中身が IP や TCP のヘッダで包まれて流れている」構造が見えます。今この入れ子のイメージがあると、そこで慌てません。**今は「封筒が入れ子になっている」だけ掴めば 7割 OK**。

## 7. 【実務接続】「どの層が怪しいか」の地図を持つ

第1章の締めくくりとして、**この章で得た地図が、実務でどう武器になるか**を言語化します。これが Phase 3 の各章末で必ずやる「実務接続」の第1回です。

現場で一番よく聞くセリフは、たぶんこれです。

> **「なんか、繋がらないんですけど」**

このとき、**層の地図を持っていない人**は「うーん…」とフリーズするか、当てずっぽうでサーバを再起動したりします。一方、**層の地図を持っている人**は、こう考えます。

> 「繋がらない」を、**下の層から順に**切り分けよう。
>
> 1. **DNS の層**:そもそも `example.com` が住所(IP)に変換できてる?(→ 第4章 / `dig`)
> 2. **IP/TCP の層**:住所は分かった。その住所に線は繋がる?(→ 第2・3章 / `ping`, `nc`)
> 3. **TLS の層**:線は繋がった。暗号化の握手は成功してる?(→ 第6章 / `openssl`, `curl -v`)
> 4. **HTTP/アプリの層**:全部繋がった。でも返ってきた中身がエラー(4xx/5xx)?(→ 第5章 / `curl -v`)

**「繋がらない」という一言を、4つの独立した問いに分解できる**。これが層で考えることの実務的な威力です。どこで詰まっているか当たりがつけば、調べる範囲も、直す場所も、相談する相手も絞れる。

具体例をいくつか(今は「ふーん」で OK。各章で詳しくやります):

- **「特定のサイトだけ繋がらない、他は繋がる」** → 名前解決かそのサイト固有の問題っぽい。まず DNS(第4章)を疑う。
- **「さっきまで動いてたのに急に全部繋がらない」** → もっと下の層(回線・ルーティング、IP レベル)を疑う。
- **「繋がるけど、ブラウザが『この接続は安全ではありません』と言う」** → TLS/証明書の層(第6章)。
- **「繋がって、レスポンスも返るが 500 が返る」** → もう HTTP から上、アプリ側の問題(第5章)。ネットワークは正常。

> **これが第9章への伏線**です。第9章「つながらない・遅いの調べ方」は、この地図を完成させ、各層に対応する調査ツール(`ping` / `dig` / `curl` / `nc` / `traceroute`)を実際に使い分ける総まとめの章です。**第1章で地図の枠を持ち、各章でマスを1つずつ埋め、第9章で地図を完成させる** ―― これが Phase 3 の全体設計です。

## 8. この章のまとめ

- **ネットワークは「層(レイヤ)」の積み重ね**でできている。郵便と同じで、上の層は下の運び方を知らず、下の層は上の中身を知らない。**決まった接点だけでやりとりする**。
- 層に分ける嬉しさは3つ:**①各層が自分の仕事に集中できる ②ある層を取り替えても他が壊れない(=抽象化。Phase 2 と同じ) ③問題を層ごとに切り分けられる(実務で最重要)**。
- **`socket` は「ネットワーク版の I/O」**。Phase 2 のファイル read/write が、遠くの相手に向いただけ。`HttpClient` はその生のソケットを隠している。**Phase 3 は Phase 2 の直接の続き**。
- データは **パケット**という小分けで運ばれる。順番がずれたり失われたりする問題は、第3章 TCP が解決する。
- 層のモデルは **TCP/IP(4層)** を軸に。下ほど物理、上ほどアプリ。あなたの HTTP は一番上。**丸暗記より「隣の層としか話さない」感覚**。
- 各層は **カプセル化(封筒の入れ子)** で重なる。送信で包み、受信で開ける。
- **実務の武器**:「繋がらない」を **DNS → TCP → TLS → HTTP** と層ごとに切り分ける地図。これが第9章で完成する。

## 次章の予告

**第2章 IP とルーティング ― 「住所」で届く仕組み**

第1章で地図を広げました。次は地図の下の方、**「そもそも、どうやって世界のどこかのマシンに届くのか?」** ―― IP アドレスという「住所」と、バケツリレーで届く「ルーティング」の話に入ります。

第2章では、`ping` や `traceroute` という**手を動かすツール**が初登場します。「自分のパケットが、実際に世界のどこを経由して相手に届いているか」を、自分の目で覗きます。ここから Phase 3 は一気に楽しくなります。

そして第2章で押さえる大事な感覚が **「IP は "届く保証" をしない(ベストエフォート)」**。この不便さを知っておくと、第3章 TCP の「取りこぼさない・順番を揃える」ありがたみが、体で分かります。
