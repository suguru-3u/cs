## 1. 生のソケット ― 第1章の伏線を、今度は本当に書く

第1章で「`socket` はネットワーク版の I/O」と学び、コードは眺めるだけでした。今、実際に書きます。登場人物は2つ:

- **`ServerSocket`**: サーバ側。「接続を待ち受ける」ための口。
- **`Socket`**: クライアント側(と、サーバが受け付けた1本1本の接続)。読み書きする取っ手。

### エコーサーバ ― 送られた文字をそのまま返すだけ

「エコー(こだま)サーバ」は、送られてきた行をそのまま返すだけの、通信の Hello World です。**以下は実際に走らせた本物の Kotlin コード**です。

```kotlin
import java.net.ServerSocket
import java.net.Socket
import kotlin.concurrent.thread

fun main() {
    // ① エコーサーバを別スレッドで起動(送られてきた行をそのまま返す)
    val server = ServerSocket(0)          // 0 = 空きポートを自動割当
    val port = server.localPort
    thread(isDaemon = true) {
        val client = server.accept()      // ← クライアントの接続を待つ(ブロッキング)
        client.getInputStream().bufferedReader().use { reader ->
            client.getOutputStream().bufferedWriter().use { writer ->
                reader.forEachLine { line ->
                    writer.write("echo: $line\n"); writer.flush()
                }
            }
        }
    }

    // ② クライアントとして、その自作サーバに接続して喋る(第1章の生ソケット)
    Socket("localhost", port).use { socket ->
        val writer = socket.getOutputStream().bufferedWriter()
        val reader = socket.getInputStream().bufferedReader()
        for (msg in listOf("hello", "ネットワーク", "7割理解でOK")) {
            writer.write("$msg\n"); writer.flush()      // write = 送信
            val reply = reader.readLine()                // read = 受信
            println("[client] 送信: \"$msg\"  ←→  受信: \"$reply\"")
        }
    }
}
```

**実行結果(本物)**:

```
[client] 送信: "hello"        ←→  受信: "echo: hello"
[client] 送信: "ネットワーク"   ←→  受信: "echo: ネットワーク"
[client] 送信: "7割理解でOK"    ←→  受信: "echo: 7割理解でOK"
```

**通信が成立しました。** 第3章で `nc` を使って手打ちした「HTTP は TCP 上のテキスト」を、今度は**自分のコードで、サーバもクライアントも両方**書いたわけです。

### ここに Phase 2 の全部が詰まっている

このコードの各行に注目してください。**Phase 2 で学んだものばかり**です。

- `ServerSocket(0)` / `Socket(...)` ―― ネットワーク版の「ファイルを開く」(**第1章:socket = システムコール**)。
- `getInputStream()` / `getOutputStream()` への読み書き ―― **Phase 2 の I/O・read/write そのもの**。相手がファイルからネットワークに変わっただけ。
- `bufferedReader()` / `bufferedWriter()` ―― **Phase 2 第9章のバッファI/O**(実測でバッファありが約11倍速だった、あれ)。1文字ずつではなくまとめて運ぶ。
- `.use { }` ―― ソケットは**閉じるべき資源**(fd と同じ)。閉じ忘れると "ソケットリーク"。
- `accept()` と `readLine()` ―― **ブロッキング**する呼び出し。次のセクションの主役。

## 2. ブロッキング I/O ― Phase 2 第9章との再会

上のコードの `accept()` と `readLine()` には、Phase 2 第9章で学んだ重要な性質があります。**ブロッキング**です。

> **ブロッキング**: その呼び出しは、**結果が返るまでスレッドを止めて待つ**。
> - `accept()` は、**クライアントが接続してくるまで**、そのスレッドを止めて待つ。
> - `readLine()` は、**相手がデータを送ってくるまで**、そのスレッドを止めて待つ。

Phase 2 第9章で「ブロッキング vs 非同期 I/O」を学び、「ネットワークは階層の最遠端=一番遅い I/O」と知りました。**その "遅い I/O で待つ" が、今まさにこの `accept()`/`readLine()` で起きています。**

### 問題:1つのスレッドでは、1クライアントしかさばけない

上のエコーサーバは、`accept()` を1回だけ呼んでいました。もし**複数のクライアント**を相手にしたければ、`accept()` をループで回します。でも、ここに罠があります。

```kotlin
while (true) {
    val client = server.accept()   // 接続を1つ受ける
    handle(client)                 // ← この処理が終わるまで、次の accept に進めない!
}
```

`handle(client)` が **ブロッキングで時間がかかる間**(相手からの入力待ち、DB 呼び出しの待ち、など)、**サーバは次の `accept()` に進めません**。つまり **クライアントAをさばいている間、クライアントBは接続すらできず待たされる**。

これは実際に測ると、はっきり分かります。**20クライアントが同時に来て、各リクエストの処理に150msかかる**場合を、1スレッドで逐次にさばくと(本物の計測):

```
・逐次サーバ(1スレッド): 3077ms   … 20件が順番待ちで直列に(150ms × 20 ≒ 3秒)
```

20人が列に並んで、1人ずつしか処理されない。**これでは実用になりません。** どうするか? ―― 並行処理です。

## 3. 1接続1スレッド ― と、スレッドの限界(Phase 2 総動員)

一番素直な解決策は、**接続ごとに新しいスレッドを立てる**ことです。「受け付けたら、その処理は別スレッドに任せて、本体はすぐ次の `accept()` に戻る」。

```kotlin
while (true) {
    val client = server.accept()
    thread { handle(client) }      // ← 接続ごとに別スレッドで処理。本体はすぐ次へ
}
```

**Phase 2 第7章のスレッド**の出番です。これで各クライアントが**並行に**さばかれ、Bさんが Aさんの完了を待つ必要がなくなります。さっきの20クライアントを、**接続ごとにスレッド**でさばくと(本物の計測):

```
・逐次サーバ(1スレッド)          : 3077ms   … 直列
・並行サーバ(接続ごとに軽量スレッド): 184ms    … 20件が同時進行
→ 約 16 倍速い(待ち時間が重なるから)
```

**16倍。** なぜこんなに速くなるのか? ―― 各リクエストの150msは大半が**待ち時間**(I/O待ち)。**待ち時間は重ねられる**。20人分の「待ち」を同時に進めれば、全体はほぼ1人分の150msで済む。第3章で「レイテンシは待ち」と学んだ、その待ちを**並行で重ねる**のが並行サーバの本質です。

### でも、スレッドは重い ― 実測、31,820本で限界

「じゃあ接続ごとにスレッドを立てまくればいい」―― ここで **Phase 2 第7章の実測**が効いてきます。**スレッドは重い資源**でした。Phase 2 で「約3.2万本が上限」と測った、あれです。この教科書でも**実際に、OSスレッドを作れるだけ作ってみました**(本物の計測):

```
作成したOSスレッド数: 31820
  → 限界! 31820 本目で OutOfMemoryError が発生
     (これ以上OSスレッドは作れない = スレッドは重い資源)
```

**31,820本**。Phase 2 の「約3.2万本」が、ネットワークの文脈でそのまま再現されました。1本のOSスレッドは約1MBのスタックメモリを持つので、数万本でメモリを食い尽くすのです。

### これが意味すること ― 数万接続で詰まる

「1接続1スレッド」だと、**同時接続がスレッド上限(約3.2万)を超えた瞬間、新しい接続を受けられなくなる**。数万〜数十万の同時接続をさばく必要があるサーバ(チャット、通知、ゲームなど)では、この方式は破綻します。これは古くから **C10K問題**(1万接続をどうさばくか)として知られる、ネットワークプログラミングの古典的な壁です。

> **Phase 2 第8章の注意も回収**: 接続ごとのスレッドが**共有された状態**(カウンタ、キャッシュ、コレクションなど)を触ると、**競合状態**が起きます。Phase 2 で「Coroutines でも共有可変状態は競合する」と学んだ通り。サーバのハンドラで共有変数を更新するなら、ロックや並行安全なデータ構造が要る。**並行にすると速いが、共有状態には注意** ―― これも Phase 2 の知識がそのまま効く場面です(演習4で確かめます)。

## 4. なぜ軽量スレッド/非同期がネットワークで嬉しいか ― Phase 2 の核心を回収

ここが Phase 2 と Phase 3 の**合流点**、この章のクライマックスです。

### 気づき:ブロッキング中のスレッドは「何もせず待っているだけ」

もう一度、ネットワークのハンドラが時間を何に使っているか考えてください。RTT の往復待ち、ハンドシェイクの往復(第3・6章)、遅いクライアントからの入力待ち、DB や別の API への呼び出し待ち ―― **大半が「待ち」**です。

そして、**待っている間、その重いOSスレッドは何もしていません**。ただ「返事はまだかな」と、高価な資源(OSスレッド)を占有したまま突っ立っている。**3.2万本しか作れない貴重なスレッドを、"待ち" のために遊ばせている** ―― これがブロッキング1接続1スレッドの根本的なムダです。

### 解決:待つときは、OSスレッドを「手放す」

Phase 2 第9章の核心 ―― 「**非同期/Coroutines がI/Oで嬉しいのは、待ちでスレッドを手放すから**」―― が、ここで完全につながります。

> **軽量スレッド(Coroutines / 仮想スレッド)**は、I/O で待ちに入るとき、**下で動いているOSスレッドを手放す**。手放されたOSスレッドは、その間、**別の接続の処理**に使える。だから、**少数のOSスレッドで、何万・何十万もの "待っている接続" をさばける**。

「待ち」がタダ同然になるので、接続の数だけ軽量スレッドを作れます。**実際に、仮想スレッド(Java 21 の軽量スレッド)を50万本作って、全部同時に走らせてみました**(本物の計測):

```
仮想スレッド 500000 本を、全部同時に走らせて完了: 3694ms
(OSスレッドは3.2万で限界。仮想スレッドは50万本でも余裕)
```

**OSスレッドは31,820本で限界。仮想スレッドは50万本でも余裕**(しかも3.7秒で完走)。この**桁違いの差**が、「なぜ非同期/軽量スレッドがネットワークで嬉しいか」の答えです。ネットワークは待ちだらけだからこそ、待ちを軽く扱える仕組みが劇的に効く。

### コードでは、ほとんど一行の違い

嬉しいことに、JVM 21 では**仮想スレッドへの切り替えは、ほぼ一行**です。

```kotlin
// 重いOSスレッド(約3.2万で限界)
thread { handle(client) }

// 軽量な仮想スレッド(数十万でも余裕)。待ちでOSスレッドを手放す
Thread.ofVirtual().start { handle(client) }
```

`handle` の中身(ブロッキングな read/write の書き方)は**そのまま**でいい。**「ブロッキングで素直に書いたコードが、軽量スレッド上では自動でスケールする」** ―― これが仮想スレッドの狙いです。

> **Coroutines との関係**: あなたが Phase 2 で学んだ **Kotlin Coroutines** も、まったく同じ発想です。`suspend` する関数が I/O 待ちに入ると、スレッドを手放して別の仕事に回す。書き方はこうなります(概念コード):
> ```kotlin
> // Coroutines 版のイメージ(Phase 2 で学んだ軽量スレッド)
> launch { handle(client) }   // 数十万の接続を少数のスレッドでさばける
> ```
> **Coroutines も仮想スレッドも、"待ちでスレッドを手放して、少数のOSスレッドで大量の待ちをさばく" という同じ核**を持ちます。Kotlin ではどちらも使えます(Coroutines は kotlinx ライブラリ、仮想スレッドは JVM 標準)。**大事なのは書き方の違いより、"待ちを軽く扱う" という考え方**。それは Phase 2 で既に手に入れていたものです。

## 5. 実務での意味 ― あなたのフレームワークは、全部これをやっている

「エコーサーバなんて実務で書かないよ」と思うかもしれません。でも ―― **あなたが毎日使う Web フレームワーク(Spring、Ktor など)や Web サーバは、この章のコードを、まさに内部でやっています**。

- クライアントの接続を `accept()` で受け(この章)、
- 1本1本の接続を**スレッドや Coroutine に割り当てて並行にさばき**(§3・§4)、
- あなたが書いた「コントローラ/ハンドラ」を、その上で呼んでいる。

**あなたはこれまで、フレームワークが隠していた `accept()` の上で仕事をしていた**のです。第1章で「`HttpClient` は生のソケットを隠している」と学びましたが、サーバ側も同じ。フレームワークは、この章の面倒な部分を隠してくれる**層**でした(また抽象化!)。

### スレッドプールと「サーバが固まる」現象 ― 第8章への直結

実務のフレームワークは、無限にスレッドを作りはしません。**Phase 2 第7章のスレッドプール**を使います ―― あらかじめ決まった本数のスレッドを用意し、リクエストを割り当てて使い回す。3.2万の上限を踏まえた、現実的な設計です。

でも、これが**実務の重大トラブル**を生みます。

> **スレッドプール枯渇(thread pool exhaustion)**: プールの全スレッドが、**遅い下流の呼び出し(DBやAPI)で待たされてブロック**すると、**新しいリクエストを受けるスレッドが1本も残らない**。結果、サーバが「固まった」「全リクエストがタイムアウトする」状態に。実は下流の1箇所が遅いだけなのに、サーバ全体が巻き添えで停止する。

この「全スレッドが待ちで埋まって固まる」は、§4 の「ブロッキング中のスレッドは何もせず待つ」の**最悪の帰結**です。だから対策として、**「いつまでも待たない」= タイムアウト**が決定的に重要になります。**それが次章・第8章の主題**です。「なぜタイムアウトを設定するのか」の答えが、ここでもう見えていますね ―― タイムアウトが無いと、遅い1箇所がスレッドプールを食い尽くし、サーバ全体を道連れにするからです。

## 6. まとめコード ― 動く並行エコーサーバ(Kotlin)

この章の到達点として、**複数クライアントを軽量スレッドで並行にさばくエコーサーバ**の骨格を置いておきます(これも実測で動かしたものの整理版)。あなたの手元でも動かせます。

```kotlin
import java.net.ServerSocket
import java.net.Socket

fun handle(client: Socket) {
    client.use { s ->
        val reader = s.getInputStream().bufferedReader()
        val writer = s.getOutputStream().bufferedWriter()
        reader.forEachLine { line ->
            writer.write("echo: $line\n"); writer.flush()
        }
    }
}

fun main() {
    val server = ServerSocket(9000)
    println("listening on 9000")
    while (true) {
        val client = server.accept()             // ブロッキングで接続を待つ(§2)
        Thread.ofVirtual().start { handle(client) }  // 接続ごとに軽量スレッド(§4)
    }
}
```

これを起動して、別ターミナルから `nc localhost 9000`(第3章!)で繋いで文字を打つと、echo が返ってきます。**第3章の `nc` が、今度はあなたのサーバのクライアントになる** ―― 章が円環でつながりました。`Thread.ofVirtual()` のおかげで、何万接続来ても(スレッド上限に阻まれず)さばけます。

### 演習1(逐次 vs 並行の見積もり)
各リクエストの処理に 200ms かかるサーバに、50クライアントが同時接続しました。(a) 1スレッドの逐次サーバ (b) 接続ごとにスレッドを立てる並行サーバ で、全部さばき終わるまでの概算時間は?(処理時間の大半は I/O 待ちとする)

<details><summary>解答・解説</summary>

- (a) **逐次**: 50 × 200ms = **約10秒**。1件ずつ順番待ち(§2 の実測 3077ms が 20件だったのと同じ理屈)。
- (b) **並行**: 待ち時間が重なるので、**約200ms**(ほぼ1件分)。50件の "待ち" を同時に進められる(§3 の実測、20件が184msだったのと同じ)。

ポイントは「**I/O 待ちは重ねられる**」。処理が CPU をぶん回す計算主体なら重ねても速くならない(コア数が上限)が、ネットワークのように**待ち主体**なら並行が劇的に効く。第3章「レイテンシ=待ち」とつながる。
</details>

### 演習2(スレッドの限界)
「接続ごとにOSスレッドを立てる」方式で、**10万の同時接続**をさばこうとすると何が起きますか? どう解決しますか?

<details><summary>解答・解説</summary>

OSスレッドは**約3.2万本が上限**(実測31,820本)なので、10万接続には**まったく足りず**、`OutOfMemoryError` で新規接続を受けられなくなる(§3、C10K問題)。

解決:**軽量スレッド(仮想スレッド or Coroutines)**を使う。待ちでOSスレッドを手放すので、少数のOSスレッドで数十万の接続をさばける(§4、仮想スレッド50万本の実測)。コード上は `thread {}` を `Thread.ofVirtual().start {}` に変える程度。「重いOSスレッドを接続数だけ作る」発想から「待ちを軽く扱う」発想への転換が答え。
</details>

### 演習3(なぜ軽量スレッドが効くのか)
「軽量スレッドは、なぜOSスレッドより桁違いに多く作れるのか」を、"待ち" という言葉を使って説明してください。

<details><summary>解答・解説</summary>

ネットワーク処理の大半は **I/O 待ち**(相手の応答待ち)。ブロッキングなOSスレッドは、待っている間もそのOSスレッドを**占有したまま遊ばせる**ので、待っている接続の数だけ高価なOSスレッドが必要になり、3.2万で頭打ち。

軽量スレッドは、**待ちに入るとき下のOSスレッドを手放す**(§4)。手放されたOSスレッドは別の接続に使い回せるので、**少数のOSスレッドで大量の "待っている接続" を面倒みられる**。「待ちの間、高価な資源を握らない」から、桁違いにスケールする。これは Phase 2 第9章「非同期がI/Oで嬉しい=待ちでスレッドを手放す」の言い換えそのもの。
</details>

### 演習4(競合状態)★Phase 2 第8章の回収
次の並行エコーサーバは「これまでに受けた総接続数」を数えようとしています。バグはどこ? どう直す?
```kotlin
var totalConnections = 0
while (true) {
    val client = server.accept()
    Thread.ofVirtual().start {
        totalConnections++          // ← ここ
        handle(client)
    }
}
```

<details><summary>解答・解説</summary>

`totalConnections++` を**複数の軽量スレッドが同時に**実行するので、**競合状態**(Phase 2 第8章)。`++` は「読む→+1→書く」の3手順で、間に他スレッドが割り込むと**カウントを取りこぼす**。1万接続来ても、値が9千いくつになったりする。

直し方:並行安全なカウンタを使う。例:`AtomicInteger` にして `totalConnections.incrementAndGet()`、またはロックで保護する。**「並行にすると速いが、共有可変状態は競合する」**という Phase 2 の教訓が、そのままサーバ実装に効く。「軽量スレッドだから安全」ではない(Phase 2 で「Coroutines でも共有可変状態は競合する」と学んだ通り)点に注意。
</details>

## 8. 【実務接続】「サーバが固まる」の正体

第7章の実務での効き所。あなたがフレームワークで作る API サーバのトラブルは、この章の理解で見通せます。

- **「負荷が上がるとサーバが応答しなくなる」**:スレッドプール枯渇(§5)。遅い下流(DB・外部API)で全スレッドがブロックし、新規リクエストを受けられない。→ 下流のタイムアウトを短く(第8章)、遅いクエリを直す、非同期化する。
- **「一部のAPIが遅いだけなのに、全体が巻き添えで遅くなる」**:同上。1箇所の待ちがプールを食い尽くす。→ 隔離(バルクヘッド)やタイムアウト(第8章)。
- **「非同期(Coroutine)なのに速くならない/固まる」**:**Coroutine の中で、ブロッキングな処理を呼んでいる**という典型バグ。軽量スレッドの利点は「待ちで手放す」ことなのに、手放さないブロッキング呼び出しを混ぜると、その利点が消えて、むしろ少数のスレッドを塞いで悪化する。→ ブロッキング呼び出しは専用のディスパッチャ/スレッドに逃がす。
- **同時接続が多いサービス**(チャット・通知・ストリーミング):1接続1OSスレッドは3.2万で破綻(§3)。→ 軽量スレッド/非同期が必須。

**共通するのは「待ちをどう扱うか」**。ブロッキングで素朴に書くと待ちがスレッドを塞ぐ。軽量スレッド/非同期は待ちを手放す。そして「**いつまでも待たない = タイムアウト**」で待ちの暴走を止める ―― それが次章です。

## 9. この章のまとめ

- **生のソケット**:`ServerSocket`(待ち受け)と `Socket`(接続)。`getInputStream/getOutputStream` への read/write は **Phase 2 の I/O そのもの**。第1章の「socket = ネットワーク I/O」を実際に書いた。
- **ブロッキング I/O**(Phase 2 第9章):`accept()`/`readLine()` は結果が返るまでスレッドを止めて待つ。**1スレッドでは1クライアントずつ**しかさばけない(実測:20件を逐次3077ms)。
- **1接続1スレッド**で並行化すると劇的に速い(実測:20件を並行184ms、約16倍。待ちが重なるから)。だが **スレッドは重く、実測31,820本で限界**(Phase 2 の3.2万を再現)。数万接続で破綻(C10K)。
- **軽量スレッド(仮想スレッド/Coroutines)**は、**待ちでOSスレッドを手放す**(Phase 2 第9章の核心)ので、少数のOSスレッドで大量の待ちをさばける(実測:仮想スレッド50万本が3.7秒)。コードはほぼ一行の違い。
- **共有可変状態は競合する**(Phase 2 第8章):並行サーバで共有カウンタ等を触るなら、`AtomicInteger` やロックが必要。
- **実務**:フレームワークは内部でこれ(accept + 並行処理)をやっている。**スレッドプール枯渇**=遅い下流が全スレッドを塞ぎサーバが固まる → **タイムアウト(第8章)**が対策。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] `ServerSocket`/`Socket` で簡単なエコーサーバ/クライアントの流れを説明できる
- [ ] ブロッキングとは何か、1スレッドだと1クライアントずつになる理由を言える
- [ ] 1接続1スレッドが速い理由(待ちが重なる)と、限界(スレッド約3.2万)を言える
- [ ] 軽量スレッドが桁違いにスケールする理由を「待ちでスレッドを手放す」で説明できる
- [ ] 並行サーバで共有可変状態が競合すること(Phase 2 第8章)を思い出せる

**全部完璧でなくて大丈夫**。コードを一字一句書けなくてもいい。「**ソケットは read/write / ブロッキングとスレッドの限界 / 待ちを手放す軽量スレッド**」の3つが体でつながっていれば、この章は大成功です。Phase 2 の実測が、ネットワークで生きたのを実感できたなら何よりです。

## 次章の予告

**第8章 信頼性・タイムアウト・コネクションプール ― 落ちる前提で作る**

この章で何度も出た「**いつまでも待たない = タイムアウト**」、そして第2章から積み上げてきた「**接続はタダじゃない(TCP + TLS の往復コスト)**」―― この2つの伏線を、次章で本題として回収します。

第8章のテーマは **「ネットワークは失敗する前提で作る」**。遅い・返ってこない・途中で切れる ―― それが普通の世界で、どう守るか。**タイムアウト**(接続用と応答用、なぜ分ける?)、**リトライ**(第5章のべき等性!POSTを再送していい?)、**コネクションプール**(第3・6章のハンドシェイクを使い回す)、そして **サーキットブレーカー**(壊れた下流から手を引く)。あなたが実務で困っている「APIがたまにタイムアウトする」に、正面から効く章です。
