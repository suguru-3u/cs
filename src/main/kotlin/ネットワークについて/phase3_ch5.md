# Phase 3・第5章 HTTP ― Web の会話のルール

> **この章の位置づけ**: 実務直結度 **MAX** の章です。あなたがフレームワークで毎日書いている `GET` / `POST`、ステータスコード(200・404・500)、ヘッダ、Cookie ―― その正体を正面から扱います。「なんとなく使っていたもの」を「分かって使えるもの」に変えます。
>
> ここまでの積み上げが効きます。DNS で名前を IP に変え(第4章)、その IP:ポートに TCP で繋ぎ(第3章)、第3章の `nc` で「HTTP は TCP 上を流れるただのテキスト」だと体感しました。**この章は、そのテキストの文法を読み解く章**です。
>
> 手を動かすパートは **`curl -v`**。本物の HTTP のやりとりを丸ごと覗きます。
>
> **7割理解で OK**。ステータスコードやヘッダを全部暗記する必要はありません。掴んでほしいのは **①HTTP はリクエスト→レスポンスのテキスト往復 ②ステータスコードの族(2xx/4xx/5xx)の読み分け ③ステートレスと Cookie** の3点です。
>
> **目安: 4〜5時間**

## 0. 前章までの続き ― 地図のどこにいるか

TCP/IP の4層、今回はアプリケーション層の本命 **HTTP** です。

| 層 | 担当 | 状況 |
|---|---|---|
| **④ アプリケーション層** | **会話の中身 ← HTTP はここの主役** | **← 今ここ(第5章)** |
| ③ トランスポート層 | TCP / UDP(信頼できるパイプ) | 第3章 ✅ |
| ② インターネット層 | IP(住所) | 第2章 ✅ |
| ① リンク層 | 電気・電波 | (雰囲気) |

これまでの章で、通信の**下ごしらえ**が全部そろいました:

1. `example.com` を IP に変換(**DNS / 第4章**)
2. その IP の 443番ポートに TCP で接続(**第3章**)

**では、繋がった後、実際に何を喚るのか?** ―― それが **HTTP** です。下の層が「信頼できるパイプ」を用意してくれたので(第3章 TCP)、HTTP はその上で **安心してテキストの会話**をすればよい。「取りこぼしや順序」は TCP が面倒を見てくれるので、HTTP は**会話の中身だけ**に集中できます。これも第1章の「各層は自分の仕事に集中」の実例です。

## 1. 身近な例 ― 窓口でのやりとり(必ず客が先に頼む)

HTTP は **クライアント(あなたのブラウザやアプリ)とサーバの、決まった形式の会話ルール(プロトコル)**です。イメージは **役所や飲食店の窓口**です。

```
客(クライアント):「これください」(決まった言い方で頼む)  ← リクエスト
店(サーバ)   :「はい、どうぞ」or「売り切れです」    ← レスポンス
```

HTTP の大事な性質が、この窓口のたとえに全部入っています。

- **必ずクライアントが先に頼む**。サーバから勝手に話しかけてはこない(客が頼まないのに料理は出てこない)。
- **1つの頼み(リクエスト)に、1つの返事(レスポンス)**が対応する。
- **頼み方・返し方に決まった型がある**。「これください」に相当する定型フォーマットがある。

この「クライアントが要求 → サーバが応答」の1往復が、HTTP のすべての基本単位です。あなたが `client.send(request)`(第1章)を呼ぶたびに、この1往復が起きています。

## 2. HTTP はテキストの往復 ― リクエストとレスポンスの構造

第3章で `nc` を使い、`GET / HTTP/1.1 ...` と手打ちしたら返事が返ってきました。あれが証明する通り、**HTTP は人間にも読めるテキスト**(HTTP/1.1 まで)です。その文法を解剖します。

### リクエスト(クライアント → サーバ)の型

```
GET /users/42 HTTP/1.1          ← ① リクエスト行:「メソッド パス バージョン」
Host: api.example.com           ┐
Accept: application/json        ├─ ② ヘッダ(メタ情報。何行でも)
Authorization: Bearer xxxxx     ┘
                                ← ③ 空行(ヘッダの終わりの合図)
{ "name": "Tanaka" }            ← ④ ボディ(POST/PUT など。GET は普通なし)
```

### レスポンス(サーバ → クライアント)の型

```
HTTP/1.1 200 OK                 ← ① ステータス行:「バージョン コード 理由」
Content-Type: application/json  ┐
Content-Length: 22              ├─ ② ヘッダ
                                ┘
                                ← ③ 空行
{ "id": 42, "name": "..." }     ← ④ ボディ(中身の本体)
```

**リクエストもレスポンスも、構造は同じ「1行目 + ヘッダ群 + 空行 + ボディ」**です。この型さえ頭に入れば、あとは各パーツ(メソッド・ステータス・ヘッダ)を理解するだけ。次のセクションから1つずつ見ます。

> **Phase 2 との接続**: 「空行でヘッダの終わりを示す」に注目。第3章で `\r\n\r\n`(空行)を手打ちしたのを覚えていますか? これは「ここまでがヘッダ、この先がボディ」という**区切り**です。Phase 2 の I/O で「どこまでが1つの塊か」を区切る話と同じ発想。テキストのプロトコルは、こういう**区切りの取り決め**でできています。

## 3. メソッド ― 「何をしたいか」

リクエスト行の先頭の **メソッド**(HTTP 動詞)は、「そのリソースに対して何をしたいか」を表します。

| メソッド | 意味 | 例 |
|---|---|---|
| **GET** | 取得する(読むだけ) | 記事を表示、一覧を取得 |
| **POST** | 送る/新しく作る | フォーム送信、新規登録 |
| **PUT** | まるごと置き換える | プロフィール全体を更新 |
| **PATCH** | 一部だけ更新する | 名前だけ変更 |
| **DELETE** | 削除する | 投稿を消す |
| HEAD | GET のヘッダだけ欲しい(本体不要) | 更新確認・サイズ確認 |

### 安全(safe)と べき等(idempotent) ― 再送との深い関係

2つの重要な性質があります。実務の「リトライ(再試行)」に直結します。

- **安全(safe)**: 何度呼んでも**サーバの状態を変えない**。GET は安全(読むだけ)。
- **べき等(idempotent)**: 何度呼んでも**結果が同じ**。GET・PUT・DELETE はべき等(同じ DELETE を2回やっても「消えている」状態は同じ)。

**POST はべき等ではありません**。同じ POST を2回送ると、**注文が2件・登録が2回**できてしまう。

> **第3章との接続 ―― これが超重要**: 第3章で「TCP は取りこぼしを再送する」と学びました。では、**アプリのレベルでリクエストが失敗したとき、自動でリトライしていいのか?** ―― **GET なら安心してリトライできる**(何度取得しても副作用なし)。でも **POST を安易にリトライすると、二重注文の事故**になる。「タイムアウトしたからもう1回送ろう」が、決済 API では二重課金を生む。**メソッドの安全性・べき等性を知っていると、"どのリクエストならリトライしていいか" が判断できる**。これは第8章のリトライ戦略の土台です。「なんとなく GET/POST を使い分ける」から「再送安全性まで考えて使い分ける」へ。

## 4. ステータスコード ― 「結果はどうだったか」

レスポンスの1行目にある **3桁の数字**が、結果を表します。**百の位で "族" が決まる**ので、まず族で覚えるのが 7割理解の近道です。

| 族 | 意味 | 代表例 |
|---|---|---|
| **2xx** | **成功** | 200 OK、201 Created(作成成功)、204 No Content(成功だが本体なし) |
| **3xx** | **リダイレクト**(別の場所へ) | 301 恒久移動、302 一時移動、**304 Not Modified**(キャッシュ有効) |
| **4xx** | **クライアント側の間違い** | 400 不正、401 未認証、403 禁止、**404 Not Found**、429 Too Many Requests |
| **5xx** | **サーバ側の異常** | **500 Internal Server Error**、502 Bad Gateway、503 Service Unavailable、504 Gateway Timeout |

### 【実務で最重要】4xx と 5xx を読み分ける

これがこの章で一番実務に効く1点です。

> **4xx = "あなた(クライアント)の頼み方"が悪い**(404=そんなものない、401=ログインしてない、403=権限ない、400=リクエストが不正)。
> **5xx = "サーバ"の側で異常が起きた**(あなたの頼み方は正しかったかもしれないのに、サーバが処理できなかった)。

トラブルのとき、この読み分けで **どっちを直すべきか**が一瞬で分かります。**4xx が返るなら、まず自分のリクエスト(URL・パラメータ・認証ヘッダ)を見直す**。**5xx が返るなら、自分ではなくサーバ側・その先の問題**を疑う(サーバのログを見る、運営に問い合わせる)。

### 502 / 504 は "下の層" の匂いがする

5xx の中でも **502 Bad Gateway** と **504 Gateway Timeout** は特別で、第2〜3章とつながります。

- **504 Gateway Timeout**: 手前のサーバ(ロードバランサやプロキシ)が、**その奥のサーバからの返事を待ちきれずタイムアウト**した。第3章で見た「返事が来ない=timeout」が、HTTP のステータスとして表面化したもの。**奥のサーバが重い/固まっている**サイン。
- **502 Bad Gateway**: 手前のサーバが奥に繋ごうとしたら、**奥が変な応答・接続拒否**を返した。第3章の `Connection refused` 的な事態が、ゲートウェイ越しに 502 として見えている。

> **実務のコツ**: 「500 が返る」→ アプリのコードのバグを疑う。「502/504 が返る」→ **その奥のサーバとの "接続や応答待ち"** を疑う(重い処理、ダウン、タイムアウト設定)。ステータスコードは、**どの層で何が起きたかのヒント**なのです。第1章の "層で切り分ける" が、コードの数字にも表れています。

### 本物のレート制限:429 と 403 の実例

実際に `api.github.com` を叩いたら、**403** が返り、こんなヘッダが付いていました(本物):

```
HTTP/2 403
x-ratelimit-limit: 60
x-ratelimit-remaining: 0        ← 残り回数 0!
x-ratelimit-used: 60
x-ratelimit-reset: 1784762365   ← この時刻まで待てば回復
```

**「一定時間に叩ける回数」を使い切った**ため拒否されたのです(認証なしだと 60回/時)。実務で外部 API を叩いていて急に 403/429 が出たら、**レート制限**をまず疑う。そして `x-ratelimit-reset` を見れば「いつ回復するか」まで分かる。**ヘッダは宝の山**です(次のセクション)。

## 5. ヘッダ ― メタ情報の宝庫

ヘッダは「本体(ボディ)以外の付帯情報」を運びます。実際に `pypi.org` を `curl -v` で叩いた**本物のレスポンスヘッダ**を見てみましょう(一部抜粋):

```
< HTTP/2 200
< content-type: text/html; charset=UTF-8     ← 中身の種類(HTML なのか JSON なのか)
< content-length: 22848                       ← 中身のバイト数
< server: gunicorn                            ← サーバの正体(隠すことも多い)
< etag: "OojgFzMJgKAn55csMa/I6Q"              ← 中身の "指紋"(キャッシュに使う。§7)
< x-cache: MISS, HIT, HIT                     ← 途中のキャッシュに当たったか(§7)
< vary: Accept-Encoding, Cookie               ← 何によって中身が変わるか
< strict-transport-security: max-age=...      ← 「今後は必ず HTTPS で来い」(第6章)
```

よく使うヘッダを、送る側(リクエスト)/返す側(レスポンス)で挙げます。

**リクエスト側の主なヘッダ**:
- **Host**: どのサイト宛てか(下記の注目ポイント)。
- **Content-Type**: 送るボディの種類(`application/json` など)。
- **Authorization**: 認証情報(`Bearer <トークン>` など)。
- **Accept**: 受け取りたい形式(`application/json`)。
- **User-Agent**: クライアントの名乗り(`curl/8.5.0` など。実測でも見えました)。

**レスポンス側の主なヘッダ**: `Content-Type` / `Content-Length` / `Set-Cookie`(§6)/ `Cache-Control`・`ETag`(§7)/ `Location`(リダイレクト先)。

### Host ヘッダはなぜ必須か ― 1つの IP に複数のサイト

実測のリクエストにも `> Host: pypi.org` がありました。なぜ必要か? ―― **1台のサーバ(1つの IP)が、複数のサイトをホストしている**からです。IP だけでは「同じサーバ上のどのサイト宛てか」が分からない。だから HTTP は **Host ヘッダで「〇〇.com 宛て」と明示**します。

> **第4章との接続**: 第4章で「1つの名前 → 複数の IP」を見ました。今度は逆に「**1つの IP → 複数の名前(サイト)**」。サーバは Host ヘッダを見て、`a.com` 宛てと `b.com` 宛てを振り分ける(バーチャルホスト)。名前と IP は多対多の関係で、その最後の交通整理を Host ヘッダがやっているわけです。

## 6. ステートレス と Cookie ― HTTP は「忘れっぽい」

HTTP の重要な性質:**ステートレス(状態を持たない)**。

> **各リクエストは独立していて、サーバはリクエスト間で "前回のこと" を覚えていない。**

窓口のたとえで言うと、**毎回、記憶を失った店員**と話すようなもの。さっき名乗ったのに、次の頼みでは「どちら様?」と聞かれる。1回目のリクエストと2回目のリクエストの間に、サーバ側には何のつながりもない。

すると困ります。**ログインはどうやって維持するの?** 1回ログインしても、次のリクエストでサーバが忘れているなら、ページを開くたびにログインし直し…?

### Cookie ― 「これを次も持ってきてね」の札

これを解決するのが **Cookie** です。仕組みはシンプル:

```
① ログイン成功時、サーバがレスポンスで札を渡す:
   Set-Cookie: session=abc123           ← 「この札を持っていて」

② ブラウザは以降、毎回リクエストにその札を付ける:
   Cookie: session=abc123               ← 「さっきもらった札です」

③ サーバは札を見て「あ、abc123 さんね」と、誰だか思い出す
```

**忘れっぽいサーバに、クライアントが毎回 "身分証(札)" を提示する**ことで、状態(ログイン状態など)を維持する。これがセッション・ログインの正体です。あなたが毎日使う「ログインしたまま」の裏側は、この Cookie の往復です。

> **Phase 2 / 第2・3章との接続 ―― 同じ設計思想がまた**: 「ステートレスな基盤(HTTP)の上に、状態(Cookie)を後付けで乗せる」。これは第3章の「保証しない IP の上に、TCP が信頼性を乗せる」、第2章の「シンプルな IP に徹して、必要な機能は上の層で足す」とまったく同じパターンです。**基盤はシンプルに保ち、必要な機能を層で足す** ―― この設計思想が、ネットワークのあらゆる場所に繰り返し現れます。ステートレスにしておくと、サーバを何台にも増やしやすい(どのサーバがどのリクエストを受けても、札さえあれば処理できる)という**拡張性**の利点があり、それを取るためにあえて「忘れっぽく」設計しているのです。

## 7. キャッシュ関連ヘッダ ― 「変わってないなら、送り直さない」

Web が速い理由の大きな部分が **HTTP キャッシュ**です。一度取ったものを再利用し、無駄な再取得を避けます。

さっきの実測に `x-cache: MISS, HIT, HIT` がありました。これは **途中のキャッシュサーバ(CDN)に当たったか**を示しています。`HIT` は「キャッシュから返した(奥まで取りに行かなかった)」の意味 ―― だから速い。

### ETag と 304 Not Modified ― 「指紋」で無駄を省く

実測の `etag: "OojgFzMJgKAn55csMa/I6Q"` は、**中身の "指紋"** です。仕組み:

```
① 初回:サーバが中身と一緒に指紋を返す
   ETag: "abc"

② 次回:クライアントは「前と同じ指紋なら、送り直さなくていいよ」と聞く
   If-None-Match: "abc"

③ 中身が変わっていなければ、サーバは本体を送らず、こう返すだけ:
   HTTP/1.1 304 Not Modified          ← 「変わってないよ。手元のを使って」
```

**本体(重いかもしれない)を送らずに済む**ので、通信量も時間も節約できる。§4 で出た **304** の正体がこれです。「変わってないなら、まるごと送り直さない」という賢さ。

> **第4章 / Phase 2 との接続 ―― キャッシュ三度目**: DNS の TTL(第4章)、CPU/メモリのキャッシュ(Phase 2 第5章)、そして HTTP キャッシュ。**「一度取ったものを再利用し、遅い取得を避ける」**という同じ知恵が、また現れました。そして DNS の TTL と同じく、HTTP キャッシュにも `Cache-Control: max-age=...`(何秒キャッシュしていいか)があります。**"速さと鮮度のトレードオフ" は、あらゆるキャッシュに共通の緊張関係**。第4章で "It's always DNS" と言いましたが、"It's always cache"(古いキャッシュが原因)も実務あるあるです。「更新したのに反映されない」の犯人は、DNS キャッシュか HTTP キャッシュか、たいていどちらか。

## 8. HTTP/1.1 vs 2 vs 3 ― バージョンの雰囲気(7割で OK)

HTTP にはバージョンがあり、**主に「速さ(レイテンシ)」のために進化**してきました。細部は不要、雰囲気だけ。

- **HTTP/1.1**: 人間に読めるテキスト。1つの接続で **1度に1つのやりとり**しかできず、前のレスポンスを待たないと次に進めない(先頭が詰まると後ろも待つ=**ヘッドオブラインブロッキング**)。**Keep-Alive** で接続を使い回す工夫が入った(第8章)。
- **HTTP/2**: テキストをやめて**バイナリ**に。1つの接続で**複数のやりとりを同時並行**(多重化)。実測の pypi.org も `using HTTP/2` でした。1つの TCP 接続を、詰まらせずに効率よく使う。
- **HTTP/3**: なんと **TCP をやめて UDP の上**(QUIC)。第3章の「HTTP/3 は UDP の上」という予告の回収です。TCP のハンドシェイクや head-of-line 問題を、UDP + 自前の仕組みで乗り越える。

> **共通するテーマ**: どのバージョンアップも、根っこは **「接続をいかに効率よく使い回し、往復(レイテンシ)を減らすか」**。第3章で「ハンドシェイクは往復コスト」と学んだ、あの問題への継続的な戦いです。**この "接続を使い回して速くする" が、第8章(コネクションプール・Keep-Alive)の主題**になります。伏線がまた1本、第8章へ伸びました。

## 9. 【手を動かす】curl -v で本物の HTTP を覗く

HTTP を調べる最強の道具が **`curl`** です。特に **`-v`(verbose)** を付けると、**送ったリクエストと受け取ったレスポンスが丸ごと**見えます。**あなたの手元のターミナル**で叩いてみてください。

> 下記は、この教科書のために **実際に `curl -v https://pypi.org/` を実行した本物の出力**(抜粋)です。

### `-v` の読み方:`>` が送信、`<` が受信

```
> GET / HTTP/2                 ┐
> Host: pypi.org               ├─ あなたが送ったリクエスト(§2 の型そのもの)
> User-Agent: curl/8.5.0       │
> Accept: */*                  ┘

< HTTP/2 200                   ┐
< content-type: text/html; charset=UTF-8   │
< content-length: 22848        ├─ サーバが返したレスポンス(§2 の型そのもの)
< etag: "OojgFzMJgKAn55csMa/I6Q"           │
< x-cache: MISS, HIT, HIT      ┘
```

**`>` はあなたが送った行、`<` はサーバが返した行**。§2 で学んだ「1行目 + ヘッダ + 空行 + ボディ」の型が、実物でそのまま見えます。**教科書の図が、本物の通信で確認できる** ―― これが手を動かす醍醐味です。

なお `curl -v` の出力の冒頭には、DNS 解決(第4章)や TLS ハンドシェイク(第6章で扱います)の行も出ます。**1本の `curl -v` の中に、第2〜6章の全部が詰まっている**のです。今は `>` と `<`(HTTP の部分)に集中してください。

### よく使う curl オプション

```
$ curl -v URL              # リクエスト・レスポンスを丸ごと表示(一番使う)
$ curl -I URL              # ヘッダだけ欲しい(HEAD 相当)
$ curl -L URL              # リダイレクト(3xx)を自動で追う
$ curl -H "Authorization: Bearer xxx" URL     # ヘッダを付ける
$ curl -X POST -d '{"name":"Tanaka"}' -H "Content-Type: application/json" URL   # POST
$ curl -s -o /dev/null -w "%{http_code}\n" URL   # ステータスコードだけ知りたい
```

### 本物のリダイレクトを見る

`http://github.com`(暗号化なし)を叩いたら、こう返りました(本物):

```

```

**301(恒久移動)で、`https://` へ飛ばされている**。多くのサイトは「http で来ても、https に強制リダイレクトする」設定です。§5 の実測にあった `strict-transport-security`(HSTS)ヘッダも「今後は必ず HTTPS で来い」という指示でした。**なぜそこまで HTTPS にこだわるのか?** ―― それが次章第6章のテーマです。ここが第6章への自然な入口になります。

## 10. Kotlin/JVM 接続 ― 第1章の `HttpClient` が、いま完全に分かる

第1章の冒頭で、こう書きました。「`client.send(...)` の裏で何が起きているか、説明できますか?」と。**もう説明できます。** 第1章のコードに、この章の知識を全部乗せてみましょう。

```kotlin
import java.net.URI
import java.net.http.HttpClient
import java.net.http.HttpRequest
import java.net.http.HttpResponse
import java.time.Duration

fun main() {
    val client = HttpClient.newBuilder()
        .connectTimeout(Duration.ofSeconds(3))   // ← 接続(TCPハンドシェイク,第3章)の待ち上限
        .build()

    val request = HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com/users/42"))
        .header("Accept", "application/json")          // ← §5 のヘッダ
        .header("Authorization", "Bearer xxxxx")       // ← §5 の認証ヘッダ
        .timeout(Duration.ofSeconds(10))               // ← 応答が返るまでの待ち上限(接続とは別物!)
        .GET()                                         // ← §3 のメソッド(POST() 等もある)
        .build()

    val response = client.send(request, HttpResponse.BodyHandlers.ofString())

    println(response.statusCode())                     // ← §4 のステータスコード(200/404/500…)
    println(response.headers().firstValue("content-type").orElse(""))  // ← §5 のヘッダ
    println(response.body())                           // ← §2 のボディ
}
```

**第1章では意味が分からなかった各行が、今は全部読める**はずです。`.GET()` はメソッド(§3)、`statusCode()` はステータス(§4)、`.header(...)` はヘッダ(§5)、`body()` はボディ(§2)。そして裏では DNS(第4章)→ TCP 接続(第3章)→ HTTP 往復(この章)が順に起きている。**第1章の「魔法」が、5章分の知識で「理解」に変わりました。** これがロードマップを積み上げてきたご褒美です。

> **第8章への伏線(タイムアウトは2種類)**: 上のコードには **`connectTimeout`(接続の待ち上限)** と **`.timeout`(応答全体の待ち上限)** の **2つの別々のタイムアウト**があります。第2章の `isReachable`、第3章の `connect(...,3000)` に続き、タイムアウトがまた顔を出しました。「接続待ち」と「応答待ち」は**別の待ち**で、別々に設定する ―― なぜ分けるのか、どう決めるのかは第8章で本格的にやります。

## 11. 【演習】採点・解説つき

**まず自分で考えてから**開いてください。7割で OK です。

### 演習1(ステータスコードの読み分け)★実務頻出
外部 API を叩いたら、それぞれ (a) 404 (b) 401 (c) 500 (d) 504 が返りました。それぞれ「まず何を疑い、どこ(自分/サーバ)を見直す」べきですか?

<details><summary>解答・解説</summary>

- (a) **404**(4xx=クライアント側):**リクエストの URL/パスが間違っている**か、対象が存在しない。まず自分の URL を見直す。
- (b) **401**(4xx):**認証されていない**。トークンが無い/期限切れ/形式ミス。自分の `Authorization` ヘッダを見直す(403 なら「認証はできたが権限が無い」で意味が違う)。
- (c) **500**(5xx=サーバ側):**サーバ内部でエラー**。自分の頼み方は正しいかもしれない。サーバのログや運営を疑う。自分でリクエストをいじっても直らないことが多い。
- (d) **504**(5xx、Gateway Timeout):**手前のサーバが、奥のサーバの返事を待ちきれなかった**。奥の処理が重い/固まっている。第3章の timeout が HTTP に表面化したもの。

**4xx は自分、5xx はサーバ**。この一線を引けるだけで、対応の初動が正しくなります。
</details>

### 演習2(べき等性とリトライ)★第8章の伏線
ネットワークが不安定で、決済 API へのリクエストがタイムアウトしました。「失敗したかも」と思い、同じリクエストをもう1回送りました。これが **GET の場合**と **POST の場合**で、それぞれ何が問題になりえますか?

<details><summary>解答・解説</summary>

- **GET**: 安全(safe)で副作用がないので、**何度リトライしても問題なし**(残高照会を2回しても何も起きない)。
- **POST**: べき等ではないので、**二重課金・二重注文の危険**。1回目が実は成功していて「タイムアウトに見えただけ」の場合、2回目で二重に決済される。

だから、**リトライしていいかはメソッドの性質で決まる**。POST を安全にリトライするには、サーバ側に「同じ処理を2回受けても1回分にする」仕組み(冪等キー)が要る。§3 の安全性・べき等性が、そのままリトライ設計の判断材料になる。第8章のリトライ戦略の核心です。
</details>

### 演習3(ステートレスと Cookie)
「ログインしたのに、ページを移動するたびにログアウトされてしまう」というバグを調査しています。HTTP の性質を踏まえ、どのヘッダの往復を確認しますか?

<details><summary>解答・解説</summary>

HTTP は**ステートレス**なので、ログイン状態は Cookie の往復で維持している(§6)。確認すべきは:
1. ログイン時のレスポンスに **`Set-Cookie`** が返っているか(サーバが札を渡しているか)
2. 以降のリクエストに **`Cookie`** が付いているか(ブラウザが札を持ってきているか)

`curl -v` や ブラウザの開発者ツールで、この2つの往復を見る。札が渡っていない/持ってこられていない(Cookie の属性・ドメイン・有効期限・Secure/SameSite の設定ミスなど)のが典型原因。「ステートレスだから札で状態を運ぶ」と分かっていれば、見る場所が一発で決まります。
</details>

### 演習4(curl -v を読む)
`curl -v` の出力で、`>` で始まる行と `<` で始まる行は、それぞれ何を表しますか? また、レスポンスが `< HTTP/2 200` だが `< x-cache: HIT` だった場合、この応答はどこから返ってきた可能性が高いですか?

<details><summary>解答・解説</summary>

- **`>` = あなた(クライアント)が送ったリクエスト**、**`<` = サーバから受け取ったレスポンス**。
- `x-cache: HIT` は、**奥の本サーバまで行かず、途中のキャッシュ(CDN 等)から返ってきた**可能性が高い(§7)。だから速い。逆に `MISS` なら奥まで取りに行った。

`curl -v` の `>`/`<` を読めると、「自分が何を送り、相手が何を返したか」を1画面で把握でき、トラブル調査が一気に速くなります。
</details>

## 12. 【実務接続】curl -v は、あなたの一番の相棒になる

第5章の実務での効き所をまとめます。

- **ステータスコードの族(2xx/4xx/5xx)で初動を決める**:4xx なら自分のリクエスト、5xx ならサーバ、502/504 なら奥との接続・応答待ちを疑う。
- **`curl -v` で "自分が送ったもの/相手が返したもの" を丸ごと見る**:アプリのログだけでは分からない「実際に飛んだヘッダ」が見える。「フレームワークが余計なヘッダを付けていた」「認証ヘッダが抜けていた」などが一目瞭然。
- **ヘッダを読む癖をつける**:`x-ratelimit-*`(レート制限)、`x-cache`(キャッシュ)、`retry-after`、`location`(リダイレクト先)―― 症状の原因と対処が、ヘッダに書いてあることが多い。
- **「更新が反映されない」→ キャッシュを疑う**:HTTP キャッシュ(ETag/Cache-Control)か DNS キャッシュ(第4章)。`curl` に `-H "Cache-Control: no-cache"` を付けて比べると切り分けられる。

## 13. この章のまとめ

- **HTTP はクライアント→サーバのリクエスト/レスポンスのテキスト往復**。構造は「1行目 + ヘッダ + 空行 + ボディ」。必ずクライアントが先に頼む。
- **メソッド**(GET/POST/PUT/DELETE…)は「何をしたいか」。**安全・べき等**の性質が**リトライしていいか**を決める(GET は安心、POST は二重実行注意)。→ 第8章。
- **ステータスコードは族で読む**:2xx 成功 / 3xx リダイレクト(304=キャッシュ有効) / **4xx=自分の問題** / **5xx=サーバの問題**。502/504 は奥との接続・応答待ちを疑う。
- **ヘッダはメタ情報の宝庫**:Content-Type、Authorization、Host(1つの IP に複数サイト)、x-ratelimit(レート制限)など。症状の答えが書いてあることが多い。
- **HTTP はステートレス**(サーバは前回を覚えない)。**Cookie の札の往復**でログイン状態を維持。「シンプルな基盤 + 上の層で状態を足す」= 第2・3章と同じ設計思想。
- **キャッシュヘッダ**(ETag/304/Cache-Control):変わってなければ送り直さない。DNS・CPU キャッシュと同じ知恵、同じ「速さ vs 鮮度」のトレードオフ。
- **バージョン**(1.1→2→3)は「接続を効率よく使い回してレイテンシを減らす」進化。→ 第8章。
- **手を動かす**:`curl -v`(`>`送信 / `<`受信)。第1章の `HttpClient` の中身が、5章分の知識で完全に読めるようになった。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] HTTP が「リクエスト→レスポンスのテキスト往復(1行目+ヘッダ+空行+ボディ)」だと言える
- [ ] ステータスコードを族で読み、**4xx=自分 / 5xx=サーバ**を言える
- [ ] GET は安全でリトライ可、POST はべき等でないので二重実行注意、と言える
- [ ] HTTP がステートレスで、Cookie の札で状態を維持することを説明できる
- [ ] `curl -v` の `>`/`<` を読める/ヘッダから原因の当たりをつけられる

**全部完璧でなくて大丈夫**。ステータスコードやヘッダは、実務で出会うたびに調べれば自然に増えます。「**リクエスト/レスポンスの型 / 4xx・5xx の読み分け / ステートレスと Cookie**」の3つが体に入っていれば、この章は十分すぎるほど 7割クリアです。

この章の最後、`http://github.com` が **`https://` に強制リダイレクト(301)**されるのを見ました。`strict-transport-security` ヘッダも「必ず HTTPS で来い」と言っていた。**なぜ、そこまで HTTPS(あの "s")にこだわるのか?**
