# Phase 3・第8章 信頼性・タイムアウト・コネクションプール ― 落ちる前提で作る

> **この章の位置づけ**: あなたが引き継ぎ資料で挙げていた実務の悩み **「API がたまにタイムアウトする」** に、正面から効く章です。第2章から積み上げてきた2つの伏線 ―― **「接続はタダじゃない(TCP+TLS の往復コスト)」**(第3・6章)と **「いつまでも待たないためのタイムアウト」**(第7章のスレッドプール枯渇)―― を、本題として回収します。
>
> テーマは **「ネットワークは失敗する前提で作る」**。遅い・返ってこない・途中で切れる ―― それが普通の世界で、どう守るか。**タイムアウト・リトライ・コネクションプール・サーキットブレーカー**を学びます。
>
> この章でも **実測**します。接続の使い回し(プール)がどれだけ効くか、TLS 握手がどれだけ重いか、本物の数字で見せます。
>
> **7割理解で OK**。掴んでほしいのは **①ネットワークは失敗する前提 ②タイムアウトは2種類あり、なぜ必須か ③接続は使い回す(プール)** の3点。
>
> **目安: 4〜5時間**

## 0. 前章の続き ― 2つの宿題を回収する章

第7章の最後に、こう予告しました。

- 「**いつまでも待たない = タイムアウト**」が、スレッドプール枯渇を防ぐ鍵(第7章 §5)。
- 「**接続はタダじゃない**(TCP+TLS の往復コスト)」から、接続は使い回したい(第3・6章)。

この章は、この2つを本題として展開します。そして全体を貫くのは、ソフトウェア設計の重要な心構えです ―― **ネットワーク越しの呼び出しは、ローカルの関数呼び出しとは根本的に違う**。

---

## 1. 大前提 ― ネットワークは失敗する

Phase 2 までのプログラミングでは、関数を呼べば **必ず・すぐに・正しく**返ってきました。`list.add(x)` が「1秒待っても返ってこない」ことはないし、「たまに失敗する」こともない。

**ネットワーク越しの呼び出しは、これがすべて崩れます。**

| | ローカルの関数呼び出し | ネットワーク越しの呼び出し |
|---|---|---|
| 速さ | 一瞬(ナノ〜マイクロ秒) | 遅い(ミリ〜秒。RTT 分。第2章 ping) |
| 必ず返る? | 必ず返る | **返ってこないことがある**(相手が固まる・消える) |
| 失敗する? | ほぼしない | **する**(パケット消失・相手ダウン・経路断) |
| 結果は明確? | 成功か例外か明確 | **曖昧**(「成功したか失敗したか分からない」= タイムアウト) |

一番厄介なのが最後の **「成功したか分からない」** です。リクエストを送って返事が来ないとき、**相手に届かなかったのか、届いて処理されたが返事が消えたのか、区別できない**(第5章のべき等性・リトライの話に直結)。

> **心構えの転換**: ローカルのコードは「動く前提」で書きます。ネットワークのコードは **「失敗する前提」** で書きます。「遅いかもしれない」「返ってこないかもしれない」「二重に届くかもしれない」を、最初から織り込む。この章の技術(タイムアウト・リトライ・プール・サーキットブレーカー)は、**すべて「失敗する前提」への備え**です。「たまにタイムアウトする」は異常ではなく、**ネットワークの通常運転**なのです。

## 2. タイムアウト ― いつまでも待たない

失敗への一番基本の備えが **タイムアウト**です。

タイムアウトが無いと、**返事が来ない呼び出しは、永遠に待ち続けます**。そして第7章で学んだ通り、待っているスレッドは占有されたまま。永遠に待つ呼び出しは、**永遠にスレッドを塞ぐ**。だからタイムアウトは「あれば良い」ではなく **必須**です。

### タイムアウトは2種類ある(第2・3・5章の伏線回収)

第2章の `isReachable(2000)`、第3章の `connect(...,3000)`、第5章の HttpClient で、タイムアウトが何度も顔を出しました。実は、ネットワーク呼び出しには **性質の違う2つのタイムアウト**があります。

```
   ①接続タイムアウト ────→ ②応答(読み取り)タイムアウト ─────→
   (相手に繋がるまで)      (繋がった後、返事が来るまで)

   [DNS→TCP→TLS 握手]  |  [サーバが処理して、レスポンスを返す]
```

- **① 接続タイムアウト(connect timeout)**: **相手に繋がる(ハンドシェイクが完了する)まで**の待ち上限。第3章 TCP + 第6章 TLS の握手の時間。
- **② 応答タイムアウト(read timeout)**: **繋がった後、サーバが返事をよこすまで**の待ち上限。

### なぜ分けるのか ― 失敗の原因が違うから

これが実務で決定的に重要です。**2つは失敗する原因が違うので、切り分けの手がかりになる**。

- **①接続タイムアウトが発動** → **そもそも相手に繋がらない**。原因は経路断・ファイアウォール・相手ダウン・IP 間違い(第2・3章の下の層)。「繋がらない」系。
- **②応答タイムアウトが発動** → **繋がったが、サーバが返事をよこさない**。原因はサーバが重い・処理が固まっている・DB が遅い(第5章の 504 と同じ匂い)。「遅い」系。

**「API がたまにタイムアウトする」を調べるとき、まず "どっちのタイムアウトが発動したか" を見る**。①なら接続の問題(下の層)、②ならサーバ側の遅さ ―― これだけで原因の方向が半分に絞れます。多くの「たまにタイムアウト」は、**②応答タイムアウト**(下流が時々遅い)です。

### どう決めるか

- **①接続タイムアウトは短く**(数秒)。ハンドシェイクは速いはず(後述の実測で TCP+TLS ≒ 数十ms)。数秒待って繋がらないなら、繋がらないと判断していい。
- **②応答タイムアウトは、その処理の期待時間 + 余裕**。ただし **必ず有限**にする。「重い集計だから」と長くしすぎると、遅い1件が長時間スレッドを塞ぐ。RTT(第2章 ping)と、サーバの通常の処理時間を目安に。
- **絶対にやってはいけないのが「タイムアウト無限(デフォルトのまま)」**。多くのライブラリはデフォルトが無限か非常に長いので、**明示的に設定する**のが鉄則。

## 3. なぜ無限待ちが最悪か ― カスケード障害の連鎖

タイムアウトを設定しないと何が起きるか。第7章のスレッドプール枯渇が、最悪の形で現実になります。

```
下流のAPIが1つ、応答しなくなる(固まる)
   ↓
タイムアウト無しなので、呼び出したスレッドは永遠に待つ
   ↓
リクエストが来るたび、また1本スレッドが永遠に待ちに入る
   ↓
プールの全スレッドが「待ち」で埋まる(第7章の枯渇)
   ↓
新しいリクエストを受けるスレッドが0本 → サーバ全体が応答不能
   ↓
【最悪】下流の1箇所が固まっただけで、あなたのサーバ全体がダウン
```

これを **カスケード障害(連鎖的障害)** と呼びます。**タイムアウトは、この連鎖を断ち切る回路**です。タイムアウトがあれば、待っているスレッドは上限時間で解放され、他のリクエストに使える。**「1箇所の遅さを、全体の停止に波及させない」** ―― これがタイムアウトの本質的な役割です。「あって困らない」ではなく、**無いとシステムが死ぬ**。

## 4. リトライ ― もう一度試す、でも慎重に

タイムアウトや失敗が起きたとき、**もう一度試すと成功する**ことがよくあります。ネットワークの失敗の多くは一時的(パケット消失=第2章、瞬間的な過負荷)だからです。これが **リトライ**。

でも、リトライには2つの落とし穴があります。

### 落とし穴①:べき等でない操作をリトライする(第5章の回収)

第5章の演習2で扱った、あの話です。**GET・PUT・DELETE(べき等)は安心してリトライできる**が、**POST(べき等でない)を安易にリトライすると二重実行**(二重課金・二重注文)になる。

> 決済 API に POST してタイムアウト(§1 の「成功したか分からない」)。実は成功していたのに、リトライして **二重課金**。

だから **「リトライしていいか」はメソッドの性質で決まる**。POST を安全にリトライするには、サーバ側に **冪等キー**(同じキーの処理は1回だけ実行する仕組み)を持たせる。第5章の安全性・べき等性の知識が、そのままリトライ設計の判断材料になります。

### 落とし穴②:いっせいリトライで、弱った相手にとどめを刺す

サーバが過負荷で遅くなっているとき、**全クライアントが即座にリトライ**すると、ただでさえ苦しい相手に**さらにリクエストが殺到**し、完全にダウンさせてしまう(**リトライストーム / サンダリングハード**)。復旧しかけた瞬間にまた殺到して、いつまでも立ち直れない。

対策が2つあります。

- **指数バックオフ(exponential backoff)**: リトライの間隔を**だんだん長くする**(例:1秒 → 2秒 → 4秒 → 8秒)。相手に回復する時間を与える。
- **ジッター(jitter)**: バックオフに**ランダムなブレ**を足す。そうしないと、全クライアントが「きっかり2秒後」に**同時に**リトライして、結局殺到する。ランダムにずらして、リトライのタイミングを散らす。

そして **リトライ回数には上限**を設ける(例:最大3回)。無限にリトライしない。

> **まとめると**:リトライは「**べき等な操作だけ / 指数バックオフ + ジッター / 回数上限つき**」で行う。この3点セットが安全なリトライの型です。

## 5. コネクションプールと Keep-Alive ― ハンドシェイクを使い回す(第3・6章の回収)

ここで、第3章・第6章から引っ張ってきた **「接続はタダじゃない」** を、実測で回収します。

### 実測:1リクエストの時間の大半は「握手」

`https://pypi.org` に1回リクエストして、時間の内訳を測りました(本物):

```
DNS解決:      0.006s   ← 名前→IP(第4章)
TCP接続完了:  0.007s   ← ここまでで TCP ハンドシェイク完了(第3章)
TLS握手完了:  0.061s   ← ここまでで TLS ハンドシェイク完了(第6章)★ここが重い
最初のバイト: 0.089s   ← サーバの応答が返り始める
全体:         0.089s
```

**注目は TLS**。`0.061 − 0.007 = 約0.054秒(54ms)`、**全体89ms のうち、TLS 握手だけで54ms** ―― リクエスト時間の **半分以上が握手**です。第6章で「TLS は TCP の上にさらに乗る往復コスト」と学んだ、それが数字で見えました。**データを受け取る前の "準備" が、こんなに重い**のです。

### 実測:接続を使い回すと、どれだけ速いか

では、この握手を**毎回やる**のと、**1回だけやって使い回す**のとで、10リクエストを比べました(本物):

```
(A) 毎回新しい接続(10回とも TCP+TLS 握手): 0.706s
(B) 1つの接続を使い回す(握手は最初の1回だけ): 0.270s
→ 使い回すと 約2.6倍速い(握手を9回分スキップできたから)
```

**握手を省くだけで、2.6倍。** リクエストが増えるほど、この差は開きます。これが **接続を使い回す**動機です。

### Keep-Alive とコネクションプール

- **Keep-Alive**: リクエストが終わっても **TCP+TLS 接続を閉じずに開いたまま**にし、次のリクエストで**再利用**する。握手をスキップできる。
- **コネクションプール**: 確立済みの接続を**何本かプール(在庫)しておき**、リクエストのたびに借りて、使い終わったら返す。毎回作らない。

> **Phase 2 / 第7章との接続 ―― 「高コスト資源の使い回し」パターン再び**: これは第7章の **スレッドプール**とまったく同じ発想です。第3章で予告した通り ―― **「作るのが高コストな資源は、毎回作らず、使い回す」**。スレッド(作るのが重い)も、TCP+TLS 接続(握手が重い)も、同じ知恵で「プール」する。**スレッドプール**と**コネクションプール**は双子です。Phase 2 のスレッドプールを理解していれば、コネクションプールは同じ型として腑に落ちます。

### プールのサイズと、3つ目のタイムアウト

プールには**サイズ**(在庫の上限)があります。ここに罠が。

- **プールが小さすぎる** → 在庫の接続が全部使用中だと、**空くまで待つ**。ここでも待ちが発生(そして「なぜか遅い」の原因に)。
- **プールが大きすぎる** → 接続を持ちすぎて、相手サーバや自分のリソースを圧迫。

そして、「**プールから接続を借りるのを待つ時間**」にも上限(**接続取得タイムアウト**)が要ります。§2 の2つ(接続・応答)に加えて、実は **プール取得タイムアウト**という3つ目もある。「タイムアウトが効かない/なぜか詰まる」の原因が、この見落とされがちな3つ目のことも多いです。

## 6. サーキットブレーカー ― 壊れた相手から手を引く

最後の守りが **サーキットブレーカー**です。

下流のサービスが**明らかに壊れている**(連続で失敗している)とき、**それ以上呼ぶのをやめて、即座に失敗を返す**。名前の通り、電気のブレーカー(異常時に回路を遮断して家を守る)と同じ発想です。

なぜ「呼ぶのをやめる」のか。理由は2つ。

1. **無駄な待ちを避ける**:壊れた相手を呼んでも、どうせタイムアウトまで待たされて失敗する。その待ち時間、スレッドが塞がる(§3)。**呼ぶ前に即失敗**すれば、スレッドを守れる。
2. **弱った相手を休ませる**:呼び続けると、復旧しようとしている相手にとどめを刺す(§4 のリトライストームと同根)。**手を引いて回復の時間を与える**。

動作は3状態です(雰囲気だけで OK):

- **閉(closed)**: 正常。普通に呼ぶ。
- **開(open)**: 連続失敗が閾値を超えたら「開」に。**しばらく呼ばず、即失敗を返す**(相手を休ませる)。
- **半開(half-open)**: 一定時間後、**試しに1回だけ**呼んでみる。成功したら「閉」に戻す(復旧を確認)、まだダメなら「開」に戻す。

> **§4 のリトライとの役割分担**: リトライは「**一時的な**失敗」への対処(すぐ直りそうだから、もう一回)。サーキットブレーカーは「**継続的な**失敗」への対処(明らかに壊れてるから、しばらく呼ばない)。この2つは組み合わせて使います ―― 数回リトライしてもダメが続くなら、ブレーカーが開いて呼び出し自体を止める。

## 7. Kotlin/JVM 接続 ― 全部を1つの呼び出しに

これらを Kotlin の `HttpClient` に落とすと、こうなります。**第5章のコードに、この章の防御を足した**ものです。

```kotlin
import java.net.http.HttpClient
import java.net.http.HttpRequest
import java.time.Duration

// クライアントは "使い回す"(毎回 new しない)。これがコネクションプールを効かせる鍵(§5)
val client = HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(3))      // ①接続タイムアウト(§2)
    .build()

fun fetchUser(id: Int): String {
    val request = HttpRequest.newBuilder()
        .uri(java.net.URI.create("https://api.example.com/users/$id"))
        .timeout(Duration.ofSeconds(5))          // ②応答タイムアウト(§2)。①とは別物!
        .GET()                                    // GET はべき等 → リトライ可(§4)
        .build()
    return client.send(request, java.net.http.HttpResponse.BodyHandlers.ofString()).body()
}
```

ポイントを2つ。

- **`connectTimeout`(①)と `.timeout`(②)は別物**。§2 の2種類のタイムアウトが、ちゃんと別々に設定できる。
- **`client` を使い回している**(関数の外で1回だけ作る)。`HttpClient` は内部でコネクションプールを持つので、**インスタンスを使い回せば Keep-Alive とプールが自動で効きます**。

> **★ 実務で頻発するバグ**: **リクエストのたびに `HttpClient.newHttpClient()` を new する**と、プールが毎回リセットされ、**毎回 TCP+TLS 握手が走ります**(§5 の実測で見た "遅い方")。「なぜか API 呼び出しが全部遅い」の犯人がこれ、というのは本当によくある。**クライアントはアプリで1個作って使い回す**のが鉄則。あなたのコードで `new` の位置を確認してみてください。

> **リトライ・サーキットブレーカーはライブラリで**: JVM 標準にはリトライやサーキットブレーカーの高機能な仕組みは無いので、実務では **Resilience4j** のようなライブラリを使います(指数バックオフ・ジッター・ブレーカーの状態管理を提供)。今は「そういうライブラリがある」で OK。大事なのは **なぜそれが必要か**(§4・§6)を理解していること。

### 演習1(2つのタイムアウトの切り分け)★実務直結
「API がたまにタイムアウトする」。ログを見ると、失敗時は **接続タイムアウトではなく、応答タイムアウト**が発動していました。原因はどちら側(自分/経路/サーバ)にありそうで、次に何を調べますか?

<details><summary>解答・解説</summary>

**応答タイムアウトの発動 = 繋がってはいるが、サーバが返事をよこすのが遅い**(§2)。つまり経路や接続(下の層)は正常で、**問題はサーバ側の遅さ**。第5章の 504 と同じ構図。

次に調べること:サーバ側の処理時間・DB クエリの遅さ・下流のさらに別の API の遅延・サーバの過負荷(その時間帯だけ遅い?)。「たまに」なら、特定の重いリクエストや、負荷が高い時間帯を疑う。**"どっちのタイムアウトが出たか" を最初に見る**ことで、経路調査(ping/traceroute)に時間を使わず、いきなりサーバ側に絞れた ―― これがこの章の実務価値です。

もし逆に**接続タイムアウト**だったら、経路・ファイアウォール・相手ダウンを疑い、ping/nc(第2・3章)で切り分ける。
</details>

### 演習2(リトライの安全性)
次のうち、自動リトライして安全なのはどれ? 危険なのはどれ? 理由も。
(a) `GET /orders/123`(注文照会) (b) `POST /orders`(注文作成) (c) `DELETE /orders/123`(注文削除)

<details><summary>解答・解説</summary>

- (a) **安全**。GET は安全(副作用なし)。何度照会しても何も起きない。
- (b) **危険**。POST はべき等でない。リトライで**注文が二重に作られる**恐れ(§4)。安全にするには冪等キーが要る。
- (c) **安全**(べき等)。同じ削除を2回やっても「123 は消えている」状態は同じ。2回目は「もう無い」だけ。

第5章のべき等性が、リトライ設計に直結。**「べき等な操作だけ自動リトライ」**が原則。POST の自動リトライは、冪等キーなしにやってはいけない。
</details>

### 演習3(バックオフとジッター)
1000台のクライアントが、あるサーバへの呼び出しに失敗しました。全台が「1秒後にリトライ」と設定されています。何が起きますか? どう直しますか?

<details><summary>解答・解説</summary>

全1000台が **きっかり1秒後に同時にリトライ**するので、サーバに**1000リクエストが一瞬で殺到**(リトライストーム/サンダリングハード、§4)。復旧しかけたサーバが、この波でまた潰れる。

直し方:**指数バックオフ**(間隔を1→2→4秒と延ばし、相手に回復時間を与える)+ **ジッター**(各台のリトライ時刻にランダムなブレを足し、同時殺到を散らす)。「1〜1.5秒のランダム後にリトライ」のように散らせば、波が平らになる。**バックオフだけではダメで、ジッターで "同時" を崩す**のが肝。
</details>

### 演習4(コネクションプールと実測)★実務頻出
あるチームの API 呼び出しが「1件ごとに約90msかかる、内90%が同じ相手」という状態でした。コードを見ると、呼び出しのたびに新しい HTTP クライアントを生成していました。何が起きていて、どう直すと速くなりますか(§5 の実測を根拠に)?

<details><summary>解答・解説</summary>

クライアントを毎回生成しているので、**毎リクエストで TCP+TLS 握手が走っている**(§7 のバグ)。§5 の実測の通り、TLS 握手だけで約54ms ―― 90msの大半が握手。同じ相手なら、この握手は**使い回せば省ける**。

直し方:**クライアントを1個作って使い回す**(Keep-Alive/コネクションプールを効かせる)。§5 の実測では、握手を使い回すと 0.71s → 0.27s(**約2.6倍**)に。同じ相手への連続呼び出しなら、これだけで劇的に速くなる。「なぜか外部 API が全部遅い」の典型的な原因と対処。**"高コストな握手を使い回す" = スレッドプールと同じ発想**(§5)。
</details>

## 9. 【実務接続】「API がたまにタイムアウトする」を切り分ける

あなたが引き継ぎ資料で挙げていた、まさにその悩みへの実践的な切り分けです。この章の道具を順に当てます。

1. **どちらのタイムアウトが発動している?**(§2)
   - **接続タイムアウト** → 経路・ファイアウォール・相手ダウンを疑う。`ping`(第2章)・`nc`(第3章)で切り分け。
   - **応答タイムアウト** → サーバ側の遅さ(重いクエリ・過負荷・さらに下流)。多くの「たまに」はこれ。
2. **「たまに」の条件は?**
   - **負荷が高い時間帯だけ** → サーバ過負荷 or 自分側のスレッドプール/コネクションプール枯渇(§3・§5)。プールが小さくて接続待ちが起きていないか。
   - **特定のリクエストだけ** → その処理(重い集計・大きいデータ)が遅い。
   - **完全にランダム** → ネットワークの一時的な揺らぎ(パケット消失=第2章)。リトライ(§4)が効く場面。
3. **クライアントを使い回しているか?**(§5・§7)
   - 毎回 new していないか。していれば毎回握手で遅い&プールが効かない。
4. **DNS が時々遅くないか?**(第4章)
   - `dig` で名前解決の時間を見る。DNS の遅延が「たまに遅い」の隠れた原因のことも。

**対策の型**:適切な**タイムアウト2種**を明示設定 → **クライアント使い回し**でプールを効かせる → **べき等な呼び出しはバックオフ+ジッターでリトライ** → 下流が頻繁に壊れるなら**サーキットブレーカー** → 根本の**遅い下流を直す**。

> これらを、次章・第9章で `ping`/`dig`/`curl`/`nc`/`openssl` の**実際の調査フロー**として束ねます。この章で「何を疑うか(仮説)」を、第9章で「どう確かめるか(実測)」につなげます。

## 10. この章のまとめ

- **ネットワークは失敗する前提**で作る。ローカル呼び出しと違い、遅い・返らない・二重・「成功したか分からない」が普通に起きる。
- **タイムアウトは必須**。**①接続タイムアウト(繋がるまで)と ②応答タイムアウト(返事まで)は別物**で、どちらが発動したかが原因切り分けの手がかり(①=経路/接続、②=サーバの遅さ)。**無限待ちは厳禁**。
- 無限待ちは **カスケード障害**を招く(遅い1箇所が全スレッドを塞ぎ、サーバ全体停止)。タイムアウトはその連鎖を断つ回路。
- **リトライ**は「**べき等な操作だけ / 指数バックオフ + ジッター / 回数上限**」で。POST の安易なリトライは二重実行(第5章)。いっせいリトライは相手にとどめ(リトライストーム)。
- **コネクションプール / Keep-Alive**:接続確立の大半は握手(実測:TLS だけで約54ms、全体の半分超)。使い回すと速い(実測:10件で 0.71s→0.27s、約2.6倍)。**スレッドプールと同じ「高コスト資源の使い回し」**。**クライアントは使い回す**(毎回 new しないのが鉄則)。
- **サーキットブレーカー**:明らかに壊れた下流は呼ぶのをやめて即失敗(スレッドを守る+相手を休ませる)。リトライ(一時的失敗)と役割分担。
- **実務**:「たまにタイムアウト」は、まず**どちらのタイムアウトか**→**たまにの条件**→**クライアント使い回しか**→**DNS**の順で切り分ける。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] ネットワーク呼び出しがローカル呼び出しと何が違うか言える(遅い・失敗する・成功不明)
- [ ] タイムアウトが2種類(接続・応答)あり、なぜ分けると切り分けに役立つか言える
- [ ] 無限待ちがカスケード障害を招くこと、タイムアウトが必須な理由を説明できる
- [ ] リトライの3原則(べき等のみ・バックオフ+ジッター・上限)を言える
- [ ] コネクションプール=握手の使い回し、スレッドプールと同じ発想だと言える/クライアント使い回しの重要性

**全部完璧でなくて大丈夫**。特にサーキットブレーカーの状態遷移などは「そういうものがある」で十分。「**失敗する前提 / 2種類のタイムアウト / 接続は使い回す**」の3つが体に入っていれば、あなたの「たまにタイムアウト」問題への見通しは、もう格段に良くなっているはずです。
