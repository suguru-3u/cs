# Phase 3・第6章 HTTPS と TLS ― 盗み見・なりすましを防ぐ

> **この章の位置づけ**: 第2部の締めくくり。前章の最後で `http://github.com` が **`https://` に強制リダイレクト(301)**され、`strict-transport-security` ヘッダが「必ず HTTPS で来い」と言っていました。**なぜ、そこまで HTTPS(あの "s")にこだわるのか?** ―― その答えです。
>
> "s" の正体は **TLS** という暗号化のしくみ。第5章の `curl -v` の冒頭に出ていた「TLS handshake」「Server certificate」の行の意味を、この章で解き明かします。
>
> 手を動かすパートは **`openssl`**。本物の証明書の中身を覗きます。実務では「証明書の期限切れでサイト全体がダウン」「`ERR_CERT_...` エラー」「TLS ハンドシェイク失敗」など HTTPS 特有のトラブルが多く、実務直結度が高い章です。
>
> **7割理解で OK**。暗号の数学的な中身は **Phase 5「セキュリティと暗号」で深掘り**するので、この章は「**HTTPS が何を・どう守っているか**」の雰囲気で十分です。掴んでほしいのは **①HTTPS が防ぐ3つの脅威 ②証明書と認証局のしくみ ③TLS ハンドシェイクもまた往復コスト** の3点。
>
> **目安: 3〜4時間**

## 0. 前章の続き ― 地図のどこにいるか & TLS の居場所

前章で HTTP を学びました。HTTPS は「**HTTP を TLS で包んだもの**」です。層の積み重ねで言うと、TLS は **HTTP と TCP の間**に挟まります。

```
   [ HTTP  ]   ← 会話の中身(第5章)
   [ TLS   ]   ← 暗号化の層(この章)。HTTP を暗号で包む
   [ TCP   ]   ← 信頼できるパイプ(第3章)
   [ IP    ]   ← 住所で届ける(第2章)
```

つまり **HTTPS = HTTP over TLS over TCP**。あなたが `https://` にアクセスすると、TCP で繋いだ後(第3章)、**HTTP を喋る前に TLS で "暗号の握手"** をして、以降の HTTP のやりとりを全部暗号で包みます。第5章で「下ごしらえが全部そろった」と言いましたが、HTTPS では **TCP 接続の後に、もう一段 TLS の下ごしらえ**が入るわけです。

---

## 1. なぜ HTTPS が要るのか ― 暗号化なしの通信は「はがき」

第2章を思い出してください。あなたのパケットは、**たくさんのルータをバケツリレー**で経由して届きました(traceroute で見た、あの経由地点の数々)。カフェの WiFi、プロバイダ、いくつもの中継地点 ―― **その道中のどこでも、通信を覗ける可能性がある**のです。

暗号化なしの HTTP(`http://`)は、**はがき**のようなものです。

> **はがき(平文の HTTP)**: 配達の途中で、郵便局員も、配達員も、盗み見しようと思えば**中身を全部読める**。パスワードもクレジットカード番号も、書いてあれば丸見え。

これでは、ログインもショッピングも危険です。HTTPS(`https://`)は、これを **封をした金庫入りの手紙**に変えます。道中の誰が手にしても、中身は読めない。

### HTTPS が防ぐ「3つの脅威」

HTTPS(TLS)は、次の3つをまとめて防ぎます。**この3点セットが TLS の存在理由**です。

| 脅威 | 何が起きる | TLS の対策 |
|---|---|---|
| **① 盗聴** | 道中で通信を**覗き見**される(パスワード等が漏れる) | **暗号化**(§2)― 覗いても読めない |
| **② 改ざん** | 道中で通信を**書き換え**られる(振込先を差し替え等) | **完全性チェック** ― 書き換えられたら気づく |
| **③ なりすまし** | 銀行だと思ったら**偽サイト**だった | **証明書**(§3)― 本物だと確認できる |

「暗号化」だけだと ①しか防げません。**②改ざん検知と ③なりすまし対策まで含めて初めて安全**になる ―― ここが「暗号化 = HTTPS」と単純化してはいけない理由です。特に **③なりすまし対策(証明書)**は見落とされがちですが、暗号化と同じくらい重要です(偽サイト相手に暗号化して通信しても意味がないので)。

## 2. 暗号化の基本 ― 対称鍵と公開鍵(雰囲気だけ)

暗号の中身は Phase 5 で深掘りするので、ここは **2種類の鍵の "気持ち"** だけ。

### 対称鍵暗号 ― 同じ鍵で施錠・解錠(速いが、鍵の受け渡しが難問)

**同じ鍵で、ロックもアンロックもする**方式。南京錠と、その合鍵のイメージ。速くて効率がいい。でも致命的な問題があります:

> **送る側と受け取る側が、"同じ秘密の鍵" をどうやって共有するのか?** 鍵を相手に送る途中で盗聴されたら(§1 のはがき問題)、その鍵で全部読まれてしまう。**鍵の受け渡し自体が盗聴されうる**のが、対称鍵のジレンマ。

### 公開鍵暗号 ― 鍵がペアになっている(鍵の受け渡し問題を解く)

**2つで1組の鍵**を使います。

- **公開鍵**: 誰に配ってもいい鍵。これで**施錠だけ**できる。
- **秘密鍵**: 自分だけが持つ鍵。これでしか**解錠できない**。

> **たとえ**: **開いた南京錠を配り歩く**イメージ。誰でもあなたの南京錠で箱をロックして送れる。でも、**その箱を開けられるのは、鍵を持つあなただけ**。だから公開鍵は堂々と配ってよく、盗聴されても平気(施錠しかできないので)。**「配ってもいい鍵」があることで、対称鍵の "受け渡し問題" を解ける**。

### TLS の賢い合わせ技

公開鍵暗号は安全ですが、**重い(遅い)**。そこで TLS はこう組み合わせます。

> **① 重いが安全な公開鍵暗号を使って、"共有する対称鍵" を安全に受け渡す。**
> **② 以降の大量のデータは、軽くて速い対称鍵暗号で暗号化する。**

「安全だが遅い方法で、鍵だけ安全に交換 → あとは速い方法で本番」。**適材適所**です。第3章の TCP/UDP の使い分けと同じ、「目的に応じて手段を組み合わせる」発想。この鍵交換のやりとりが、次の TLS ハンドシェイク(§4)の中身です。

> **7割理解ポイント**: 「対称鍵は速いが鍵の受け渡しが難しい / 公開鍵は鍵を配れるが重い / TLS は公開鍵で鍵を渡して対称鍵で本番」―― この筋だけで十分。数学(素因数分解、楕円曲線…)は Phase 5 で。

## 3. 証明書と認証局 ― 「なりすまし」を防ぐしくみ

暗号化(§2)だけでは、実は **③なりすまし**を防げません。考えてみてください:**あなたが暗号化して通信している相手が、そもそも偽物だったら?** 偽サイトの公開鍵で暗号化して、偽サイトに秘密を送っているだけ ―― 暗号化は完璧でも、相手が偽物なら無意味です。

そこで **「この公開鍵は、確かに `pypi.org` のものだ」と保証する仕組み**が要る。それが **証明書**と **認証局(CA)**です。

### 証明書 = 公開鍵の "身分証明書"

**証明書(サーバ証明書)**は、ざっくり言うと:

> 「この公開鍵は、`pypi.org` の持ち主のものです」という**証明書き**を、**信頼できる第三者が署名して**発行したもの。

### 認証局(CA)= 身分を保証する信頼できる第三者

**認証局(CA: Certificate Authority)**は、その "保証人" です(Let's Encrypt、DigiCert など)。CA が「このサイトは本物だと確認した」と**署名**した証明書を、サーバは持っている。

> **たとえ ―― パスポート**: 誰でも「私は田中です」と口では言える(なりすまし)。でも、**国(信頼できる第三者=CA)が発行したパスポート(証明書)**があれば、本物だと信じられる。あなたのブラウザや OS は、**信頼できる国(CA)のリスト**をあらかじめ持っていて(トラストストア)、そのリストにある CA が署名した証明書なら「本物」と認める。

### 信頼の連鎖(チェーン)

第5章の `curl -v` に `Certificate level 0, 1, 2` という行がありました。これは **信頼の連鎖**です。

```
ルート CA(ブラウザが最初から信頼)
   └─署名→ 中間 CA
              └─署名→ サーバ証明書(pypi.org)
```

ブラウザは最上位のルート CA だけを信頼し、そこから連鎖的に「ルートが中間を保証、中間がサーバを保証」とたどって、末端のサーバ証明書を信頼します。**この連鎖のどこかが切れていると(中間証明書が抜けている等)、"信頼できない" と判定される** ―― これが実務の証明書トラブルの一因(§9)。

## 4. TLS ハンドシェイク ― 話す前の「暗号の握手」

いよいよ、`curl -v` の冒頭に出ていた TLS の行の正体です。**HTTP を喋る前に、TLS はそれ専用のハンドシェイク**をします。第3章の TCP ハンドシェイクの**上に、さらに**乗るイメージです。

ざっくりの流れ(TLS 1.3):

```
1. クライアント →「これらの暗号方式に対応してます」   (Client Hello)
2. サーバ     →「これを使いましょう + 私の証明書です」(Server Hello + Certificate)
3. クライアント: 証明書を CA のリストと照合して "本物か" 検証(§3)
4. 双方で、以降使う対称鍵を安全に取り決める(§2 の合わせ技)
5. 「準備OK」                                        (Finished)
   ── 以降、HTTP のやりとりを対称鍵で暗号化して流す ──
```

第5章で実際に叩いた `curl -v https://pypi.org/` の、**本物の TLS 行**がこれです:

```
* TLSv1.3 (OUT), TLS handshake, Client hello (1):        ← 上の 1
* TLSv1.3 (IN),  TLS handshake, Server hello (2):        ← 上の 2
* TLSv1.3 (IN),  TLS handshake, Certificate (11):        ← サーバが証明書を提示
* TLSv1.3 (IN),  TLS handshake, CERT verify (15):
* TLSv1.3 (IN),  TLS handshake, Finished (20):           ← 上の 5
* SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384  ← 合意した暗号方式
*  SSL certificate verify ok.                            ← 証明書の検証に成功(§3)
```

`SSL certificate verify ok.` の一行が、**「相手は本物だと確認できた」**という、なりすまし対策の成功サインです。もしここが失敗すると、ブラウザは警告を出し、通信を止めます。

### 【重要】TLS ハンドシェイクも "往復コスト" ―― 第8章へまた1本

第3章で「TCP の3ウェイハンドシェイクは往復コスト」と学びました。**HTTPS では、その TCP ハンドシェイクの後に、さらに TLS ハンドシェイクの往復が乗ります**。

```
接続確立の全体像(HTTPS):
   TCP ハンドシェイク(第3章)  ≒ 1往復
 + TLS ハンドシェイク(この章) ≒ 1往復(TLS 1.3 の場合。1.2 は約2往復)
 ───────────────────────────
 = データを送る前に、これだけの往復が必要
```

RTT が 80ms なら、接続確立だけで **TCP + TLS = 160ms 以上**。第3章の「接続はタダじゃない」が、HTTPS では**さらに高くつく**わけです。

> **これが第8章の伏線を強化する**: 「接続を1回張るのに TCP + TLS の往復がかかる」なら、**毎回張り直すのは、平文 HTTP 以上にもったいない**。だから接続を使い回す(Keep-Alive / コネクションプール)価値が、HTTPS では一段と大きい。第8章でこの話が本題になります。ちなみに TLS 1.3 は往復を減らし(以前の 1.2 より速い)、再接続時はさらに省略(セッション再開)する工夫もあります ―― これも「往復を減らして速くする」という HTTP/2・HTTP/3 と同じ戦いの一部です。

## 5. TLS のバージョンと "s" の完成

- **SSL**: TLS の古い名前(SSL 2.0/3.0)。**もう使われません**(脆弱性のため廃止)。でも慣習で今も「SSL 証明書」などと呼ばれます。名前は SSL でも、実体は TLS。
- **TLS 1.2**: 長く主流だったバージョン。ハンドシェイクが約2往復。
- **TLS 1.3**: 現在の主流。**より速く(1往復)・より安全**(古い危険な暗号方式を廃止)。実測の pypi.org も TLSv1.3 でした。

**`https://` の "s" の正体は、この TLS のしくみ全体**(暗号化 + 完全性 + 証明書による認証)です。前章で見た `http`→`https` リダイレクトや HSTS ヘッダは、「はがき(平文)ではなく、必ず封をした手紙(TLS)で来い」という要求だったわけです。

## 6. 【手を動かす】openssl で証明書を覗く

証明書や TLS を調べる定番ツールが **`openssl`** です。**あなたの手元のターミナル**で叩いてみてください。

### 実際のサーバの証明書とハンドシェイクを見る:`s_client`

```
$ openssl s_client -connect pypi.org:443 -servername pypi.org </dev/null
```

これで、§4 のハンドシェイクの様子、提示された証明書チェーン、検証結果が表示されます。出力の中の **`Verify return code: 0 (ok)`** が「検証成功」の合図(0 以外なら何か問題あり)。

> ⚠️ 注:企業や学校のネットワーク、一部のセキュリティ製品の裏では、通信を**代理(中継)する装置**が独自の証明書を挟むことがあります。その場合、`issuer`(発行元)に見慣れない名前が出ることも。**あなたの普通の環境では、`issuer` に Let's Encrypt や DigiCert など本物の CA 名**が見えるはずです。

### 証明書ファイルの中身を読む:`x509 -text`

証明書の各フィールドを人間が読める形にします。以下は **この教科書のためにデモ用の証明書を作り、実際に openssl で覗いた本物の出力**です(構造を見るのが目的なので、自作の証明書を使っています):

```
$ openssl x509 -in cert.pem -noout -subject -issuer -dates -ext subjectAltName

subject = CN = example.local, O = Demo Corp        ← ① 誰のための証明書か
issuer  = CN = example.local, O = Demo Corp        ← ② 誰が署名したか(発行元 CA)
notBefore = Jul 22 23:17:09 2026 GMT               ┐
notAfter  = Oct 20 23:17:09 2026 GMT               ┘ ③ 有効期間(ここ超えると期限切れ!)
X509v3 Subject Alternative Name:
    DNS:example.local, DNS:www.example.local        ← ④ この証明書が有効なホスト名
Signature Algorithm: sha256WithRSAEncryption        ← ⑤ 署名の方式
Public-Key: (2048 bit)                              ← ⑥ 公開鍵(§2)
```

**読み方(実務で見る場所)**:
- **subject**: この証明書が「誰(どのドメイン)のものか」。
- **issuer**: 「誰が署名したか」=発行元の CA。**subject と issuer が同じなら "自己署名"**(下記)。
- **notBefore / notAfter**: **有効期間**。この `notAfter` を過ぎると **期限切れ**。実務の証明書トラブル No.1(§9)。
- **Subject Alternative Name (SAN)**: この証明書が有効な**ホスト名の一覧**。ここに載っていないホスト名でアクセスすると "名前が一致しない" エラーになる。

### 「信頼されない証明書」はこう見える(本物のエラー)

上のデモ証明書は **subject と issuer が同じ**でした。これは自分で自分に署名した **自己署名証明書**で、CA の保証がありません。検証すると、こうなります(本物の出力):

```
$ openssl verify cert.pem
CN = example.local, O = Demo Corp
error 18 at 0 depth lookup: self-signed certificate       ← 「自己署名だ」と検知
error cert.pem: verification failed                        ← 検証失敗
```

ブラウザで自己署名や信頼されない証明書のサイトを開くと「この接続は安全ではありません」の警告が出ますが、**その裏でこの検証失敗が起きている**わけです。§3 の「CA の保証がないと信頼されない」が、目に見えました。

### 期限切れを確認する:`-checkend`

```
$ openssl x509 -in cert.pem -noout -checkend 0
Certificate will not expire        ← 今は有効

$ openssl x509 -in cert.pem -noout -checkend 99999999   # 約3年後もまだ有効か?
Certificate will expire            ← その頃には期限切れ(このデモ証明書は90日有効)
```

`-checkend <秒>` は「今から◯秒後に切れているか」を判定します。監視スクリプトで「30日後に切れる証明書」を検知するのに使えます(§9 の実務ネタ)。

## 7. Kotlin/JVM 接続 ― HTTPS は自動、でも JVM 特有の証明書トラブルがある

嬉しいことに、JVM で HTTPS を使うのは簡単です。**第5章の `HttpClient` のコードで、URL を `https://` にするだけ**。TLS ハンドシェイクも証明書検証も、JVM が裏で自動でやってくれます。

```kotlin
// 第5章のコードと同じ。https:// にするだけで、TLS が自動で働く
val request = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users/42"))  // ← s が付いただけ
    .GET()
    .build()
val response = client.send(request, HttpResponse.BodyHandlers.ofString())
```

### ★ JVM 特有の落とし穴:トラストストアと `PKIX path building failed`

ただし、**JVM は自前の「信頼する CA のリスト(トラストストア、`cacerts`)」**を持っていて、OS やブラウザのものとは**別**です。ここが実務の罠になります。

> あなたの Kotlin アプリが、社内 API(社内で立てた独自 CA の証明書)に繋ごうとすると、こんな例外が出る:
> **`SSLHandshakeException: PKIX path building failed ... unable to find valid certification path to requested target`**

これは **「その証明書を署名した CA が、JVM のトラストストアに入っていない」**という意味(§3 の "信頼できる CA のリストにない")。ブラウザでは開けるのに JVM アプリだけ失敗する、というのは、**トラストストアが別物**だから起きます。正しい対処は **社内 CA の証明書を JVM のトラストストアに追加**すること。

> **⚠️ やってはいけない対処**: ネットで「証明書検証を全部無効にする(trust-all)」コードが見つかりますが、**これは §1 の ③なりすまし対策を丸ごと捨てる行為**で、本番では重大なセキュリティ穴です。「検証エラーが出たから検証をオフにする」は、鍵がかからないから鍵を捨てるようなもの。**正しくは、信頼すべき CA を足す**。この判断ができるのが、仕組みを理解している人の強みです。

> **Phase 2 との接続**: 「JVM が独自のトラストストアを持つ」のは、第4章の「JVM が独自の DNS キャッシュを持つ」と同じ構図です。**JVM は OS の上にもう一枚、自分のレイヤを持っている** ―― DNS も証明書も、OS とは別に JVM 内で管理される。この "JVM という層" の存在を知っておくと、「他のツールでは平気なのに JVM アプリだけおかしい」系のトラブルに強くなります。

### 演習1(3つの脅威と対策)
公衆 WiFi のカフェで、暗号化なしの `http://` でログインしました。理論上、悪意ある同席者は何ができてしまいますか? また `https://` なら、それぞれどう防がれますか?

<details><summary>解答・解説</summary>

平文 HTTP(はがき)なので、同じ WiFi の同席者は:
- **①盗聴**:あなたのパスワードやセッション Cookie を**そのまま読める**(§1)。
- **②改ざん**:あなたへのレスポンスを**書き換え**られる(偽の画面を見せる等)。
- **③なりすまし**:偽のログインページに誘導できる。

`https://` なら、①は暗号化で読めなくなり、②は完全性チェックで書き換えを検知、③は証明書で本物のサイトか確認できる(§1 の3点セット)。**"公衆 WiFi で http は危険" の理由が、仕組みで説明できる**。だから今のサイトは軒並み HTTPS 必須(前章の強制リダイレクト)なのです。
</details>

### 演習2(対称鍵と公開鍵)
「安全なのに重い公開鍵暗号」と「軽いのに鍵の受け渡しが難しい対称鍵暗号」。TLS はこの2つをどう組み合わせて、両方の欠点を回避していますか?

<details><summary>解答・解説</summary>

**① まず公開鍵暗号(安全)を使って、"これから使う対称鍵" を安全に受け渡す。② 以降の大量データは対称鍵暗号(高速)で暗号化する**(§2)。

公開鍵の「鍵を配れる」利点で対称鍵の受け渡し問題を解き、対称鍵の「速い」利点で本番データを効率よく守る。**適材適所の合わせ技**。「なぜ2種類あるの?」の答えは「それぞれの欠点を補い合うため」。
</details>

### 演習3(証明書トラブル)★実務頻出
ある朝、自社サービスに繋がらなくなり、ブラウザが「この接続は安全ではありません(`NET::ERR_CERT_DATE_INVALID`)」と表示。サービス自体は動いているようです。何が起きた可能性が高く、どう確認しますか?

<details><summary>解答・解説</summary>

**証明書の期限切れ**の可能性が高い(`CERT_DATE_INVALID`=日付が無効=有効期間外)。§6 の `notAfter` を過ぎた状態。サーバは動いていても、証明書が切れると**ブラウザが接続を拒否**するので "サイト全体がダウン" に見える。

確認:`openssl s_client -connect ドメイン:443` で証明書の日付を見る、または `openssl x509 ... -checkend 0`。対処は**証明書の更新(再発行)**。これは実際に世界中で頻発する障害で、「証明書の自動更新(Let's Encrypt の certbot 等)や、期限監視を仕込んでおく」のが定石(§6 の -checkend が監視に使える)。**"サーバは生きてるのに繋がらない" の代表例が証明書期限切れ**、と知っているだけで初動が段違い。
</details>

### 演習4(JVM の落とし穴)
ブラウザや `curl` では問題なく開ける社内 API に、あなたの Kotlin アプリからアクセスすると `PKIX path building failed` で失敗します。原因と、やってはいけない対処・正しい対処は?

<details><summary>解答・解説</summary>

原因:社内 API の証明書を署名した CA(社内 CA など)が、**JVM のトラストストア(`cacerts`)に入っていない**(§7)。ブラウザ/curl は OS のトラストストアを使うので通るが、JVM は別物なので失敗する。

- **やってはいけない**:証明書検証を無効化(trust-all)。なりすまし対策を捨てる重大な穴。
- **正しい対処**:社内 CA の証明書を JVM のトラストストアに**追加**する(`keytool` 等)。

「JVM は OS とは別に、自分の DNS キャッシュ・トラストストアを持つ」を知っていれば、この手の "アプリだけおかしい" を素早く切り分けられます。
</details>

## 9. 【実務接続】HTTPS のトラブルは "証明書" 絡みが多い

第6章の実務での効き所。HTTPS 特有のトラブルは、**証明書**に起因するものが大半です。

- **証明書の期限切れ**(最頻出):`notAfter` 超過。サーバは生きているのに繋がらない。→ 更新。**自動更新と期限監視**を仕込むのが定石(`-checkend`)。
- **信頼されない CA / 自己署名**(`ERR_CERT_AUTHORITY_INVALID`、`PKIX path building failed`):CA がトラストストアに無い(§3, §7)。→ 正しい CA を足す(検証を切らない)。中間証明書の設定漏れも定番原因。
- **ホスト名の不一致**(`ERR_CERT_COMMON_NAME_INVALID`):証明書の SAN に、アクセスしたホスト名が無い(§6)。→ 証明書を正しいドメインで再発行。
- **時計のズレ**:あなたのマシンの時刻が大幅にずれていると、有効期間内の証明書も「まだ有効でない/期限切れ」と誤判定される。→ マシンの時刻(NTP)を確認。地味だが実際にある。
- **TLS バージョン/暗号方式の不一致**:古すぎるクライアントと新しいサーバ(またはその逆)で、共通の方式が無く**ハンドシェイク失敗**。→ `openssl s_client -tls1_2` 等でバージョンを指定して切り分け。

### 切り分けの地図 ― ④の内側の "TLS 握手" が埋まった

第5章で「④ curl の中の TLS ハンドシェイク部分は第6章で」と予告しました。いま埋まります。

```
① dig(第4章)      → 名前は IP に変換できる?
② ping(第2章)      → その IP まで経路は届く?
③ nc(第3章)        → 目的のポートは開いてる? refused/timeout?
④ curl -v(第5章)   → HTTP のやりとりは? ステータス・ヘッダは?
   └ openssl s_client(この章) → TLS 握手は成功? 証明書は有効?  ← 今ここ完成
```

**「繋がらない」を層で切り分ける地図が、これでほぼ完成**しました。`curl -v` は途中で TLS 握手が失敗すると、そこで止まって理由を教えてくれます(証明書エラーなど)。より詳しく証明書だけ調べたいときは `openssl s_client`。第9章で、この①〜④+TLS を **1つの実践フロー**に束ね、さらに「**遅い**」の調べ方も加えて総仕上げします。

## 10. この章のまとめ ― 第2部 完走

- **HTTPS = HTTP over TLS over TCP**。TLS は HTTP と TCP の間で、HTTP を暗号で包む。
- 平文 HTTP は**はがき**(道中で読める)。HTTPS が防ぐのは **①盗聴(暗号化)②改ざん(完全性)③なりすまし(証明書)** の3点セット。暗号化だけでは不十分で、③証明書が同じくらい大事。
- **暗号の2種類**:対称鍵(速いが鍵の受け渡しが難)/公開鍵(鍵を配れるが重い)。TLS は**公開鍵で対称鍵を渡し、本番は対称鍵**の合わせ技。数学は Phase 5 で。
- **証明書と CA**:証明書は公開鍵の身分証、CA は保証人(パスポートを発行する国)。ブラウザ/OS/JVM は信頼する CA のリスト(トラストストア)を持つ。信頼は連鎖(ルート→中間→サーバ)。
- **TLS ハンドシェイク**は、TCP ハンドシェイクの**上にさらに乗る往復コスト**。HTTPS の接続確立は平文より高くつく → だから接続の使い回し(第8章)が一層効く。
- **手を動かす**:`openssl s_client`(握手と証明書を見る)、`openssl x509 -text`(証明書の中身:subject/issuer/有効期間/SAN)、`verify`・`-checkend`(自己署名検知・期限確認)。
- **JVM の落とし穴**:JVM は独自トラストストアを持つ → `PKIX path building failed`。対処は CA を足す(検証を切らない)。DNS キャッシュに続く "JVM という層" の話。
- **実務**:HTTPS トラブルは証明書絡みが多い(期限切れが最頻出)。切り分け地図の TLS 握手部分が完成。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] 対称鍵と公開鍵の違い、TLS が両者をどう組み合わせるか、ざっくり言える
- [ ] 証明書と CA を「パスポートと発行する国」で説明できる
- [ ] TLS ハンドシェイクが TCP の上にさらに乗る往復コストだと言える(→ 接続使い回しの価値)
