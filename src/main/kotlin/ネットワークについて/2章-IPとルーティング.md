> **この章の位置づけ**: 第1章で広げた地図の、**下の方(インターネット層)**に降りていきます。テーマは **「そもそも、どうやって世界のどこかのマシンにデータが届くのか?」**。答えは **IP アドレス(住所)** と **ルーティング(バケツリレー)** です。

## 0. 前章の続き ― 地図のどこにいるか

第1章で、ネットワークは層の積み重ねだと学びました。TCP/IP の4層はこうでした(下から):

| 層 | 担当 | 学ぶ章 |
|---|---|---|
| ④ アプリケーション層 | 会話の中身(HTTP など) | 第4〜6章 |
| ③ トランスポート層 | 取りこぼさず/速く、相手プロセスへ | 第3章 |
| **② インターネット層** | **どのマシンに届けるか(住所)** | **← 今ここ(第2章)** |

第1章の郵便のたとえで言うと、今日の話は **「宛先の住所を書けば、なぜか届く」** の "住所" と "届くしくみ" の部分です。あなたがフレームワークで `https://example.com` と書くとき、この "example.com" は最終的に **IP アドレスという住所**に変換され(その変換は第4章 DNS)、その住所目がけてパケットが運ばれます。今日はその **「住所目がけて運ばれる」**の中身を開けます。

## 1. 身近な例 ― 住所があるから、届く

友人に手紙が届くのは、封筒に **住所**が書いてあるからです。「東京都〇〇区△△ 1-2-3」。この住所が世界で(ほぼ)一意だから、郵便網はあなたの家を特定できる。

ネットワークも同じで、世界中のマシンには **住所**が割り振られています。それが **IP アドレス** です。あなたのパケットの封筒(第1章のカプセル化を思い出してください)には、**「宛先 IP アドレス」と「送り主 IP アドレス」** が書かれています。宛先を見て、ネットワークはそのマシンへ運ぶ。送り主を見て、相手は返事を送り返せる。

> 郵便の住所と1つだけ違うのは、**IP アドレスは「場所」ではなく「ネットワーク上の位置」に振られる**点です。同じノート PC でも、家の WiFi と会社の LAN では違う IP になります。「引っ越すと住所が変わる」に近い感覚です(この含みは後で NAT の話に効きます)。

## 2. IP アドレスとは ― マシンの住所

### IPv4 ― 32ビットの住所(今も主役)

一番よく見る IP アドレスは、こういう形です。

```
192.168.1.10
203.0.113.42
```

これは **IPv4** という方式で、**32ビット(0/1 が32個)** の数字です。32個の 0/1 だと人間には読めないので、**8ビットずつ4つに区切って、それぞれ 10進数(0〜255)にして、ドットで繋いで**表記します。

```
11000000.10101000.00000001.00001010   ← 本当の姿(32ビット)
   192   .   168  .    1   .   10      ← 人間向けの表記
```

各区切りは 0〜255 の 256通り。だから `256.1.1.1` のような表記は不正です(255 が上限)。

32ビットで表せる住所の総数は **約 43億(2の32乗)個**。……これ、**世界の人口より少ない**んです。スマホ・PC・IoT 機器…と考えると、43億では全然足りません。この「住所が枯渇する」問題への対処が、後述の **プライベート IP + NAT** と、次の **IPv6** です。

### IPv6 ― 128ビットの住所(枯渇対策の本命)

**IPv6** は住所を **128ビット**に拡張した方式です。表記はこんな形(16進数をコロン区切り)。

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

128ビットで表せる数は **約 340澗(かん)個**(3.4×10³⁸)。もはや事実上無限で、「地球上の砂粒1つ1つに住所を振っても余る」とよく例えられます。枯渇の心配はありません。

> **7割理解ポイント**: IPv6 の細かい表記ルール(`::` で 0 を省略できる、等)は **今は覚えなくて OK**。「IPv4 は数が足りないので、その桁数を大幅に増やしたのが IPv6」という **雰囲気だけ**掴めば十分です。実務でも、まだ IPv4 に触る場面の方が多いです。

### Kotlin/JVM で覗く ― `InetAddress`

JVM では、IP アドレスは `java.net.InetAddress` という型で表されます。「名前 → 住所」の変換(第4章 DNS の先取り)もここでできます。

```kotlin
import java.net.InetAddress

fun main() {
    // 名前(ドメイン)を住所(IP)に変換する ← 中で DNS が動いている(第4章)
    val addr = InetAddress.getByName("example.com")
    println(addr.hostAddress)   // 例: 93.184.216.34 のような IPv4

    // 1つの名前に複数の住所が紐づくこともある(冗長化・負荷分散)
    InetAddress.getAllByName("github.com").forEach { println(it.hostAddress) }

    // 自分自身の住所
    println(InetAddress.getLocalHost().hostAddress)  // 例: 192.168.1.10
}
```

> `getByName("example.com")` の1行の中で、実は「名前を住所に変換する問い合わせ(DNS)」が走っています。第1章で見た `HttpClient` の裏側も、まずこの変換から始まっていたわけです。**あなたが書く1行の裏に、いくつもの層が動いている** ―― この感覚を積み重ねていきます。

## 3. どうやって届くのか ― ルーティング(バケツリレー)

住所が分かったとして、**あなたのマシンから相手のマシンまで、1本の線で直結しているわけではありません**。世界中のマシンと1対1で配線するのは不可能です。

では、どう届くのか。**ルータ**という中継機器が、**バケツリレー**で運びます。

### イメージ:郵便の集配

あなたが東京から沖縄の友人に手紙を出すとき、**あなたは沖縄までの全経路を知りません**。近所のポストに入れるだけです。あとは:

```
あなた → 地元の郵便局 → 地域の集配センター → 空港 → 沖縄の集配センター → 相手の郵便局 → 友人
```

**各郵便局は「次にどこへ渡せば、より宛先に近づくか」だけを知っていて、最終目的地までの全経路は誰も知らない**。それでも、各段階が「宛先に近づく方向」へ渡し続ければ、いつか届く。

ネットワークも同じです。あなたのパケットは:

```
あなたのPC → 自宅ルータ → プロバイダのルータ → … → いくつものルータ … → 相手側のルータ → 相手サーバ
```

**各ルータは、パケットの「宛先 IP」を見て、「次にどのルータへ渡せば宛先に近づくか」を判断して、隣へ手渡す**。これを繰り返して届く。この「次にどこへ渡すか」の判断を **ルーティング**、判断の材料表を **ルーティングテーブル**と呼びます。

> **ここが第1章の "層" の威力**: あなた(アプリを書く人)は、この中継の全経路をまったく知らずに済んでいます。「宛先 IP を書いて渡す」だけ。経路選びは下の層(ルータ群)が勝手にやってくれる。上の層は下の複雑さを知らなくていい ―― 第1章の原則そのものです。

### 1パケットずつ、別々の道を通ることもある

第1章で「データはパケットに小分けされて運ばれる」と学びました。ここで少し不思議な事実を足します。**同じ相手に送った複数のパケットが、それぞれ違う経路を通ることがあります**。あるパケットは経路A、別のパケットは(混雑を避けて)経路B、というふうに。

すると当然、こういうことが起きます:

- **後から送ったパケットが、先に着く**(順番が入れ替わる)
- **どこかのルータが混雑していて、パケットが捨てられる**(消える)

「え、それ困るのでは?」―― はい、困ります。そしてそれを解決するのが次章 TCP です。今はまず「**IP のレベルでは、順番も到着も保証されていない**」という事実を受け止めてください。これが次のセクションの核心です。

## 4. 【この章の核心】IP は「届く保証」をしない ― ベストエフォート

第2章で **一番大事な1点**です。ここだけは 7割ではなく、しっかり掴んでください。

> **IP は「ベストエフォート(最善は尽くすが、保証はしない)」で配送する。**
> パケットは **失われることも、順番が入れ替わることも、重複することもある**。IP はそれを直してくれない。

郵便でたとえると、**IP は「普通郵便」**です。ポストに入れれば大抵は届くけれど、

- 配送中に紛失しても、郵便局は教えてくれない
- 「届きましたよ」の確認も返ってこない
- 2通出したら、順番が前後して届くこともある

一方、大事な書類には **「書留(かきとめ)」** を使いますよね。追跡番号があり、受け取りの確認が返り、紛失したら分かる。**この "書留" の役割を、IP の上に乗って提供するのが次章の TCP** です。

```
   [ TCP ]  ← 書留:取りこぼしを検知して再送・順番を揃える(第3章)
   [ IP  ]  ← 普通郵便:とりあえず宛先へ運ぶ。保証はしない(この章)
```

> **なぜ IP はあえて保証しないのか?** 「全部のパケットに確認を取って、失敗したら再送して…」を **ネットワークの根っこ(IP)でやると、遅くて複雑になりすぎる**からです。だから IP は「運ぶことだけ」に徹してシンプルに速く保ち、**「保証が必要な人は上の層(TCP)で足してね」** という役割分担にしている。これも第1章の「各層は自分の仕事に集中する」の実例です。
>
> そして「保証が要らない人(多少パケットが欠けても、とにかく速い方がいい動画やゲーム)」は、TCP を乗せずに **UDP** を使う ―― これも次章の話です。**この章の "ベストエフォート" が、次章 TCP と UDP の分岐の出発点**になります。

## 5. プライベート IP・グローバル IP・NAT

さっき「IPv4 は約43億個しかなく、足りない」と言いました。IPv6 が本命の解決策ですが、**IPv4 のまま数の不足をしのぐ仕組み**が、実務でも毎日関わってくるので押さえます。

### 住所を「内側」と「外側」で分ける

IP アドレスには、**そのまま世界で使う住所(グローバル IP)** と、**ある建物の内側だけで使う住所(プライベート IP)** の区別があります。

プライベート IP には、こういう決まった範囲が予約されています(見覚えありませんか?):

```
10.0.0.0     〜 10.255.255.255
172.16.0.0   〜 172.31.255.255
192.168.0.0  〜 192.168.255.255   ← 自宅ルータでよく見るやつ
```

**プライベート IP は「建物の内側専用」なので、別の建物では同じ番号を使い回せます**。あなたの家の `192.168.1.10` と、隣の家の `192.168.1.10` は、別物として共存できる。世界で一意である必要がないから、43億問題を回避できるわけです。

> **たとえ**: 会社の内線番号と同じです。「内線 105」は各社にあり、社内では通じるけれど、外から直接はかけられない。外からかけるには、まず会社の **代表番号(=グローバル IP)** にかけて、受付が内線へ繋ぐ。

### NAT ― 内側の住所と外側の住所を翻訳する

では、内側(プライベート IP)のマシンが、外(インターネット)と通信するには? ここで **NAT(Network Address Translation、アドレス変換)** が登場します。

自宅のルータは、**1つのグローバル IP** を持っています(プロバイダから借りている)。家の中の全デバイス(PC、スマホ、テレビ…)は、外に出るとき **ルータのグローバル IP に化けて**通信し、返事が来たらルータが **正しい内側のデバイスへ振り分けます**。この「内側の住所 ⇄ 外側の住所」の翻訳が NAT です。

```
[ スマホ 192.168.1.11 ]  ┐
[ PC   192.168.1.10 ]  ├─→ [ 自宅ルータ / NAT ] ──→ インターネット
[ TV   192.168.1.20 ]  ┘        (グローバル IP: 203.0.113.42)
      内側の住所                     この1つの外側住所を全員で共有
```

**実務での "あるある"**:自分の PC でIPを調べると `192.168.x.x`(内側)なのに、「確認くん」的なサイトで見ると全然違う IP(外側=ルータのグローバル IP)が表示される ―― これは NAT のせいです。**あなたの PC がインターネットから見えている住所は、実は PC 自身の住所ではなく、ルータの住所**なのです。

> **7割理解ポイント**: NAT の細かい種類や、外から内側へ繋ぐ「ポート開放」の話は、今は不要です。「**家の中は使い回せる内側住所、外に出るときルータが1つの外側住所に翻訳する**」という骨格だけで十分。この感覚は、後で「なぜサーバは外から繋げるのに、家の PC は外から繋ぎにくいのか」を理解する土台になります。

## 6. 【手を動かす】ping と traceroute で "届く様子" を覗く

いよいよ Phase 3 初の手を動かすパートです。**あなたの手元のターミナル**(Mac / Linux はそのまま、Windows は PowerShell かコマンドプロンプト)で叩いてみてください。

> ⚠️ 下記の出力は **代表的な例**です。**あなたの環境では数値・経路が変わります**(それが正しい)。自分の目で「本当にこうなる」を確かめるのが、この章の狙いです。

### ping ― 「そこに居ますか?」の疎通確認

`ping` は、相手に小さなパケットを送って **「返ってくるか」「往復にどれくらいかかるか」** を測るツールです。ネットワーク疎通確認の一番基本の道具です。

```
$ ping -c 4 github.com
```

出力の例(あなたの環境では数値が変わります):

```
PING github.com (20.27.177.113): 56 data bytes
64 bytes from 20.27.177.113: icmp_seq=0 ttl=115 time=12.3 ms
64 bytes from 20.27.177.113: icmp_seq=1 ttl=115 time=11.8 ms
64 bytes from 20.27.177.113: icmp_seq=2 ttl=115 time=13.1 ms
64 bytes from 20.27.177.113: icmp_seq=3 ttl=115 time=12.0 ms

--- github.com ping statistics ---
4 packets transmitted, 4 packets received, 0.0% packet loss
round-trip min/avg/max = 11.8/12.3/13.1 ms
```

**読み方**:

- `github.com (20.27.177.113)` ―― まず名前が **IP(住所)に変換**された(DNS が動いた証拠。第4章)。
- `time=12.3 ms` ―― **往復にかかった時間(RTT: Round-Trip Time)**。パケットが相手まで行って戻るまで。**これがレイテンシ(遅延)の実感値**です(第8章で深掘り)。近いサーバは数ミリ秒、地球の裏側だと 200ms 超えることも。
- `0.0% packet loss` ―― **パケットの取りこぼし率**。ここが 0% でないと、ネットワークのどこかで欠けている(まさに §4 のベストエフォート)。
- `ttl=115` ―― TTL(Time To Live)。パケットが「あと何ホップ生きられるか」の残り。ルータを1つ通るごとに1減り、0 になると捨てられる(無限ループ防止)。

> **実務での使い方**: 「繋がらない」ときの **第一手**がこれ。ping が通れば「少なくとも住所までは届いている」。ping が全く返らない/loss が高いなら、IP/経路/ファイアウォールを疑う。ping は通るのにアプリが繋がらないなら、問題はもっと上の層(TCP/TLS/HTTP)にある ―― という **切り分けの起点**です(第1章の "層の地図" が早速効きます)。
>
> 注:一部のサーバは ping(ICMP)にわざと返事しない設定なので、「ping が返らない=壊れている」とは限りません。あくまで手がかりの1つです。

### traceroute ― バケツリレーの経路を1ホップずつ見る

`traceroute`(Windows は `tracert`)は、**§3 のバケツリレーを実際に可視化**します。宛先までに **どのルータを何個経由したか**を、1ホップずつ表示します。

```
$ traceroute github.com          （Windows: tracert github.com）
```

出力の例(あなたの環境では経路が全く変わります):

```
traceroute to github.com (20.27.177.113), 30 hops max
 1  192.168.1.1              1.2 ms    ← 自宅ルータ(プライベート IP!)
 2  10.xxx.xxx.1            8.5 ms    ← プロバイダの中の機器
 3  xxx.xxx.xxx.xxx        10.1 ms    ← プロバイダの上位
 4  * * *                            ← 応答しない(見えない)ルータもある
 5  xxx.xxx.xxx.xxx        11.9 ms
 ...
 9  20.27.177.113          12.3 ms    ← 到着(github)
```

**読み方**:

- **1行 = 1ルータ(1ホップ)**。§3 の「バケツリレー」が、そのまま行数になって見えています。
- **1行目が `192.168.1.1`**(プライベート IP)―― これは §5 の **自宅ルータ**。最初の1歩は必ず家のルータ、という §5 の内容が実際に現れています。
- **`* * *`** ―― そのルータが traceroute に返事をしない設定なだけで、**壊れてはいません**。飛ばして次を見ればOK。
- 各行の時間は、そのルータまでの往復時間。**急に時間が跳ね上がる行**があれば、そこが混雑・遠距離の境目です。

> **実務での使い方**: 「途中までは行くが、ある地点から先に進まない」ときに、**どこで止まっているか**が見えます。例えば自宅ルータ(1行目)すら返らないなら家の中の問題、プロバイダの途中で止まるなら回線側の問題、といった当たりがつく。**§3 の理屈が、§6 のツールで目に見える** ―― これが Phase 3 の醍醐味です。

### 演習1(住所の分類)
次の IP アドレスを「プライベート IP」と「グローバル IP」に分けてください。
`10.0.5.3` / `8.8.8.8` / `192.168.0.1` / `203.0.113.10` / `172.20.1.1`

<details><summary>解答・解説</summary>

- **プライベート**: `10.0.5.3`(10.系)、`192.168.0.1`(192.168.系)、`172.20.1.1`(172.16〜31 の範囲内)
- **グローバル**: `8.8.8.8`(Google の公開 DNS。世界で一意)、`203.0.113.10`

判定のコツ:§5 の3つの範囲(`10.*` / `172.16〜31.*` / `192.168.*`)に入っていればプライベート、そうでなければグローバル。`172` は **16〜31 の範囲だけ**がプライベートなので `172.20` は該当(`172.40` ならグローバル)。ここは間違えやすいので、7割で OK です。
</details>

### 演習2(ping の読み取り)
ある ping の結果が `10 packets transmitted, 6 packets received, 40.0% packet loss` でした。何が起きていて、どの層を疑いますか?

<details><summary>解答・解説</summary>

**4割のパケットが取りこぼされています**。まさに §4 の「IP はベストエフォート、パケットは失われる」が現実に起きている状態。原因の候補は、経路上のルータの混雑、電波の弱い WiFi、回線の不安定さなど、**IP/リンク層(下の層)**が怪しい。

実務的には「たまに遅い・たまに失敗する」の温床です。ここで大事なのは、**loss があると、この上に乗る TCP が "取りこぼしを再送" するために余計に時間がかかり、結果としてアプリから見て「遅い」「タイムアウトする」**という形で症状が出る点(第3章・第8章へ繋がる)。
</details>

### 演習3(ベストエフォートの含意)
「IP は順番を保証しない」。では、大きなファイルを送るとき、受け取った側は届いたパケットをどう正しく組み立てればよいでしょう? どんな情報があれば解決できそうですか?(仕組みを知らなくてOK。**設計を想像**してみてください)

<details><summary>解答・解説</summary>

各パケットに **「これは何番目のかけら」という通し番号**が付いていれば、受け取り側は順番がバラバラに届いても、番号順に並べ直して元のファイルに戻せます。また「番号5だけ来ていない」と分かれば、**それだけ再送**を要求できます。

**まさにこれが次章 TCP がやっていること**です(シーケンス番号と再送)。あなたが今「こうすれば解決できそう」と想像した設計を、TCP は実際にやっています。**自分で仕組みを想像してから答え合わせをすると、TCP が『自然な発明』として腑に落ちます**。この演習は第3章の完全な伏線です。
</details>

### 演習4(traceroute の解釈)
traceroute の途中に `* * *` の行が3つ続きましたが、その先の行はちゃんと時間が表示され、最終的に宛先に到達しました。この `* * *` は「障害」でしょうか?

<details><summary>解答・解説</summary>

**障害ではありません**。§6 で触れた通り、`* * *` は「そのルータが traceroute の問い合わせにわざと返事をしない設定」なだけ。その先が正常に表示され、宛先まで届いているなら、パケットは問題なくそのルータを通過しています。**"見えない=壊れている" ではない**、が要点。ツールの出力を鵜呑みにせず、意味を分かって読む練習です。
</details>

## 8. Kotlin/JVM 接続 ― 「疎通確認」をコードで

`ping` 相当のことは JVM からもできます(ただし OS の ping コマンドとは仕組みが少し違い、環境によっては権限の都合で使えないこともあります)。

```kotlin
import java.net.InetAddress

fun main() {
    val host = InetAddress.getByName("github.com")

    // タイムアウト 2000ms で「到達可能か」を確認(ping に近い疎通確認)
    val reachable = host.isReachable(2000)
    println("${host.hostAddress} reachable = $reachable")
}
```

> **Phase 2 との接続**: `isReachable(2000)` の **2000(ミリ秒のタイムアウト)** に注目。Phase 2 の第9章で「ネットワークは階層の最遠端=一番遅い I/O」と学びました。**遅い相手を待つときは "いつまで待つか" を決める必要がある** ―― これが **タイムアウト**の発想の芽です。第8章で「タイムアウトをなぜ・どう設定するか」を本格的にやりますが、その最初の顔がここに出ています。ネットワークのコードには、ほぼ必ずタイムアウトが付いて回ります。

## 9. 【実務接続】この章の知識は「どの場面」で効くか

Phase 3 では毎章、最後に「この知識は "繋がらない・遅い" のどこで効くか」を言語化します。第2章の分:

- **「繋がらない」の切り分けの、下から2番目のマス**を担当します。第1章の地図で言うと、DNS(住所変換)の次、TCP の手前。
- **ping が通る/通らない**で、「少なくとも住所までは届いているか」を最速で判定できる。ping が通ればネットワークの下の方は生きている → 問題は上の層(TCP/TLS/HTTP)に絞れる。
- **traceroute** で、「どこまで行けて、どこから先に進めないか」が見える。自宅ルータで止まる/プロバイダで止まる/相手の直前で止まる、で原因の場所が変わる。
- **packet loss / RTT** は、「たまに失敗する」「なんか遅い」の**下の層の証拠**。loss があれば TCP の再送で遅くなり、RTT が大きければそもそもの往復に時間がかかっている(第8章のレイテンシへ)。
- **NAT の知識**は、「自分の PC の IP と、外から見える IP が違う」「家の PC は外から繋ぎにくい」といった日常の謎を説明してくれる。

これらは第9章「つながらない・遅いの調べ方」で、他の層の道具(`dig`/`curl`/`nc`)と組み合わせて **切り分けフロー**にまとめます。今は「ping と traceroute という2つの道具を、地図のこのマスで使う」と覚えておけば十分です。

## 10. この章のまとめ

- **IP アドレス**は「マシンの住所」。**IPv4** は32ビット(`192.168.1.10` のような表記、約43億個で足りない)、**IPv6** は128ビットで枯渇を解決(雰囲気だけでOK)。
- パケットは **ルーティング(バケツリレー)** で届く。各ルータは「宛先に近づく次の一手」だけを知っていて、全経路は誰も知らない ―― これも第1章の "層で複雑さを隠す" の実例。
- **【最重要】IP はベストエフォート**。パケットは失われ・入れ替わり・重複しうる。IP は "普通郵便"。**"書留" の保証は上の層(TCP)が足す** ―― 次章の出発点。
- **プライベート IP / グローバル IP / NAT**:内側は住所を使い回し、外に出るときルータが1つの外側住所に翻訳する。だから自分の IP と外から見える IP が違う。
- **手を動かす**:`ping`(疎通・RTT・loss)、`traceroute`(経路のバケツリレーを可視化)。§3 の理屈が §6 で目に見えた。
- **JVM 接続**:`InetAddress`(住所への変換・疎通確認)。`isReachable(2000)` の **タイムアウト**が、第8章への芽。

- [ ] ルーティングを「バケツリレー(各ルータは次の一手だけ知る)」で説明できる
