> 手を動かすパートは **`nc`(netcat)**。TCP 接続を自分の手で張って、第1章の「socket = ネットワーク I/O」を、コマンド1つで体感します。

## 0. 前章の続き ― 地図のどこにいるか

TCP/IP の4層、今回は下から3番目の**トランスポート層**です。

| 層 | 担当 | 学ぶ章 |
|---|---|---|
| ④ アプリケーション層 | 会話の中身(HTTP など) | 第4〜6章 |
| **③ トランスポート層** | **取りこぼさず/速く、相手の "プログラム" へ** | **← 今ここ(第3章)** |
| ② インターネット層 | どのマシンに届けるか(住所=IP) | 第2章 ✅ |
| ① リンク層 | 電気・電波で隣へ運ぶ | (雰囲気だけ) |

第2章の IP は **「どのマシンに」** 届けるところまでを担当しました。でも、それだけでは足りないことが2つあります。

1. **マシンには、たくさんのプログラムが動いている**。届いたデータを「どのプログラムに渡すか」が要る。 → **ポート番号**(§1)
2. **IP は取りこぼす**。「取りこぼさない」保証が欲しい場面がある。 → **TCP**(§2〜3)。逆に「速さ優先で保証は要らない」場面もある。 → **UDP**(§4)

この章で、第2章までの「マシンまで届ける」が、「**マシンの中の、目的のプログラムまで、必要な品質で届ける**」に完成します。

## 1. ポート番号 ― 「どのプログラムに届けるか」

第2章で「IP アドレス = マシンの住所」と学びました。でも、1台のマシンでは同時にたくさんのプログラムが動いています ―― Web サーバ、データベース、SSH、メール…。**IP だけでは「どのプログラムに渡すか」が決まりません**。

そこで **ポート番号** が登場します。

> **たとえ**: IP アドレスが **マンションの住所**なら、ポート番号は **部屋番号**です。「〇〇マンション(IP)の 443号室(ポート)」まで指定して、はじめて届け先が1つに決まる。

だから通信相手は、必ず **「IP アドレス : ポート番号」のペア**で指定します。第1章で書いたこのコードを思い出してください。

```kotlin
Socket("example.com", 80)   // ← この 80 がポート番号!
```

この `80` こそポート番号でした。第1章では説明せず流しましたが、いま回収されました。

### よく使うポート(ウェルノウンポート)

ポート番号は 0〜65535 まであります。そのうち **有名なサービスには "定番の番号"** が決まっています。覚える必要はありませんが、実務でよく見るものだけ挙げます。

| ポート | サービス | 備考 |
|---|---|---|
| 80 | HTTP | 暗号化なしの Web(第5章) |
| 443 | HTTPS | 暗号化ありの Web(第6章)。今の主役 |
| 53 | DNS | 名前解決(第4章)。UDP をよく使う |
| 22 | SSH | サーバへのログイン |
| 5432 / 3306 | PostgreSQL / MySQL | DB(Phase 4 で再会) |

> **実務での効き所**: `https://example.com` と書くと、ブラウザは **自動的にポート 443** に繋ぎます(「443 = HTTPS の定番」だから省略できる)。`http://` なら 80。もし変なポートで動くサーバに繋ぐなら `http://example.com:8080` のように **明示**します。「ポートを明示的に指定した URL を見たことがある」なら、それはこの仕組みです。

## 2. TCP ― 信頼できない土台の上に、信頼性を作る

いよいよこの章の心臓部です。第2章の演習3で想像した設計を、TCP がどう実現しているか見ます。

### TCP が提供する「4つの保証」

TCP は、**取りこぼす IP(普通郵便)の上に乗って**、次のような品質を提供します。

- **順序保証**: 送った順にデータが届く(バラバラに届いても TCP が並べ直す)
- **再送(欠落の回復)**: 途中で消えたパケットを検知して、送り直す
- **重複の排除**: 同じデータが二重に届いても1回分にする
- **フロー制御**: 受け手が処理しきれない量を、送り手が一気に送りつけない調整

結果として、TCP を使うアプリ(HTTP など上の層)は、**「相手に write したものは、順番通り・取りこぼしなく相手に届く」** という、まるで信頼できる1本のパイプが繋がっているかのように扱えます。

### どうやって? ― 通し番号(シーケンス番号)と確認応答(ACK)

第2章の演習3の答えそのものです。

1. TCP は送るデータの各かけらに **通し番号(シーケンス番号)** を振る。
2. 受け取った側は「〇番まで受け取ったよ」と **確認応答(ACK)** を返す。
3. 送った側は、一定時間 ACK が返ってこないかけらを **「消えた」と判断して再送**する。
4. 受け取った側は、番号順に **並べ直して**上の層に渡す(バラバラに着いても整列)。

```
   送信側                          受信側
   ── データ #1 ──────────────→   受け取った
   ← ACK #1 まで受領 ────────────   「1番OK」
   ── データ #2 ──────────────→   （途中で消失! 届かない）
   （ACK #2 が来ない…タイムアウト）
   ── データ #2 再送 ─────────→   受け取った
   ← ACK #2 まで受領 ────────────   「2番もOK」
```

> **たとえ**: TCP は **書留(かきとめ)**。追跡番号(シーケンス番号)があり、受領確認(ACK)が返り、届かなければ送り直す。第2章の「IP=普通郵便、TCP=書留」がここで完成します。

### Phase 2 との接続 ― これは「抽象化」そのもの

ここが CS 全体を貫く美しい構造です。

> **信頼できない下位層(IP)の上に、信頼できる層(TCP)を作る。** そして上の層(HTTP)には、下が信頼できないことを **一切見せない**。

これは Phase 2 第1章で学んだ **抽象化**とまったく同じ発想です。Phase 2 では「OS がバラバラなハードウェアの違いを隠し、アプリは統一的に扱える」と学びました。TCP も同じで、**"取りこぼす現実(IP)" を隠し、上の層には "信頼できるパイプ" という理想を見せる**。

**下の層の欠点を、上の層に見せないように覆い隠す** ―― この「覆い隠し」こそが抽象化の正体です。ネットワークでも OS でも、同じパターンが繰り返されている。これに気づくと、CS の別々に見えた分野が地続きになります。

## 3. 3ウェイハンドシェイク ― 話す前に「もしもし」を3回

TCP は、データを送り始める前に、まず **接続を確立**します。いきなり話し出すのではなく、お互いに「聞こえてますか?」を確認し合う。これが **3ウェイハンドシェイク**です。

### 電話のたとえ

電話をかけたとき、いきなり用件を話し出さず、こうしますよね。

```
あなた:「もしもし」                    → 相手に「話しかけてますよ」    (SYN)
相手 :「はい、もしもし、聞こえます」    → 「聞こえてる、そっちは?」    (SYN + ACK)
あなた:「はい、聞こえます。では…」      → 「こちらも聞こえた、話すよ」  (ACK)
```

3回のやりとりで、**双方が「送れる」「受け取れる」を相互に確認**してから、本題に入る。TCP もこれと同じ3ステップ(SYN → SYN-ACK → ACK)で接続を確立します。フラグ名(SYN/ACK)は覚えなくて OK。**「話す前に往復して確認する」感覚**だけ掴んでください。

### 【重要】ハンドシェイクは "往復のコスト" ―― これが第8章の伏線

ここが実務的に超重要です。**データを1バイトも送る前に、この往復(約1.5往復)が必要**です。つまり **接続確立にはレイテンシ(遅延)のコストがかかる**。

第2章の `ping` で見た **RTT(往復時間)**を思い出してください。仮に相手までの RTT が **100ms** なら、ハンドシェイクだけで **約150ms**、最初のデータが飛ぶ前に消費されます。地球の裏側のサーバなら、この "話す前の準備" だけで体感できる遅さになる。

> **実務での効き所(第8章への直結)**: 「毎回のリクエストで、いちいちこのハンドシェイクをやり直したら遅い」―― だから、**一度張った接続を使い回す**発想が生まれます。それが **Keep-Alive** や **コネクションプール**(第8章)。「接続確立はタダじゃない(往復コストがある)」を今ここで腹落ちさせておくと、第8章で「なぜプールが要るのか」が一発で分かります。
>
> **Phase 2 との接続**: これは第7章のスレッドプールと同じ構図です。「作るのが高コストな資源(スレッド/TCP接続)は、毎回作らず使い回す」。**高コストな資源を使い回す**という同じパターンが、スレッドでもコネクションでも現れます。

> ちなみに接続を閉じるときは4ステップ(FIN のやりとり)を踏みます。細部は不要ですが、「**開くときも閉じるときも、往復のやりとりが要る**」= 接続はタダじゃない、とだけ覚えておいてください。

## 4. UDP ― 速さのために、保証を捨てる

TCP は "書留" で安心ですが、**確認・再送・順序整えのぶん、手間と遅延がかかります**。世の中には「多少取りこぼしても、とにかく速い方がいい」通信もあります。そこで使うのが **UDP** です。

UDP は、**ハンドシェイクなし・確認なし・再送なし・順序整えなし**。ポート番号だけ付けて、パケットを投げっぱなしにします。ほぼ「IP + ポート番号」の薄い層です。取りこぼしは起きるが、**軽くて速い**。

> **たとえ**: UDP は **生放送(ライブ配信)やラジオ**。一瞬音が途切れても、放送を止めて「今の言い直して」とは頼みません。番組は進み続ける。**"今" が大事で、過去のかけらの回収は諦める**のが UDP。TCP(書留)が「全部確実に届けること」を優先するのと対照的です。

### UDP が向いている場面

- **動画・音声のライブ配信、ビデオ通話・音声通話**: 1フレーム欠けても、止めて再送するより、次の映像を出す方が良い。再送を待つ間に会話が固まる方が困る。
- **オンラインゲーム**: 0.1秒前の位置情報を再送してもらっても無意味。最新だけ欲しい。
- **DNS の問い合わせ(第4章)**: 「この名前の IP は?」という小さな1往復。取りこぼしたら問い合わせ直せばよく、TCP のハンドシェイク往復をかけるより UDP で軽く済ませる方が速い。

## 5. TCP か UDP か ― 使い分けの原則

| | **TCP** | **UDP** |
|---|---|---|
| 保証 | 順序・再送・重複排除あり(書留) | なし(生放送) |
| 接続確立 | 3ウェイハンドシェイク(往復コスト) | なし(投げっぱなし) |
| 速さ・軽さ | 手間があるぶん重い | 軽くて速い |
| 向く用途 | Web(HTTP)、ファイル転送、メール、**大半の API** | ライブ配信、通話、ゲーム、DNS |
| ひとことで | **取りこぼしが許されないなら TCP** | **遅延が命で、多少の欠けは許せるなら UDP** |

実務で書く通信の **大半は TCP**(HTTP が TCP の上に乗っているため)。UDP を直接触るのは、配信・ゲーム・DNS のような特定領域です。まず「**普段の HTTP は TCP の上**」を押さえておけば十分です。

> **小さな注記(今は雰囲気だけ)**: 最新の **HTTP/3** は、実は **UDP の上**に QUIC という仕組みを乗せ、「UDP の軽さ + TCP 相当の信頼性」を自前で再発明しています。「TCP か UDP か」は絶対の二択ではなく、目的に応じて組み合わせ直せる ―― という話は、第5章・第8章でちらっと触れます。今は「そういう新しい流れもある」で流して OK。

## 6. 【手を動かす】nc で TCP 接続を自分の手で張る

Phase 3 で一番ワクワクするパートかもしれません。**`nc`(netcat)** を使うと、TCP 接続を手で張って、キーボードから直接プロトコルを喋れます。第1章で Kotlin で書いた「生のソケット通信」の、コマンドライン版です。

> 下記はこの教科書を書いた環境で **実際に実行した本物の出力**です(localhost で小さなサーバを立てて試しています)。**あなたの手元でも、外部のサーバ相手に同じことができます**。

### 実演①:TCP でHTTPを手打ちする

ターミナルで `nc <ホスト> <ポート>` と打つと TCP 接続が張られ、そのままキーボードで打った文字が相手に送られます。HTTP のリクエストを手で打ってみます。

```
$ printf 'GET / HTTP/1.1\r\nHost: localhost\r\n\r\n' | nc localhost 9999
```

実際の出力(本物):

```
HTTP/1.1 200 OK
Content-Length: 13

Hello via TCP
```

**何が起きたか**:
- `nc localhost 9999` で、**§3 の3ウェイハンドシェイクが裏で完了**し、TCP 接続が張られた。
- あなたが打った `GET / HTTP/1.1 ...` が、その接続に **write** され、
- サーバの返事が **read** されて表示された。

これは第1章の Kotlin コードの `getOutputStream().write(...)` と `getInputStream().read(...)` を、**コマンド1つでやった**のと同じです。**「HTTP は、TCP という信頼できるパイプの上を流れる、ただのテキスト」**という第5章の予告が、もう目に見えています。手で打った GET が通じるくらい、HTTP はシンプルなテキストの約束事なのです。

### 実演②:ポートが開いているか確認する(`-z`)

`nc -z -v <ホスト> <ポート>` で、「そのポートで誰かが待ち受けているか(TCP 接続を受け付けるか)」だけを確認できます。実際の出力(本物):

```
$ nc -z -v localhost 9997        （誰かが待ち受けている場合）
Connection to localhost (127.0.0.1) 9997 port [tcp/*] succeeded!

$ nc -z -v localhost 9996        （誰も待ち受けていない場合）
nc: connect to localhost (127.0.0.1) port 9996 (tcp) failed: Connection refused
```

この **`succeeded!` と `Connection refused` の違い**が、実務のトラブル切り分けで極めて重要になります(§9 で詳述)。

> **やってみよう**: あなたの手元で、`nc -z -v github.com 443`(HTTPS のポートが開いているか)や `nc -z -v github.com 22` を試してみてください。開いているポート・閉じているポートで、メッセージが変わるはずです。**「マシン(IP)には届くが、そのポートは開いていない」**という状態を、自分の目で見られます。

## 7. Kotlin/JVM 接続 ― TCP は `Socket`、UDP は `DatagramSocket`

JVM では、TCP と UDP で使うクラスが分かれています。

```kotlin
import java.net.InetSocketAddress
import java.net.Socket

fun main() {
    Socket().use { socket ->
        // ハンドシェイク(§3)に「いつまで待つか」の上限 = 接続タイムアウト
        socket.connect(InetSocketAddress("example.com", 80), 3000)  // 3秒でハンドシェイク打ち切り
        println("connected: ${socket.remoteSocketAddress}")
        // ここから getOutputStream()/getInputStream() で read/write(第1章と同じ)
    }
}
```

- **TCP は `java.net.Socket`**(第1章で使ったやつ)。`DatagramSocket` が UDP 用ですが、実務で直接使う場面は少ないので今は「UDP 用の別クラスがある」程度で OK。
- `connect(..., 3000)` の **3000(ミリ秒)は接続タイムアウト**。**これは §3 のハンドシェイクの待ち時間の上限**です。第2章の `isReachable(2000)` に続き、また **タイムアウト**が出てきました。

> **Phase 2 との接続 & 第8章への伏線**: ネットワークのコードには、**種類の違うタイムアウトが複数**あります。「接続(ハンドシェイク)の待ち時間」と「データが返ってくるまでの待ち時間(読み取りタイムアウト)」は別物です。この区別は第8章で「なぜタイムアウトを分けて設定するのか」として深掘りします。今は「**接続確立にも、いつまで待つかの上限が要る**(ハンドシェイクは往復コストだから)」とだけ掴んでください。

## 8. 【演習】採点・解説つき

### 演習1(ポートの理解)
`https://api.example.com` にブラウザでアクセスしたとき、接続先の「IP:ポート」のポート番号は何番になりますか? また、なぜあなたは URL にポートを書かずに済んだのですか?

<details><summary>解答・解説</summary>

ポートは **443**(HTTPS の定番ポート)。`https://` というスキームが「443 を使う」を意味しているので、ブラウザが自動で 443 を補い、URL に `:443` を書かずに済みます。もし `http://` なら 80。変則的なポートのサーバなら `https://api.example.com:8443` のように明示が必要です。要点は「**URL のスキーム(http/https)が、デフォルトのポートを決めている**」。
</details>

### 演習2(TCP か UDP か)
次の3つは TCP と UDP のどちらが向いていますか? 理由も一言で。
(a) 銀行の残高照会 API (b) リアルタイム対戦ゲームの自キャラ位置の送信 (c) 大きな ZIP ファイルのダウンロード

<details><summary>解答・解説</summary>

- (a) **TCP**。金額が1バイトでも欠けたら大問題。取りこぼしが許されない → 書留。
- (b) **UDP**。0.1秒前の位置を再送されても無意味で、最新だけ欲しい。遅延が命 → 生放送。
- (c) **TCP**。ファイルは1バイトでも欠けたら壊れる。全部確実に届く必要がある → 書留。

原則「**取りこぼしが許されない=TCP / 遅延が命で欠けは許せる=UDP**」を当てはめるだけです。
</details>

### 演習3(ハンドシェイクのコスト計算)★第8章の伏線
あるサーバまでの RTT(往復時間)が **80ms** です。1回のリクエストのたびに新しく TCP 接続を張ると、**データを送り始める前**に、ハンドシェイクだけで概ね何 ms かかりますか? そして、これを繰り返し何百回もするサービスは、どうすれば速くできますか?

<details><summary>解答・解説</summary>

3ウェイハンドシェイクは約1.5往復なので、**80ms × 1.5 ≒ 120ms**。データを1バイト送る前に、毎回この約120msが消えます。100回リクエストすれば、ハンドシェイクだけで **合計12秒**もの "準備時間" を浪費する計算です。

速くするには、**一度張った接続を使い回す**(Keep-Alive / コネクションプール)。使い回せば2回目以降はハンドシェイクを省略できます。**これがまさに第8章のコネクションプールの動機**です。「接続はタダじゃない」を数字で実感できましたか? この感覚が、第8章の設定を "なぜやるか分かって" 使えるかどうかを分けます。
</details>

### 演習4(nc の結果を読む)★第9章の伏線
`nc -z -v example.com 443` を実行したら `Connection refused` と出ました。一方 `ping example.com` は正常に返ってきます。何が起きていると考えられますか?

<details><summary>解答・解説</summary>

**マシン(IP)までは届いている**(ping が返る=IP/経路は生きている)。しかし **443番ポートで待ち受けているプログラムがない、またはファイアウォールで塞がれている**(refused)。つまり問題は IP/経路の層ではなく、**その上(ポート/TCP/サーバ設定)の層**にある。

これが第1章の「層で切り分ける」の実践です。ping(IP 層)と nc(TCP/ポート層)で **層を分けて確かめる**ことで、「マシンには届くが、目的のプログラムに繋がらない」と原因を絞り込めました。§9 と第9章で、この切り分けを体系化します。
</details>

## 9. 【実務接続】"refused" と "timeout" の違いが読めると、切り分けが速い

第3章の実務での効き所は、**接続がうまくいかないときの「エラーの読み分け」**です。実務でよく遭遇する2つを区別できると、原因の当たりが段違いに速くなります。

### Connection refused(接続拒否)
`nc: ... Connection refused` ―― これは **「マシンには届いたが、そのポートで誰も待ち受けていない(または明示的に拒否された)」**。§6 で本物を見ました。意味するところは:

- 相手マシン自体は**生きている**(だから "拒否" という返事が返せる)
- でも **目的のプログラムが動いていない/違うポート/ファイアウォールで即拒否**
- → 疑うべきは **サーバ側のプロセスやポート設定**。ネットワーク経路は問題ないことが多い。

### Connection timed out(接続タイムアウト)
`nc` や接続が **無反応のまま時間切れ**になる ―― これは **「そもそも返事が返ってこない」**。意味は refused と全然違う:

- パケットが**相手に届いていない、または相手が黙殺している**
- → 疑うべきは **経路・ファイアウォールでのドロップ・相手マシンのダウン・IP 間違い**。もっと下の層(第2章 IP/経路)寄り。

> **切り分けの型**: 「繋がらない」と言われたら、
> 1. `ping`(第2章)→ IP/経路は生きてる?
> 2. `nc -z -v host port`(この章)→ 目的のポートは開いてる? **refused なのか timeout なのか**?
> 3. その先(TLS/HTTP)は第6章・第5章の道具へ
>
> **refused = 上の層(サーバ設定)寄り、timeout = 下の層(経路/FW)寄り** ―― この読み分けだけで、調べる方向が半分に絞れます。これらは第9章「つながらない・遅いの調べ方」で、全層の道具とあわせて総まとめします。

## 10. この章のまとめ

- **ポート番号**は「どのプログラムに届けるか」。IP(マンションの住所)+ ポート(部屋番号)で届け先が確定。`https://` は 443、`http://` は 80 が定番。
- **TCP** は、取りこぼす IP(普通郵便)の上に **信頼性(順序・再送・重複排除)** を作る=**書留**。仕組みは **通し番号(シーケンス番号)+ 確認応答(ACK)+ 再送**。第2章演習3であなたが想像した設計そのもの。
- これは Phase 2 の **抽象化**(下位層の欠点を上位層に見せず覆い隠す)と同じ構造。
- **3ウェイハンドシェイク**(話す前の「もしもし」3回)で接続を確立。**往復コスト(レイテンシ)がかかる** → だから接続は使い回したい(**第8章 コネクションプールの動機**)。
- **UDP** は保証を捨てて速さを取る=**生放送**。配信・通話・ゲーム・DNS 向き。
- 使い分け:**取りこぼし厳禁なら TCP / 遅延が命で欠けを許せるなら UDP**。普段の HTTP は TCP の上。
- **手を動かす**:`nc` で TCP 接続を手打ち(HTTP が TCP 上のテキストだと体感)、`-z` でポート開放確認。
- **実務**:**refused(相手は生きてるがポートが閉/拒否=上の層寄り)** と **timeout(返事すら来ない=下の層寄り)** の読み分け。

### 7割理解チェック(雰囲気で言えれば OK)

- [ ] ポート番号が「どのプログラムに届けるか」だと言える(IP=住所、ポート=部屋番号)
- [ ] TCP が「取りこぼさない書留」で、通し番号+確認+再送で実現していると言える
- [ ] TCP が「信頼できない IP の上に信頼性を作る抽象化」だと Phase 2 と結びつけられる
- [ ] 3ウェイハンドシェイクに **往復コスト**があり、それが接続使い回し(第8章)の動機だと言える
- [ ] TCP と UDP の使い分けを、書留 vs 生放送で説明できる
- [ ] `nc` で TCP 接続を張れる/refused と timeout の意味の違いを言える

**全部完璧でなくて大丈夫**。特にハンドシェイクのフラグ名や UDP の細部は流して OK。「**TCP=書留 / 接続はタダじゃない / TCP か UDP か**」の3つが体に入っていれば十分です。
